# 🤖 バイブコーディング ワークフロー

> 「バイブコーディング」は、望むものを自然言語で説明し AI エージェントに実装させる方式だ。AI コーディングエージェントで作業するとき、またはエージェントが見当違いの場所を直したり文脈を失ったりするときに読む。

成功の 90% は **コンテキストエンジニアリング** — 何を、どれだけ、どの順序で与えるかにかかっている。

## 1. 核心原則

- 作業は一度に一つの小さく検証可能な単位で要求する — '会員登録フォーム + 検証 + エラー表示' のように 1 PR サイズに分割する。
- 直し方(how)ではなく望む動作・症状(what)を説明する — 'ページネーションが挿入中に重複を返す' が 'WHERE 句を追加して' よりも正確な結果を生む。
- 必要なファイル・ドキュメントだけをコンテキストに入れる(@ファイル, @docs) — 全部ダンプするとかえって精度が下がる。
- 作業が変われば新しいセッション/新しいコンテキストで始める — 長いセッションはエージェントが前の文脈を失う。
- Accept All の前に統合 diff を一度ざっと見る — 間違ったファイル修正を 5 秒で捕まえる方が 30 分のデバッグより安い。
- 企画書(PRD)・仕様を先に与えてからコードを生成させる — 仕様が具体的なほど幻覚・手戻りが減る。
- エージェントが詰まったら、より大きなプロンプトではなくより小さなステップに分割し、失敗ログ・エラーメッセージをそのまま貼り付ける。

## 2. ルール

### 2-1. 小さく分割する (Scope Control)

一度に「ショッピングモールを作って」ではなく 1 PR サイズで:

```
❌ 전체 결제 시스템을 구현해줘
✅ 1) 장바구니 담기 API + 단위 테스트
   2) 결제 요청 화면 (금액 표시 + 약관 동의)
   3) 결제 웹훅 수신 + 멱등 처리
```

小さな単位はレビュー可能で、失敗してもロールバック範囲が小さい。

### 2-2. 動作を説明する (What, not How)

```
❌ pagination 쿼리에 WHERE id > :cursor 추가해
✅ 목록을 스크롤하는 중에 항목이 새로 삽입되면 같은 항목이
   두 번 나온다. 커서 기반으로 안정적으로 페이징되게 해줘.
```

解決策を指定するとエージェントのより良い代替案を塞ぐ。**症状・期待動作**を述べる。

### 2-3. コンテキストは精密に

- `@파일명` で関連ファイルだけ、`@docs` で規約、`@web` で最新情報。
- ❌ 全部入れない — ノイズが増えると見当違いの場所を直す。
- ✅ 企画書(PRD)・API 仕様を先に与えると幻覚が大きく減る。(← Harness が作ってくれる成果物)

### 2-4. セッション衛生

作業テーマが変わったら **新しいセッション**。長い会話は前の文脈を失い見当違いの仮定をする。

### 2-5. レビューして受け入れる

`Accept All` の前に **統合 diff を一度ざっと見る。** 間違ったファイル・削除・シークレット露出を 5 秒で捕まえる方が 30 分のデバッグより安い。(セキュリティ・レビューは 'AI 生成コードレビュー' スキル参照。)

## 3. よくある間違い

- プロンプトを大きくせず **ステップをさらに分割する。** エージェントが詰まったら、より大きなプロンプトは答えではない。
- エラーメッセージ・失敗ログを **そのまま** 貼り付ける — 要約せず原文を。
- 二、三度同じ場所で詰まったらアプローチを変える(別のライブラリ/構造)。
