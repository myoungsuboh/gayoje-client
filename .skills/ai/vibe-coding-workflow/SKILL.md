# 🤖 바이브 코딩 워크플로우

> "바이브 코딩"은 원하는 것을 자연어로 설명하고 AI 에이전트가 구현하게 하는 방식이다. AI 코딩 에이전트로 작업할 때, 또는 에이전트가 엉뚱한 곳을 고치거나 맥락을 잃을 때 읽는다.

성공의 90%는 **컨텍스트 엔지니어링** — 무엇을, 얼마나, 어떤 순서로 주느냐에 달려 있다.

## 1. 핵심 원칙

- 작업은 한 번에 하나의 작고 검증 가능한 단위로 요청한다 — '회원가입 폼 + 검증 + 에러표시'처럼 한 PR 크기로 쪼갠다.
- 고치는 방법(how)이 아니라 원하는 동작·증상(what)을 설명한다 — '페이지네이션이 삽입 중 중복 반환됨'이 'WHERE 절 추가해'보다 정확한 결과를 낳는다.
- 필요한 파일·문서만 컨텍스트에 넣는다(@파일, @docs) — 전부 덤프하면 오히려 정확도가 떨어진다.
- 작업이 바뀌면 새 세션/새 컨텍스트로 시작한다 — 긴 세션은 에이전트가 앞 맥락을 잃는다.
- Accept All 전에 통합 diff를 한 번 훑는다 — 잘못된 파일 수정을 5초에 잡는 게 30분 디버깅보다 싸다.
- 기획서(PRD)·명세를 먼저 주고 코드를 생성시킨다 — 명세가 구체적일수록 환각·재작업이 준다.
- 에이전트가 막히면 더 큰 프롬프트가 아니라 더 작은 단계로 쪼개고, 실패 로그·에러 메시지를 그대로 붙여준다.

## 2. 규칙

### 2-1. 작게 쪼갠다 (Scope Control)

한 번에 "쇼핑몰 만들어줘"가 아니라 한 PR 크기로:

```
❌ 전체 결제 시스템을 구현해줘
✅ 1) 장바구니 담기 API + 단위 테스트
   2) 결제 요청 화면 (금액 표시 + 약관 동의)
   3) 결제 웹훅 수신 + 멱등 처리
```

작은 단위는 검토 가능하고, 실패해도 롤백 범위가 작다.

### 2-2. 동작을 설명한다 (What, not How)

```
❌ pagination 쿼리에 WHERE id > :cursor 추가해
✅ 목록을 스크롤하는 중에 항목이 새로 삽입되면 같은 항목이
   두 번 나온다. 커서 기반으로 안정적으로 페이징되게 해줘.
```

해결책을 지정하면 에이전트의 더 나은 대안을 막는다. **증상·기대 동작**을 말한다.

### 2-3. 컨텍스트는 정밀하게

- `@파일명` 으로 관련 파일만, `@docs` 로 규약, `@web` 으로 최신 정보.
- ❌ 전부 넣지 않는다 — 노이즈가 늘면 엉뚱한 곳을 고친다.
- ✅ 기획서(PRD)·API 명세를 먼저 주면 환각이 크게 준다. (← Harness 가 만들어주는 산출물)

### 2-4. 세션 위생

작업 주제가 바뀌면 **새 세션**. 긴 대화는 앞 맥락을 잃고 엉뚱한 가정을 한다.

### 2-5. 검토하고 받는다

`Accept All` 전에 **통합 diff를 한 번 훑는다.** 잘못된 파일·삭제·시크릿 노출을 5초에 잡는 게 30분 디버깅보다 싸다. (보안·검토는 'AI 생성 코드 검토' 스킬 참조.)

## 3. 흔한 실수

- 프롬프트를 키우지 말고 **단계를 더 쪼갠다.** 에이전트가 막히면 더 큰 프롬프트가 답이 아니다.
- 에러 메시지·실패 로그를 **그대로** 붙여준다 — 요약하지 말고 원문을.
- 두세 번 같은 자리에서 막히면 접근을 바꾼다(다른 라이브러리/구조).

## 4. 체크리스트

- [ ] 작업을 한 PR 크기의 작고 검증 가능한 단위로 쪼갰는가
- [ ] how(해결책)가 아니라 what(증상·기대 동작)을 설명했는가
- [ ] 필요한 파일·문서만 컨텍스트에 넣었는가 (전부 덤프 금지)
- [ ] 작업 주제가 바뀔 때 새 세션으로 시작했는가
- [ ] Accept All 전에 통합 diff를 훑었는가
- [ ] 막혔을 때 프롬프트를 키우지 않고 단계를 더 쪼갰는가
