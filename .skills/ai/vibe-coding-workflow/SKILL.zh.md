# 🤖 氛围编程工作流

> "氛围编程" 是一种用自然语言描述想要的东西、让 AI 代理来实现的方式。当用 AI 编码代理工作时，或代理改错地方、丢失上下文时阅读。

成功的 90% 在于 **上下文工程** — 提供什么、提供多少、以什么顺序提供。

## 1. 核心原则

- 一次只请求一个小而可验证的单位 — 像 '注册表单 + 校验 + 错误显示' 那样拆成一个 PR 大小。
- 描述期望的行为·症状(what)而非修复方法(how) — '分页在插入过程中返回重复项' 比 '加个 WHERE 子句' 产出更准确的结果。
- 只把必要的文件·文档放入上下文(@文件, @docs) — 全部倾倒反而降低准确度。
- 任务改变时用新会话/新上下文开始 — 长会话会让代理丢失前面的上下文。
- 在 Accept All 之前把合并 diff 通览一遍 — 用 5 秒抓住改错文件比 30 分钟调试更划算。
- 先给出企划书(PRD)·规格再生成代码 — 规格越具体，幻觉·返工越少。
- 代理卡住时不要把提示变大，而是拆成更小的步骤，并原样粘贴失败日志·错误信息。

## 2. 规则

### 2-1. 拆小 (Scope Control)

不要一次 "给我做个商城"，而是按一个 PR 大小:

```
❌ 전체 결제 시스템을 구현해줘
✅ 1) 장바구니 담기 API + 단위 테스트
   2) 결제 요청 화면 (금액 표시 + 약관 동의)
   3) 결제 웹훅 수신 + 멱등 처리
```

小单位可审查，即使失败回滚范围也小。

### 2-2. 描述行为 (What, not How)

```
❌ pagination 쿼리에 WHERE id > :cursor 추가해
✅ 목록을 스크롤하는 중에 항목이 새로 삽입되면 같은 항목이
   두 번 나온다. 커서 기반으로 안정적으로 페이징되게 해줘.
```

指定解决方案会堵住代理更好的替代方案。陈述 **症状·期望行为**。

### 2-3. 上下文要精确

- `@파일명` 只放相关文件，`@docs` 放规约，`@web` 放最新信息。
- ❌ 不要全部放入 — 噪音增多就会改错地方。
- ✅ 先给出企划书(PRD)·API 规格能大幅减少幻觉。(← Harness 为你生成的产物)

### 2-4. 会话卫生

工作主题改变时 **开新会话**。长对话会丢失前面的上下文并做出错误假设。

### 2-5. 审查后再接受

在 `Accept All` 之前 **把合并 diff 通览一遍。** 用 5 秒抓住改错文件·删除·密钥泄露比 30 分钟调试更划算。(安全·审查参见 'AI 生成代码审查' 技能。)

## 3. 常见错误

- 不要把提示变大，而要 **进一步拆分步骤。** 代理卡住时，更大的提示不是答案。
- **原样** 粘贴错误信息·失败日志 — 不要总结，给出原文。
- 在同一处卡住两三次就改变方法(换库/换结构)。

## 4. 检查清单

- [ ] 是否把工作拆成一个 PR 大小、小而可验证的单位
- [ ] 是否描述了 what(症状·期望行为)而非 how(解决方案)
- [ ] 是否只把必要的文件·文档放入上下文 (禁止全部倾倒)
- [ ] 工作主题改变时是否用新会话开始
- [ ] 在 Accept All 之前是否通览了合并 diff
- [ ] 卡住时是否不把提示变大而进一步拆分步骤
