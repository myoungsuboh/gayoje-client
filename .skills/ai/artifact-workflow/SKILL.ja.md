# 📁 アーティファクトベースのワークフロー (Artifact Workflow)

> 計画と進捗をチャット欄にばらまかず、アーティファクトファイル(作業用成果物ファイル)で管理する。長い計画はファイルに分離してチャットをきれいに保ち、作業前に計画をファイルに残し、進捗はファイルで追跡し、完了時に要約を残す。新しい作業を始めるときや進捗を報告するときに読む。特定のエージェントツール(ファイルパス・メタデータ形式)に依存しない汎用標準である。

## 1. 核心原則

- **チャットはきれいに、記録はファイルに** — 5行以上の長い計画・複雑なステップ手順はチャットに直接出さない。アーティファクトファイルに書き、チャットにはファイルの場所 + 一行要約だけを残す。
- **作業前に計画を文書化** — 新しい作業を始める前に、計画をまずアーティファクトファイルとして書く(`implementation-plan` を参照)。頭の中・チャットだけに留めない。
- **進捗はファイルで追跡** — 進捗はチェックボックス(`[ ]` → `[x]`)でアーティファクトファイルに表示し、ユーザーがいつでもファイル一つで現状を確認できるようにする。
- **単一の情報源(Single Source of Truth)** — 計画・進捗・結果はファイルが基準。同じ内容をチャットとファイルに二重に長く書かない。
- **ツール中立** — アーティファクトは任意の `artifacts/` ディレクトリに置いた普通のファイルでよい。特定のエージェントツール専用のパスやメタデータ形式に依存しない(ツール別の具体形式は付録参照)。

## 2. ルール

### 2-1. 長い計画はファイルへ、チャットは一行要約 (Clean Chat)

- 5行以上の長い計画や複雑なステップ手順をチャットに丸ごと貼り付けない。
- アーティファクトファイルを作成/更新し、チャットには**ファイルの場所 + 一行要約**だけを残す。

```text
// ❌ 禁止 — 20行の計画をチャットに丸ごと貼り付け
"計画です:
 1. ...
 2. ...
 ... (20行)"

// ✅ 推奨 — ファイルに分離、チャットには場所 + 一行要約
"ご依頼の機能を分析し、計画を artifacts/router_migration_plan.md に作成しました。確認のうえ承認してください。"
```

### 2-2. 作業前に計画アーティファクトを作成 (Documented Planning)

- 新しい作業を始める前に、計画アーティファクトをまず作成する(作成形式は `implementation-plan` スキルに従う)。
- 任意の `artifacts/` ディレクトリの下に意味の分かるファイル名で置く(例: `artifacts/router_migration_plan.md`)。
- 含める内容: 作業目標、事前確認事項、フェーズ別の実装計画。

```text
// ❌ 禁止 — 計画なしでいきなり実装開始
start coding...

// ✅ 推奨 — 計画ファイルが先、その次に実装
write artifacts/<task>_plan.md  →  実装開始
```

### 2-3. 進捗はチェックボックスで追跡 (Task-Centric Tracking)

- リアルタイムの進捗は進捗追跡用のアーティファクトファイルで管理する。
- フェーズ別の進捗をチェックボックスで更新し、ファイルを見るだけで現状が分かるようにする。
- 更新タイミング: 各フェーズ完了時または重要な状態変更時。

```text
// ✅ 推奨 — ファイル内でチェックボックスで進捗表示
## 進捗
- [x] Phase 1: スキーマ定義
- [x] Phase 2: ハンドラ実装
- [ ] Phase 3: テスト
```

### 2-4. 完了時に要約アーティファクトで締める (Summary)

- 作業が終わったら要約アーティファクトを残し、チャットを締める。
- 含める: 最終成果物の場所、変更内容の要約、テスト結果。

> 全体の流れは **計画(2-2) → 追跡(2-3) → 要約(2-4)** の一行に要約される。ステップ別のチャット出力文言のツール別例は付録を参照する。

### 2-5. 即答が必要な場合だけ短いテキスト

- ユーザーがチャットで即座のフィードバックを求める短い応答にのみテキストを直接書く。
- それ以外の長く構造的な内容はすべてアーティファクトファイルに移す。

## 3. よくある間違い

- **長い計画をチャットに丸ごと貼り付け** → 会話が散らかり後で見つけにくい。ファイルに分離し一行要約だけを残す。
- **計画なしでいきなり実装** → 範囲がぶれ、ユーザー承認の地点が消える。作業前に計画アーティファクトをまず作る。
- **進捗をチャットだけで報告** → 現状を一目で見る単一の情報源がない。チェックボックスでファイルで追跡する。
- **計画・進捗をチャットとファイルに二重作成** → 両者が食い違う。ファイルを単一の情報源とし、チャットは要約のみ。
- **完了要約の欠落** → 成果物の場所・変更要約・テスト結果が散らばる。完了時に要約アーティファクトで締める。
- **特定ツール専用のパス/メタデータに依存** → 他のエージェントツールに移すと壊れる。任意の `artifacts/` ディレクトリ + 普通のファイルを基準とし、ツール別形式は付録のように分離する。

## 4. チェックリスト

- [ ] 作業開始前に計画アーティファクトを作成したか(形式は `implementation-plan` 参照)
- [ ] 計画を任意の `artifacts/` ディレクトリに意味のあるファイル名で保存したか
- [ ] 進捗をアーティファクトファイルのチェックボックス(`[x]`)で更新しているか
- [ ] 各フェーズ完了/状態変更の時点で進捗を更新したか
- [ ] 完了時に成果物の場所・変更要約・テスト結果を含む要約アーティファクトを作成したか
- [ ] 計画・進捗・結果の単一の情報源がファイルか(チャットと二重作成していないか)

## 付録: ツール別の例

> 以下は特定のエージェントツールでの具体的な適用例である。上記1〜4の原則・ルールが標準であり、付録はツール別の適用事例にすぎない。**他のエージェントツールはチームが同じパターンで追加する。**

### Gemini Antigravity

Gemini Antigravity 環境では、アーティファクトを次の規約で扱う。

- **タイプ区別**: 計画は `implementation_plan` タイプ、進捗追跡は `task` タイプのアーティファクトとして生成する。
- **保存パス**: アーティファクトファイルは `./.gemini/antigravity/artifacts/` 配下に明確なファイル名で置く(例: `./.gemini/antigravity/artifacts/router_migration_plan.md`)。
- **メタデータ**: アーティファクトファイルは常に `IsArtifact: true` と適切な `ArtifactMetadata` を含めて生成/修正する。
- **チャット出力例**:
  - 計画: "✅ ご依頼の機能を分析し、実装計画書(`.gemini/antigravity/artifacts/task_name_plan.md`)を作成しました。確認のうえ承認してください。"
  - 進捗: "🟡 Phase 1 完了。進捗(`.gemini/antigravity/artifacts/task_tracking.md`)を更新しました。次のステップに進みます。"
