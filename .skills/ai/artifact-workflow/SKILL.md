# 📁 아티팩트 기반 워크플로우 (Artifact Workflow)

> 계획과 진행 상황을 채팅창에 흩뿌리지 않고 아티팩트 파일(작업용 산출물 파일)로 관리한다. 긴 계획은 파일로 분리해 채팅을 깔끔히 유지하고, 작업 전 계획을 파일로 남기며, 진행은 파일에서 추적하고, 완료 시 요약을 남긴다. 새 작업을 시작하거나 진행을 보고할 때 읽는다. 특정 에이전트 도구(파일 경로·메타데이터 형식)에 종속되지 않는 범용 표준이다.

## 1. 핵심 원칙

- **채팅은 깨끗하게, 기록은 파일에** — 5줄 이상의 긴 계획·복잡한 단계별 절차는 채팅에 직접 노출하지 않는다. 아티팩트 파일에 적고, 채팅에는 파일 위치 + 한 줄 요약만 남긴다.
- **작업 전에 계획을 문서화** — 새 작업을 시작하기 전, 계획을 아티팩트 파일로 먼저 작성한다(`implementation-plan` 참고). 머릿속·채팅에만 두지 않는다.
- **진행은 파일에서 추적** — 진척도는 체크박스(`[ ]` → `[x]`)로 아티팩트 파일에 표시해, 사용자가 언제든 파일 하나로 현재 상태를 확인하게 한다.
- **단일 출처(Single Source of Truth)** — 계획·진행·결과는 파일이 기준이다. 같은 내용을 채팅과 파일에 이중으로 길게 적지 않는다.
- **도구 중립** — 아티팩트는 임의의 `artifacts/` 디렉토리에 둔 일반 파일이면 된다. 특정 에이전트 도구의 전용 경로나 메타데이터 형식에 의존하지 않는다(도구별 구체 형식은 부록 참고).

## 2. 규칙

### 2-1. 긴 계획은 파일로, 채팅은 한 줄 요약 (Clean Chat)

- 5줄 이상의 긴 계획이나 복잡한 단계별 절차는 채팅에 통째로 붙여넣지 않는다.
- 아티팩트 파일을 만들고/갱신하고, 채팅에는 **파일 위치 + 한 줄 요약**만 남긴다.

```text
// ❌ 금지 — 20줄짜리 계획을 채팅에 통째로 붙여넣음
"계획입니다:
 1. ...
 2. ...
 ... (20줄)"

// ✅ 권장 — 파일로 분리, 채팅엔 위치 + 한 줄 요약
"요청 기능을 분석해 계획을 artifacts/router_migration_plan.md 에 작성했습니다. 확인 후 승인해 주세요."
```

### 2-2. 작업 전 계획 아티팩트 작성 (Documented Planning)

- 새 작업을 시작하기 전, 계획 아티팩트를 먼저 만든다(작성 형식은 `implementation-plan` 스킬을 따른다).
- 임의의 `artifacts/` 디렉토리 아래에 의미가 드러나는 파일명으로 둔다 (예: `artifacts/router_migration_plan.md`).
- 포함 내용: 작업 목표, 사전 확인 사항, 단계별(Phase) 구현 계획.

```text
// ❌ 금지 — 계획 없이 곧장 구현 시작
start coding...

// ✅ 권장 — 계획 파일 먼저, 그다음 구현
write artifacts/<task>_plan.md  →  구현 시작
```

### 2-3. 진행은 체크박스로 추적 (Task-Centric Tracking)

- 실시간 진행 상황은 진행 추적용 아티팩트 파일로 관리한다.
- Phase별 진척도를 체크박스로 갱신해, 파일만 보면 현재 상태를 알 수 있게 한다.
- 갱신 시점: 각 Phase 완료 시 또는 중요한 상태 변경 시.

```text
// ✅ 권장 — 파일 안에서 체크박스로 진척 표시
## 진행 상황
- [x] Phase 1: 스키마 정의
- [x] Phase 2: 핸들러 구현
- [ ] Phase 3: 테스트
```

### 2-4. 완료 시 요약 아티팩트로 마무리 (Summary)

- 작업이 끝나면 요약 아티팩트를 남기고 채팅을 마무리한다.
- 포함: 최종 결과물 위치, 변경 사항 요약, 테스트 결과.

> 전체 흐름은 **계획(2-2) → 추적(2-3) → 요약(2-4)** 한 줄로 요약된다. 단계별 채팅 출력 문구의 도구별 예시는 부록을 참고한다.

### 2-5. 즉답이 필요한 경우만 짧은 텍스트

- 사용자가 채팅에서 즉각적인 피드백을 원하는 짧은 응답에만 텍스트를 직접 쓴다.
- 그 외 길고 구조적인 내용은 모두 아티팩트 파일로 옮긴다.

## 3. 흔한 실수

- **긴 계획을 채팅에 통째로 붙여넣음** → 대화가 지저분해지고 나중에 다시 찾기 어렵다. 파일로 분리하고 한 줄 요약만 남긴다.
- **계획 없이 곧장 구현** → 범위가 흔들리고 사용자 승인 지점이 사라진다. 작업 전 계획 아티팩트를 먼저 만든다.
- **진행을 채팅에만 보고** → 현재 상태를 한눈에 볼 단일 출처가 없다. 체크박스로 파일에서 추적한다.
- **계획·진행을 채팅과 파일에 이중 작성** → 둘이 어긋난다. 파일을 단일 출처로 두고 채팅은 요약만.
- **완료 요약 누락** → 결과물 위치·변경 요약·테스트 결과가 흩어진다. 완료 시 요약 아티팩트로 마무리한다.
- **특정 도구 전용 경로/메타데이터에 의존** → 다른 에이전트 도구로 옮기면 깨진다. 임의의 `artifacts/` 디렉토리 + 일반 파일을 기준으로 하고, 도구별 형식은 부록처럼 분리한다.

## 4. 체크리스트

- [ ] 5줄 이상 계획을 채팅에 통째로 노출하지 않고, 파일 위치 + 한 줄 요약만 남겼는가
- [ ] 작업 시작 전 계획 아티팩트를 만들었는가 (형식은 `implementation-plan` 참고)
- [ ] 계획을 임의의 `artifacts/` 디렉토리에 의미 있는 파일명으로 저장했는가
- [ ] 진행 상황을 아티팩트 파일의 체크박스(`[x]`)로 갱신하는가
- [ ] 각 Phase 완료/상태 변경 시점에 진행을 갱신했는가
- [ ] 완료 시 결과물 위치·변경 요약·테스트 결과를 담은 요약 아티팩트를 작성했는가
- [ ] 계획·진행·결과의 단일 출처가 파일인가 (채팅과 이중 작성하지 않았는가)

## 부록: 도구별 예시

> 아래는 특정 에이전트 도구에서의 구체적 적용 예시다. 위 1~4의 원칙·규칙이 표준이고, 부록은 도구별 적용 사례일 뿐이다. **다른 에이전트 도구는 팀이 같은 패턴으로 추가한다.**

### Gemini Antigravity

Gemini Antigravity 환경에서는 아티팩트를 다음 규약으로 다룬다.

- **타입 구분**: 계획은 `implementation_plan` 타입, 진행 추적은 `task` 타입 아티팩트로 생성한다.
- **저장 경로**: 아티팩트 파일은 `./.gemini/antigravity/artifacts/` 하위에 명확한 파일명으로 둔다 (예: `./.gemini/antigravity/artifacts/router_migration_plan.md`).
- **메타데이터**: 아티팩트 파일은 항상 `IsArtifact: true`와 적절한 `ArtifactMetadata`를 포함하여 생성/수정한다.
- **채팅 출력 예시**:
  - 계획: "✅ 요청하신 기능을 분석하여 구현 계획서(`.gemini/antigravity/artifacts/task_name_plan.md`)를 작성했습니다. 확인 후 승인해 주세요."
  - 진행: "🟡 Phase 1 완료. 진행 상황(`.gemini/antigravity/artifacts/task_tracking.md`)을 업데이트했습니다. 다음 단계를 진행합니다."
