# 📁 基于工件的工作流 (Artifact Workflow)

> 不把计划与进度散落在聊天窗口里，而是用工件文件(工作用产出文件)来管理。把长计划分离到文件以保持聊天整洁，作业前在文件里留下计划，进度在文件中追踪，完成时留下总结。开始新任务或报告进度时阅读。它是不绑定任何特定智能体工具(文件路径·元数据格式)的通用标准。

## 1. 核心原则

- **聊天保持整洁，记录放进文件** — 5 行以上的长计划·复杂的分步流程不直接暴露在聊天里。写入工件文件，聊天里只留下文件位置 + 一行摘要。
- **作业前先把计划文档化** — 开始新任务前，先把计划写成工件文件(参见 `implementation-plan`)。不要只放在脑中·聊天里。
- **进度在文件中追踪** — 用复选框(`[ ]` → `[x]`)在工件文件中标记进度，让用户随时通过一个文件确认当前状态。
- **单一事实来源(Single Source of Truth)** — 计划·进度·结果以文件为准。不要把同一内容在聊天和文件里双重冗长地写。
- **工具中立** — 工件只需是放在任意 `artifacts/` 目录下的普通文件即可。不依赖特定智能体工具的专用路径或元数据格式(工具专属的具体格式参见附录)。

## 2. 规则

### 2-1. 长计划进文件，聊天给一行摘要 (Clean Chat)

- 不要把 5 行以上的长计划或复杂的分步流程整段粘贴进聊天。
- 创建/更新工件文件，聊天里只留下**文件位置 + 一行摘要**。

```text
// ❌ 禁止 — 把 20 行的计划整段粘贴进聊天
"计划如下:
 1. ...
 2. ...
 ... (20 行)"

// ✅ 推荐 — 分离到文件，聊天给位置 + 一行摘要
"我分析了所请求的功能，并把计划写在 artifacts/router_migration_plan.md。请确认后批准。"
```

### 2-2. 作业前编写计划工件 (Documented Planning)

- 开始新任务前，先创建计划工件(编写格式遵循 `implementation-plan` 技能)。
- 放在任意 `artifacts/` 目录下，用含义清晰的文件名(例: `artifacts/router_migration_plan.md`)。
- 包含内容: 作业目标、事前确认事项、分阶段(Phase)实现计划。

```text
// ❌ 禁止 — 没有计划就直接开始实现
start coding...

// ✅ 推荐 — 先计划文件，再实现
write artifacts/<task>_plan.md  →  开始实现
```

### 2-3. 用复选框追踪进度 (Task-Centric Tracking)

- 实时进度用进度追踪工件文件来管理。
- 用复选框更新各阶段进度，使只看文件就能知道当前状态。
- 更新时机: 每个阶段完成时或重要状态变更时。

```text
// ✅ 推荐 — 在文件内用复选框标记进度
## 进度
- [x] Phase 1: 定义 schema
- [x] Phase 2: 实现 handler
- [ ] Phase 3: 测试
```

### 2-4. 完成时用总结工件收尾 (Summary)

- 作业结束后，留下总结工件并收尾聊天。
- 包含: 最终产出物位置、变更内容摘要、测试结果。

> 整个流程可概括为一行: **计划(2-2) → 追踪(2-3) → 总结(2-4)**。各步骤聊天输出措辞的工具专属示例参见附录。

### 2-5. 仅在需要即时回答时用短文本

- 仅对用户希望在聊天中获得即时反馈的简短回应直接写文本。
- 其余冗长而结构化的内容全部移入工件文件。

## 3. 常见错误

- **把长计划整段粘贴进聊天** → 对话变乱且日后难以查找。分离到文件，只留一行摘要。
- **没有计划就直接实现** → 范围漂移，用户批准节点消失。作业前先建计划工件。
- **进度只在聊天里报告** → 没有一眼看清当前状态的单一来源。用复选框在文件中追踪。
- **计划·进度在聊天和文件双重编写** → 两者会不一致。以文件为单一来源，聊天只做摘要。
- **缺少完成总结** → 产出物位置·变更摘要·测试结果被打散。完成时用总结工件收尾。
- **依赖特定工具的专用路径/元数据** → 换到别的智能体工具就会失效。以任意 `artifacts/` 目录 + 普通文件为基准，工具专属格式像附录那样分离。

## 4. 检查清单

- [ ] 没有把 5 行以上的计划整段暴露在聊天里，只留下了文件位置 + 一行摘要
- [ ] 作业开始前创建了计划工件(格式参见 `implementation-plan`)
- [ ] 把计划保存在任意 `artifacts/` 目录下、用有意义的文件名
- [ ] 用工件文件中的复选框(`[x]`)更新进度
- [ ] 在每个阶段完成/状态变更的时点更新了进度
- [ ] 完成时编写了包含产出物位置·变更摘要·测试结果的总结工件
- [ ] 计划·进度·结果的单一来源是文件(没有在聊天里双重编写)

## 附录: 工具专属示例

> 以下是特定智能体工具上的具体应用示例。上面 1~4 的原则·规则是标准，附录只是工具专属的应用案例。**其他智能体工具由团队按相同模式添加。**

### Gemini Antigravity

在 Gemini Antigravity 环境中，按以下约定处理工件。

- **类型区分**: 计划生成为 `implementation_plan` 类型，进度追踪生成为 `task` 类型工件。
- **保存路径**: 工件文件放在 `./.gemini/antigravity/artifacts/` 下，用清晰的文件名(例: `./.gemini/antigravity/artifacts/router_migration_plan.md`)。
- **元数据**: 工件文件始终包含 `IsArtifact: true` 和适当的 `ArtifactMetadata` 来创建/修改。
- **聊天输出示例**:
  - 计划: "✅ 我分析了您请求的功能，并编写了实现计划书(`.gemini/antigravity/artifacts/task_name_plan.md`)。请确认后批准。"
  - 进度: "🟡 Phase 1 完成。我已更新进度(`.gemini/antigravity/artifacts/task_tracking.md`)。即将进入下一步。"
