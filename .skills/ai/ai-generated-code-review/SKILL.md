# 🤖 AI 생성 코드 검토·보안

> AI가 만든 코드를 머지·배포 전에 안전하게 거른다. 바이브 코딩으로 코드를 빠르게 생성하되 인증·결제·권한·의존성 위험을 검토할 때 읽는다.

바이브 코딩은 코드를 빠르게 쏟아내지만, **AI 코드는 그럴듯하게 틀린다.**
속도를 살리면서 사고를 막는 핵심은 "받기 전 검토"를 체크리스트로 자동화하는 것이다.

## 1. 핵심 원칙
- 인증·결제·권한·개인정보 처리 코드는 AI 출력을 절대 그대로 신뢰하지 않고 사람이 한 줄씩 검토한다.
- AI가 추가한 의존성(import/package)이 실존하고 신뢰할 수 있는지 확인한다 — 존재하지 않는 패키지명 환각(슬롭스쿼팅 표적)을 차단한다.
- 커밋 전 시크릿 스캔을 돌린다 — AI가 하드코딩한 API 키·비밀번호·토큰이 `.env` 밖으로 새지 않게 한다.
- AI 생성 코드도 입력 검증·권한 체크·에러 처리가 있는지 OWASP 기준으로 확인한다 (AI는 happy-path만 짜는 경향).
- 변경된 코드 경로에 테스트가 있는지 확인하고, AI가 테스트를 통과시키려 단언을 약화시키지 않았는지 본다.
- 대량 변경은 통합 diff를 파일 단위로 훑어 의도치 않은 파일 수정·삭제를 잡는다.
- 보안 도구(secret scanning, SAST, 의존성 취약점 스캔)를 CI에 넣어 사람 검토를 보강한다.

## 2. 규칙

### 2-1. AI 코드에 흔한 함정

| 함정 | 증상 | 대응 |
|---|---|---|
| 시크릿 하드코딩 | API 키·비번을 코드에 직접 | 커밋 전 secret 스캔 + .env 강제 |
| 환각 의존성 | 존재하지 않는 패키지 import | 설치/실존 확인 (슬롭스쿼팅 표적) |
| happy-path만 | 입력 검증·에러 처리 누락 | OWASP 입력검증·권한 체크 확인 |
| 단언 약화 | 테스트 통과시키려 assert 삭제/완화 | 테스트 diff 검토 |
| 과도한 권한 | 전체 접근 토큰·`chmod 777` | 최소 권한 원칙 확인 |

### 2-2. 절대 그대로 믿지 않는 영역

다음은 AI 출력을 **사람이 한 줄씩** 검토한다:

- ✅ **인증/인가** — 토큰 검증, 세션, 권한 분기
- ✅ **결제** — 금액 계산, 멱등성, 웹훅 서명 검증
- ✅ **개인정보(PII)** — 저장·로깅·삭제 경로
- ✅ **파괴적 작업** — DB 마이그레이션, 대량 삭제, 배포

### 2-3. CI로 보강

사람 검토는 빠뜨린다 — 다음을 CI에 넣어 기계가 1차로 거른다:

- ✅ **Secret scanning** (gitleaks, trufflehog)
- ✅ **SAST** (Semgrep, CodeQL)
- ✅ **의존성 취약점·실존 검사** (osv-scanner, npm audit)

> 빠른 생성 + 자동 게이트 + 위험 영역 사람 검토 = 바이브 코딩을 프로덕션에 안전하게 올리는 공식.

## 3. 흔한 실수
- ❌ 시크릿을 코드에 하드코딩 → 커밋 전 secret 스캔 + `.env` 강제로 차단.
- ❌ 존재하지 않는 패키지 import(환각) → 설치/실존 확인 (슬롭스쿼팅 표적).
- ❌ happy-path만 작성, 입력 검증·에러 처리 누락 → OWASP 기준으로 확인.
- ❌ 테스트 통과시키려 단언 삭제/완화 → 테스트 diff 검토.
- ❌ 전체 접근 토큰·`chmod 777` 등 과도한 권한 → 최소 권한 원칙 확인.

## 4. 체크리스트
- [ ] 통합 diff를 파일 단위로 훑었다 (의도 외 수정/삭제 없음)
- [ ] 추가된 의존성이 실존하고 신뢰할 수 있다
- [ ] 시크릿 스캔 통과 (하드코딩 키 0)
- [ ] 인증/결제/PII 코드는 사람이 직접 읽었다
- [ ] 변경 경로에 테스트가 있고, 단언이 약화되지 않았다
- [ ] 입력 검증·권한 체크·에러 처리가 있다
