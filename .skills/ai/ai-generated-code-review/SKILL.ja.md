# 🤖 AI生成コードのレビュー・セキュリティ

> AIが作成したコードをマージ・デプロイ前に安全にフィルタリングする。バイブコーディングでコードを速く生成しつつ、認証・決済・権限・依存関係のリスクをレビューするときに読む。

バイブコーディングはコードを速く吐き出すが、**AIコードはもっともらしく間違える。**
スピードを生かしながら事故を防ぐ要点は、「受け取る前のレビュー」をチェックリストとして自動化することだ。

## 1. 核心原則
- 認証・決済・権限・個人情報を扱うコードは、AI出力を決してそのまま信頼せず、人が一行ずつレビューする。
- AIが追加した依存関係(import/package)が実在し信頼できるか確認する — 存在しないパッケージ名の幻覚(スロップスクワッティングの標的)を遮断する。
- コミット前にシークレットスキャンを実行する — AIがハードコードしたAPIキー・パスワード・トークンが`.env`の外へ漏れないようにする。
- AI生成コードにも入力検証・権限チェック・エラー処理があるかをOWASP基準で確認する(AIはhappy-pathだけ書く傾向がある)。
- 変更されたコードパスにテストがあるか確認し、AIがテストを通すためにアサーションを弱めていないかを見る。
- 大量変更は統合diffをファイル単位で見て、意図しないファイルの変更・削除を捕捉する。
- セキュリティツール(secret scanning, SAST, 依存関係の脆弱性スキャン)をCIに入れて人のレビューを補強する。

## 2. ルール

### 2-1. AIコードに多い落とし穴

| 落とし穴 | 症状 | 対応 |
|---|---|---|
| シークレットのハードコード | APIキー・パスワードをコードに直接 | コミット前のシークレットスキャン + .env強制 |
| 幻覚による依存関係 | 存在しないパッケージのimport | インストール/実在確認(スロップスクワッティングの標的) |
| happy-pathのみ | 入力検証・エラー処理の欠落 | OWASPの入力検証・権限チェックを確認 |
| アサーションの弱体化 | テストを通すためにassertを削除/緩和 | テストdiffをレビュー |
| 過剰な権限 | 全アクセストークン・`chmod 777` | 最小権限の原則を確認 |

### 2-2. 決してそのまま信じない領域

次のAI出力は**人が一行ずつ**レビューする:

- ✅ **認証/認可** — トークン検証、セッション、権限分岐
- ✅ **決済** — 金額計算、冪等性、Webhook署名検証
- ✅ **個人情報(PII)** — 保存・ロギング・削除のパス
- ✅ **破壊的操作** — DBマイグレーション、大量削除、デプロイ

### 2-3. CIで補強

人のレビューは見落とす — 次をCIに入れて機械が一次フィルタリングする:

- ✅ **Secret scanning** (gitleaks, trufflehog)
- ✅ **SAST** (Semgrep, CodeQL)
- ✅ **依存関係の脆弱性・実在チェック** (osv-scanner, npm audit)

> 速い生成 + 自動ゲート + リスク領域の人によるレビュー = バイブコーディングを本番に安全に載せる公式。

## 3. よくある間違い
- ❌ シークレットをコードにハードコード → コミット前のシークレットスキャン + `.env`強制で遮断。
- ❌ 存在しないパッケージのimport(幻覚) → インストール/実在確認(スロップスクワッティングの標的)。
- ❌ happy-pathのみ作成、入力検証・エラー処理の欠落 → OWASP基準で確認。
- ❌ テストを通すためにアサーションを削除/緩和 → テストdiffをレビュー。
- ❌ 全アクセストークン・`chmod 777`など過剰な権限 → 最小権限の原則を確認。

## 4. チェックリスト
- [ ] 統合diffをファイル単位で見た(意図外の変更/削除なし)
- [ ] 追加された依存関係が実在し信頼できる
- [ ] シークレットスキャン通過(ハードコードキー0)
- [ ] 認証/決済/PIIコードは人が直接読んだ
- [ ] 変更パスにテストがあり、アサーションが弱められていない
- [ ] 入力検証・権限チェック・エラー処理がある
