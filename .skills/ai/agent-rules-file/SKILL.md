# 🤖 AI 에이전트 규칙 파일 (AGENTS.md)

> AI 코딩 에이전트가 리포 루트의 규칙 파일을 읽고 프로젝트 규약대로 작업하게 만든다. 새 프로젝트를 시작하거나 바이브 코딩 컨텍스트를 정비할 때, 에이전트가 명령어·컨벤션을 몰라 헤맬 때 읽는다.

AI 코딩 에이전트(Cursor, Claude Code, Windsurf, Codex, Copilot)는 리포 루트의 **AGENTS.md** 를
네이티브로 읽어 프로젝트별 작업 지침으로 삼는다. 2026년 6만+ 공개 리포가 채택한 사실상 표준이다.

## 1. 핵심 원칙

- 리포 루트에 `AGENTS.md`(또는 `CLAUDE.md`/`.cursorrules`)를 두고 프로젝트 개요·기술 스택·셋업 명령어·코드 스타일·테스트 방법·보안 주의사항 섹션을 둔다.
- 에이전트가 코드에서 스스로 알 수 없는 것만 적는다 — 빌드/실행/테스트 명령어, 컨벤션의 '왜', 하지 말아야 할 것(반례)에 집중한다.
- 추상적 설명 세 문단보다 실제 코드 스니펫 한 개가 낫다 — 원하는 패턴을 예시 코드로 보여준다.
- 파괴적·비가역 작업(DB 마이그레이션, 배포, 시크릿 변경, 대량 삭제)은 '사람 확인 없이 실행 금지'로 명시한다.
- 규칙 파일은 사람이 직접 큐레이션한다 — 자동 생성으로 방대해지면 토큰만 먹고 정확도가 떨어진다. 짧고 정확하게 유지한다.
- 여러 에이전트를 쓰면 `AGENTS.md` 를 단일 진실원으로 두고 `CLAUDE.md`/`.cursorrules` 는 심볼릭 링크나 참조로 통일한다.

## 2. 규칙

### 2-1. 왜 필요한가

에이전트는 코드를 읽어 *구조*는 파악하지만, **빌드/실행/테스트 명령어**, **컨벤션의 의도**,
**하면 안 되는 것**은 추론할 수 없다. 이걸 매 세션 반복 설명하지 않으려면 파일로 고정한다.

### 2-2. 권장 구조 (커뮤니티 수렴형)

```markdown
# <프로젝트명>

## Project Overview
한 문단으로 무엇을 만드는 서비스인지.

## Tech Stack
- Frontend: Vue 3 + Vite + Vuetify
- Backend: Spring Boot 3 (Java 17)
- DB: PostgreSQL + MyBatis

## Setup / Commands
- 설치: `pnpm i`
- 개발: `pnpm dev`
- 테스트: `pnpm test` (반드시 통과 후 커밋)
- 빌드: `pnpm build`

## Code Style
- 컴포넌트 PascalCase, 변수/함수 camelCase
- 커밋: `type(scope): 요약` (Conventional Commits)
- (반례) any 타입 금지 — unknown + 좁히기 사용

## Testing
- 변경한 코드 경로의 테스트를 추가/갱신하고 전체 그린 확인 후 커밋

## Security
- 시크릿은 .env (커밋 금지). 결제/인증 로직은 사람 검토 필수
- DB 마이그레이션·배포·대량 삭제는 사람 확인 없이 실행 금지
```

### 2-3. 작성 원칙

1. **에이전트가 모르는 것만 적는다.** 코드로 자명한 건 빼고, 명령어·의도·금지사항에 집중.
2. **반례(counterexample)가 가장 가치 있다.** "any 금지", "이 폴더는 자동 생성이라 수정 금지" 같은 직관에 반하는 규칙.
3. **스니펫 > 설명.** 원하는 패턴을 코드로 보여주면 장황한 설명보다 정확히 따른다.
4. **짧게 유지.** 방대한 자동 생성 파일은 토큰만 먹고 오히려 정확도를 떨어뜨린다 — 사람이 큐레이션.

### 2-4. 여러 에이전트 단일화

```bash
# ✅ 권장 — AGENTS.md 를 단일 진실원으로 두고 나머지는 심볼릭 링크
ln -s AGENTS.md CLAUDE.md
ln -s AGENTS.md .cursorrules
```

## 3. 흔한 실수

- 파일 하나에 수백 줄의 일반론(에이전트가 이미 아는 React 문법 등)을 채운다 → 토큰만 먹고 정확도가 떨어진다.
- 자주 바뀌는 세부 구현을 적는다 → 금방 stale 해진다. 안정적인 규약·명령어 위주로 적는다.
- 파괴적 작업의 금지 규칙을 빼먹는다 → 에이전트가 DB 마이그레이션·배포를 임의 실행한다.
- 에이전트별 규칙 파일을 따로 관리한다 → 내용이 어긋난다. 단일 진실원 + 링크로 통일한다.
