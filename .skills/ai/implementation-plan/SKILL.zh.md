# 📋 Implementation Plan & Progress Tracking Guide

> 在开始新的功能开发或缺陷修复之前制定实现计划并追踪进度状况的标准。在需要把握任务全局并与团队共享进度时阅读。

## 1. 核心原则
- 在编码之前先制定实现计划,防止迷失方向。
- 把大任务拆分为 UI·状态管理·API 对接等小单元。
- 用标准标记(`[ ]`/`[-]`/`[x]`/`[!]`)可视化并共享进度状态。
- 在计划阶段提前发现卡点和问题。

## 2. 规则

### 2-1. 为什么要编写实现计划?(给初级开发者的提示)
1. **防止迷失方向:** 如果从编码开始,容易陷入实现细节而忘记目的。先制定计划能起到导航的作用。
2. **拆分任务 (Divide & Conquer):** 看起来令人发怵的大任务,只要拆分为 UI、状态管理、API 对接等小单元,就会变成可以着手的大小。
3. **可视化进度:** 可以轻松地向团队成员(尤其是带教前辈)共享自己一整天做了什么、目前卡在哪里。
4. **提前阻断问题:** 在制定计划的阶段,就能提前发现诸如"咦?这个数据从哪里取?"之类的卡点。

### 2-2. 进度状态(Status)标记标准
* `[ ]` ⚪ **TODO**: 尚未开始
* `[-]` 🟡 **IN PROGRESS**: 正在进行
* `[x]` 🟢 **DONE**: 工作完成且通过自测
* `[!]` 🔴 **BLOCKED**: 因外部因素导致工作受阻

### 2-3. Implementation Plan 标准模板

#### 1. 📝 任务概述 (Task Overview)
* **任务目标:** (例:实现通用日期时间选择器 `<AppDateTimePicker>` 组件)
* **相关问题/工单:** `#123`
* **预计排期:** `2026.03.27 ~ 2026.03.29`

#### 2. 🔍 事前确认事项 (Prerequisites)
- [ ] 确认 Figma 设计稿
- [ ] 确认后端 API 规格书(Swagger)是否已更新
- [ ] 弄清是否存在已有的相似组件(避免重复开发)

#### 3. 🏃‍♂️ 分步实现计划 (Step-by-step Plan)

- **Phase 1: 搭建 UI 及组件骨架**
  - [x] 🟢 创建文件
  - [-] 🟡 基础 UI 标记

- **Phase 2: 状态管理及逻辑对接**
  - [ ] ⚪ 处理 `v-model` 绑定
  - [ ] ⚪ 添加 Validation 逻辑

- **Phase 3: API 对接**
  - [ ] ⚪ Mock 数据测试
  - [ ] ⚪ 实际 API 对接及错误处理

#### 4. ✅ 最终验证 (QA & Testing)
- [ ] ⚪ 确认桌面端及移动端环境 UI
- [ ] ⚪ 确认无控制台错误/警告
- [ ] ⚪ 确认无 `npm run build` 错误

## 3. 常见错误
- ❌ 不做计划直接编码 → 丢失目的而乱转。先制定分步计划。
- ❌ 把庞大的阶段当作一整块 → 看不见进度·问题。按 UI/状态/API 单元拆分。
- ❌ 缺少分阶段的完成(验证)标准 → "做完了"的判断含糊。在每个阶段放入验证项。
- ❌ 放任 BLOCKED 不管·不共享 → 前辈不知道卡点。明示外部因素并共享。
- ❌ 只制定计划而不更新状态 → 文档与现实脱节。每次进展都更新标记。
- ❌ 不考虑回滚·风险 → 失败时没有应对。

## 4. 检查清单
- [ ] 在编码之前编写了任务概述(目标·问题·排期)
- [ ] 检查了事前确认事项(设计稿、API 规格、相似组件)
- [ ] 把任务按 Phase 单元(UI/状态/API)进行了拆分
- [ ] 对每个项目应用标准状态标记(`[ ]`/`[-]`/`[x]`/`[!]`)
- [ ] 对 BLOCKED 项目明示外部因素并共享
- [ ] 通过了最终验证(桌面端/移动端 UI、控制台错误、`npm run build`)
