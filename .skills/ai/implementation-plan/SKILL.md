# 📋 Implementation Plan & Progress Tracking Guide

> 새 기능 개발이나 버그 수정을 시작하기 전 구현 계획을 세우고 진행 상황을 추적하기 위한 표준. 작업의 큰 그림을 잡고 진척을 팀과 공유해야 할 때 읽습니다.

## 1. 핵심 원칙
- 코딩 전에 구현 계획을 먼저 세워 길 잃음을 방지한다.
- 큰 작업은 UI·상태 관리·API 연동 등 작은 단위로 쪼갠다.
- 진행 상태를 표준 표기(`[ ]`/`[-]`/`[x]`/`[!]`)로 시각화해 공유한다.
- 계획 단계에서 막히는 부분과 이슈를 미리 발견한다.

## 2. 규칙

### 2-1. 왜 구현 계획을 작성해야 하나요? (주니어 개발자를 위한 팁)
1. **길 잃음 방지:** 코딩부터 시작하면 세부 구현에 빠져 목적을 잊기 쉽습니다. 계획을 먼저 세우면 내비게이션 역할을 해줍니다.
2. **작업 쪼개기 (Divide & Conquer):** 막막해 보이는 큰 작업도 UI, 상태 관리, API 연동 등 작은 단위로 쪼개면 해볼 만한 크기가 됩니다.
3. **진척도 시각화:** 내가 하루 종일 무엇을 했는지, 현재 어디에 멈춰있는지 팀원(특히 사수)에게 쉽게 공유할 수 있습니다.
4. **이슈 사전 차단:** 계획을 세우는 단계에서 '어? 이 데이터는 어디서 가져오지?' 같은 막히는 부분을 미리 발견할 수 있습니다.

### 2-2. 진행 상태(Status) 표기 표준
* `[ ]` ⚪ **TODO**: 아직 시작하지 않음
* `[-]` 🟡 **IN PROGRESS**: 현재 작업 중
* `[x]` 🟢 **DONE**: 작업 완료 및 자체 테스트 통과
* `[!]` 🔴 **BLOCKED**: 외부 요인으로 인해 작업이 막힘

### 2-3. Implementation Plan 표준 템플릿

#### 1. 📝 작업 요약 (Task Overview)
* **작업 목표:** (예: 공통 날짜 선택기 `<AppDateTimePicker>` 컴포넌트 구현)
* **관련 이슈/티켓:** `#123`
* **예상 일정:** `2026.03.27 ~ 2026.03.29`

#### 2. 🔍 사전 확인 사항 (Prerequisites)
- [ ] Figma 디자인 시안 확인
- [ ] 백엔드 API 명세서(Swagger) 업데이트 여부 확인
- [ ] 기존 유사 컴포넌트 존재 여부 파악 (중복 개발 방지)

#### 3. 🏃‍♂️ 단계별 구현 계획 (Step-by-step Plan)

- **Phase 1: UI 및 컴포넌트 뼈대 잡기**
  - [x] 🟢 파일 생성
  - [-] 🟡 기본 UI 마크업

- **Phase 2: 상태 관리 및 로직 연동**
  - [ ] ⚪ `v-model` 바인딩 처리
  - [ ] ⚪ Validation 로직 추가

- **Phase 3: API 연동**
  - [ ] ⚪ Mock 데이터 테스트
  - [ ] ⚪ 실제 API 연동 및 에러 핸들링

#### 4. ✅ 최종 검증 (QA & Testing)
- [ ] ⚪ 데스크톱 및 모바일 환경 UI 확인
- [ ] ⚪ 콘솔 에러/경고 없는지 확인
- [ ] ⚪ `npm run build` 에러 없는지 확인

## 3. 흔한 실수
- ❌ 계획 없이 바로 코딩 → 목적을 잃고 헤맨다. 먼저 단계 계획을 세운다.
- ❌ 거대 단계를 한 덩어리로 → 진척·이슈가 안 보인다. UI/상태/API 단위로 쪼갠다.
- ❌ 단계별 완료(검증) 기준 누락 → "끝났다" 판단이 모호. 각 단계에 검증 항목을 둔다.
- ❌ BLOCKED를 방치·미공유 → 사수가 막힘을 모른다. 외부 요인을 명시하고 공유한다.
- ❌ 계획만 세우고 상태를 갱신 안 함 → 문서가 현실과 괴리. 진행마다 표기를 갱신한다.
- ❌ 롤백·리스크를 고려 안 함 → 실패 시 대응이 없다.

## 4. 체크리스트
- [ ] 코딩 전에 작업 요약(목표·이슈·일정)을 작성했다
- [ ] 사전 확인 사항(디자인 시안, API 명세, 유사 컴포넌트)을 점검했다
- [ ] 작업을 Phase 단위(UI/상태/ API)로 쪼갰다
- [ ] 각 항목에 표준 상태 표기(`[ ]`/`[-]`/`[x]`/`[!]`)를 적용한다
- [ ] BLOCKED 항목은 외부 요인을 명시하고 공유한다
- [ ] 최종 검증(데스크톱/모바일 UI, 콘솔 에러, `npm run build`)을 통과했다
