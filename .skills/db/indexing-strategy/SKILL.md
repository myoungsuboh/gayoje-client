# 📇 인덱싱 전략 (Indexing Strategy)

> 인덱스 유형 선택과 설계·유지 관리를 통일한다. 새 인덱스를 만들거나 느린 쿼리를 튜닝하거나 미사용 인덱스를 정리할 때 읽는다. 특정 DB 엔진/도구에 종속되지 않는 범용 표준이다. 쿼리 자체를 다시 쓰는 방법은 query-optimization 스킬을 참고한다.

## 1. 핵심 원칙
- **인덱스는 읽기를 빠르게, 쓰기를 느리게 한다**: 모든 쓰기(INSERT/UPDATE/DELETE)는 인덱스도 함께 갱신한다. WHERE·JOIN·ORDER BY에 실제로 쓰이는 컬럼에 집중하고, 과잉 생성을 피한다.
- **측정 후 설계한다**: 추측으로 인덱스를 만들지 말고, 실행계획(EXPLAIN류)으로 느린 접근(풀스캔 등)을 먼저 확인한 뒤 인덱스를 추가하고, 다시 실행계획으로 효과를 검증한다.
- **복합 인덱스는 컬럼 순서가 전부다**: 선택도(cardinality)가 높은 컬럼과 등가(=) 조건 컬럼을 앞에, 범위(`>`,`<`,`BETWEEN`)·정렬 컬럼을 뒤에 둔다. 인덱스는 "왼쪽 접두사(leftmost prefix)"만 활용되므로 쿼리 조건 순서와 맞춘다.
- **컬럼을 가공하면 인덱스를 못 탄다**: WHERE 절에서 컬럼에 함수·연산을 씌우거나(`UPPER(col)`, `col + 1`) 앞부분이 와일드카드인 패턴(`'%suffix'`)을 쓰면 인덱스가 무력화된다 — 식 자체를 인덱싱하거나 쿼리를 바꾼다.
- **꼭 필요한 만큼만, 조건부로 좁힌다**: 일부 행만 조회 대상이면 조건부(부분) 인덱스로 크기를 줄이고, 조회에 필요한 컬럼을 인덱스에 포함(커버링)하면 테이블 접근 없이 인덱스만으로 응답할 수 있다.
- **미사용 인덱스는 부채다**: 한 번도 사용되지 않는 인덱스는 쓰기 비용·저장 공간만 잡아먹는다. 사용 통계를 주기적으로 확인해 제거한다.
- **운영 테이블 인덱스 생성은 락을 조심한다**: 큰 운영 테이블에 인덱스를 만들면 쓰기 락이 걸려 장애가 될 수 있다 — DB가 제공하는 온라인/무중단 인덱스 생성 옵션을 쓴다.

## 2. 규칙

### 2-1. 인덱스 대상은 WHERE·JOIN·ORDER BY 컬럼으로 한정
- 실제 쿼리가 거르거나·조인하거나·정렬하는 컬럼에만 인덱스를 둔다.
- "혹시 몰라서" 모든 컬럼에 인덱스를 거는 일은 금지 — 쓰기마다 모든 인덱스를 갱신하느라 느려진다.

```text
// ❌ 금지 — 쿼리에서 쓰지도 않는 컬럼까지 전부 인덱스
index(col1); index(col2); index(col3); ... // 테이블의 거의 모든 컬럼

// ✅ 권장 — 실제 조회 패턴이 쓰는 컬럼만
index(orders.user_id)          // WHERE user_id = ?
index(orders.status)           // WHERE status = ?
```

### 2-2. 인덱스 유형은 접근 패턴으로 고른다
대부분의 등가·범위·정렬은 기본(정렬형, 흔히 B-tree) 인덱스로 충분하다. 아래는 개념적 선택 기준이며, 실제 지원 유형·문법은 DB마다 다르다(부록 참고).

| 유형(개념) | 언제 쓰나 |
|------|-----------|
| 정렬형 기본 인덱스 (B-tree류) | 등가·범위·정렬 — 가장 보편적 |
| 복합(다중 컬럼) 인덱스 | 여러 컬럼을 함께 거르는 조건 |
| 조건부(부분) 인덱스 | 특정 조건의 행만 자주 조회할 때 (예: 활성 행만) |
| 커버링 인덱스 | 조회 컬럼을 인덱스가 모두 포함해 테이블 접근을 없앨 때 |
| 식(함수) 인덱스 | `lower(email)`처럼 가공된 값으로 조회할 때 |
| 등가 전용 인덱스 (해시류) | 정렬·범위 없이 등가 비교만 필요할 때 |

### 2-3. 복합 인덱스 컬럼 순서
등가 조건·선택도 높은 컬럼을 앞에, 범위·정렬 컬럼을 뒤에 두고, 쿼리 조건 순서와 일치시킨다. 인덱스는 왼쪽 접두사부터만 활용된다.

```sql
-- ✅ 권장 — 등가(=) 컬럼 먼저, 정렬 컬럼은 뒤, 쿼리 조건 순서와 일치
CREATE INDEX idx_orders_user_status_created ON orders (user_id, status, created_at DESC);

-- 위 인덱스를 타는 쿼리
SELECT * FROM orders
 WHERE user_id = ? AND status = 'paid'
 ORDER BY created_at DESC;
```

```text
// ❌ 금지 — 인덱스가 (a, b, c)인데 쿼리는 b만 조건
//          왼쪽 접두사(a)를 건너뛰어 인덱스를 못 탄다
index(a, b, c)  ↔  WHERE b = ?      // a 없이 b만 → 활용 불가
```

### 2-4. 컬럼을 가공한 조건은 인덱스를 무력화한다
WHERE 절에서 컬럼 자체를 함수·연산으로 감싸면 일반 인덱스를 타지 못한다. 식을 그대로 인덱싱(식 인덱스)하거나 쿼리를 바꾼다.

```sql
-- ❌ 금지 — 컬럼을 함수로 감싸면 일반 인덱스 무력화
SELECT * FROM users WHERE lower(email) = 'a@b.com';   -- email 인덱스를 못 탐
SELECT * FROM orders WHERE created_at + 1 > ?;        -- 연산으로 무력화

-- ✅ 권장 — 식 자체를 인덱싱하거나, 컬럼을 가공하지 않도록 조건을 변형
-- (식 인덱스: lower(email) 을 인덱싱)
-- (범위 변형: created_at > ? - 1  처럼 컬럼은 그대로 두고 상수쪽을 가공)
```

### 2-5. 패턴 검색과 인덱스
앞부분이 고정된 접두 검색은 인덱스를 활용하지만, 앞이 와일드카드면 풀스캔이다. 후방/부분 일치가 잦으면 전문 검색(full-text) 기능이나 검색 엔진을 고려한다.

```sql
-- ✅ 권장 — 접두 검색은 인덱스 활용
SELECT * FROM users WHERE name LIKE 'Kim%';

-- ❌ 금지 — 앞이 와일드카드면 풀스캔, 전문 검색 엔진 고려
SELECT * FROM users WHERE name LIKE '%Kim';
```

### 2-6. 실행계획(EXPLAIN)으로 검증
인덱스를 추가하기 전·후로 실행계획을 보고, 풀스캔이 인덱스 접근으로 바뀌는지 확인한다. 구체적 문법·출력은 DB마다 다르다(부록 참고).

```sql
-- 표준 진입점 (출력 형식·옵션은 DB마다 다름)
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 공통 확인 항목:
--  · 풀스캔(Full/Seq Scan) → 인덱스 접근(Index Scan 등)으로 바뀌는지
--  · 옵티마이저의 예상 행 수 vs 실제 행 수 차이 (크면 통계 갱신 필요)
--  · 실제로 의도한 인덱스를 쓰는지
```

### 2-7. 커버링 인덱스 (인덱스 온리 접근)
조회에 필요한 컬럼을 인덱스가 모두 담고 있으면 테이블 본체에 접근하지 않고 인덱스만으로 응답한다. 자주 함께 조회되는 소수 컬럼에 효과적이다. (포함 컬럼 지정 문법은 DB마다 다름 → 부록)

```text
// 개념 — 조회 컬럼(b, c)을 인덱스에 함께 담으면 테이블 접근 생략
index on t(a) + 포함(b, c)   ↔   SELECT a, b, c FROM t WHERE a = ?
```

### 2-8. 미사용 인덱스 감지·제거
어떤 쿼리에도 쓰이지 않는 인덱스는 쓰기 비용·저장 공간만 차지한다. DB의 사용 통계(스캔 횟수 등)를 주기적으로 확인해, 0회로 오래 유지된 인덱스는 제거 대상에 올린다. (통계 조회 방법은 DB마다 다름 → 부록)

```text
// 개념 — 사용 횟수 0이고 충분히 관찰된 인덱스를 제거 후보로
인덱스별 사용횟수 조회 → 사용횟수 = 0 인 것 → 삭제 후보 검토
```

### 2-9. 운영 테이블 인덱스 생성은 락을 피한다
큰 운영 테이블에 평범하게 인덱스를 만들면 생성 동안 쓰기 락이 걸려 장애가 될 수 있다. DB가 제공하는 온라인/무중단 인덱스 생성 옵션을 사용한다. (해당 옵션 문법은 DB마다 다름 → 부록)

```text
// ❌ 금지 — 큰 운영 테이블에 락 거는 일반 생성
일반 인덱스 생성  → 생성 동안 테이블 쓰기 락 → 장애 위험

// ✅ 권장 — DB가 제공하는 온라인/무중단 생성 옵션 사용
무중단 인덱스 생성 옵션  → 락 없이 백그라운드 구축
```

## 3. 흔한 실수
- **모든 컬럼에 인덱스 남발** → 쓰기 성능 저하와 저장 공간 낭비. 실제 조회 패턴 컬럼만 인덱싱한다.
- **복합 인덱스 컬럼 순서를 쿼리와 다르게 배치** → 왼쪽 접두사를 못 맞춰 인덱스를 타지 못한다.
- **WHERE 절에서 컬럼을 함수·연산으로 가공** → 일반 인덱스 무력화. 식 인덱스를 쓰거나 조건을 변형한다.
- **앞이 와일드카드인 LIKE에 인덱스 기대** (`'%suffix'`) → 풀스캔. 전문 검색을 고려한다.
- **인덱스를 만들고 실행계획으로 검증하지 않음** → 옵티마이저가 안 쓰는 인덱스를 방치하게 된다.
- **미사용 인덱스 방치** → 불필요한 쓰기·저장 비용. 사용 통계로 주기 점검한다.
- **운영 테이블에 락 거는 일반 생성** → 인덱스 생성 중 쓰기 락으로 장애. 무중단 생성 옵션을 쓴다.

## 4. 체크리스트
- [ ] WHERE·JOIN·ORDER BY 컬럼에 인덱스가 있고 과잉 생성이 없는가
- [ ] 복합 인덱스 컬럼 순서가 등가/선택도/정렬·쿼리 순서(왼쪽 접두사)와 맞는가
- [ ] WHERE 조건에서 컬럼을 함수·연산으로 가공해 인덱스를 죽이지 않았는가
- [ ] 앞이 와일드카드인 패턴 검색을 인덱스로 처리하려 하지 않았는가 (전문 검색 고려)
- [ ] 인덱스 추가 전·후로 실행계획(EXPLAIN)으로 인덱스 접근 전환을 확인했는가
- [ ] 자주 함께 조회되는 컬럼은 커버링으로 테이블 접근을 줄였는가
- [ ] 미사용 인덱스(사용 횟수 0)를 점검·제거했는가
- [ ] 운영 테이블 인덱스를 무중단/온라인 옵션으로 생성했는가

## 부록: 스택별 예시

> 아래는 위 1~4 표준의 PostgreSQL 문법 예시다(개념·선택 기준은 본문 참조). 팀 DB에 맞게 추가한다(예: MySQL/InnoDB, Oracle, SQL Server, MongoDB 등).

### PostgreSQL

```sql
-- 인덱스 유형별 문법 (유형 선택 기준은 2-2)
CREATE INDEX ON t(col);                       -- B-tree (기본): 범위·등가·정렬
CREATE INDEX ON t(a, b);                       -- Composite: 다중 컬럼
CREATE INDEX ON t(col) WHERE active = true;    -- Partial: 조건부 행만
CREATE INDEX ON t(a) INCLUDE (b, c);           -- Covering: INCLUDE 로 비검색 컬럼 포함
CREATE INDEX USING HASH ON t(col);             -- Hash: 등가 전용

-- 실행계획 검증 (2-6) — actual vs estimated 차이 크면 ANALYZE, Buffers 로 캐시 히트율
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM users WHERE email = 'test@example.com';

-- 미사용 인덱스 감지 (2-8) — idx_scan = 0 인 것을 크기순으로
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0 AND schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;

-- 무중단 인덱스 생성 (2-9) — CONCURRENTLY: 락 없이 생성
CREATE INDEX CONCURRENTLY idx_orders_user ON orders(user_id);
```
