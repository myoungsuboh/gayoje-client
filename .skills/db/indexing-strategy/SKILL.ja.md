# 📇 インデックス戦略 (Indexing Strategy)

> インデックス種別の選択と設計・保守を統一する。新しいインデックスを作成したり遅いクエリをチューニングしたり、未使用インデックスを整理するときに読む。特定のDBエンジン/ツールに依存しない汎用標準である。クエリ自体を書き直す方法は query-optimization スキルを参照する。

## 1. 核心原則
- **インデックスは読み取りを速く、書き込みを遅くする**: すべての書き込み(INSERT/UPDATE/DELETE)はインデックスも一緒に更新する。WHERE・JOIN・ORDER BYで実際に使われるカラムに集中し、作りすぎを避ける。
- **測定してから設計する**: 推測でインデックスを作らず、実行計画(EXPLAIN系)で遅いアクセス(フルスキャン等)をまず確認したうえでインデックスを追加し、再び実行計画で効果を検証する。
- **複合インデックスはカラムの順序がすべてだ**: 選択度(cardinality)の高いカラムと等価(=)条件のカラムを前に、範囲(`>`,`<`,`BETWEEN`)・ソートのカラムを後ろに置く。インデックスは「左端接頭辞(leftmost prefix)」のみ活用されるため、クエリ条件の順序に合わせる。
- **カラムを加工するとインデックスが効かない**: WHERE句でカラムに関数・演算をかけたり(`UPPER(col)`, `col + 1`)、先頭がワイルドカードのパターン(`'%suffix'`)を使うとインデックスが無効化される — 式自体をインデックス化するか、クエリを変える。
- **必要な分だけ、条件付きで絞る**: 一部の行だけが照会対象なら条件付き(部分)インデックスでサイズを減らし、照会に必要なカラムをインデックスに含める(カバリング)とテーブルにアクセスせずインデックスだけで応答できる。
- **未使用インデックスは負債だ**: 一度も使われないインデックスは書き込みコスト・ストレージ容量だけを食う。利用統計を定期的に確認して削除する。
- **本番テーブルのインデックス作成はロックに注意する**: 大きな本番テーブルにインデックスを作ると書き込みロックがかかり障害になりうる — DBが提供するオンライン/無停止インデックス作成オプションを使う。

## 2. ルール

### 2-1. インデックス対象はWHERE・JOIN・ORDER BYのカラムに限定
- 実際のクエリが絞り込む・結合する・ソートするカラムにのみインデックスを置く。
- 「念のため」すべてのカラムにインデックスを張ることは禁止 — 書き込みのたびにすべてのインデックスを更新して遅くなる。

```text
// ❌ 禁止 — クエリで使いもしないカラムまで全部インデックス
index(col1); index(col2); index(col3); ... // テーブルのほぼ全カラム

// ✅ 推奨 — 実際の照会パターンが使うカラムだけ
index(orders.user_id)          // WHERE user_id = ?
index(orders.status)           // WHERE status = ?
```

### 2-2. インデックス種別はアクセスパターンで選ぶ
ほとんどの等価・範囲・ソートは基本(ソート型、多くはB-tree)インデックスで十分だ。以下は概念的な選択基準であり、実際にサポートされる種別・構文はDBごとに異なる(付録参照)。

| 種別(概念) | いつ使うか |
|------|-----------|
| ソート型基本インデックス (B-tree系) | 等価・範囲・ソート — 最も一般的 |
| 複合(複数カラム)インデックス | 複数カラムを一緒に絞り込む条件 |
| 条件付き(部分)インデックス | 特定条件の行だけを頻繁に照会するとき(例: アクティブな行のみ) |
| カバリングインデックス | 照会カラムをインデックスがすべて含み、テーブルアクセスをなくすとき |
| 式(関数)インデックス | `lower(email)`のように加工された値で照会するとき |
| 等価専用インデックス (ハッシュ系) | ソート・範囲なしに等価比較だけが必要なとき |

### 2-3. 複合インデックスのカラム順序
等価条件・選択度の高いカラムを前に、範囲・ソートのカラムを後ろに置き、クエリ条件の順序と一致させる。インデックスは左端接頭辞からのみ活用される。

```sql
-- ✅ 推奨 — 等価(=)カラムが先、ソートカラムは後、クエリ条件の順序と一致
CREATE INDEX idx_orders_user_status_created ON orders (user_id, status, created_at DESC);

-- 上のインデックスを使うクエリ
SELECT * FROM orders
 WHERE user_id = ? AND status = 'paid'
 ORDER BY created_at DESC;
```

```text
// ❌ 禁止 — インデックスは(a, b, c)なのにクエリはbだけが条件
//          左端接頭辞(a)を飛ばしてインデックスを使えない
index(a, b, c)  ↔  WHERE b = ?      // aなしでbだけ → 活用不可
```

### 2-4. カラムを加工した条件はインデックスを無効化する
WHERE句でカラム自体を関数・演算で包むと通常のインデックスを使えない。式をそのままインデックス化(式インデックス)するか、クエリを変える。

```sql
-- ❌ 禁止 — カラムを関数で包むと通常のインデックスが無効化
SELECT * FROM users WHERE lower(email) = 'a@b.com';   -- emailインデックスを使えない
SELECT * FROM orders WHERE created_at + 1 > ?;        -- 演算で無効化

-- ✅ 推奨 — 式自体をインデックス化するか、カラムを加工しないように条件を変形
-- (式インデックス: lower(email) をインデックス化)
-- (範囲変形: created_at > ? - 1 のようにカラムはそのままで定数側を加工)
```

### 2-5. パターン検索とインデックス
先頭が固定された接頭検索はインデックスを活用するが、先頭がワイルドカードならフルスキャンだ。後方/部分一致が多いなら全文検索(full-text)機能や検索エンジンを検討する。

```sql
-- ✅ 推奨 — 接頭検索はインデックス活用
SELECT * FROM users WHERE name LIKE 'Kim%';

-- ❌ 禁止 — 先頭がワイルドカードならフルスキャン、全文検索エンジンを検討
SELECT * FROM users WHERE name LIKE '%Kim';
```

### 2-6. 実行計画(EXPLAIN)で検証
インデックスを追加する前後で実行計画を見て、フルスキャンがインデックスアクセスに変わるか確認する。具体的な構文・出力はDBごとに異なる(付録参照)。

```sql
-- 標準的な入口 (出力形式・オプションはDBごとに異なる)
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 共通の確認項目:
--  · フルスキャン(Full/Seq Scan) → インデックスアクセス(Index Scan等)に変わるか
--  · オプティマイザの予想行数 vs 実際の行数の差(大きければ統計の更新が必要)
--  · 実際に意図したインデックスを使うか
```

### 2-7. カバリングインデックス (インデックスオンリーアクセス)
照会に必要なカラムをインデックスがすべて持っていれば、テーブル本体にアクセスせずインデックスだけで応答する。頻繁に一緒に照会される少数のカラムに効果的だ。(含めるカラムを指定する構文はDBごとに異なる → 付録)

```text
// 概念 — 照会カラム(b, c)をインデックスに一緒に持たせるとテーブルアクセスを省略
index on t(a) + 含む(b, c)   ↔   SELECT a, b, c FROM t WHERE a = ?
```

### 2-8. 未使用インデックスの検知・削除
どのクエリにも使われないインデックスは書き込みコスト・ストレージ容量だけを占める。DBの利用統計(スキャン回数等)を定期的に確認し、0回で長く維持されたインデックスは削除対象に挙げる。(統計の照会方法はDBごとに異なる → 付録)

```text
// 概念 — 使用回数0で十分に観察されたインデックスを削除候補に
インデックスごとの使用回数を照会 → 使用回数 = 0 のもの → 削除候補を検討
```

### 2-9. 本番テーブルのインデックス作成はロックを避ける
大きな本番テーブルに普通にインデックスを作ると作成中に書き込みロックがかかり障害になりうる。DBが提供するオンライン/無停止インデックス作成オプションを使う。(該当オプションの構文はDBごとに異なる → 付録)

```text
// ❌ 禁止 — 大きな本番テーブルにロックをかける通常作成
通常のインデックス作成  → 作成中にテーブル書き込みロック → 障害リスク

// ✅ 推奨 — DBが提供するオンライン/無停止作成オプションを使う
無停止インデックス作成オプション  → ロックなしでバックグラウンド構築
```

## 3. よくあるミス
- **すべてのカラムにインデックス乱発** → 書き込み性能の低下とストレージ容量の浪費。実際の照会パターンのカラムだけをインデックス化する。
- **複合インデックスのカラム順序をクエリと違えて配置** → 左端接頭辞を合わせられずインデックスを使えない。
- **WHERE句でカラムを関数・演算で加工** → 通常のインデックス無効化。式インデックスを使うか条件を変形する。
- **先頭がワイルドカードのLIKEにインデックスを期待** (`'%suffix'`) → フルスキャン。全文検索を検討する。
- **インデックスを作って実行計画で検証しない** → オプティマイザが使わないインデックスを放置することになる。
- **未使用インデックスの放置** → 不要な書き込み・ストレージコスト。利用統計で定期点検する。
- **本番テーブルにロックをかける通常作成** → インデックス作成中の書き込みロックで障害。無停止作成オプションを使う。

## 4. チェックリスト
- [ ] WHERE・JOIN・ORDER BYのカラムにインデックスがあり、作りすぎがないか
- [ ] 複合インデックスのカラム順序が等価/選択度/ソート・クエリ順序(左端接頭辞)と合っているか
- [ ] WHERE条件でカラムを関数・演算で加工してインデックスを殺していないか
- [ ] 先頭がワイルドカードのパターン検索をインデックスで処理しようとしていないか(全文検索を検討)
- [ ] インデックス追加の前後で実行計画(EXPLAIN)でインデックスアクセスへの切り替えを確認したか
- [ ] 頻繁に一緒に照会されるカラムはカバリングでテーブルアクセスを減らしたか
- [ ] 未使用インデックス(使用回数0)を点検・削除したか
- [ ] 本番テーブルのインデックスを無停止/オンラインオプションで作成したか

## 付録: スタック別の例

> 以下は上記1〜4標準のPostgreSQL構文の例だ(概念・選択基準は本文参照)。チームのDBに合わせて追加する(例: MySQL/InnoDB, Oracle, SQL Server, MongoDB等)。

### PostgreSQL

```sql
-- インデックス種別ごとの構文 (種別選択の基準は2-2)
CREATE INDEX ON t(col);                       -- B-tree (基本): 範囲・等価・ソート
CREATE INDEX ON t(a, b);                       -- Composite: 複数カラム
CREATE INDEX ON t(col) WHERE active = true;    -- Partial: 条件付きの行のみ
CREATE INDEX ON t(a) INCLUDE (b, c);           -- Covering: INCLUDE で非検索カラムを含める
CREATE INDEX USING HASH ON t(col);             -- Hash: 等価専用

-- 実行計画の検証 (2-6) — actual vs estimated の差が大きければ ANALYZE、Buffers でキャッシュヒット率
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM users WHERE email = 'test@example.com';

-- 未使用インデックスの検知 (2-8) — idx_scan = 0 のものをサイズ順に
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0 AND schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;

-- 無停止インデックス作成 (2-9) — CONCURRENTLY: ロックなしで作成
CREATE INDEX CONCURRENTLY idx_orders_user ON orders(user_id);
```
