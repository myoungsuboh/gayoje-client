# 🔐 密钥管理标准 (Secrets Management)

> 将 API 密钥·密码·证书·令牌等敏感信息放在代码之外，在信任边界内注入，扫描提交·历史以阻止暴露，并定期更换。在处理密钥、决定配置注入方式、防止提交前暴露时阅读。这是不依赖特定语言/工具/云的通用标准。（通过环境变量注入·校验配置值的方法本身请参考 env-config。）

## 1. 核心原则

底层思路只有一个 — **应用只「接收并使用」密钥，并不「持有」它们。** 进入代码的密钥即视为已泄漏。由此引出以下几点。

- **放在外部并接受注入**: 密钥在运行时从执行环境或专用管理器注入，环境越往上（本地→共享→生产）越使用具有更强管控的存储。
- **自动阻止暴露**: 版本控制中只放占位/示例值，扫描提交·CI·过往历史，使真实密钥无法泄入。
- **不作为永久资产保留**: 定期且在疑似泄漏时立即更换（轮换），启动时校验缺失（fail-fast），不在日志·错误中留下痕迹。

具体实践在 §2 规则中以 ✅/❌ 处理。

## 2. 规则

### 2-1. 不在代码·配置中硬编码
- 不将密钥直接写入源码、配置文件、容器镜像、注释。
- 哪怕只被提交·部署过一次的密钥也视为「已暴露」并立即轮换。

```text
// ❌ 금지 — 시크릿을 코드에 박음
apiKey = "sk_live_8f2a...";

// ✅ 권장 — 실행 환경/시크릿 매니저에서 주입받아 사용
apiKey = readSecret("PAYMENT_API_KEY")   // 값은 코드 밖에 존재
```

### 2-2. 按环境分离存储
- 环境越往上越使用管控更强的存储。本地明文文件仅限单个开发者范围，共享·生产则放到具备访问管控·审计之处。
- 由团队就「何处放何物」达成一致并一贯执行。

```text
// 저장소를 민감도에 맞게 분리 (도구는 팀 스택에 맞게)
로컬 개발 : 로컬 평문 파일 (버전 관리 제외, 최소 권한)
팀 공유   : 비밀번호/시크릿 공유 도구 (접근 통제)
스테이징/운영 : 전용 시크릿 관리 서비스 (접근 통제 + 감사 로그)
```

### 2-3. 不向版本控制提交真实密钥
- 含真实密钥值的密钥文件排除于版本控制之外（登记到忽略列表）。
- 改为共享只展示键名·结构的**占位/示例值**文件，使新人知道该填什么。

```text
// ❌ 금지 — 실제 시크릿이 담긴 파일을 커밋
commit secrets.local   // 이력에 영구히 남는다

// ✅ 권장 — 무시 목록에 실제 파일 등록 + 더미 예시만 공유
ignore: secrets.local
commit: secrets.example   // DATABASE_URL=... (더미값)
```

### 2-4. 以提交·历史扫描阻断泄漏
- 在**提交时点**（本地钩子）与 **CI 阶段**扫描密钥模式，阻止以暴露状态被合并。
- 不仅扫描新提交，也定期扫描**整个过往历史**。对已混入的密钥要找出并轮换·移除。

```text
// ✅ 권장 — 두 겹으로 차단
커밋 전(훅) : 변경분에서 시크릿 패턴 탐지 → 발견 시 커밋 차단
CI          : 머지 전 동일 스캔으로 우회 방지
이력 스캔   : 과거 커밋 전체 주기적 스캔 → 발견 시 로테이션
```

### 2-5. 启动时校验必需密钥 (fail-fast)
- 应用在启动时确认所需密钥是否全部注入，缺失则立即停止。
- 不让其以空值·默认值悄悄运行，到运行时很久之后才崩溃。

```text
// ❌ 금지 — 없으면 빈 값으로 진행하다 한참 뒤 실패
key = readSecret("JWT_SECRET")   // 없으면 null로 계속 진행

// ✅ 권장 — 시작 시 필수 목록을 한 번에 검증
required = [DATABASE_URL, JWT_SECRET, ...]
missing  = required where not provided
if missing: abort("필수 시크릿 누락: " + missing)
```

### 2-6. 定期轮换与失效自动化
- 密钥按既定周期更换，并具备疑似泄漏时可立即更换的流程。
- 尽量采用零停机更换（新发 → 切换 → 弃用旧密钥），并自动化失效·续期以减少人工。

```text
// ✅ 권장 — 무중단 교체 + 자동화
1) 새 시크릿 발급/적용
2) 트래픽을 새 시크릿으로 전환
3) 구 시크릿 폐기
+ 만료 임박 알림과 갱신을 자동화
```

### 2-7. 不在日志·错误中暴露密钥
- 脱敏处理，使密钥不被打印到日志·异常消息·调试输出·对外发出的 URL/追踪信息中。
- 也对 CI/构建日志施加脱敏，使密钥不以明文残留。

```text
// ❌ 금지 — 시크릿이 로그로 샘
log("connecting with token=" + token)

// ✅ 권장 — 마스킹
log("connecting with token=****")
```

## 3. 常见错误

仅摘取知道规则却仍频繁踩中的陷阱（省略规则的重复）。

- **"从提交里删了就安全"的错觉** → 已提交的密钥会永久留在历史中。别止于删文件，务必**轮换**。
- **忘记 CI/构建日志** → 即便运行时日志脱敏，密钥也常以明文泄入构建·CI 输出。脱敏要两边都做。
- **所有环境用同一密钥** → 图省事而共享，一处被攻破即全部被攻破。按环境分离·发放。
- **以空值悄悄运行** → 不校验密钥缺失就会以默认值·null 运转，到很久之后才崩溃。启动时 fail-fast。

## 4. 检查清单
- [ ] 是否不在代码·配置·镜像中硬编码密钥而是**从外部注入**
- [ ] 是否按环境（本地·共享·生产）分离了**与敏感度相称的存储**
- [ ] 是否不向版本控制提交真实密钥而只共享**占位/示例值**
- [ ] 是否以**提交钩子 + CI + 历史扫描**阻断密钥泄漏
- [ ] 是否在启动时**校验必需密钥**并在缺失时立即失败
- [ ] 是否自动化了**定期轮换**与失效告警/续期，并能在泄漏时立即弃用
- [ ] 日志·错误·CI 输出中的密钥是否已**脱敏**而不暴露
- [ ] 是否每个环境使用**不同的密钥**（使一处泄漏不蔓延至整体）

## 附录: 按技术栈的示例

> 以下为参考用实现示例。按团队所用技术栈（密钥管理器·扫描器·CI 等）以相同模式补充。上述 1~4 的原则·规则才是标准，附录只是其应用案例。

### 按工具的示例

#### 密钥存储层级 (示例)

按环境分离密钥的存储位置。

```
개발: .env (로컬, .gitignore)
     → 팀 공유: 1Password / Bitwarden / Vault
스테이징: CI/CD 환경 변수 (GitHub Actions Secrets)
프로덕션: HashiCorp Vault / AWS Secrets Manager / GCP Secret Manager
```

#### `.env` 文件管理 (dotenv 惯例)

含真实密钥的 `.env` 禁止提交，只提交占位值的 `.env.example`。

```bash
# ❌ 금지 — 실제 시크릿 파일을 레포에 커밋

# ✅ 권장
# .gitignore
.env
.env.local
.env.production

# 레포에 커밋 — 더미값으로
# .env.example
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
REDIS_URL=redis://localhost:6379
JWT_SECRET=your-secret-key-here
```

#### 密钥扫描配置 (detect-secrets / truffleHog)

以 pre-commit 钩子与历史扫描阻断密钥泄漏。

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
```

```bash
# truffleHog로 히스토리 전체 스캔
trufflehog git file://. --since-commit HEAD~50
```

#### 环境变量注入模式 (Python)

启动时校验必需环境变量以尽早捕获缺失。

```python
# ❌ 금지 — 시크릿을 코드에 하드코딩
JWT_SECRET = "super-secret-key"

# ✅ 권장 — 시작 시 필수 환경 변수 검증
import os

REQUIRED_VARS = ["DATABASE_URL", "JWT_SECRET", "REDIS_URL"]
missing = [v for v in REQUIRED_VARS if not os.environ.get(v)]
if missing:
    raise EnvironmentError(f"필수 환경 변수 누락: {missing}")
```

#### 密钥轮换自动化 (示例)

按密钥种类自动化零停机更换流程。

- API 密钥: 失效前 7 天告警 → 发放新密钥 → 停用旧密钥
- DB 密码: Blue-Green 方式 — 应用新密码 → 切换连接 → 删除旧密码
- TLS 证书: Let's Encrypt auto-renew 或 ACM 自动续期
