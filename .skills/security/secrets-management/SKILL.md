# 🔐 시크릿 관리 표준 (Secrets Management)

> API 키·비밀번호·인증서·토큰 같은 민감 정보를 코드 밖에 두고, 신뢰 경계 안에서 주입하며, 커밋·이력을 스캔해 노출을 막고, 정기적으로 교체한다. 시크릿을 다루거나 설정 주입 방식을 정할 때, 커밋 전 노출을 막을 때 읽는다. 특정 언어/도구/클라우드에 종속되지 않는 범용 표준이다. (환경 변수로 설정값을 주입·검증하는 방법 자체는 env-config 참고.)

## 1. 핵심 원칙

바탕에 깔린 생각은 하나다 — **애플리케이션은 시크릿을 "받아서 쓸" 뿐, "담고 있지" 않는다.** 코드에 들어간 시크릿은 이미 유출된 것으로 본다. 여기서 다음이 따라온다.

- **밖에 두고 주입받는다**: 시크릿은 실행 환경이나 전용 매니저에서 런타임에 주입하고, 환경이 올라갈수록(로컬→공유→운영) 더 강한 통제를 가진 저장소를 쓴다.
- **노출을 자동으로 막는다**: 버전 관리에는 더미/예시 값만 두고, 커밋·CI·과거 이력을 스캔해 실제 시크릿이 새어 들어가지 못하게 한다.
- **영구 자산으로 두지 않는다**: 정기적으로, 유출 의심 시 즉시 교체(로테이션)하고, 시작 시 누락을 검증(fail-fast)하며, 로그·에러에 흔적을 남기지 않는다.

구체적 실천은 §2 규칙에서 ✅/❌로 다룬다.

## 2. 규칙

### 2-1. 코드·설정에 하드코딩하지 않는다
- 시크릿을 소스, 설정 파일, 컨테이너 이미지, 주석에 직접 적지 않는다.
- 한 번이라도 커밋·배포된 시크릿은 "노출됐다"고 보고 즉시 로테이션한다.

```text
// ❌ 금지 — 시크릿을 코드에 박음
apiKey = "sk_live_8f2a...";

// ✅ 권장 — 실행 환경/시크릿 매니저에서 주입받아 사용
apiKey = readSecret("PAYMENT_API_KEY")   // 값은 코드 밖에 존재
```

### 2-2. 환경별로 저장소를 분리한다
- 환경이 올라갈수록 통제가 강한 저장소를 쓴다. 로컬 평문 파일은 개발자 1인 범위로만, 공유·운영은 접근 통제·감사가 되는 곳으로.
- "어디에 무엇을 두는지"를 팀이 합의해 일관되게 적용한다.

```text
// 저장소를 민감도에 맞게 분리 (도구는 팀 스택에 맞게)
로컬 개발 : 로컬 평문 파일 (버전 관리 제외, 최소 권한)
팀 공유   : 비밀번호/시크릿 공유 도구 (접근 통제)
스테이징/운영 : 전용 시크릿 관리 서비스 (접근 통제 + 감사 로그)
```

### 2-3. 버전 관리에 실제 시크릿을 커밋하지 않는다
- 실제 값이 담긴 시크릿 파일은 버전 관리에서 제외한다(무시 목록 등록).
- 대신 키 이름·구조만 보여주는 **더미/예시 값** 파일을 공유해 신규자가 무엇을 채워야 하는지 알게 한다.

```text
// ❌ 금지 — 실제 시크릿이 담긴 파일을 커밋
commit secrets.local   // 이력에 영구히 남는다

// ✅ 권장 — 무시 목록에 실제 파일 등록 + 더미 예시만 공유
ignore: secrets.local
commit: secrets.example   // DATABASE_URL=... (더미값)
```

### 2-4. 커밋·이력 스캐닝으로 유출을 차단한다
- **커밋 시점**(로컬 훅)과 **CI 단계**에서 시크릿 패턴을 스캔해, 노출된 채로 머지되는 것을 막는다.
- 새 커밋뿐 아니라 **과거 이력 전체**도 주기적으로 스캔한다. 이미 들어간 시크릿은 찾아서 로테이션·제거한다.

```text
// ✅ 권장 — 두 겹으로 차단
커밋 전(훅) : 변경분에서 시크릿 패턴 탐지 → 발견 시 커밋 차단
CI          : 머지 전 동일 스캔으로 우회 방지
이력 스캔   : 과거 커밋 전체 주기적 스캔 → 발견 시 로테이션
```

### 2-5. 시작 시 필수 시크릿을 검증한다 (fail-fast)
- 애플리케이션은 부팅 시 필요한 시크릿이 모두 주입됐는지 확인하고, 없으면 즉시 멈춘다.
- 빈 값·기본값으로 조용히 동작해 런타임 한참 뒤에 터지지 않게 한다.

```text
// ❌ 금지 — 없으면 빈 값으로 진행하다 한참 뒤 실패
key = readSecret("JWT_SECRET")   // 없으면 null로 계속 진행

// ✅ 권장 — 시작 시 필수 목록을 한 번에 검증
required = [DATABASE_URL, JWT_SECRET, ...]
missing  = required where not provided
if missing: abort("필수 시크릿 누락: " + missing)
```

### 2-6. 정기 로테이션과 만료 자동화
- 시크릿은 정해진 주기로 교체하고, 유출 의심 시 즉시 교체할 수 있는 절차를 갖춘다.
- 가능하면 무중단 교체(신규 발급 → 전환 → 구 시크릿 폐기)로, 만료·갱신을 자동화해 사람 손을 줄인다.

```text
// ✅ 권장 — 무중단 교체 + 자동화
1) 새 시크릿 발급/적용
2) 트래픽을 새 시크릿으로 전환
3) 구 시크릿 폐기
+ 만료 임박 알림과 갱신을 자동화
```

### 2-7. 로그·에러에 시크릿을 노출하지 않는다
- 로그·예외 메시지·디버그 출력·외부로 나가는 URL/추적 정보에 시크릿이 찍히지 않게 마스킹한다.
- CI/빌드 로그에도 시크릿이 평문으로 남지 않도록 마스킹을 적용한다.

```text
// ❌ 금지 — 시크릿이 로그로 샘
log("connecting with token=" + token)

// ✅ 권장 — 마스킹
log("connecting with token=****")
```

## 3. 흔한 실수

규칙을 알면서도 자주 빠지는 함정만 추린다(규칙의 반복은 생략).

- **"커밋에서 지웠으니 안전" 착각** → 커밋된 시크릿은 이력에 영구히 남는다. 파일을 지우는 걸로 끝내지 말고 반드시 **로테이션**한다.
- **CI/빌드 로그를 잊음** → 런타임 로그는 마스킹해도 빌드·CI 출력에 평문으로 새는 경우가 많다. 마스킹은 양쪽 모두에.
- **모든 환경에 같은 시크릿** → 편하다고 공유하면 한 곳이 뚫릴 때 전부 뚫린다. 환경별로 분리·발급한다.
- **빈 값으로 조용히 동작** → 시크릿 누락을 검증하지 않으면 기본값·null로 돌다 한참 뒤에 터진다. 부팅 시 fail-fast.

## 4. 체크리스트
- [ ] 시크릿을 코드·설정·이미지에 하드코딩하지 않고 **외부에서 주입**하는가
- [ ] 환경별(로컬·공유·운영)로 **민감도에 맞는 저장소**를 분리했는가
- [ ] 버전 관리에 실제 시크릿을 커밋하지 않고 **더미/예시 값만** 공유하는가
- [ ] **커밋 훅 + CI + 이력 스캔**으로 시크릿 유출을 차단하는가
- [ ] 부팅 시 **필수 시크릿을 검증**하고 없으면 즉시 실패하는가
- [ ] **정기 로테이션**과 만료 알림/갱신을 자동화했고, 유출 시 즉시 폐기할 수 있는가
- [ ] 로그·에러·CI 출력에 시크릿이 **마스킹**되어 노출되지 않는가
- [ ] 환경마다 **다른 시크릿**을 쓰는가 (한 곳 유출이 전체로 번지지 않게)

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(시크릿 매니저·스캐너·CI 등)에 맞게 같은 패턴으로 추가한다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다.

### 도구별 예시

#### 시크릿 저장 계층 (예시)

환경마다 시크릿 저장 위치를 분리한다.

```
개발: .env (로컬, .gitignore)
     → 팀 공유: 1Password / Bitwarden / Vault
스테이징: CI/CD 환경 변수 (GitHub Actions Secrets)
프로덕션: HashiCorp Vault / AWS Secrets Manager / GCP Secret Manager
```

#### `.env` 파일 관리 (dotenv 관례)

실제 시크릿이 담긴 `.env`는 커밋 금지, 더미값 `.env.example`만 커밋한다.

```bash
# ❌ 금지 — 실제 시크릿 파일을 레포에 커밋

# ✅ 권장
# .gitignore
.env
.env.local
.env.production

# 레포에 커밋 — 더미값으로
# .env.example
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
REDIS_URL=redis://localhost:6379
JWT_SECRET=your-secret-key-here
```

#### 시크릿 스캐닝 설정 (detect-secrets / truffleHog)

pre-commit 훅과 히스토리 스캔으로 시크릿 유출을 차단한다.

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
```

```bash
# truffleHog로 히스토리 전체 스캔
trufflehog git file://. --since-commit HEAD~50
```

#### 환경 변수 주입 패턴 (Python)

시작 시 필수 환경 변수를 검증해 누락을 조기에 잡는다.

```python
# ❌ 금지 — 시크릿을 코드에 하드코딩
JWT_SECRET = "super-secret-key"

# ✅ 권장 — 시작 시 필수 환경 변수 검증
import os

REQUIRED_VARS = ["DATABASE_URL", "JWT_SECRET", "REDIS_URL"]
missing = [v for v in REQUIRED_VARS if not os.environ.get(v)]
if missing:
    raise EnvironmentError(f"필수 환경 변수 누락: {missing}")
```

#### 시크릿 로테이션 자동화 (예시)

시크릿 종류별로 무중단 교체 절차를 자동화한다.

- API 키: 만료 7일 전 알림 → 새 키 발급 → 구 키 비활성화
- DB 비밀번호: Blue-Green 방식 — 새 비밀번호 적용 → 연결 전환 → 구 비밀번호 삭제
- TLS 인증서: Let's Encrypt auto-renew 또는 ACM 자동 갱신
