# 🔐 シークレット管理標準 (Secrets Management)

> APIキー・パスワード・証明書・トークンといった機微情報をコードの外に置き、信頼境界の中で注入し、コミット・履歴をスキャンして露出を防ぎ、定期的に交換する。シークレットを扱うとき、設定の注入方式を決めるとき、コミット前の露出を防ぐときに読む。特定の言語/ツール/クラウドに依存しない汎用標準だ。（環境変数で設定値を注入・検証する方法自体は env-config 参照。）

## 1. 核心原則

根底にある考えは一つ — **アプリケーションはシークレットを「受け取って使う」だけで、「保持して」はいない。** コードに入ったシークレットはすでに漏洩したものとみなす。ここから次が従う。

- **外に置いて注入を受ける**: シークレットは実行環境または専用マネージャーからランタイムに注入し、環境が上がるほど（ローカル→共有→運用）より強い統制を持つストアを使う。
- **露出を自動で防ぐ**: バージョン管理にはダミー/例の値だけを置き、コミット・CI・過去の履歴をスキャンして実際のシークレットが漏れ込まないようにする。
- **永久資産として置かない**: 定期的に、漏洩が疑われる場合は直ちに交換（ローテーション）し、起動時に欠落を検証（fail-fast）し、ログ・エラーに痕跡を残さない。

具体的な実践は§2のルールで✅/❌で扱う。

## 2. ルール

### 2-1. コード・設定にハードコーディングしない
- シークレットをソース、設定ファイル、コンテナイメージ、コメントに直接書かない。
- 一度でもコミット・デプロイされたシークレットは「露出した」とみなし、直ちにローテーションする。

```text
// ❌ 금지 — 시크릿을 코드에 박음
apiKey = "sk_live_8f2a...";

// ✅ 권장 — 실행 환경/시크릿 매니저에서 주입받아 사용
apiKey = readSecret("PAYMENT_API_KEY")   // 값은 코드 밖에 존재
```

### 2-2. 環境ごとにストアを分離する
- 環境が上がるほど統制の強いストアを使う。ローカル平文ファイルは開発者1人の範囲のみ、共有・運用はアクセス統制・監査が行われる場所へ。
- 「どこに何を置くか」をチームで合意し一貫して適用する。

```text
// 저장소를 민감도에 맞게 분리 (도구는 팀 스택에 맞게)
로컬 개발 : 로컬 평문 파일 (버전 관리 제외, 최소 권한)
팀 공유   : 비밀번호/시크릿 공유 도구 (접근 통제)
스테이징/운영 : 전용 시크릿 관리 서비스 (접근 통제 + 감사 로그)
```

### 2-3. バージョン管理に実際のシークレットをコミットしない
- 実際の値が入ったシークレットファイルはバージョン管理から除外する（無視リストに登録）。
- 代わりにキー名・構造だけを示す**ダミー/例の値**ファイルを共有し、新規参加者が何を埋めるべきか分かるようにする。

```text
// ❌ 금지 — 실제 시크릿이 담긴 파일을 커밋
commit secrets.local   // 이력에 영구히 남는다

// ✅ 권장 — 무시 목록에 실제 파일 등록 + 더미 예시만 공유
ignore: secrets.local
commit: secrets.example   // DATABASE_URL=... (더미값)
```

### 2-4. コミット・履歴スキャニングで漏洩を遮断する
- **コミット時点**（ローカルフック）と**CI段階**でシークレットパターンをスキャンし、露出したままマージされるのを防ぐ。
- 新規コミットだけでなく**過去の履歴全体**も定期的にスキャンする。すでに入ったシークレットは見つけてローテーション・除去する。

```text
// ✅ 권장 — 두 겹으로 차단
커밋 전(훅) : 변경분에서 시크릿 패턴 탐지 → 발견 시 커밋 차단
CI          : 머지 전 동일 스캔으로 우회 방지
이력 스캔   : 과거 커밋 전체 주기적 스캔 → 발견 시 로테이션
```

### 2-5. 起動時に必須シークレットを検証する (fail-fast)
- アプリケーションは起動時に必要なシークレットがすべて注入されたか確認し、なければ直ちに停止する。
- 空の値・デフォルト値で静かに動作し、ランタイムのずっと後で破綻しないようにする。

```text
// ❌ 금지 — 없으면 빈 값으로 진행하다 한참 뒤 실패
key = readSecret("JWT_SECRET")   // 없으면 null로 계속 진행

// ✅ 권장 — 시작 시 필수 목록을 한 번에 검증
required = [DATABASE_URL, JWT_SECRET, ...]
missing  = required where not provided
if missing: abort("필수 시크릿 누락: " + missing)
```

### 2-6. 定期ローテーションと失効の自動化
- シークレットは定められた周期で交換し、漏洩が疑われる場合に直ちに交換できる手順を備える。
- 可能なら無停止交換（新規発行 → 切替 → 旧シークレット廃棄）で、失効・更新を自動化して人手を減らす。

```text
// ✅ 권장 — 무중단 교체 + 자동화
1) 새 시크릿 발급/적용
2) 트래픽을 새 시크릿으로 전환
3) 구 시크릿 폐기
+ 만료 임박 알림과 갱신을 자동화
```

### 2-7. ログ・エラーにシークレットを露出しない
- ログ・例外メッセージ・デバッグ出力・外部へ出るURL/トレース情報にシークレットが出力されないようマスキングする。
- CI/ビルドログにもシークレットが平文で残らないようマスキングを適用する。

```text
// ❌ 금지 — 시크릿이 로그로 샘
log("connecting with token=" + token)

// ✅ 권장 — 마스킹
log("connecting with token=****")
```

## 3. よくある間違い

ルールを知りながらも頻繁に陥る罠だけを抜粋する（ルールの繰り返しは省略）。

- **「コミットから消したから安全」という錯覚** → コミットされたシークレットは履歴に永久に残る。ファイルを消して終わりにせず必ず**ローテーション**する。
- **CI/ビルドログを忘れる** → ランタイムログはマスキングしてもビルド・CI出力に平文で漏れる場合が多い。マスキングは両方に。
- **すべての環境に同じシークレット** → 楽だからと共有すると一箇所が破られたとき全部破られる。環境ごとに分離・発行する。
- **空の値で静かに動作** → シークレットの欠落を検証しなければデフォルト値・nullで回り、ずっと後で破綻する。起動時にfail-fast。

## 4. チェックリスト
- [ ] シークレットをコード・設定・イメージにハードコーディングせず**外部から注入**しているか
- [ ] 環境ごと（ローカル・共有・運用）に**機微度に合うストア**を分離したか
- [ ] バージョン管理に実際のシークレットをコミットせず**ダミー/例の値だけ**を共有しているか
- [ ] **コミットフック + CI + 履歴スキャン**でシークレット漏洩を遮断しているか
- [ ] 起動時に**必須シークレットを検証**し、なければ直ちに失敗するか
- [ ] **定期ローテーション**と失効通知/更新を自動化し、漏洩時に直ちに廃棄できるか
- [ ] ログ・エラー・CI出力にシークレットが**マスキング**され露出しないか
- [ ] 環境ごとに**異なるシークレット**を使うか（一箇所の漏洩が全体に波及しないよう）

## 付録: スタック別の例

> 以下は参考用の実装例だ。チームが使うスタック（シークレットマネージャー・スキャナー・CIなど）に合わせて同じパターンで追加する。上記1〜4の原則・ルールが標準であり、付録はその適用事例にすぎない。

### ツール別の例

#### シークレット保存階層 (例)

環境ごとにシークレットの保存場所を分離する。

```
개발: .env (로컬, .gitignore)
     → 팀 공유: 1Password / Bitwarden / Vault
스테이징: CI/CD 환경 변수 (GitHub Actions Secrets)
프로덕션: HashiCorp Vault / AWS Secrets Manager / GCP Secret Manager
```

#### `.env` ファイル管理 (dotenv 慣習)

実際のシークレットが入った `.env` はコミット禁止、ダミー値の `.env.example` だけをコミットする。

```bash
# ❌ 금지 — 실제 시크릿 파일을 레포에 커밋

# ✅ 권장
# .gitignore
.env
.env.local
.env.production

# 레포에 커밋 — 더미값으로
# .env.example
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
REDIS_URL=redis://localhost:6379
JWT_SECRET=your-secret-key-here
```

#### シークレットスキャニング設定 (detect-secrets / truffleHog)

pre-commit フックと履歴スキャンでシークレット漏洩を遮断する。

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
```

```bash
# truffleHog로 히스토리 전체 스캔
trufflehog git file://. --since-commit HEAD~50
```

#### 環境変数注入パターン (Python)

起動時に必須環境変数を検証して欠落を早期に捕捉する。

```python
# ❌ 금지 — 시크릿을 코드에 하드코딩
JWT_SECRET = "super-secret-key"

# ✅ 권장 — 시작 시 필수 환경 변수 검증
import os

REQUIRED_VARS = ["DATABASE_URL", "JWT_SECRET", "REDIS_URL"]
missing = [v for v in REQUIRED_VARS if not os.environ.get(v)]
if missing:
    raise EnvironmentError(f"필수 환경 변수 누락: {missing}")
```

#### シークレットローテーション自動化 (例)

シークレット種別ごとに無停止交換手順を自動化する。

- APIキー: 失効7日前に通知 → 新キー発行 → 旧キー無効化
- DBパスワード: Blue-Green方式 — 新パスワード適用 → 接続切替 → 旧パスワード削除
- TLS証明書: Let's Encrypt auto-renew または ACM 自動更新
