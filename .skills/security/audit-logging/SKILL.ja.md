# 🔒 セキュリティ監査ログ & イベント追跡

> セキュリティイベントを構造化された監査ログとして残し、インシデントを追跡し異常の兆候を検知する。認証・認可・機微データ変更・管理機能を作るときや、監査ログの構造・保存・アラートを決めるときに読む。

## 1. 中核となる原則

- 認証イベント(ログイン成功・失敗・ログアウト・トークン更新)と認可失敗は必ず監査ログに記録する。
- 監査ログは変更不可能(append-only)で保存し、アプリケーションログとは分離したストレージに保管する。
- 監査ログ項目にはタイムスタンプ(UTC)・行為者(user_id・IP)・対象(resource_id)・行為・結果を含める。
- 監査ログに個人情報を露出させず、機微な値はマスキングするかハッシュで参照する。
- ログイン失敗の反復・異常な時間帯のアクセス・大容量データ照会など異常パターンにアラートを設定する。

## 2. ルール

### 2-1. 監査ログ必須イベント

以下のカテゴリのイベントは漏れなく記録する。

| カテゴリ | イベント例 |
|----------|-------------|
| 認証 | ログイン成功/失敗、ログアウト、パスワード変更、MFA試行 |
| 認可 | 権限不足のアクセス試行、ロール変更 |
| データ変更 | 作成・更新・削除(機微リソース) |
| 管理行為 | ユーザー作成/削除、設定変更、権限付与 |
| システム | サービス起動/停止、設定ファイル変更 |

### 2-2. 監査ログ構造(必須フィールド + append-only記録)

```python
from dataclasses import dataclass
from datetime import datetime, timezone

@dataclass
class AuditEvent:
    timestamp: str     # ISO 8601 UTC
    event_type: str    # "AUTH_LOGIN_FAILED"
    actor_id: str      # "user_abc123" 또는 "system"
    actor_ip: str      # "192.168.1.1"
    resource_type: str # "User", "Order", "File"
    resource_id: str   # "order_xyz789"
    action: str        # "READ", "CREATE", "UPDATE", "DELETE"
    result: str        # "SUCCESS", "FAILURE", "DENIED"
    detail: dict       # 추가 컨텍스트 (민감 정보 제외)

def record_audit(event: AuditEvent):
    # append-only 스토리지에 기록
    audit_db.insert_one(asdict(event))
    # SIEM으로 스트리밍
    siem_client.send(event)
```

### 2-3. 機微情報のマスキング(個人情報の露出禁止)

```python
# ❌ 금지 — 비밀번호·토큰 등 민감 값을 평문으로 detail에 담음
detail = {"password": raw_password, "token": access_token}

# ✅ 권장 — 민감 값은 마스킹하거나 해시로 참조
detail = {"password": "***", "token_hash": sha256(access_token)}
```

### 2-4. 異常検知アラートルール

```python
ALERT_RULES = [
    # 5분 내 로그인 실패 5회 이상
    {"event": "AUTH_LOGIN_FAILED", "count": 5, "window_sec": 300},
    # 1시간 내 대용량 데이터 조회 (>1000 rows)
    {"event": "DATA_EXPORT", "threshold": 1000, "window_sec": 3600},
    # 근무 시간 외(22:00~06:00) 관리 행위
    {"event": "ADMIN_ACTION", "time_range": "22:00-06:00"},
]
```

### 2-5. ログ保存ポリシー

| ログ種別 | 保存期間 | 理由 |
|-----------|-----------|------|
| 認証ログ | 1年 | セキュリティ調査 |
| データ変更ログ | 3年 | 法的要件 |
| 管理行為ログ | 5年 | 規制遵守 |
| 一般アクセスログ | 90日 | ストレージコスト |

## 3. よくある間違い

- 監査ログをアプリケーションログと混在して保存 → 改ざん・削除リスク、追跡性の喪失。
- detailにパスワード・トークン・マイナンバーなど機微な値を平文で記録。
- 認可失敗(権限不足のアクセス)を記録せず、攻撃の試みを見逃す。
- アラートなしでログを溜めるだけ → 異常の兆候を事後にしか発見できない。
- 保存期間未設定 → 法的要件違反または不要なストレージコスト。
