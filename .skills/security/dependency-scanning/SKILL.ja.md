# 🛡️ 依存関係セキュリティスキャン (SCA)

> サードパーティ依存関係に潜む既知の脆弱性を自動的に検出・追跡・パッチし、深刻なものはビルド段階で止める。CIパイプラインにスキャンを組み込む、依存関係の自動更新を設定する、新しいライブラリを採用するときに読む。特定のパッケージマネージャ(npm/pip/Maven など)やCIツール(GitHub Actions/Dependabot など)に依存しない汎用標準である。バージョン固定・範囲ポリシーなど依存関係そのものの管理ルールは `dependency-management` を併せて見る。

## 1. 基本原則

出発点は一つだ — **取り込むすべてのサードパーティパッケージ(およびそれがさらに引き込む推移的依存)が攻撃面である。**「他人が作ったから安全」とは仮定しない。ここから次が導かれる。

- **自動的・継続的に検出する**: 脆弱性DBは毎日更新されるため、一度の点検で終わらない。CIと定期スケジュールで繰り返し実行し、新たに公開された脆弱性も捕捉する。
- **深刻なら止め、パッチは取り込む**: 一定の深刻度以上はビルドでゲーティングし(警告だけでは誰も見ない)、セキュリティパッチは自動化ツールにPRとして上げさせ、人はレビュー・マージに集中する。
- **攻撃面を小さく・再現可能に**: ロックファイルをコミットしてどこでも同じ依存ツリーになるようにし、本番にはランタイム依存だけを入れ、新しいライブラリは取り込む前に検証する。

具体的な実践は §2 のルールで ✅/❌ により扱う。

## 2. ルール

### 2-1. 外部依存を無条件に信頼しない
- 「広く使われているパッケージだから安全」と仮定せず、直接・推移的依存の両方をスキャン対象とみなす。
- 脆弱性は自分が直接呼ぶパッケージだけでなく、そのパッケージが引き込む深い依存からも出る。推移的依存まで含めて検出する。

```text
// ❌ 禁止 — 「有名だから」点検なしにそのまま追加して終わり
add some-popular-pkg            // どんな脆弱性が付いてくるか知らないまま使用

// ✅ 推奨 — 直接・推移的依存をすべて脆弱性スキャン対象に
scan(all dependencies incl. transitive) → 通過したものだけ使用
```

### 2-2. 脆弱性スキャンをCIに統合し、深刻ならビルドを止める
- 依存関係の脆弱性スキャンをPR/ビルドごとに自動実行し、一度きりでなく定期スケジュールでも再実行して、コードが変わらなくても新しい脆弱性を捕捉する。
- 定めた深刻度しきい値(例: CRITICAL)以上が見つかったらパイプラインを**失敗**させる。警告だけ残して通過させない。
- スキャン段階をパイプラインのどこに配置し、失敗時に後続をどう遮断するか(段階の直列化・fail-fast)は `CI/CD パイプライン標準` に従う。この標準は「何をどの深刻度で止めるか」(スキャナ・しきい値・ゲーティングポリシー)を定める。

```text
// ❌ 禁止 — スキャン結果をログに出すだけで常に通過
scan(deps); build_status = SUCCESS    // CRITICALがあってもそのままデプロイ

// ✅ 推奨 — しきい値以上ならパイプライン失敗(ゲーティング)
result = scan(deps)
if result.maxSeverity >= CRITICAL: fail build   // 本番への流入を遮断
```

### 2-3. セキュリティパッチを自動提案で受け取る
- 依存関係の自動更新ツールを有効化し、脆弱な依存のセキュリティパッチを自動変更リクエスト(PR/MR)として受け取る。
- セキュリティ更新は別にまとめて優先レビュー・マージする。人が脆弱性通知を手動で追わないようにする。

```text
// ❌ 禁止 — 人が通知を手動追跡、結局パッチ漏れ・遅延
"誰かたまに依存バージョンを上げるだろう"      // セキュリティパッチが何ヶ月も放置

// ✅ 推奨 — 自動化ツールがパッチPRを上げ、人はレビュー・マージ
auto-update tool → opens PR(セキュリティパッチ) → レビュー後マージ
```

### 2-4. 脆弱性の優先順位と対応期限(SLA)
- 標準的な深刻度スコア(例: CVSS)に従って対応期限を定め、致命的なものから処理する。
- **「深刻なほど早く」が原則で、具体的な期限はチームが定める。** 下の表は出発点の例にすぎないので、リスク許容度に合わせてしきい値・期限を調整する。

| 深刻度(例: CVSS) | 等級 | 対応期限 — 例(チームが調整) |
|-----------------|--------|----------------|
| 9.0–10.0 | CRITICAL | 最速で(例: 数日以内) |
| 7.0–8.9 | HIGH | 速やかに(例: 1〜2週間以内) |
| 4.0–6.9 | MEDIUM | 計画に反映(例: 1ヶ月以内) |
| 0.1–3.9 | LOW | 次のリリース |

### 2-5. 新しいライブラリの採用基準
- 新しいライブラリは採用前にメンテナンス活性度・人気度・ライセンス・既存の脆弱性を確認する。
- 放置された(長く更新されない、イシューに応答がない)パッケージは、将来の脆弱性にパッチを受けられないため採用しない。

```text
// ❌ 禁止 — 最終更新が古い・イシュー放置・脆弱性未確認のパッケージをそのまま追加
add some-abandoned-pkg

// ✅ 推奨 — 採用前に点検してから追加
// - 最終更新: 最近(例: 6ヶ月以内)
// - イシュー/PRの応答速度: 合理的
// - 使用量/人気度: プロジェクトの性格に適合
// - ライセンス: 会社ポリシーに適合(制約的ライセンスに注意)
// - 既知の脆弱性なし(採用前にスキャンで事前確認)
scan(candidate) → 異常なければ add trusted-pkg
```

### 2-6. ロックファイルのコミットと本番/開発依存の分離
- 正確な解決バージョンを固定したロックファイルを常に構成管理にコミットし、再現可能なビルドを保証する。
- 本番成果物にはランタイムで必要な依存だけを含め、ビルド・テスト・リント用の開発依存は除外する。

```text
// ❌ 禁止 — ロックファイル未コミット + 本番に開発依存を含む
ignore(lockfile)                 // 環境ごとに異なるバージョン → 再現不可
build(prod, include dev deps)    // 不要な攻撃面の拡大

// ✅ 推奨 — ロックファイルをコミット + ランタイム依存だけを本番に
commit(lockfile)                 // どこでビルドしても同じ依存ツリー
build(prod, runtime deps only)   // 開発依存を除外
```

## 3. よくある間違い

ルールを知っていても陥りやすい罠だけを抜き出す(ルールの繰り返しは省略)。

- **直接依存だけを見る** → 脆弱性は自分が呼んだパッケージがさらに引き込む推移的依存からより頻繁に出る。スキャン範囲に推移的依存を必ず含める。
- **スキャン結果を警告だけで処理** → CRITICALがあってもビルドを通過させると誰も見ない。しきい値以上はゲーティングで止めてこそ意味がある。
- **一度通過すれば終わりと考える** → コードを変えなくても新しい脆弱性が毎日公開される。定期スケジュールのスキャンがなければそのまま放置される。
- **深刻度を無視した一括対応** → 優先順位なしに処理すると致命的な脆弱性への対応が遅れる。深刻度別の期限(§2-4)で並べる。

## 4. チェックリスト
- [ ] 直接・推移的依存をすべて脆弱性スキャン対象とみなしているか
- [ ] CIに依存関係スキャンを統合し、しきい値の深刻度以上ならビルドを**失敗**させるか
- [ ] コード変更がなくても定期スケジュールでスキャンを再実行するか
- [ ] 自動更新ツールでセキュリティパッチの変更リクエスト(PR/MR)を受け取るか
- [ ] 深刻度(例: CVSS)に合った対応期限(SLA)を定めて守るか
- [ ] ロックファイルを構成管理にコミットして再現可能なビルドを保証するか
- [ ] 本番成果物から開発依存を除外するか
- [ ] 新しいライブラリの採用前にメンテナンス活性度・人気度・ライセンス・既存の脆弱性を確認するか

## 付録: スタック別の例

> 以下は参考用の実装例である。上記1〜4の原則・ルールが標準であり、付録はその適用事例にすぎない。チームが使うスタック(パッケージマネージャ・CIツール)に合った例を同じパターンで追加する。

### ツール別の例 (npm/pip/Dependabot など)

#### CIパイプライン統合

**npm (Node.js)**
```yaml
# GitHub Actions
- name: Security Audit
  run: npm audit --audit-level=critical
  # CRITICAL脆弱性があれば終了コード1 → ビルド失敗
```

**Python**
```yaml
- name: Pip Audit
  run: |
    pip install pip-audit
    pip-audit --requirement requirements.txt --fail-on CRITICAL
```

**Java (Maven)**
```yaml
- name: OWASP Dependency Check
  run: mvn org.owasp:dependency-check-maven:check -DfailBuildOnCVSS=9
```

#### Dependabot 自動更新
```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    groups:
      security-updates:
        applies-to: security-updates
        patterns: ["*"]

  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
```

#### ライブラリ採用前の点検 (npm の例)

採用基準は §2-5 参照。npm での事前スキャンコマンド:

```bash
# ❌ 放置パッケージを点検なしに追加
npm install some-abandoned-pkg

# ✅ 採用前に脆弱性を事前確認してから追加 (snyk test などで代替可能)
npm audit && npm install trusted-pkg
```

#### ロックファイルのコミットと依存の分離 (npm の例)

原則は §2-6 参照。npm でのコマンド:

```bash
# ✅ ロックファイルをコミット (poetry.lock, requirements.txt も同様)
git add package-lock.json

# ✅ 本番ビルドで devDependencies を除外
npm ci --omit=dev
```
