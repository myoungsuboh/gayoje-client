# 🛡️ 의존성 보안 스캐닝 (SCA)

> 서드파티 의존성에 숨은 알려진 취약점을 자동으로 탐지·추적·패치하고, 심각한 것은 빌드 단계에서 막는다. CI 파이프라인에 스캐닝을 붙이거나, 의존성 자동 업데이트를 설정하거나, 새 라이브러리를 채택할 때 읽는다. 특정 패키지 매니저(npm/pip/Maven 등)나 CI 도구(GitHub Actions/Dependabot 등)에 종속되지 않는 범용 표준이다. 버전 고정·범위 정책 등 의존성 자체 관리 규칙은 `dependency-management`를 함께 본다.

## 1. 핵심 원칙

출발점은 하나다 — **끌어다 쓰는 모든 서드파티 패키지(와 그것이 다시 끌어오는 전이 의존성)가 공격 표면이다.** "남이 만들었으니 안전"은 가정하지 않는다. 여기서 다음이 따라온다.

- **자동으로·지속적으로 탐지한다**: 취약점 DB는 매일 갱신되므로 1회성 점검으로 끝나지 않는다. CI와 정기 스케줄로 반복 실행해 새로 공개된 취약점도 잡는다.
- **심각하면 막고, 패치는 받아 온다**: 일정 심각도 이상은 빌드에서 게이팅하고(경고만 하면 아무도 안 본다), 보안 패치는 자동화 도구가 PR로 올리게 해 사람은 검토·머지에 집중한다.
- **공격 표면을 작게·재현 가능하게**: 잠금 파일을 커밋해 어디서든 같은 의존성 트리가 나오게 하고, 프로덕션엔 런타임 의존성만 넣으며, 새 라이브러리는 들이기 전에 검증한다.

구체적 실천은 §2 규칙에서 ✅/❌로 다룬다.

## 2. 규칙

### 2-1. 외부 의존성을 무조건 신뢰하지 않는다
- "널리 쓰이는 패키지니까 안전하다"고 가정하지 말고, 직접·전이 의존성 모두를 스캐닝 대상으로 본다.
- 취약점은 내가 직접 부르는 패키지뿐 아니라 그 패키지가 끌어오는 깊은 의존성에서도 나온다. 전이 의존성까지 포함해 탐지한다.

```text
// ❌ 금지 — "유명하니까" 점검 없이 그냥 추가하고 끝
add some-popular-pkg            // 어떤 취약점이 딸려오는지 모른 채 사용

// ✅ 권장 — 직접·전이 의존성을 모두 취약점 스캐닝 대상으로
scan(all dependencies incl. transitive) → 통과한 것만 사용
```

### 2-2. 취약점 스캐닝을 CI에 통합하고 심각하면 빌드를 막는다
- 의존성 취약점 스캐닝을 PR/빌드마다 자동으로 돌리고, 1회성이 아니라 정기 스케줄로도 다시 돌려 코드가 안 바뀌어도 새 취약점을 잡는다.
- 정한 심각도 임계값(예: CRITICAL) 이상이 발견되면 파이프라인을 **실패**시킨다. 경고만 남기고 통과시키지 않는다.
- 스캔 단계를 파이프라인 어디에 배치하고 실패 시 후속을 어떻게 차단할지(단계 직렬화·fail-fast)는 `CI/CD 파이프라인 표준`을 따른다. 이 표준은 "무엇을 어느 심각도에서 막을지"(스캐너·임계·게이팅 정책)를 정한다.

```text
// ❌ 금지 — 스캔 결과를 로그로만 찍고 항상 통과
scan(deps); build_status = SUCCESS    // CRITICAL이 있어도 그냥 배포

// ✅ 권장 — 임계값 이상이면 파이프라인 실패(게이팅)
result = scan(deps)
if result.maxSeverity >= CRITICAL: fail build   // 프로덕션 유입 차단
```

### 2-3. 보안 패치를 자동 제안으로 받는다
- 의존성 자동 업데이트 도구를 활성화해, 취약한 의존성의 보안 패치를 자동 변경 요청(PR/MR)으로 받는다.
- 보안 업데이트는 따로 묶어 우선 검토·머지한다. 사람이 취약점 공지를 수동으로 좇지 않게 한다.

```text
// ❌ 금지 — 사람이 공지를 수동 추적, 결국 패치 누락·지연
"누가 가끔 의존성 버전 올리겠지"      // 보안 패치가 몇 달씩 방치

// ✅ 권장 — 자동화 도구가 패치 PR을 올리고, 사람은 검토·머지
auto-update tool → opens PR(보안 패치) → 검토 후 머지
```

### 2-4. 취약점 우선순위와 대응 기한(SLA)
- 표준 심각도 점수(예: CVSS)에 따라 대응 기한을 정하고, 치명적인 것부터 처리한다.
- **"심각할수록 빨리"가 원칙이고, 구체적 기한은 팀이 정한다.** 아래 표는 출발점 예시일 뿐이니 위험 허용 수준에 맞춰 임계값·기한을 조정한다.

| 심각도(예: CVSS) | 등급 | 대응 기한 — 예시(팀이 조정) |
|-----------------|--------|----------------|
| 9.0–10.0 | CRITICAL | 가장 빠르게(예: 수일 내) |
| 7.0–8.9 | HIGH | 빠르게(예: 1~2주 내) |
| 4.0–6.9 | MEDIUM | 계획에 반영(예: 한 달 내) |
| 0.1–3.9 | LOW | 다음 릴리즈 |

### 2-5. 새 라이브러리 채택 기준
- 새 라이브러리는 채택 전 유지보수 활성도·인기도·라이선스·기존 취약점을 확인한다.
- 방치된(오래 갱신 안 된, 이슈 응답 없는) 패키지는 미래의 취약점에 패치를 못 받으므로 채택하지 않는다.

```text
// ❌ 금지 — 마지막 갱신 오래됨·이슈 방치·취약점 미확인 패키지를 그냥 추가
add some-abandoned-pkg

// ✅ 권장 — 채택 전 점검 후 추가
// - 마지막 갱신: 최근(예: 6개월 이내)
// - 이슈/PR 응답 속도: 합리적
// - 사용량/인기도: 프로젝트 성격에 적합
// - 라이선스: 회사 정책에 부합 (제약적 라이선스 주의)
// - 알려진 취약점 없음 (채택 전 스캔으로 사전 확인)
scan(candidate) → 이상 없으면 add trusted-pkg
```

### 2-6. 잠금 파일 커밋과 프로덕션/개발 의존성 분리
- 정확한 해석 버전을 박제한 잠금 파일을 항상 형상관리에 커밋해 재현 가능한 빌드를 보장한다.
- 프로덕션 산출물에는 런타임에 필요한 의존성만 포함하고, 빌드·테스트·린트용 개발 의존성은 제외한다.

```text
// ❌ 금지 — 잠금 파일 미커밋 + 프로덕션에 개발 의존성 포함
ignore(lockfile)                 // 환경마다 다른 버전 → 재현 불가
build(prod, include dev deps)    // 불필요한 공격 표면 확대

// ✅ 권장 — 잠금 파일 커밋 + 런타임 의존성만 프로덕션에
commit(lockfile)                 // 어디서 빌드해도 같은 의존성 트리
build(prod, runtime deps only)   // 개발 의존성 제외
```

## 3. 흔한 실수

규칙을 알면서도 자주 빠지는 함정만 추린다(규칙의 반복은 생략).

- **직접 의존성만 본다** → 취약점은 내가 부른 패키지가 다시 끌어오는 전이 의존성에서 더 자주 나온다. 스캔 범위에 전이 의존성을 반드시 포함한다.
- **스캔 결과를 경고로만 처리** → CRITICAL이 있어도 빌드를 통과시키면 아무도 안 본다. 임계값 이상은 게이팅으로 막아야 의미가 있다.
- **한 번 통과하면 끝이라 생각** → 코드를 안 바꿔도 새 취약점이 매일 공개된다. 정기 스케줄 스캔이 없으면 그대로 방치된다.
- **심각도 무시한 일괄 대응** → 우선순위 없이 처리하면 치명적 취약점 대응이 늦어진다. 심각도별 기한(§2-4)으로 줄을 세운다.

## 4. 체크리스트
- [ ] 직접·전이 의존성을 모두 취약점 스캐닝 대상으로 보는가
- [ ] CI에 의존성 스캐닝을 통합하고 임계 심각도 이상이면 빌드를 **실패**시키는가
- [ ] 코드 변경이 없어도 정기 스케줄로 스캔을 다시 돌리는가
- [ ] 자동 업데이트 도구로 보안 패치 변경 요청(PR/MR)을 받는가
- [ ] 심각도(예: CVSS)에 맞는 대응 기한(SLA)을 정하고 지키는가
- [ ] 잠금 파일을 형상관리에 커밋해 재현 가능한 빌드를 보장하는가
- [ ] 프로덕션 산출물에서 개발 의존성을 제외하는가
- [ ] 새 라이브러리 채택 전 유지보수 활성도·인기도·라이선스·기존 취약점을 확인하는가

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다. 팀이 쓰는 스택(패키지 매니저·CI 도구)에 맞는 예시를 같은 패턴으로 추가한다.

### 도구별 예시 (npm/pip/Dependabot 등)

#### CI 파이프라인 통합

**npm (Node.js)**
```yaml
# GitHub Actions
- name: Security Audit
  run: npm audit --audit-level=critical
  # CRITICAL 취약점 있으면 종료코드 1 → 빌드 실패
```

**Python**
```yaml
- name: Pip Audit
  run: |
    pip install pip-audit
    pip-audit --requirement requirements.txt --fail-on CRITICAL
```

**Java (Maven)**
```yaml
- name: OWASP Dependency Check
  run: mvn org.owasp:dependency-check-maven:check -DfailBuildOnCVSS=9
```

#### Dependabot 자동 업데이트
```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    groups:
      security-updates:
        applies-to: security-updates
        patterns: ["*"]

  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
```

#### 라이브러리 채택 전 점검 (npm 예시)

채택 기준은 §2-5 참조. npm에서의 사전 스캔 명령:

```bash
# ❌ 방치 패키지를 점검 없이 추가
npm install some-abandoned-pkg

# ✅ 채택 전 취약점 사전 확인 후 추가 (snyk test 등으로 대체 가능)
npm audit && npm install trusted-pkg
```

#### 잠금 파일 커밋과 의존성 분리 (npm 예시)

원칙은 §2-6 참조. npm에서의 명령:

```bash
# ✅ 잠금 파일 커밋 (poetry.lock, requirements.txt 도 마찬가지)
git add package-lock.json

# ✅ 프로덕션 빌드에서 devDependencies 제외
npm ci --omit=dev
```
