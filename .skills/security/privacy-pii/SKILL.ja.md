# 🔐 個人情報保護 & PII 管理 (GDPR/PIPA)

> 個人情報（PII）をGDPR・個人情報保護法（PIPA）に沿って安全に扱う。収集は最小に、保存は安全に（パスワードは一方向ハッシュ・機微情報は暗号化）、ログには識別情報を残さず、情報主体の削除要求を定められた期限内に履行する。ユーザー情報を収集・保存・処理・削除する機能を作るまたは直すときに読む。特定の言語/ストレージに依存しない汎用標準だ。

## 1. 目的

- 個人情報を扱うすべての機能が**言語・フレームワーク・ストレージに関わらず**同じ保護水準を持つようにする。
- 「どこまで収集してよいか、パスワード・機微情報をどう保存するか、ログに何を残してはいけないか、削除要求が来たら何を消すべきか」を毎回考え直さなくて済むよう基準を固定する。
- 事故（漏洩・法令違反）時に責任範囲を縮小し、履行の事実を証明できるようにする。

## 2. 核心原則

この標準を貫く考え方は三つだ。具体的な実践は§3のルールで✅/❌で扱う。

- **収集は少なく、保管は短く**: 本当に必要な最小限のみを告知した目的の範囲内で収集し（「念のため」禁止）、項目ごとの保存期間が過ぎたら自動的に廃棄する。
- **保存は復元不可能に**: パスワードは一方向ハッシュのみ、機微情報は暗号化 + 最小権限で置く。漏洩してもそれ自体では役に立たない形が原則だ。
- **処理は証拠を残せるように**: 同意・削除などの処理行為を監査ログに残し、情報主体の削除要求は散在するすべての複製まで期限内に履行する。

## 3. ルール (✅ 推奨 / ❌ 禁止)

### 3-1. 収集は最小限、目的の範囲内のみ

- 画面・フォーム・APIに「あれば良さそうだから」フィールドを追加しない。収集項目ごとに「これがなぜ必要か」に答えられなければならない。
- 収集目的を告知・記録し、その目的外の処理は別途の根拠なく行わない。

```text
// ❌ 금지 — 쓸 데 없는데 "혹시 몰라" 다 받음
signup(name, email, password, 주민번호, 직장, 가족관계, 연소득)

// ✅ 권장 — 서비스에 필요한 최소 항목만, 목적과 함께
signup(email, password)            // 인증에 필요한 최소
// 추가 수집이 필요하면 → 목적 고지 + 동의 기록
```

| 原則 | 何をするか（実装方向） |
|------|------------------------|
| 目的制限 | 収集目的を明示し同意/根拠を記録 |
| 最小収集 | 不要なフィールドを削除、不要なら最初から受け取らない |
| 正確性 | 訂正経路を提供、最新状態を維持 |
| 保存制限 | 項目ごとの保存期間設定、満了時に自動削除・匿名化 |
| 完全性・機密性 | 暗号化、アクセス制御、監査ログ |

### 3-2. パスワードは一方向ハッシュのみで保存

- パスワードは**ソルトを含む一方向ハッシュ**（現代的なパスワードハッシュアルゴリズム）のみで保存し、検証は「入力値を同じ方式でハッシュして比較」する。
- 平文保存、可逆（復号可能）暗号化保存、自作・弱いハッシュ（速い汎用ハッシュ）での保存はしない。

```text
// ❌ 금지 — 평문 또는 복원 가능한 형태로 저장
save(password)                       // 평문 → 유출 시 그대로 노출
save(reversibleEncrypt(password))    // 복호화 가능 → 사실상 평문과 같음

// ✅ 권장 — 솔트 포함 단방향 해시로 저장, 비교로 검증
save(hash(password, salt))           // 저장 시
verify = hash(input, salt) == stored // 검증 시 (복원하지 않음)
```

> 具体的なアルゴリズム選択・パラメータ（作業コストなど）とライブラリ例は**付録**を参照する。

### 3-3. 機微情報は暗号化 + アクセス権限の最小化

- 住民番号・カード番号・医療情報などの機微データは保存時に暗号化し、**実際に必要な役割のみ**が読めるようアクセス範囲を狭める（ストレージレベルのカラム/フィールド単位の権限または別途保管）。
- 暗号化キーはデータと分離して安全に管理する（→ `secrets-management`）。

```text
// ❌ 금지 — 민감정보를 평문으로, 누구나 읽을 수 있게 저장
record.ssn = "900101-1234567"        // 평문 + 전 직원 조회 가능

// ✅ 권장 — 암호화 저장 + 필요한 역할만 접근
record.ssn = encrypt("900101-1234567")   // 저장은 암호문
// 조회 권한은 최소 역할로 제한, 키는 별도 관리
```

### 3-4. ログ・観測データにPIIを残さない

- メール・電話番号・IP・識別子などをログ・トレース・監視データに平文で記録しない。
- デバッグなどでやむを得ない場合は**記録の直前にマスキング**する（例: `u***@example.com`、電話の中間桁を隠す）。

```text
// ❌ 금지 — 식별정보를 로그에 평문으로
log("사용자 요청: user@example.com / 010-1234-5678")

// ✅ 권장 — 기록 전에 마스킹
log("사용자 요청: " + mask(email) + " / " + mask(phone))
//  → "u***@example.com / 010-****-5678"
```

> マスキング関数の実装例は**付録**参照。監査ログ自体の構造・保存は `audit-logging` に従う。

### 3-5. 保存期間 · 自動廃棄

- データごとに保存期間を定め、満了したデータは自動的に削除または匿名化する。
- 「いつか使うかも」という理由で無期限に保管しない。

```text
// ✅ 권장 — 항목별 보존 정책 + 만료 시 자동 처리
retention: 결제기록 = 5년(법정), 접속로그 = 6개월, 마케팅 동의 철회 시 = 즉시 삭제
expired → 자동 삭제 또는 익명화 (수동 의존 X)
```

### 3-6. 削除権（情報主体の権利）の履行 — どこに散在していてもすべて

- 削除要求を受けたら**定められた期限内に**（例: 30日以内）関連データを完全削除するか、保存が必要な集計は匿名化する。
- メインストレージだけ消して終わらせない。**複製・集計・サードパーティ・キャッシュ・バックアップ**まですべてを対象にする。
- 削除の履行自体を監査ログに残して証拠とする（法的要件）。

```text
// ✅ 권장 — 삭제 요청 처리의 표준 절차
onDeleteRequest(userId):
  1) 식별 가능 데이터 삭제          // 메인 저장소
  2) 서드파티에 삭제 요청 전파       // 메일·분석·알림 등
  3) 보존 필요한 집계는 익명화        // 식별자 제거 후 통계만 유지
  4) 삭제 이벤트를 감사 로그에 기록    // "언제 누구를 지웠는지" 증빙

// ❌ 금지 — 메인만 지우고 나머지 방치
onDeleteRequest(userId):
  delete(mainStore, userId)         // 서드파티·집계·백업에 잔존 → 법 위반
```

> ストレージごとの削除/匿名化実装（例: ドキュメントDBのフィールド削除）例は**付録**参照。削除ログ構造は `audit-logging`、アクセス権限は `authn-authz` に従う。

## 4. よくある間違い

ルールを知っていると思いながらも頻繁に陥る罠だけを抜粋する（ルールの繰り返しは省略）。

- **「削除したから安全」という錯覚** → メインストレージだけ消して複製・集計・サードパーティ・キャッシュ・バックアップに残れば法令違反だ。散在するすべての場所が対象だ。
- **「仮名化すれば終わり」という錯覚** → 他のデータと結合して再識別が可能なら依然として個人情報だ。結合リスクまで見て匿名化だ。
- **可逆暗号化を一方向と誤認** → 復号可能なら事実上平文だ。パスワードは必ず一方向ハッシュ。
- **デバッグ中に一時的にPIIをログ記録** → 「ちょっとだけ」がそのまま残り、ログ漏洩時に露出する。記録前のマスキングは例外なく。
- **履行を記録しない** → 同意・削除を実際に行っても監査ログがなければ証明できない。

## 5. チェックリスト

- [ ] 収集項目がサービス提供に本当に必要な**最小限**か（目的告知・同意/根拠記録）
- [ ] パスワードを**ソルト含む一方向ハッシュ**で保存しているか（平文・可逆暗号化・弱いハッシュではない）
- [ ] 機微データを**暗号化**し、**必要な役割のみ**アクセスするよう権限を狭めたか
- [ ] ログ・トレースデータにPIIが平文で残らないよう**マスキング**したか
- [ ] 項目ごとの**保存期間**を定め、満了時に自動削除・匿名化するか
- [ ] 削除要求時に**メイン・サードパーティ・集計・バックアップ**まで削除または匿名化するか
- [ ] 同意・削除などの処理行為を**監査ログ**で証拠化しているか
- [ ] 暗号化キーをデータと分離して安全に管理しているか（→ `secrets-management`）

## 付録: スタック別の例

> 以下は参考用の実装例だ。チームが使うスタック（ハッシュライブラリ、ストレージ種別など）に合う例を同じパターンで追加する。上記1〜5の目的・原則・ルールが標準であり、付録はその適用事例にすぎない。

### 実装例（ハッシュライブラリ / MongoDB など）

#### パスワードハッシュ — bcrypt (Python)

```python
# ❌ 금지 — 평문 또는 가역 암호화 저장
db.save(password)                       # 평문
db.save(reversible_encrypt(password))   # 복호화 가능 → 유출 시 그대로 노출
```
```python
# ✅ 권장 — bcrypt 단방향 해시
import bcrypt

# 저장
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# 검증
bcrypt.checkpw(password.encode(), hashed)
```

#### パスワードハッシュ — Argon2 (Python)

```python
# ✅ 권장 — Argon2
# 아래 파라미터는 예시값이다. 환경(하드웨어·지연 허용치)에 맞게 조정하고,
# 최신 권장치는 OWASP Password Storage Cheat Sheet를 참고한다.
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=1)
hashed = ph.hash(password)
ph.verify(hashed, password)
```

#### ログマスキング (Python)

```python
# ❌ 금지 — 로그에 개인식별 정보 평문 기록
logger.info(f"사용자 요청: {user_email}")   # user@example.com 그대로 노출
```
```python
# ✅ 권장 — 기록 전 마스킹
import re

def mask_pii(text: str) -> str:
    # 이메일 마스킹: user@example.com → u***@example.com
    text = re.sub(r'(\w{1})\w+@', r'\1***@', text)
    # 전화번호 마스킹
    text = re.sub(r'(\d{3})-\d{4}-(\d{4})', r'\1-****-\2', text)
    return text

logger.info(mask_pii(f"사용자 요청: {user_email}"))
```

#### データ削除権の実装 (MongoDB)

```python
# ✅ 권장 — 식별정보 삭제 + 서드파티 정리 + 집계 익명화 + 감사 로그
async def delete_user_data(user_id: str):
    # 1. 개인 식별 가능 데이터 삭제
    await db.users.delete_one({"_id": user_id})
    # 2. 서드파티 서비스 삭제 요청 (이메일 서비스, 분석 등)
    await email_service.delete_contact(user_email)
    # 3. 집계 데이터는 익명화 후 보존
    await db.orders.update_many(
        {"user_id": user_id},
        {"$unset": {"user_id": ""}, "$set": {"user_anonymized": True}}
    )
    # 4. 삭제 이벤트 감사 로그 (법적 요건)
    await audit_log.record("USER_DELETED", user_id, timestamp=now())
```
