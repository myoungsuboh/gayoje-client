# 🔐 个人信息保护 & PII 管理 (GDPR/PIPA)

> 按照 GDPR·个人信息保护法（PIPA）安全地处理个人信息（PII）。收集要最小，存储要安全（密码用单向哈希·敏感信息加密），日志中不留识别信息，并在规定期限内履行信息主体的删除请求。在创建或修改收集·存储·处理·删除用户信息的功能时阅读。这是不依赖特定语言/存储的通用标准。

## 1. 目的

- 让所有处理个人信息的功能**无论语言·框架·存储**都具有同等的保护水平。
- 固定标准，使"可以收集到什么程度、密码·敏感信息如何存储、日志中不可留下什么、收到删除请求时应删除什么"无需每次重新斟酌。
- 在事故（泄漏·违法）时缩小责任范围，并使履行事实可举证。

## 2. 核心原则

贯穿这套标准的思路有三个。具体实践在 §3 规则中以 ✅/❌ 处理。

- **收得少，存得短**: 仅在告知的目的范围内收集确有必要的最小限度（禁止"以防万一"），各项目超过保存期限后自动废弃。
- **存储不可还原**: 密码仅用单向哈希，敏感信息加密 + 最小权限。原则是即使泄漏其本身也无法利用的形态。
- **处理可举证**: 将同意·删除等处理行为留存于审计日志，信息主体的删除请求须在期限内履行到散落的所有副本。

## 3. 规则 (✅ 推荐 / ❌ 禁止)

### 3-1. 收集最小限度，仅在目的范围内

- 不因"有了好像不错"而在界面·表单·API 中添加字段。每个收集项目都应能回答"为什么需要它"。
- 告知·记录收集目的，目的之外的处理无另行依据则不进行。

```text
// ❌ 금지 — 쓸 데 없는데 "혹시 몰라" 다 받음
signup(name, email, password, 주민번호, 직장, 가족관계, 연소득)

// ✅ 권장 — 서비스에 필요한 최소 항목만, 목적과 함께
signup(email, password)            // 인증에 필요한 최소
// 추가 수집이 필요하면 → 목적 고지 + 동의 기록
```

| 原则 | 做什么（实现方向） |
|------|------------------------|
| 目的限制 | 明示收集目的并记录同意/依据 |
| 最小收集 | 删除不必要的字段，不需要则从一开始就不收 |
| 准确性 | 提供更正途径，保持最新状态 |
| 存储限制 | 按项目设置保存期限，到期时自动删除·匿名化 |
| 完整性·机密性 | 加密、访问控制、审计日志 |

### 3-2. 密码仅以单向哈希存储

- 密码仅以**含盐的单向哈希**（现代密码哈希算法）存储，验证时"以相同方式哈希输入值并比较"。
- 不以明文存储、不以可逆（可解密）加密存储、不以自制·弱哈希（快速通用哈希）存储。

```text
// ❌ 금지 — 평문 또는 복원 가능한 형태로 저장
save(password)                       // 평문 → 유출 시 그대로 노출
save(reversibleEncrypt(password))    // 복호화 가능 → 사실상 평문과 같음

// ✅ 권장 — 솔트 포함 단방향 해시로 저장, 비교로 검증
save(hash(password, salt))           // 저장 시
verify = hash(input, salt) == stored // 검증 시 (복원하지 않음)
```

> 具体的算法选择·参数（工作成本等）与库示例请参考**附录**。

### 3-3. 敏感信息加密 + 最小化访问权限

- 身份证号·卡号·医疗信息等敏感数据在存储时加密，并将访问范围收窄到**仅实际需要的角色**可读（存储层面的列/字段级权限或单独保管）。
- 加密密钥与数据分离并安全管理（→ `secrets-management`）。

```text
// ❌ 금지 — 민감정보를 평문으로, 누구나 읽을 수 있게 저장
record.ssn = "900101-1234567"        // 평문 + 전 직원 조회 가능

// ✅ 권장 — 암호화 저장 + 필요한 역할만 접근
record.ssn = encrypt("900101-1234567")   // 저장은 암호문
// 조회 권한은 최소 역할로 제한, 키는 별도 관리
```

### 3-4. 日志·观测数据中不留 PII

- 不在日志·追踪·监控数据中以明文记录邮箱·电话号·IP·标识符等。
- 因调试等不得已时，**在记录前脱敏**（例: `u***@example.com`，遮蔽电话中间数位）。

```text
// ❌ 금지 — 식별정보를 로그에 평문으로
log("사용자 요청: user@example.com / 010-1234-5678")

// ✅ 권장 — 기록 전에 마스킹
log("사용자 요청: " + mask(email) + " / " + mask(phone))
//  → "u***@example.com / 010-****-5678"
```

> 脱敏函数实现示例见**附录**。审计日志本身的结构·保存遵循 `audit-logging`。

### 3-5. 保存期限 · 自动废弃

- 为每项数据设定保存期限，到期数据自动删除或匿名化。
- 不以"也许某天会用"为由无限期保存。

```text
// ✅ 권장 — 항목별 보존 정책 + 만료 시 자동 처리
retention: 결제기록 = 5년(법정), 접속로그 = 6개월, 마케팅 동의 철회 시 = 즉시 삭제
expired → 자동 삭제 또는 익명화 (수동 의존 X)
```

### 3-6. 删除权（信息主体权利）的履行 — 无论散落何处皆须处理

- 收到删除请求后，**在规定期限内**（例: 30 日内）彻底删除相关数据，或将需要保存的聚合数据匿名化。
- 不要只清主存储就完事。**副本·聚合·第三方·缓存·备份**都要纳入对象。
- 将删除履行本身留存于审计日志以作举证（法定要求）。

```text
// ✅ 권장 — 삭제 요청 처리의 표준 절차
onDeleteRequest(userId):
  1) 식별 가능 데이터 삭제          // 메인 저장소
  2) 서드파티에 삭제 요청 전파       // 메일·분석·알림 등
  3) 보존 필요한 집계는 익명화        // 식별자 제거 후 통계만 유지
  4) 삭제 이벤트를 감사 로그에 기록    // "언제 누구를 지웠는지" 증빙

// ❌ 금지 — 메인만 지우고 나머지 방치
onDeleteRequest(userId):
  delete(mainStore, userId)         // 서드파티·집계·백업에 잔존 → 법 위반
```

> 各存储的删除/匿名化实现（例: 文档 DB 的字段删除）示例见**附录**。删除日志结构遵循 `audit-logging`，访问权限遵循 `authn-authz`。

## 4. 常见错误

仅摘取自认为懂规则却仍频繁踩中的陷阱（省略规则的重复）。

- **"删了就安全"的错觉** → 只清主存储而在副本·聚合·第三方·缓存·备份中残留即属违法。散落的所有位置都是对象。
- **"假名化就完事"的错觉** → 若与其他数据结合可再识别则仍是个人信息。要看到结合风险才算匿名化。
- **将可逆加密误认为单向** → 可解密则事实上等同明文。密码必须用单向哈希。
- **调试中临时记录 PII** → "就一会儿"原样留下，日志泄漏时即被暴露。记录前脱敏，无一例外。
- **未记录履行** → 即便实际做了同意·删除，没有审计日志便无法举证。

## 5. 检查清单

- [ ] 收集项目是否为提供服务确有必要的**最小限度**（已告知目的·记录同意/依据）
- [ ] 密码是否以**含盐的单向哈希**存储（非明文·可逆加密·弱哈希）
- [ ] 是否将敏感数据**加密**并将访问权限收窄到**仅必要角色**
- [ ] 是否对日志·追踪数据进行**脱敏**以免 PII 以明文残留
- [ ] 是否按项目设定了**保存期限**并在到期时自动删除·匿名化
- [ ] 删除请求时是否在**主存储·第三方·聚合·备份**范围内删除或匿名化
- [ ] 是否以**审计日志**对同意·删除等处理行为加以举证
- [ ] 是否将加密密钥与数据分离并安全管理（→ `secrets-management`）

### 实现示例（哈希库 / MongoDB 等）

#### 密码哈希 — bcrypt (Python)

```python
# ❌ 금지 — 평문 또는 가역 암호화 저장
db.save(password)                       # 평문
db.save(reversible_encrypt(password))   # 복호화 가능 → 유출 시 그대로 노출
```
```python
# ✅ 권장 — bcrypt 단방향 해시
import bcrypt

# 저장
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# 검증
bcrypt.checkpw(password.encode(), hashed)
```

#### 密码哈希 — Argon2 (Python)

```python
# ✅ 권장 — Argon2
# 아래 파라미터는 예시값이다. 환경(하드웨어·지연 허용치)에 맞게 조정하고,
# 최신 권장치는 OWASP Password Storage Cheat Sheet를 참고한다.
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=1)
hashed = ph.hash(password)
ph.verify(hashed, password)
```

#### 日志脱敏 (Python)

```python
# ❌ 금지 — 로그에 개인식별 정보 평문 기록
logger.info(f"사용자 요청: {user_email}")   # user@example.com 그대로 노출
```
```python
# ✅ 권장 — 기록 전 마스킹
import re

def mask_pii(text: str) -> str:
    # 이메일 마스킹: user@example.com → u***@example.com
    text = re.sub(r'(\w{1})\w+@', r'\1***@', text)
    # 전화번호 마스킹
    text = re.sub(r'(\d{3})-\d{4}-(\d{4})', r'\1-****-\2', text)
    return text

logger.info(mask_pii(f"사용자 요청: {user_email}"))
```

#### 数据删除权实现 (MongoDB)

```python
# ✅ 권장 — 식별정보 삭제 + 서드파티 정리 + 집계 익명화 + 감사 로그
async def delete_user_data(user_id: str):
    # 1. 개인 식별 가능 데이터 삭제
    await db.users.delete_one({"_id": user_id})
    # 2. 서드파티 서비스 삭제 요청 (이메일 서비스, 분석 등)
    await email_service.delete_contact(user_email)
    # 3. 집계 데이터는 익명화 후 보존
    await db.orders.update_many(
        {"user_id": user_id},
        {"$unset": {"user_id": ""}, "$set": {"user_anonymized": True}}
    )
    # 4. 삭제 이벤트 감사 로그 (법적 요건)
    await audit_log.record("USER_DELETED", user_id, timestamp=now())
```
