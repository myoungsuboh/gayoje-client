# 🔐 개인정보 보호 & PII 관리 (GDPR/PIPA)

> 개인정보(PII)를 GDPR·개인정보보호법(PIPA)에 맞게 안전하게 다룬다. 수집은 최소로, 저장은 안전하게(비밀번호는 단방향 해시·민감정보는 암호화), 로그에는 식별정보를 남기지 않고, 정보주체의 삭제 요청을 정해진 기한 안에 이행한다. 사용자 정보를 수집·저장·처리·삭제하는 기능을 만들거나 고칠 때 읽는다. 특정 언어/저장소에 종속되지 않는 범용 표준이다.

## 1. 목적

- 개인정보를 다루는 모든 기능이 **언어·프레임워크·저장소와 무관하게** 같은 보호 수준을 갖도록 한다.
- "어디까지 수집해도 되나, 비밀번호·민감정보를 어떻게 저장하나, 로그에 무엇을 남기면 안 되나, 삭제 요청이 오면 무엇을 지워야 하나"를 매번 다시 고민하지 않도록 기준을 고정한다.
- 사고(유출·법 위반) 시 책임 범위를 줄이고, 이행 사실을 증빙할 수 있게 한다.

## 2. 핵심 원칙

이 표준을 관통하는 사고방식은 세 가지다. 구체적 실천은 §3 규칙에서 ✅/❌로 다룬다.

- **수집은 적게, 보관은 짧게**: 꼭 필요한 최소한만 고지한 목적 안에서 수집하고("혹시 몰라" 금지), 항목별 보존 기간이 지나면 자동 폐기한다.
- **저장은 복원 불가능하게**: 비밀번호는 단방향 해시로만, 민감정보는 암호화 + 최소 권한으로 둔다. 유출돼도 그 자체로는 쓸모없는 형태가 원칙이다.
- **처리는 증빙 가능하게**: 동의·삭제 등 처리 행위를 감사 로그로 남기고, 정보주체의 삭제 요청은 흩어진 모든 사본까지 기한 내 이행한다.

## 3. 규칙 (✅ 권장 / ❌ 금지)

### 3-1. 수집은 최소한, 목적 안에서만

- 화면·폼·API에 "있으면 좋을 것 같아서" 필드를 추가하지 않는다. 수집 항목마다 "이게 왜 필요한가"에 답할 수 있어야 한다.
- 수집 목적을 고지·기록하고, 그 목적 밖의 처리는 별도 근거 없이 하지 않는다.

```text
// ❌ 금지 — 쓸 데 없는데 "혹시 몰라" 다 받음
signup(name, email, password, 주민번호, 직장, 가족관계, 연소득)

// ✅ 권장 — 서비스에 필요한 최소 항목만, 목적과 함께
signup(email, password)            // 인증에 필요한 최소
// 추가 수집이 필요하면 → 목적 고지 + 동의 기록
```

| 원칙 | 무엇을 한다 (구현 방향) |
|------|------------------------|
| 목적 제한 | 수집 목적을 명시하고 동의/근거를 기록 |
| 최소 수집 | 불필요한 필드 제거, 필요 없으면 처음부터 안 받음 |
| 정확성 | 정정 경로 제공, 최신 상태 유지 |
| 저장 제한 | 항목별 보존 기간 설정, 만료 시 자동 삭제·익명화 |
| 무결성·기밀성 | 암호화, 접근 제어, 감사 로그 |

### 3-2. 비밀번호는 단방향 해시로만 저장

- 비밀번호는 **솔트가 포함된 단방향 해시**(현대적 비밀번호 해시 알고리즘)로만 저장하고, 검증은 "입력값을 같은 방식으로 해시해 비교"한다.
- 평문 저장, 가역(복호화 가능) 암호화 저장, 자체 제작·약한 해시(빠른 일반 해시)로 저장하지 않는다.

```text
// ❌ 금지 — 평문 또는 복원 가능한 형태로 저장
save(password)                       // 평문 → 유출 시 그대로 노출
save(reversibleEncrypt(password))    // 복호화 가능 → 사실상 평문과 같음

// ✅ 권장 — 솔트 포함 단방향 해시로 저장, 비교로 검증
save(hash(password, salt))           // 저장 시
verify = hash(input, salt) == stored // 검증 시 (복원하지 않음)
```

> 구체적인 알고리즘 선택·파라미터(작업 비용 등)와 라이브러리 예시는 **부록**을 참고한다.

### 3-3. 민감정보는 암호화 + 접근 권한 최소화

- 주민번호·카드번호·의료정보 등 민감 데이터는 저장 시 암호화하고, **실제로 필요한 역할만** 읽도록 접근 범위를 좁힌다(저장소 차원의 컬럼/필드 단위 권한 또는 별도 보관).
- 암호화 키는 데이터와 분리해 안전하게 관리한다(→ `secrets-management`).

```text
// ❌ 금지 — 민감정보를 평문으로, 누구나 읽을 수 있게 저장
record.ssn = "900101-1234567"        // 평문 + 전 직원 조회 가능

// ✅ 권장 — 암호화 저장 + 필요한 역할만 접근
record.ssn = encrypt("900101-1234567")   // 저장은 암호문
// 조회 권한은 최소 역할로 제한, 키는 별도 관리
```

### 3-4. 로그·관측 데이터에 PII를 남기지 않는다

- 이메일·전화번호·IP·식별자 등을 로그·추적·모니터링 데이터에 평문으로 기록하지 않는다.
- 디버깅 등으로 불가피하면 **기록 직전에 마스킹**한다(예: `u***@example.com`, 전화 중간자리 가림).

```text
// ❌ 금지 — 식별정보를 로그에 평문으로
log("사용자 요청: user@example.com / 010-1234-5678")

// ✅ 권장 — 기록 전에 마스킹
log("사용자 요청: " + mask(email) + " / " + mask(phone))
//  → "u***@example.com / 010-****-5678"
```

> 마스킹 함수 구현 예시는 **부록** 참고. 감사 로그 자체의 구조·보존은 `audit-logging`을 따른다.

### 3-5. 보존 기간 · 자동 폐기

- 데이터마다 보존 기간을 정하고, 만료된 데이터는 자동으로 삭제 또는 익명화한다.
- "언젠가 쓸지도" 라는 이유로 무기한 보관하지 않는다.

```text
// ✅ 권장 — 항목별 보존 정책 + 만료 시 자동 처리
retention: 결제기록 = 5년(법정), 접속로그 = 6개월, 마케팅 동의 철회 시 = 즉시 삭제
expired → 자동 삭제 또는 익명화 (수동 의존 X)
```

### 3-6. 삭제권(정보주체 권리) 이행 — 어디에 흩어졌든 전부

- 삭제 요청을 받으면 **정해진 기한 안에**(예: 30일 이내) 관련 데이터를 완전 삭제하거나, 보존이 필요한 집계는 익명화한다.
- 메인 저장소만 지우고 끝내지 않는다. **사본·집계·서드파티·캐시·백업**까지 모두 대상으로 한다.
- 삭제 이행 자체를 감사 로그로 남겨 증빙한다(법적 요건).

```text
// ✅ 권장 — 삭제 요청 처리의 표준 절차
onDeleteRequest(userId):
  1) 식별 가능 데이터 삭제          // 메인 저장소
  2) 서드파티에 삭제 요청 전파       // 메일·분석·알림 등
  3) 보존 필요한 집계는 익명화        // 식별자 제거 후 통계만 유지
  4) 삭제 이벤트를 감사 로그에 기록    // "언제 누구를 지웠는지" 증빙

// ❌ 금지 — 메인만 지우고 나머지 방치
onDeleteRequest(userId):
  delete(mainStore, userId)         // 서드파티·집계·백업에 잔존 → 법 위반
```

> 저장소별 삭제/익명화 구현(예: 문서 DB의 필드 제거) 예시는 **부록** 참고. 삭제 로그 구조는 `audit-logging`, 접근 권한은 `authn-authz`를 따른다.

## 4. 흔한 실수

규칙을 안다고 생각하면서도 자주 빠지는 함정만 추린다(규칙의 반복은 생략).

- **"삭제했으니 안전" 착각** → 메인 저장소만 지우고 사본·집계·서드파티·캐시·백업에 잔존하면 법적 위반이다. 흩어진 모든 위치가 대상이다.
- **"가명화하면 끝" 착각** → 다른 데이터와 결합해 재식별이 가능하면 여전히 개인정보다. 결합 위험까지 봐야 익명화다.
- **가역 암호화를 단방향으로 오인** → 복호화 가능하면 사실상 평문이다. 비밀번호는 반드시 단방향 해시.
- **디버그 중 임시로 PII 로깅** → "잠깐만"이 그대로 남아 로그 유출 시 노출된다. 기록 전 마스킹은 예외 없이.
- **이행을 기록하지 않음** → 동의·삭제를 실제로 했어도 감사 로그가 없으면 증빙할 수 없다.

## 5. 체크리스트

- [ ] 수집 항목이 서비스 제공에 꼭 필요한 **최소한**인가 (목적 고지·동의/근거 기록)
- [ ] 비밀번호를 **솔트 포함 단방향 해시**로 저장하는가 (평문·가역 암호화·약한 해시 아님)
- [ ] 민감 데이터를 **암호화**하고 **필요한 역할만** 접근하도록 권한을 좁혔는가
- [ ] 로그·추적 데이터에 PII가 평문으로 남지 않도록 **마스킹**했는가
- [ ] 항목별 **보존 기간**을 정하고 만료 시 자동 삭제·익명화하는가
- [ ] 삭제 요청 시 **메인·서드파티·집계·백업**까지 삭제 또는 익명화하는가
- [ ] 동의·삭제 등 처리 행위를 **감사 로그**로 증빙하는가
- [ ] 암호화 키를 데이터와 분리해 안전하게 관리하는가 (→ `secrets-management`)

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(해싱 라이브러리, 저장소 종류 등)에 맞는 예시를 같은 패턴으로 추가한다. 위 1~5의 목적·원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다.

### 구현 예시 (해싱 라이브러리 / MongoDB 등)

#### 비밀번호 해싱 — bcrypt (Python)

```python
# ❌ 금지 — 평문 또는 가역 암호화 저장
db.save(password)                       # 평문
db.save(reversible_encrypt(password))   # 복호화 가능 → 유출 시 그대로 노출
```
```python
# ✅ 권장 — bcrypt 단방향 해시
import bcrypt

# 저장
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# 검증
bcrypt.checkpw(password.encode(), hashed)
```

#### 비밀번호 해싱 — Argon2 (Python)

```python
# ✅ 권장 — Argon2
# 아래 파라미터는 예시값이다. 환경(하드웨어·지연 허용치)에 맞게 조정하고,
# 최신 권장치는 OWASP Password Storage Cheat Sheet를 참고한다.
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=1)
hashed = ph.hash(password)
ph.verify(hashed, password)
```

#### 로그 마스킹 (Python)

```python
# ❌ 금지 — 로그에 개인식별 정보 평문 기록
logger.info(f"사용자 요청: {user_email}")   # user@example.com 그대로 노출
```
```python
# ✅ 권장 — 기록 전 마스킹
import re

def mask_pii(text: str) -> str:
    # 이메일 마스킹: user@example.com → u***@example.com
    text = re.sub(r'(\w{1})\w+@', r'\1***@', text)
    # 전화번호 마스킹
    text = re.sub(r'(\d{3})-\d{4}-(\d{4})', r'\1-****-\2', text)
    return text

logger.info(mask_pii(f"사용자 요청: {user_email}"))
```

#### 데이터 삭제 권리 구현 (MongoDB)

```python
# ✅ 권장 — 식별정보 삭제 + 서드파티 정리 + 집계 익명화 + 감사 로그
async def delete_user_data(user_id: str):
    # 1. 개인 식별 가능 데이터 삭제
    await db.users.delete_one({"_id": user_id})
    # 2. 서드파티 서비스 삭제 요청 (이메일 서비스, 분석 등)
    await email_service.delete_contact(user_email)
    # 3. 집계 데이터는 익명화 후 보존
    await db.orders.update_many(
        {"user_id": user_id},
        {"$unset": {"user_id": ""}, "$set": {"user_anonymized": True}}
    )
    # 4. 삭제 이벤트 감사 로그 (법적 요건)
    await audit_log.record("USER_DELETED", user_id, timestamp=now())
```
