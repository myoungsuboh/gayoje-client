# 📊 テストカバレッジポリシー (Coverage Policy)

> 行数ではなくビジネスロジック経路基準で意味のあるカバレッジを測定し、コード種別ごとに目標を差別化して適用し、CIでゲーティングする。カバレッジ目標・閾値を定めたり、CIゲーティングを付けたり、テストの空白を点検するときに読む。特定の測定ツール/言語に依存しない汎用標準だ。

## 1. 核心原則
- **カバレッジは目的ではなく信号**: %は「テストが届いていないコード」を示す指標にすぎず、品質の証拠ではない。数値ではなく「どこをもっと見るべきか」を指し示すために使う。
- **行ではなく経路**: 実行された行数ではなく、ビジネスロジックの分岐・境界経路が検証されたかを見る。
- **一律基準ではなく文脈別目標**: コード種別(バグコスト・補完層)に応じて目標を分け、測定価値のないコードは分母から除く。
- **強制は人ではなくCIが**: 閾値と新規コードの低下遮断をパイプラインに置き、人が毎回気を配らなくても回帰を防ぐ。

## 2. 規則

### 2-1. 行ではなくビジネスロジック経路基準で目標を立てる
- カバレッジ%を埋めるために断言(assertion)のないテストや意味のない呼び出しを書かない。
- 核心は「このコードが実行されたか」ではなく「このコードの動作が検証されたか」だ。

```text
// ❌ 禁止 — 行を踏むだけで検証がない (カバレッジは上がるが欠陥を捕まえられない)
test: callDiscount(order)            // 呼び出しのみ、結果断言なし

// ✅ 推奨 — 動作(経路)を断言で検証
test: callDiscount(vipOrder) == 20%  // VIP分岐の結果を検証
test: callDiscount(normalOrder) == 0 // 一般分岐の結果を検証
```

### 2-2. コード種別ごとの差別化目標
- 単一の%目標を全コードに一律適用しない。バグコスト・補完手段(統合/E2E)に応じて目標を分ける。
- 以下の数値は**出発点の例(チームが調整)**にすぎず、固定基準ではない。相対的優先順位(核心>補完層)だけを引き継ぐ。

| コード種別 | 最低目標(出発点) | 理由 |
|-----------|-----------|------|
| ビジネスサービス(核心ロジック) | 80% | バグコストが高い |
| ユーティリティ/ヘルパー | 90% | 再利用性が高くエラー伝播リスク |
| API/入口ルーター | 60% | 統合テストが補完 |
| UIコンポーネント | 60% | E2Eが補完 |
| 生成コード/マイグレーション/設定 | 除外 | 保守・検証の対象外 |

```text
// ❌ 禁止 — 全コード一律80%、核心ロジックとボイラープレートを同一扱い
all: 80%

// ✅ 推奨 — 種別ごとの差別化
business: 80%, util: 90%, ui/router: 60%, generated: 除外
```

### 2-3. 行ではなくブランチ(分岐)カバレッジを見る
- 行カバレッジ100%でも`if/else`・`switch`・`try/catch`・三項・短絡評価の一部経路が踏まれないとバグが隠れる。
- 閾値に行だけでなく**ブランチ**項目を含め、見逃した分岐に境界条件テストを追加する。

```text
ブランチカバレッジ = 条件分岐(if/else/switch/try-catch など)の各経路が実行されたか

// ❌ 行カバレッジは100%だがelse分岐が未テスト
if (isValid) {
  doSomething();   // テスト済み
} else {
  handleError();   // ← 未テスト — バグが隠れうる
}

// ✅ 両分岐とも検証 (isValid=true / false ケースをそれぞれ)
```

### 2-4. 測定価値のないファイルは除外する
- テストファイル・ストーリー・入口・プラグイン・生成コード・マイグレーション・設定ファイルは測定対象から除く。
- これらのファイルを含めると分母が膨らんだり縮んだりして目標達成可否が歪み、レポートがノイズで覆われる。

```text
// ✅ 推奨 — 除外リストを明示 (ツール非依存の概念)
exclude:
  - テスト/スペックファイル (*.test.*, *.spec.*)
  - ストーリー/フィクスチャ
  - 入口/ブートストラップ (アプリ起動点)
  - マイグレーション / 生成コード
  - 設定ファイル
```

### 2-5. CIで閾値をゲーティングする
- カバレッジ閾値未達ならビルドを失敗(終了コード非0)させる — レポートだけ出力して通過させない。
- ゲーティングは人が毎回確認しなくても強制されるようパイプラインに置く。

```text
// ❌ 禁止 — カバレッジを出力するだけで通過 (ゲートなし)
run tests with coverage → 常に成功

// ✅ 推奨 — 閾値未達ならビルド失敗
run tests with coverage AND fail-under(threshold)
  → 未達時に終了コード1 → ビルド失敗
```

### 2-6. 新規コードのカバレッジ低下をPR段階で遮断する
- 全体%ゲートだけでは、新たに追加された未検証コードが既存カバレッジに希釈されて通過しうる。
- 変更分(diff)/新規コード基準で低下幅に限界を置き、カバレッジが下がるPRをマージ前に止める。

```text
// ✅ 推奨 — 絶対閾値 + 変更分の低下限界を併せて
gate: 全体 ≥ 目標  AND  PR 変更分カバレッジ低下 ≤ 許容幅
  → 未検証の新規コードが混ざったPRを遮断
```

## 3. よくある間違い

規則(§2)を破ったときに現れる症状 — 見えたら該当規則に戻る。

- **数値が目標になる** → 断言のない呼び出しで%だけ上げ欠陥を捕まえられない(2-1)。行100%なのに`else`・例外分岐が空(2-3)。
- **分母が歪む** → 単一%一律適用で核心ロジックとボイラープレートが混ざるか(2-2)、生成・設定ファイルが紛れ込み目標がノイズに埋もれる(2-4)。
- **ゲートが新規コードを見逃す** → 全体%ゲートだけ置くと新規未検証コードが希釈されて通過し(2-6)、出力だけだと閾値が強制されず徐々に崩れる(2-5)。

## 4. チェックリスト
- [ ] カバレッジ目標を行数ではなく**ビジネスロジック経路**基準で立てたか
- [ ] コード種別ごとに**差別化目標**を設定したか (核心ロジックを高く、統合/E2E補完層を低く)
- [ ] 生成コード・マイグレーション・設定・入口・テストファイルを測定から**除外**したか
- [ ] 行だけでなく**ブランチ(分岐)カバレッジ**を閾値に含め確認したか
- [ ] CIで閾値未達時に**ビルドが失敗**するか (出力だけ X)
- [ ] **新規コード/変更分のカバレッジ低下**をPR段階で遮断するか
- [ ] 見逃した分岐・境界条件にテストを追加したか (数値埋め用テスト X)

## 付録: スタック別の例

> 以下は参考用の実装例だ。チームが使うスタック(例: Jest/nyc、Go cover、JaCoCo、Coverallsなど)に合った例を同じパターンで追加する。上記1~4の原則・規則が標準であり、付録はその適用事例にすぎない。

### ツール別の例 (Vitest / pytest-cov / Codecov など)

#### Vitest (Vue/JS)

閾値を明示し、テスト・ストーリー・入口・プラグインなど測定価値のないファイルは除外する。

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: "v8",
      reporter: ["text", "html", "lcov"],
      thresholds: {
        lines: 80,
        branches: 75,
        functions: 80,
        statements: 80,
      },
      exclude: [
        "node_modules/",
        "src/**/*.test.*",
        "src/**/*.stories.*",
        "src/main.ts",
        "src/plugins/",
      ],
    },
  },
});
```

#### pytest-cov (Python)

`--cov-fail-under`で閾値未達時にビルドを失敗させ、マイグレーション・テスト・設定は測定から除く。

```ini
# pyproject.toml
[tool.pytest.ini_options]
addopts = "--cov=app --cov-report=html --cov-report=term-missing --cov-fail-under=80"

[tool.coverage.run]
omit = [
    "*/migrations/*",
    "*/tests/*",
    "*/__init__.py",
    "*/config.py",
]
```

#### CIゲーティング + Codecov (GitHub Actions)

閾値未達ならビルドを失敗させ、新規コードのカバレッジ低下はPR段階で遮断する。

```yaml
# GitHub Actions
- name: 테스트 & 커버리지
  run: |
    npm test -- --coverage
    # 커버리지 미달 시 종료 코드 1 → 빌드 실패

- name: Codecov 업로드
  uses: codecov/codecov-action@v4
  with:
    token: ${{ secrets.CODECOV_TOKEN }}
    fail_ci_if_error: true
    threshold: 5  # 5% 이상 커버리지 하락 시 PR 차단
```
