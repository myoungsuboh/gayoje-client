# 📊 测试覆盖率政策 (Coverage Policy)

> 以业务逻辑路径而非行数为基准测量有意义的覆盖率,按代码类型差异化应用目标,并在CI中门禁。在设定覆盖率目标·阈值、加上CI门禁、或检查测试空白时阅读。这是不依赖特定测量工具/语言的通用标准。

## 1. 核心原则
- **覆盖率是信号而非目的**: %只是揭示"测试未触及的代码"的指标,并非质量的证据。用它来指向"该多看哪里",而非当作一个数字。
- **是路径而非行**: 看的是业务逻辑的分支·边界路径是否被验证,而非执行的行数。
- **不是一刀切而是按上下文的目标**: 按代码类型(缺陷成本·补偿层)划分目标,把无测量价值的代码从分母中剔除。
- **强制由CI而非人**: 把阈值与新代码下降阻断放入流水线,使人不必每次操心也能防止回归。

## 2. 规则

### 2-1. 以业务逻辑路径而非行为基准设定目标
- 不要为了填满覆盖率%而编写没有断言(assertion)的测试或无意义的调用。
- 核心不是"这段代码是否被执行",而是"这段代码的行为是否被验证"。

```text
// ❌ 禁止 — 只踏过行而没有验证 (覆盖率上升但抓不到缺陷)
test: callDiscount(order)            // 仅调用,无结果断言

// ✅ 推荐 — 用断言验证行为(路径)
test: callDiscount(vipOrder) == 20%  // 验证VIP分支结果
test: callDiscount(normalOrder) == 0 // 验证普通分支结果
```

### 2-2. 按代码类型的差异化目标
- 不要把单一%目标一律应用于全部代码。按缺陷成本·补偿手段(集成/E2E)划分目标。
- 以下数值只是**起点示例(团队调整)**,并非固定标准。只沿用相对优先级(核心>补偿层)。

| 代码类型 | 最低目标(起点) | 理由 |
|-----------|-----------|------|
| 业务服务(核心逻辑) | 80% | 缺陷成本高 |
| 工具/辅助 | 90% | 复用性高、错误传播风险 |
| API/入口路由 | 60% | 集成测试补偿 |
| UI组件 | 60% | E2E补偿 |
| 生成代码/迁移/配置 | 排除 | 非维护·验证对象 |

```text
// ❌ 禁止 — 全部代码一律80%,核心逻辑与样板同等对待
all: 80%

// ✅ 推荐 — 按类型差异化
business: 80%, util: 90%, ui/router: 60%, generated: 排除
```

### 2-3. 看分支(分支)覆盖率而非行
- 即使行覆盖率100%,若`if/else`·`switch`·`try/catch`·三元·短路求值的部分路径未被踏过,缺陷会隐藏。
- 在阈值中不仅含行还含**分支**项,并对遗漏的分支添加边界条件测试。

```text
分支覆盖率 = 条件分支(if/else/switch/try-catch 等)的各路径是否被执行

// ❌ 行覆盖率为100%但else分支未测试
if (isValid) {
  doSomething();   // 已测试
} else {
  handleError();   // ← 未测试 — 缺陷可能隐藏
}

// ✅ 两个分支都验证 (isValid=true / false 各自的情况)
```

### 2-4. 排除无测量价值的文件
- 把测试文件·故事·入口·插件·生成代码·迁移·配置文件从测量对象中剔除。
- 包含这些文件会使分母膨胀或缩小,扭曲目标是否达成,并使报告被噪声淹没。

```text
// ✅ 推荐 — 明示排除列表 (与工具无关的概念)
exclude:
  - 测试/规格文件 (*.test.*, *.spec.*)
  - 故事/夹具
  - 入口/引导 (应用启动点)
  - 迁移 / 生成代码
  - 配置文件
```

### 2-5. 在CI中对阈值门禁
- 覆盖率阈值未达则让构建失败(退出码非0) — 不要只输出报告就放行。
- 把门禁放入流水线,使其不必人每次确认也被强制。

```text
// ❌ 禁止 — 只输出覆盖率就放行 (无门禁)
run tests with coverage → 总是成功

// ✅ 推荐 — 阈值未达则构建失败
run tests with coverage AND fail-under(threshold)
  → 未达时退出码1 → 构建失败
```

### 2-6. 在PR阶段阻断新代码的覆盖率下降
- 仅靠整体%门禁,新添加的未验证代码可能被既有覆盖率稀释而通过。
- 以变更部分(diff)/新代码为基准对下降幅度设上限,在合并前拦住覆盖率下降的PR。

```text
// ✅ 推荐 — 绝对阈值 + 变更部分下降上限一起
gate: 整体 ≥ 目标  AND  PR 变更部分覆盖率下降 ≤ 允许幅度
  → 阻断混入未验证新代码的PR
```

## 3. 常见错误

违反规则(§2)时出现的症状 — 看到就回到相应规则。

- **数值变成目标** → 用无断言的调用只把%抬高、抓不到缺陷(2-1)。行100%但`else`·异常分支为空(2-3)。
- **分母被扭曲** → 单一%一律应用使核心逻辑与样板混杂(2-2),或生成·配置文件混入使目标埋没于噪声(2-4)。
- **门禁漏掉新代码** → 只放整体%门禁时新的未验证代码被稀释而通过(2-6),只输出则阈值不被强制而逐渐崩塌(2-5)。

## 4. 检查清单
- [ ] 是否以**业务逻辑路径**而非行数为基准设定了覆盖率目标
- [ ] 是否按代码类型设置了**差异化目标** (核心逻辑高、集成/E2E补偿层低)
- [ ] 是否将生成代码·迁移·配置·入口·测试文件从测量中**排除**
- [ ] 是否在阈值中不仅含行还含**分支(分支)覆盖率**并加以确认
- [ ] CI中阈值未达时**构建是否失败** (而非仅输出)
- [ ] 是否在PR阶段阻断**新代码/变更部分的覆盖率下降**
- [ ] 是否对遗漏的分支·边界条件添加了测试 (而非凑数用的测试)

## 附录: 按栈的示例

> 以下是供参考的实现示例。按团队所用的栈(例: Jest/nyc、Go cover、JaCoCo、Coveralls等)以相同模式追加示例。上述1~4的原则·规则为标准,附录只是其应用案例。

### 按工具的示例 (Vitest / pytest-cov / Codecov 等)

#### Vitest (Vue/JS)

明示阈值,并排除测试·故事·入口·插件等无测量价值的文件。

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: "v8",
      reporter: ["text", "html", "lcov"],
      thresholds: {
        lines: 80,
        branches: 75,
        functions: 80,
        statements: 80,
      },
      exclude: [
        "node_modules/",
        "src/**/*.test.*",
        "src/**/*.stories.*",
        "src/main.ts",
        "src/plugins/",
      ],
    },
  },
});
```

#### pytest-cov (Python)

用`--cov-fail-under`在阈值未达时让构建失败,并将迁移·测试·配置从测量中剔除。

```ini
# pyproject.toml
[tool.pytest.ini_options]
addopts = "--cov=app --cov-report=html --cov-report=term-missing --cov-fail-under=80"

[tool.coverage.run]
omit = [
    "*/migrations/*",
    "*/tests/*",
    "*/__init__.py",
    "*/config.py",
]
```

#### CI门禁 + Codecov (GitHub Actions)

阈值未达则让构建失败,新代码的覆盖率下降在PR阶段阻断。

```yaml
# GitHub Actions
- name: 테스트 & 커버리지
  run: |
    npm test -- --coverage
    # 커버리지 미달 시 종료 코드 1 → 빌드 실패

- name: Codecov 업로드
  uses: codecov/codecov-action@v4
  with:
    token: ${{ secrets.CODECOV_TOKEN }}
    fail_ci_if_error: true
    threshold: 5  # 5% 이상 커버리지 하락 시 PR 차단
```
