# 📊 테스트 커버리지 정책 (Coverage Policy)

> 라인 수가 아니라 비즈니스 로직 경로 기준으로 의미 있는 커버리지를 측정하고, 코드 유형별로 목표를 차등 적용하며, CI에서 게이팅한다. 커버리지 목표·임계치를 정하거나, CI 게이팅을 붙이거나, 테스트 공백을 점검할 때 읽는다. 특정 측정 도구/언어에 종속되지 않는 범용 표준이다.

## 1. 핵심 원칙
- **커버리지는 목적이 아니라 신호**: %는 "테스트가 닿지 않은 코드"를 드러내는 지표일 뿐 품질의 증거가 아니다. 숫자가 아니라 "어디를 더 봐야 하는지"를 가리키는 데 쓴다.
- **라인이 아니라 경로**: 실행된 라인 수가 아니라 비즈니스 로직의 분기·경계 경로가 검증됐는지를 본다.
- **한 줄 기준이 아니라 맥락별 목표**: 코드 유형(버그 비용·보완 계층)에 따라 목표를 나누고, 측정 가치 없는 코드는 분모에서 뺀다.
- **강제는 사람이 아니라 CI가**: 임계치와 새 코드 하락 차단을 파이프라인에 두어, 사람이 매번 챙기지 않아도 회귀를 막는다.

## 2. 규칙

### 2-1. 라인이 아닌 비즈니스 로직 경로 기준으로 목표를 잡는다
- 커버리지 %를 채우려고 단언(assertion) 없는 테스트나 의미 없는 호출을 작성하지 않는다.
- 핵심은 "이 코드가 실행됐는가"가 아니라 "이 코드의 동작이 검증됐는가"다.

```text
// ❌ 금지 — 라인만 밟고 검증이 없음 (커버리지는 오르지만 결함을 못 잡음)
test: callDiscount(order)            // 호출만, 결과 단언 없음

// ✅ 권장 — 동작(경로)을 단언으로 검증
test: callDiscount(vipOrder) == 20%  // VIP 분기 결과를 검증
test: callDiscount(normalOrder) == 0 // 일반 분기 결과를 검증
```

### 2-2. 코드 유형별 차등 목표
- 단일 % 목표를 전 코드에 일괄 적용하지 않는다. 버그 비용·보완 수단(통합/E2E)에 따라 목표를 나눈다.
- 아래 수치는 **출발점 예시(팀이 조정)** 일 뿐, 고정 기준이 아니다. 상대적 우선순위(핵심>보완 계층)만 가져간다.

| 코드 유형 | 최소 목표(출발점) | 이유 |
|-----------|-----------|------|
| 비즈니스 서비스(핵심 로직) | 80% | 버그 비용 높음 |
| 유틸리티/헬퍼 | 90% | 재사용성 높아 오류 전파 위험 |
| API/진입점 라우터 | 60% | 통합 테스트가 보완 |
| UI 컴포넌트 | 60% | E2E가 보완 |
| 생성 코드/마이그레이션/설정 | 제외 | 유지보수·검증 대상 아님 |

```text
// ❌ 금지 — 전 코드 일괄 80%, 핵심 로직과 보일러플레이트를 동일 취급
all: 80%

// ✅ 권장 — 유형별 차등
business: 80%, util: 90%, ui/router: 60%, generated: 제외
```

### 2-3. 라인이 아닌 브랜치(분기) 커버리지를 본다
- 라인 커버리지 100%여도 `if/else`·`switch`·`try/catch`·삼항·단축평가의 일부 경로가 안 밟히면 버그가 숨는다.
- 임계치에 라인뿐 아니라 **브랜치** 항목을 포함하고, 놓친 분기에 경계 조건 테스트를 추가한다.

```text
브랜치 커버리지 = 조건 분기(if/else/switch/try-catch 등)의 각 경로가 실행됐는지

// ❌ 라인 커버리지는 100%지만 else 분기 미테스트
if (isValid) {
  doSomething();   // 테스트됨
} else {
  handleError();   // ← 미테스트 — 버그 숨을 수 있음
}

// ✅ 두 분기 모두 검증 (isValid=true / false 케이스 각각)
```

### 2-4. 측정 가치 없는 파일은 제외한다
- 테스트 파일·스토리·진입점·플러그인·생성 코드·마이그레이션·설정 파일은 측정 대상에서 뺀다.
- 이런 파일을 포함하면 분모가 부풀거나 줄어 목표 달성 여부가 왜곡되고, 리포트가 노이즈로 덮인다.

```text
// ✅ 권장 — 제외 목록을 명시 (도구 무관 개념)
exclude:
  - 테스트/스펙 파일 (*.test.*, *.spec.*)
  - 스토리/픽스처
  - 진입점/부트스트랩 (앱 시작점)
  - 마이그레이션 / 생성 코드
  - 설정 파일
```

### 2-5. CI에서 임계치를 게이팅한다
- 커버리지 임계치 미달이면 빌드를 실패(종료 코드 비0)시킨다 — 리포트만 출력하고 통과시키지 않는다.
- 게이팅은 사람이 매번 확인하지 않아도 강제되도록 파이프라인에 둔다.

```text
// ❌ 금지 — 커버리지를 출력만 하고 통과 (게이트 없음)
run tests with coverage → 항상 성공

// ✅ 권장 — 임계치 미달이면 빌드 실패
run tests with coverage AND fail-under(threshold)
  → 미달 시 종료 코드 1 → 빌드 실패
```

### 2-6. 새 코드의 커버리지 하락을 PR 단계에서 차단한다
- 전체 % 게이트만으로는 새로 추가된 미검증 코드가 기존 커버리지에 희석돼 통과할 수 있다.
- 변경분(diff)/새 코드 기준으로 하락 폭에 한계를 두어, 커버리지가 떨어지는 PR을 머지 전에 막는다.

```text
// ✅ 권장 — 절대 임계치 + 변경분 하락 한계를 함께
gate: 전체 ≥ 목표  AND  PR 변경분 커버리지 하락 ≤ 허용폭
  → 미검증 새 코드가 섞인 PR 차단
```

## 3. 흔한 실수

규칙(§2)을 어겼을 때 나타나는 증상들 — 보이면 해당 규칙으로 돌아간다.

- **숫자가 목표가 됨** → 단언 없는 호출로 %만 올려 결함을 못 잡는다 (2-1). 라인 100%인데 `else`·예외 분기가 비어 있다 (2-3).
- **분모가 왜곡됨** → 단일 % 일괄 적용으로 핵심 로직과 보일러플레이트가 뒤섞이거나(2-2), 생성·설정 파일이 끼어 목표가 노이즈에 묻힌다(2-4).
- **게이트가 새 코드를 놓침** → 전체 % 게이트만 두면 새 미검증 코드가 희석돼 통과하고(2-6), 출력만 하면 임계치가 강제되지 않아 서서히 무너진다(2-5).

## 4. 체크리스트
- [ ] 커버리지 목표를 라인 수가 아닌 **비즈니스 로직 경로** 기준으로 잡았는가
- [ ] 코드 유형별로 **차등 목표**를 설정했는가 (핵심 로직 높게, 통합/E2E 보완 계층 낮게)
- [ ] 생성 코드·마이그레이션·설정·진입점·테스트 파일을 측정에서 **제외**했는가
- [ ] 라인뿐 아니라 **브랜치(분기) 커버리지**를 임계치에 포함하고 확인했는가
- [ ] CI에서 임계치 미달 시 **빌드가 실패**하는가 (출력만 X)
- [ ] **새 코드/변경분의 커버리지 하락**을 PR 단계에서 차단하는가
- [ ] 놓친 분기·경계 조건에 테스트를 추가했는가 (숫자 채우기용 테스트 X)

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(예: Jest/nyc, Go cover, JaCoCo, Coveralls 등)에 맞는 예시를 같은 패턴으로 추가한다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다.

### 도구별 예시 (Vitest / pytest-cov / Codecov 등)

#### Vitest (Vue/JS)

임계치를 명시하고, 테스트·스토리·진입점·플러그인 등 측정 가치가 없는 파일은 제외한다.

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: "v8",
      reporter: ["text", "html", "lcov"],
      thresholds: {
        lines: 80,
        branches: 75,
        functions: 80,
        statements: 80,
      },
      exclude: [
        "node_modules/",
        "src/**/*.test.*",
        "src/**/*.stories.*",
        "src/main.ts",
        "src/plugins/",
      ],
    },
  },
});
```

#### pytest-cov (Python)

`--cov-fail-under`로 임계치 미달 시 빌드를 실패시키고, 마이그레이션·테스트·설정은 측정에서 뺀다.

```ini
# pyproject.toml
[tool.pytest.ini_options]
addopts = "--cov=app --cov-report=html --cov-report=term-missing --cov-fail-under=80"

[tool.coverage.run]
omit = [
    "*/migrations/*",
    "*/tests/*",
    "*/__init__.py",
    "*/config.py",
]
```

#### CI 게이팅 + Codecov (GitHub Actions)

임계치 미달이면 빌드를 실패시키고, 새 코드의 커버리지 하락은 PR 단계에서 차단한다.

```yaml
# GitHub Actions
- name: 테스트 & 커버리지
  run: |
    npm test -- --coverage
    # 커버리지 미달 시 종료 코드 1 → 빌드 실패

- name: Codecov 업로드
  uses: codecov/codecov-action@v4
  with:
    token: ${{ secrets.CODECOV_TOKEN }}
    fail_ci_if_error: true
    threshold: 5  # 5% 이상 커버리지 하락 시 PR 차단
```
