# 🎲 フレーキーテスト管理 (Flaky Test Management)

> コード変更なしに合格/不合格が行き来するテスト(flaky)を検出し、隔離し、根本原因を直す。CI が間欠的に赤くなる、再実行すると通る、「あれは元々たまに壊れます」という言葉が出るときに読む。

フレーキーテストの本当の被害は一度の失敗ではなく **信頼の崩壊** である — 赤信号を「またあれだろう」と無視し始めると、本物のリグレッションも一緒に埋もれる。だからフレーキーは「気に障るノイズ」ではなく **直すべきバグ** として扱う。多くの場合、原因はテストの **非決定性**(時間・順序・並行性・外部依存・共有状態)であり、`sleep` を増やしたりリトライを塗りたくったりするのは症状を隠すだけだ。

## 1. コア原則

- **フレーキーはバグだ** — 無視したり盲目的にリトライしたりしない。green を信じられなくなると、テストスイート全体が無力化する。
- **原因はたいてい非決定性** — 固定待機、テスト間の順序依存、並行性の競合、実ネットワーク・時間、共有状態の未整理が主犯だ。
- **隔離するが必ず追跡** — メインパイプラインを塞いではならないので隔離(quarantine)するが、**チケットなしで放置は禁止**。隔離は一時しのぎであって解決ではない。
- **根本原因を直す** — `sleep` を増やす・リトライ回数を上げるのではなく、決定性を確保する。
- **リトライは慎重に、計測とともに** — 無制限・グローバルな自動リトライはフレーキーを隠す。限定的に使い、リトライ率を記録する。
- **流入を止めて測定する** — flaky rate を追跡し、新たに入ってくるフレーキーをレビュー・CI で弾く。

## 2. ルール

### 2-1. まず原因を分類する

リトライする前に **なぜ揺れるのか** をまず見る。

| 原因 | 兆候 | 処方 |
|---|---|---|
| 固定待機(time-based) | `sleep(n)` 後の断言、遅い環境でのみ失敗 | 条件ベースの待機 (2-2) |
| 順序依存 | 単独実行は通過、特定の順序で失敗 | テスト隔離・独立 (2-3) |
| 並行性の競合 | たまにのみ失敗、負荷時に増加 | 同期・決定的スケジュール・隔離 |
| 実外部依存 | ネットワーク/外部 API の不安定に左右 | モック/スタブ・サンドボックス (2-4) |
| 時間・乱数 | 真夜中・月末・特定の seed で失敗 | 時計・乱数の固定 (2-4) |
| 共有状態 | 前のテストの残留物に左右 | setup/teardown で初期化 (2-3) |

### 2-2. 固定待機禁止 → 条件ベースの待機

```text
❌ 禁止 — 任意の時間待機(遅ければ失敗、速ければ無駄)
   click("保存"); sleep(3000); expect(toast).visible

✅ 推奨 — 望む状態になるまで待機(ポーリング/イベント)
   click("保存"); await waitFor(() => toast.isVisible())   // 条件充足で直ちに進行、タイムアウト上限
```

### 2-3. テストを隔離し順序に独立させる

```text
❌ 禁止 — 前のテストが作ったデータ/グローバル状態に依存
✅ 推奨 — 各テストが自分の状態を作って整理(setup/teardown)、共有シングルトンを reset
         無作為な順序で回して順序依存を積極的に暴き出す
```

### 2-4. 時間・乱数・ネットワークを固定する

```text
❌ 禁止 — now()/Math.random()/実 HTTP に依存 → 環境・時点に揺れる
✅ 推奨 — 偽の時計(fake timer/clock)・固定 seed・ネットワークモック/スタブで決定化
         (単体・統合の隔離は `unit-testing`・`integration-testing`)
```

### 2-5. 隔離(quarantine)するがチケットで追跡する

```text
❌ 禁止 — @Disabled / test.skip だけ付けて終わり → 永遠に忘れられる、カバレッジの穴
✅ 推奨 — 隔離タグ(例: @Tag("flaky"))でメインゲートから分離 + 追跡チケット必須
         隔離は「今ビルドを解放する一時しのぎ」、期限と担当を定める
```

### 2-6. リトライは限定的に、隠さない

```text
❌ 禁止 — グローバル無制限の自動リトライ → フレーキーが永遠に隠れる
✅ 推奨 — リトライ上限を狭く(例: 1〜2回)しつつ、リトライで通ったケースを記録・集計
         リトライ率が上がれば通知 → 根本原因の作業へ(リトライは緩和であって解決ではない)
```

## 3. よくある間違い

- ❌ 赤信号を「またあれ」と再実行で流す → 本物のリグレッションも一緒に埋もれる
- ❌ `sleep` 時間を増やして「解決」 → 遅くなるだけで、より遅い環境で再発
- ❌ グローバル自動リトライでフレーキーを永久に隠蔽
- ❌ 隔離だけしてチケットなしで放置 → カバレッジに静かな穴
- ❌ 単独実行のみ確認してマージ → 順序依存・共有状態のフレーキーを捕まえられない
- ❌ 実ネットワーク・時計に依存するテストをそのまま放置

> **適用のコツ**: CI でテストを無作為な順序で回し、マージ前に変更されたテストを複数回繰り返し実行してフレーキーを入口で弾く。決定性の原則は `unit-testing`・`integration-testing`、E2E のフレーキー(最も多い)は `e2e-testing` も併せて見る。

## 4. チェックリスト

- [ ] リトライ/スキップの前に原因(時間・順序・並行性・外部依存・共有状態)を分類したか
- [ ] 固定 `sleep` の代わりに条件ベースの待機を使うか
- [ ] テストが互いに独立で(共有状態を初期化)、無作為な順序でも通るか
- [ ] 時間・乱数・ネットワークを固定(fake timer・seed・モック)したか
- [ ] 隔離したテストに追跡チケット・担当・期限があるか(放置禁止)
- [ ] リトライは上限が狭く、リトライで通ったケースを計測・可視化するか

## 付録: ツール別の例

> 以下はツール別の隔離・リトライ設定例である。チームが使うランナーに合わせて追加する。上の 1〜4 の原則が標準であり、付録は適用例にすぎない。

### Playwright
```ts
// playwright.config.ts — CI でのみ限定的なリトライ + 追跡用レポート
export default defineConfig({
  retries: process.env.CI ? 2 : 0,   // 無制限禁止
  reporter: [['html'], ['list']],
})
test.fixme(isFlaky, 'フレーキー追跡: ISSUE-123')   // 隔離 + チケット明示
```

### Jest / Vitest
```ts
test.skip('決済の並行性 — フレーキー, ISSUE-123', async () => { /* ... */ })
// 自動リトライ(jest.retryTimes, ファイル/describe スコープ)の乱用禁止 — 使うならリトライ通過を別途集計
```

### JUnit 5 (JVM)
```java
@Tag("flaky")   // CI メインゲートから除外するタグ + 追跡チケット(ISSUE-123)
@Test void paymentConcurrency() { /* ... */ }
```
