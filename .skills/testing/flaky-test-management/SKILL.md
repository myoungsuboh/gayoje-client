# 🎲 플래키 테스트 관리 (Flaky Test Management)

> 코드 변화 없이 통과/실패가 오락가락하는 테스트(flaky)를 탐지하고, 격리하고, 근본 원인을 고친다. CI가 간헐적으로 빨개지거나, 재실행하면 통과하거나, "그건 원래 가끔 깨져요"라는 말이 나올 때 읽는다.

플래키 테스트의 진짜 피해는 한 번의 실패가 아니라 **신뢰의 붕괴**다 — 빨간불을 "또 그거겠지" 하고 무시하기 시작하면, 진짜 회귀도 같이 묻힌다. 그래서 플래키는 "거슬리는 잡음"이 아니라 **고쳐야 할 버그**로 다룬다. 대부분 원인은 테스트의 **비결정성**(시간·순서·동시성·외부 의존·공유 상태)이며, `sleep`을 늘리거나 재시도를 떡칠하는 건 증상만 가린다.

## 1. 핵심 원칙

- **플래키는 버그다** — 무시하거나 맹목적으로 재시도하지 않는다. green을 못 믿게 되면 테스트 스위트 전체가 무력해진다.
- **원인은 대개 비결정성** — 고정 대기, 테스트 간 순서 의존, 동시성 경쟁, 실제 네트워크·시간, 공유 상태 미정리가 주범이다.
- **격리하되 반드시 추적** — 메인 파이프라인을 막으면 안 되니 격리(quarantine)하되, **티켓 없이 방치 금지**. 격리는 임시 조치지 해결이 아니다.
- **근본 원인을 고친다** — `sleep` 늘리기·재시도 횟수 올리기가 아니라 결정성을 확보한다.
- **재시도는 신중히, 계측과 함께** — 무제한·전역 자동 재시도는 플래키를 숨긴다. 제한적으로 쓰고 재시도율을 기록한다.
- **유입을 막고 측정한다** — flaky rate를 추적하고, 새로 들어오는 플래키를 리뷰·CI에서 걸러낸다.

## 2. 규칙

### 2-1. 원인부터 분류한다

재시도하기 전에 **왜 흔들리는지**부터 본다.

| 원인 | 신호 | 처방 |
|---|---|---|
| 고정 대기(time-based) | `sleep(n)` 후 단언, 느린 환경에서만 실패 | 조건 기반 대기 (2-2) |
| 순서 의존 | 단독 실행은 통과, 특정 순서에서 실패 | 테스트 격리·독립 (2-3) |
| 동시성 경쟁 | 가끔만 실패, 부하 시 증가 | 동기화·결정적 스케줄·격리 |
| 실제 외부 의존 | 네트워크/외부 API 불안정에 좌우 | 모킹/스텁·샌드박스 (2-4) |
| 시간·랜덤 | 자정·월말·특정 seed에 실패 | 시계·난수 고정 (2-4) |
| 공유 상태 | 앞 테스트 잔여물에 좌우 | setup/teardown로 초기화 (2-3) |

### 2-2. 고정 대기 금지 → 조건 기반 대기

```text
❌ 금지 — 임의 시간 대기 (느리면 실패, 빠르면 낭비)
   click("저장"); sleep(3000); expect(toast).visible

✅ 권장 — 원하는 상태가 될 때까지 대기 (폴링/이벤트)
   click("저장"); await waitFor(() => toast.isVisible())   // 조건 충족 시 즉시 진행, 타임아웃 상한
```

### 2-3. 테스트를 격리하고 순서에 독립시킨다

```text
❌ 금지 — 앞 테스트가 만든 데이터/전역 상태에 의존
✅ 권장 — 각 테스트가 자기 상태를 만들고 정리(setup/teardown), 공유 싱글톤 reset
         무작위 순서로 돌려 순서 의존을 적극적으로 들춰낸다
```

### 2-4. 시간·랜덤·네트워크를 고정한다

```text
❌ 금지 — now()/Math.random()/실제 HTTP에 의존 → 환경·시점에 흔들림
✅ 권장 — 가짜 시계(fake timer/clock)·고정 seed·네트워크 모킹/스텁으로 결정화
         (단위·통합 격리는 `unit-testing`·`integration-testing`)
```

### 2-5. 격리(quarantine)하되 티켓으로 추적한다

```text
❌ 금지 — @Disabled / test.skip 만 달고 끝 → 영원히 잊힘, 커버리지 구멍
✅ 권장 — 격리 태그(예: @Tag("flaky")) 로 메인 게이트에서 분리 + 추적 티켓 필수
         격리는 "지금 빌드를 풀어주는 임시 조치", 마감 기한과 담당을 정한다
```

### 2-6. 재시도는 제한적으로, 숨기지 않는다

```text
❌ 금지 — 전역 무제한 자동 재시도 → 플래키가 영원히 가려짐
✅ 권장 — 재시도 상한을 좁게(예: 1~2회) 두되, 재시도로 통과한 케이스를 기록·집계
         재시도율이 오르면 알림 → 근본 원인 작업으로 (재시도는 완화지 해결이 아님)
```

## 3. 흔한 실수

- ❌ 빨간불을 "또 그거"라며 재실행으로 넘김 → 진짜 회귀도 함께 묻힘
- ❌ `sleep` 시간을 늘려 "해결" → 느려지기만 하고 더 느린 환경에서 재발
- ❌ 전역 자동 재시도로 플래키를 영구히 은폐
- ❌ 격리만 하고 티켓 없이 방치 → 커버리지에 조용한 구멍
- ❌ 단독 실행만 확인하고 머지 → 순서 의존·공유 상태 플래키를 못 잡음
- ❌ 실제 네트워크·시계에 의존하는 테스트를 그대로 둠

> **적용 팁**: CI에서 테스트를 무작위 순서로 돌리고, 머지 전 변경된 테스트를 여러 번 반복 실행해 플래키를 입구에서 거른다. 결정성 원칙은 `unit-testing`·`integration-testing`, E2E의 플래키(가장 흔함)는 `e2e-testing`과 함께 본다.

## 4. 체크리스트

- [ ] 재시도/스킵 전에 원인(시간·순서·동시성·외부의존·공유상태)을 분류했는가
- [ ] 고정 `sleep` 대신 조건 기반 대기를 쓰는가
- [ ] 테스트가 서로 독립적이고(공유 상태 초기화), 무작위 순서로도 통과하는가
- [ ] 시간·랜덤·네트워크를 고정(fake timer·seed·모킹)했는가
- [ ] 격리한 테스트에 추적 티켓·담당·기한이 있는가 (방치 금지)
- [ ] 재시도는 상한이 좁고, 재시도로 통과한 케이스를 계측·가시화하는가

## 부록: 도구별 예시

> 아래는 도구별 격리·재시도 설정 예시다. 팀이 쓰는 러너에 맞게 추가한다. 위 1~4의 원칙이 표준이고, 부록은 적용 사례일 뿐이다.

### Playwright
```ts
// playwright.config.ts — CI에서만 제한적 재시도 + 추적용 리포트
export default defineConfig({
  retries: process.env.CI ? 2 : 0,   // 무제한 금지
  reporter: [['html'], ['list']],
})
test.fixme(isFlaky, '플래키 추적: ISSUE-123')   // 격리 + 티켓 명시
```

### Jest / Vitest
```ts
test.skip('결제 동시성 — 플래키, ISSUE-123', async () => { /* ... */ })
// 자동 재시도(jest.retryTimes, 파일/describe 스코프) 남용 금지 — 쓰면 재시도 통과를 별도 집계
```

### JUnit 5 (JVM)
```java
@Tag("flaky")   // CI 메인 게이트에서 제외하는 태그 + 추적 티켓(ISSUE-123)
@Test void paymentConcurrency() { /* ... */ }
```
