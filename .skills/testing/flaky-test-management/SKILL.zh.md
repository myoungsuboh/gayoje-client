# 🎲 脆弱测试管理 (Flaky Test Management)

> 检测、隔离并根因修复在代码无变化的情况下时而通过时而失败的测试(flaky)。在 CI 间歇性变红、重跑就过、或听到"那个本来就偶尔会坏"时阅读。

脆弱测试真正的危害不是一次失败,而是 **信任的崩塌** — 一旦开始把红灯当作"八成又是那个"而忽视,真正的回归也会被一起埋没。所以脆弱测试不被当作"碍眼的噪声",而是 **要修复的 bug**。多数情况下,原因是测试的 **非确定性**(时间·顺序·并发·外部依赖·共享状态),而加大 `sleep` 或猛堆重试只是掩盖症状。

## 1. 核心原则

- **脆弱测试是 bug** — 不要忽视或盲目重试。一旦不再相信 green,整个测试套件就失去意义。
- **原因多半是非确定性** — 固定等待、测试间顺序依赖、并发竞争、真实网络·时间、共享状态未清理是主犯。
- **隔离但必须追踪** — 不能堵住主流水线所以隔离(quarantine),但**禁止无工单放置**。隔离是临时措施而非解决。
- **修复根因** — 不是加大 `sleep`·提高重试次数,而是确保确定性。
- **重试要谨慎,并带计量** — 无限制·全局自动重试会掩盖脆弱测试。有限使用并记录重试率。
- **拦住流入并测量** — 追踪 flaky rate,在评审·CI 中拦截新进来的脆弱测试。

## 2. 规则

### 2-1. 先分类原因

在重试之前先看 **为什么会抖动**。

| 原因 | 信号 | 处方 |
|---|---|---|
| 固定等待(time-based) | `sleep(n)` 后断言,仅在慢环境失败 | 基于条件的等待 (2-2) |
| 顺序依赖 | 单独运行通过,特定顺序失败 | 测试隔离·独立 (2-3) |
| 并发竞争 | 偶尔失败,负载时增多 | 同步·确定性调度·隔离 |
| 真实外部依赖 | 受网络/外部 API 不稳定左右 | 模拟/桩·沙箱 (2-4) |
| 时间·随机 | 在午夜·月末·特定 seed 失败 | 固定时钟·随机数 (2-4) |
| 共享状态 | 受上一个测试残留物左右 | 用 setup/teardown 初始化 (2-3) |

### 2-2. 禁止固定等待 → 基于条件的等待

```text
❌ 禁止 — 等待任意时间(慢则失败,快则浪费)
   click("保存"); sleep(3000); expect(toast).visible

✅ 推荐 — 等到所需状态(轮询/事件)
   click("保存"); await waitFor(() => toast.isVisible())   // 条件满足即刻继续,设超时上限
```

### 2-3. 隔离测试并使其与顺序无关

```text
❌ 禁止 — 依赖上一个测试创建的数据/全局状态
✅ 推荐 — 每个测试创建并清理自己的状态(setup/teardown),重置共享单例
         以随机顺序运行,积极揭露顺序依赖
```

### 2-4. 固定时间·随机·网络

```text
❌ 禁止 — 依赖 now()/Math.random()/真实 HTTP → 随环境·时点抖动
✅ 推荐 — 用假时钟(fake timer/clock)·固定 seed·网络模拟/桩使其确定化
         (单元·集成的隔离见 `unit-testing`·`integration-testing`)
```

### 2-5. 隔离(quarantine)但用工单追踪

```text
❌ 禁止 — 只贴 @Disabled / test.skip 就完事 → 永远被遗忘,覆盖率漏洞
✅ 推荐 — 用隔离标签(如 @Tag("flaky"))从主门禁分离 + 必须有追踪工单
         隔离是"现在解开构建的临时措施",定下期限与负责人
```

### 2-6. 重试要有限,不要掩盖

```text
❌ 禁止 — 全局无限制自动重试 → 脆弱测试被永久掩盖
✅ 推荐 — 把重试上限定窄(如 1~2 次),但记录·汇总靠重试才通过的用例
         重试率上升则告警 → 转入根因工作(重试是缓解而非解决)
```

## 3. 常见错误

- ❌ 把红灯当"又是那个"重跑了事 → 真正的回归也一起被埋没
- ❌ 加大 `sleep` 时间来"解决" → 只是变慢,在更慢的环境里复发
- ❌ 用全局自动重试永久隐藏脆弱测试
- ❌ 只隔离而无工单放置 → 覆盖率里悄然的漏洞
- ❌ 只确认单独运行就合并 → 抓不住顺序依赖·共享状态的脆弱测试
- ❌ 把依赖真实网络·时钟的测试原样留着

> **应用要点**: 在 CI 中以随机顺序运行测试,合并前对改动的测试反复运行多次,在入口处拦截脆弱测试。确定性原则见 `unit-testing`·`integration-testing`,E2E 的脆弱测试(最常见)请一并看 `e2e-testing`。

## 4. 检查清单

- [ ] 在重试/跳过之前是否对原因(时间·顺序·并发·外部依赖·共享状态)做了分类
- [ ] 是否用基于条件的等待代替固定 `sleep`
- [ ] 测试是否彼此独立(初始化共享状态),并且以随机顺序也能通过
- [ ] 是否固定了时间·随机·网络(fake timer·seed·模拟)
- [ ] 隔离的测试是否有追踪工单·负责人·期限(禁止放置)
- [ ] 重试上限是否窄,并对靠重试通过的用例进行计量·可视化

## 附录: 各工具示例

> 以下是各工具的隔离·重试配置示例。按团队所用的运行器补充。上面 1~4 的原则才是标准,附录只是应用案例。

### Playwright
```ts
// playwright.config.ts — 仅在 CI 中有限重试 + 追踪用报告
export default defineConfig({
  retries: process.env.CI ? 2 : 0,   // 禁止无限制
  reporter: [['html'], ['list']],
})
test.fixme(isFlaky, '脆弱测试追踪: ISSUE-123')   // 隔离 + 明示工单
```

### Jest / Vitest
```ts
test.skip('支付并发 — 脆弱, ISSUE-123', async () => { /* ... */ })
// 禁止滥用自动重试(jest.retryTimes, 文件/describe 作用域) — 用了就把重试通过单独汇总
```

### JUnit 5 (JVM)
```java
@Tag("flaky")   // 在 CI 主门禁排除的标签 + 追踪工单(ISSUE-123)
@Test void paymentConcurrency() { /* ... */ }
```
