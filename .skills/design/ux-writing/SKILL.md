# ✍️ UX 라이팅 & 마이크로카피

> UI의 모든 텍스트를 사용자 중심으로 통일한다. 새 화면의 문구를 쓰거나 버튼·오류·로딩·빈 상태 텍스트를 정할 때 읽는다.

## 1. 핵심 원칙
- 버튼 텍스트는 동사로 시작하는 행동 지향 문구를 사용한다 (예: '저장하기', '삭제', '시작하기').
- 오류 메시지는 원인과 해결 방법을 함께 제공하고, 기술 용어(500 Error, null, undefined)를 노출하지 않는다.
- 로딩 상태는 '불러오는 중...', '저장 중...'처럼 진행 중임을 명시해 응답이 있다는 것을 알린다.
- 빈 상태(Empty State)에는 이유 + 행동 유도 문구를 제공하고, '데이터 없음'만 표시하지 않는다.
- 국제화(i18n) 대상 텍스트는 하드코딩하지 않고 반드시 번역 키로 관리한다.

## 2. 규칙

### 2-1. 버튼 & 레이블
| ❌ 나쁜 예 | ✅ 좋은 예 |
|---------|---------|
| 확인 | 저장하기 |
| 취소 | 돌아가기 |
| 삭제 | 계정 삭제 |
| Error | 저장에 실패했습니다 |

- **구체성**: 무엇을 하는 버튼인지 명확히
- **일관성**: 동일 행동에는 동일 단어 (저장 vs 저장하기를 혼용 금지)
- **간결성**: 최대 3~4단어

### 2-2. 오류 메시지 패턴
```
❌ 나쁜 예: "Error 422: Validation failed"
✅ 좋은 예: "이메일 형식이 올바르지 않습니다. 예: example@domain.com"

❌ 나쁜 예: "네트워크 오류"
✅ 좋은 예: "연결에 실패했습니다. 인터넷 연결을 확인하고 다시 시도해 주세요."
```

오류 메시지 구성:
1. **무엇이** 잘못됐는지 (원인)
2. **어떻게** 해결하는지 (행동)
3. 필요시 도움말 링크

### 2-3. 빈 상태 (Empty State) 템플릿
```
[이유] 아직 {항목}이 없어요.
[유도] {첫 번째 항목}을 만들어 시작해 보세요.
[행동] [+ {항목} 만들기] 버튼
```

### 2-4. 플레이스홀더 작성
- 입력 예시 사용 (`예: hello@example.com`) — 형식 힌트
- 레이블의 반복 금지 (`이름 입력` → 레이블이 이미 `이름`이므로 예시만)
- 보조 힌트는 `<small>` 또는 `helper-text`로 레이블 아래에 표시

## 3. 흔한 실수
- '확인'·'취소'처럼 모호한 버튼 → 무슨 일이 일어나는지 알 수 없다.
- 같은 행동에 '저장'과 '저장하기'를 혼용 → 일관성이 깨진다.
- 기술 오류 코드(500, null)를 그대로 노출 → 사용자가 해결법을 모른다.
- 빈 상태에 '데이터 없음'만 표시 → 다음 행동을 유도하지 못한다.
- 텍스트 하드코딩 → 국제화·문구 변경이 어려워진다.

## 4. 체크리스트
- [ ] 버튼이 동사로 시작하는 구체적 행동 문구인가
- [ ] 동일 행동에 동일 단어를 일관되게 썼는가
- [ ] 오류 메시지에 원인 + 해결 방법이 있고 기술 용어가 없는가
- [ ] 빈 상태에 이유 + 행동 유도 문구가 있는가
- [ ] i18n 대상 텍스트를 번역 키로 관리했는가
