# ✍️ UXライティング & マイクロコピー

> UIのすべてのテキストをユーザー中心で統一する。新しい画面の文言を書くとき、またはボタン・エラー・ローディング・空状態のテキストを決めるときに読む。

## 1. 基本原則
- ボタンのテキストは動詞で始まる行動志向の文言を使う(例:「保存する」「削除」「始める」)。
- エラーメッセージは原因と解決方法を併せて提示し、技術用語(500 Error、null、undefined)を露出しない。
- ローディング状態は「読み込み中...」「保存中...」のように進行中であることを明示し、応答があることを知らせる。
- 空状態(Empty State)には理由 + 行動を促す文言を提供し、「データなし」だけを表示しない。
- 国際化(i18n)対象のテキストはハードコードせず、必ず翻訳キーで管理する。

## 2. ルール

### 2-1. ボタン & ラベル
| ❌ 悪い例 | ✅ 良い例 |
|---------|---------|
| 確認 | 保存する |
| キャンセル | 戻る |
| 削除 | アカウント削除 |
| Error | 保存に失敗しました |

- **具体性**: 何をするボタンなのかを明確に
- **一貫性**: 同じ行動には同じ言葉(保存 vs 保存するの混用禁止)
- **簡潔性**: 最大3〜4語

### 2-2. エラーメッセージのパターン
```
❌ 나쁜 예: "Error 422: Validation failed"
✅ 좋은 예: "이메일 형식이 올바르지 않습니다. 예: example@domain.com"

❌ 나쁜 예: "네트워크 오류"
✅ 좋은 예: "연결에 실패했습니다. 인터넷 연결을 확인하고 다시 시도해 주세요."
```

エラーメッセージの構成:
1. **何が** 間違っているか(原因)
2. **どのように** 解決するか(行動)
3. 必要に応じてヘルプリンク

### 2-3. 空状態(Empty State)テンプレート
```
[이유] 아직 {항목}이 없어요.
[유도] {첫 번째 항목}을 만들어 시작해 보세요.
[행동] [+ {항목} 만들기] 버튼
```

### 2-4. プレースホルダーの記述
- 入力例を使う(`예: hello@example.com`)— 形式のヒント
- ラベルの繰り返し禁止(`名前を入力` → ラベルが既に `名前` なので例示のみ)
- 補助ヒントは `<small>` または `helper-text` でラベルの下に表示

## 3. よくある間違い
- 「確認」「キャンセル」のような曖昧なボタン → 何が起きるか分からない。
- 同じ行動に「保存」と「保存する」を混用 → 一貫性が崩れる。
- 技術的なエラーコード(500、null)をそのまま露出 → ユーザーが解決法を知らない。
- 空状態に「データなし」だけを表示 → 次の行動を促せない。
- テキストのハードコード → 国際化・文言変更が難しくなる。

## 4. チェックリスト
- [ ] ボタンが動詞で始まる具体的な行動文言になっているか
- [ ] 同じ行動に同じ言葉を一貫して使ったか
- [ ] エラーメッセージに原因 + 解決方法があり技術用語がないか
- [ ] 空状態に理由 + 行動を促す文言があるか
- [ ] i18n対象テキストを翻訳キーで管理したか
