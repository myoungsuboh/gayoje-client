# 🗂️ 상태 관리 (State Management)

> 어떤 값을 로컬에 둘지 전역에 둘지 경계를 분명히 하고, 같은 데이터의 출처를 하나로 모으며(단일 출처), 파생 값은 저장하지 않고 계산하고, 변경은 정해진 액션을 통해서만 하며, 영속은 꼭 필요한 범위로만(민감정보 제외) 한다. 전역 상태를 설계하거나 상태를 어디에 둘지·어떻게 구독·영속할지 정할 때 읽는다. 특정 프레임워크/라이브러리에 종속되지 않는 범용 표준이다.

## 1. 목적

클라이언트 상태가 여기저기 복제되고, 같은 값이 두 곳에서 따로 갱신되며, 컴포넌트가 전역 스토어를 직접 휘저으면 디버깅이 불가능해진다. 이 표준은 "무엇을 어디에 두고, 누가 어떻게 바꾸며, 무엇을 남길지"의 공통 기준을 정해 상태 흐름을 예측 가능하게 만든다. 라이브러리 선택(전역 스토어든 컨텍스트든 시그널이든)과 무관하게 적용된다.

## 2. 핵심 원칙

- **로컬 vs 전역 경계**: 한 컴포넌트(또는 그 자식)만 쓰는 값은 로컬에 둔다. 여러 화면·여러 컴포넌트가 공유하고 화면 이동 후에도 살아 있어야 하는 값만 전역으로 올린다. "혹시 몰라서" 전역에 올리지 않는다.
- **단일 출처(single source of truth)**: 같은 데이터는 한곳에서만 소유한다. 같은 값을 여러 스토어/여러 컴포넌트에 복제해두고 각자 갱신하지 않는다 — 복제본은 반드시 서로 어긋난다.
- **파생 상태는 저장하지 말고 계산**: 다른 상태로부터 계산할 수 있는 값(합계·필터 결과·플래그)은 별도로 저장하지 말고 원본에서 파생시킨다. 파생값을 따로 저장하면 원본과 동기화하는 책임이 생기고 곧 깨진다.
- **변경은 액션을 통해서만**: 상태는 정해진 변경 통로(액션/리듀서/메서드)로만 바꾼다. 컴포넌트가 전역 상태 객체를 직접 수정하지 않는다. 변경 지점을 한곳에 모아야 추적·테스트가 가능하다.
- **읽기는 구독으로, 반응성을 유지**: 컴포넌트는 필요한 상태만 구독해서 읽는다. 상태를 꺼내 쓸 때 반응성(변경 시 갱신되는 연결)이 끊기지 않게 한다 — 값을 한 번 복사해 박제하면 이후 변경이 화면에 반영되지 않는다.
- **영속 범위 최소화(민감정보 제외)**: 새로고침 후에도 살아남아야 하는 값(테마, 비휘발성 사용자 설정 등)만 골라 영속화한다. 토큰·비밀번호 같은 민감정보, 그리고 서버에서 다시 받아오면 되는 캐시성 데이터는 영구 저장소에 함부로 남기지 않는다.
- **상태 모듈화**: 전역 상태를 한 덩어리로 두지 말고 도메인 단위(사용자·장바구니·설정 등)로 나눈다. 모듈마다 명확한 책임과 일관된 이름을 갖게 한다.

## 3. 규칙

### 3-1. 로컬에 둘지 전역에 둘지 먼저 정한다

- 한 컴포넌트 안에서만 쓰는 입력값·열림/닫힘·임시 플래그는 로컬에 둔다.
- 여러 화면이 공유하고 라우트 이동 후에도 유지돼야 하는 값만 전역으로 올린다.

```text
// ❌ 금지 — 한 컴포넌트만 쓰는 토글을 전역 스토어에 올림
globalStore.isDropdownOpen = true   // 다른 화면과 무관한데 전역 오염

// ✅ 권장 — 지역값은 로컬, 공유값만 전역
local:  isDropdownOpen          // 이 컴포넌트 안에서만
global: currentUser, cart       // 여러 화면이 공유
```

### 3-2. 같은 데이터는 단일 출처로

- 같은 값을 두 스토어/두 컴포넌트에 각각 들고 따로 갱신하지 않는다.
- 다른 곳이 필요하면 출처를 구독하게 하거나 식별자만 들고 출처에서 조회한다.

```text
// ❌ 금지 — 같은 user를 두 곳이 각자 보관 → 곧 불일치
authStore.user   = {...}
profileStore.user = {...}        // 복제본, 갱신 누락 시 어긋남

// ✅ 권장 — 한 곳이 소유, 나머지는 참조/구독
authStore.user = {...}           // 단일 출처
profileView → reads authStore.user
```

### 3-3. 파생 상태는 저장하지 말고 계산한다

- 다른 상태로부터 도출되는 값은 저장 필드로 두지 말고 파생(계산)으로 표현한다.
- 굳이 저장하면 원본 변경 때마다 수동 동기화가 필요하고, 그 동기화는 빠진다.

```text
// ❌ 금지 — 합계를 따로 저장 → items 바뀔 때마다 직접 갱신해야 함
state: { items, totalCount }     // totalCount가 items와 어긋날 수 있음

// ✅ 권장 — 합계는 items에서 파생
state:   { items }
derived: totalCount = items.length
```

### 3-4. 변경은 정해진 액션을 통해서만

- 전역 상태는 액션/리듀서/메서드 같은 정해진 통로로만 변경한다.
- 컴포넌트가 전역 상태 객체의 필드를 직접 대입해 바꾸지 않는다.

```text
// ❌ 금지 — 컴포넌트가 전역 상태를 직접 수정
component:  store.cart.items.push(item)   // 변경 지점이 사방에 흩어짐

// ✅ 권장 — 액션 한 곳을 통해 변경
store.addToCart(item)                     // 추적·테스트·로깅 가능
```

### 3-5. 읽을 때 반응성을 유지한다

- 컴포넌트는 필요한 상태만 골라 구독한다(과구독은 불필요한 갱신을 부른다).
- 상태를 꺼내 쓸 때 반응 연결이 끊기지 않게 한다 — 그 시점 값을 복사해 박제하면 이후 변경이 반영되지 않는다.

```text
// ❌ 금지 — 시점 값을 복사 → 이후 변경이 화면에 안 옴
const u = store.user          // 박제된 스냅샷

// ✅ 권장 — 반응 연결을 유지한 채 구독
const u = subscribe(store, s => s.user)   // user 변경 시 갱신됨
```

### 3-6. 영속은 꼭 필요한 것만, 민감정보는 제외

- 새로고침 후 유지가 꼭 필요한 값만 영속화 대상으로 명시한다.
- 토큰·비밀번호 등 민감정보, 서버에서 다시 받으면 되는 캐시성 데이터는 영구 저장소에 남기지 않는다.

```text
// ❌ 금지 — 스토어 전체를 통째로 영속화(민감정보 포함)
persist: entire authStore        // accessToken까지 디스크에 남음

// ✅ 권장 — 비민감·유지 필요 항목만 골라 영속화
persist: [theme, locale]         // 토큰은 제외(메모리/보안 저장소)
```

### 3-7. 상태를 도메인 단위로 모듈화한다

- 전역 상태를 하나의 거대한 객체로 두지 말고 도메인별로 나눈다.
- 모듈마다 일관된 이름 규칙을 정해 어디에 무엇이 있는지 예측 가능하게 한다.

```text
// ❌ 금지 — 모든 게 한 덩어리
globalStore = { user, cart, theme, modalOpen, searchText, ... }

// ✅ 권장 — 도메인별 분리 + 일관된 이름
userStore / cartStore / settingsStore
```

## 4. 흔한 실수

- **전부 전역에 올림** → 한 컴포넌트만 쓰는 값까지 전역으로 올려 결합도가 치솟고 추적이 어려워진다. 공유·유지가 필요한 값만 전역으로.
- **같은 값을 여러 곳에 복제** → 단일 출처를 어기고 각자 갱신하다 불일치가 난다. 한 곳이 소유하고 나머지는 참조한다.
- **파생값을 저장 필드로 둠** → 원본과 동기화가 빠져 어긋난다. 파생은 계산으로 표현한다.
- **컴포넌트가 전역 상태를 직접 수정** → 변경 지점이 흩어져 디버깅이 불가능. 액션을 통해서만 바꾼다.
- **반응성 끊고 값 복사** → 구독 없이 시점 값을 박제해 이후 변경이 화면에 반영되지 않는다.
- **스토어 전체 영속화** → 민감정보(토큰 등)와 캐시성 데이터까지 디스크에 남긴다. 유지가 꼭 필요한 비민감 항목만 영속화한다.
- **거대한 단일 스토어** → 모든 상태를 한 덩어리에 넣어 책임이 뒤엉킨다. 도메인 단위로 나눈다.
- **서버 데이터를 전역 상태로 떠안음** → 서버에서 받아오면 되는 데이터를 수동으로 전역에 복제·동기화한다. 서버 상태는 가능하면 데이터 패칭/캐싱 계층(`server-state`)에 맡기고 전역 상태는 진짜 클라이언트 상태에 한정한다.

## 5. 체크리스트

- [ ] 이 값이 **로컬인지 전역인지** 기준(공유·라우트 이동 후 유지)으로 정했는가
- [ ] 같은 데이터를 **단일 출처**로 한 곳만 소유하는가 (복제 후 각자 갱신 금지)
- [ ] 다른 상태로 계산 가능한 값을 **파생(계산)**으로 두었는가 (저장 필드로 중복 금지)
- [ ] 상태 변경을 **정해진 액션/통로**로만 하는가 (컴포넌트의 직접 수정 금지)
- [ ] 컴포넌트가 필요한 상태만 구독하고 **반응성**이 끊기지 않게 읽는가
- [ ] 영속화는 **유지 필요·비민감 항목만** 대상으로 했는가 (토큰 등 민감정보 제외)
- [ ] 전역 상태를 **도메인 단위로 모듈화**하고 일관된 이름을 따랐는가
- [ ] 서버에서 받아오는 데이터를 전역 상태로 수동 복제하지 않았는가

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(예: React/Redux Toolkit·Zustand, Vue/Pinia, Angular/NgRx, Svelte stores 등)에 맞는 예시를 같은 패턴으로 추가한다. 위 1~5의 목적·원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다. (서버 상태 캐싱은 별도 주제이며 `server-state`를 참고한다.)

### Vue 3 (Pinia)

> 본문 2~3의 원칙·규칙이 표준이다. Pinia에서는 다음 매핑으로 적용한다 — 새 코드를 지어내기보다 어떤 API가 어떤 원칙을 구현하는지만 짚는다.

- **도메인 모듈화(본문 3-7)** → `defineStore`로 스토어를 나누고 `use[Name]Store` 명명을 따른다(파일은 `src/store` 등).
- **영속 범위 최소화(본문 3-6)** → `pinia-plugin-persistedstate`로 유지가 필요한 비민감 값만 영속화한다(토큰 제외).
- **반응성 유지(본문 3-5)** → 스토어 값을 구조 분해할 때 `storeToRefs`로 반응 연결이 끊기지 않게 한다.

```javascript
import { useUserStore } from '@/utils/user-store'
import { storeToRefs } from 'pinia'

const userStore = useUserStore()
const { user, sessionId } = storeToRefs(userStore)   // 반응성 유지
```
