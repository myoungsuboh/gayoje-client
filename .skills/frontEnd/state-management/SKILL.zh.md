# 🗂️ 状态管理 (State Management)

> 明确哪个值放在本地、哪个放在全局的边界,将同一数据的来源汇聚为一个(单一来源),派生值不保存而是计算,变更只通过既定的 action 进行,持久化只限定在确有必要的范围内(排除敏感信息)。在设计全局状态,或决定状态放在哪里·如何订阅·如何持久化时阅读。这是不依赖于特定框架/库的通用标准。

## 1. 目的

当客户端状态被到处复制、同一值在两处各自更新、组件直接搅动全局 store 时,调试就变得不可能。本标准通过设定“把什么放在哪里、由谁如何更改、保留什么”的共同基准,使状态流变得可预测。它与库的选择(无论全局 store、context 还是 signals)无关地适用。

## 2. 核心原则

- **本地 vs 全局边界**: 只被一个组件(或其子组件)使用的值放在本地。只有被多个画面·多个组件共享、且画面跳转后仍需存活的值才提升为全局。不要“以防万一”就提升为全局。
- **单一来源(single source of truth)**: 同一数据只在一处拥有。不要把同一值复制到多个 store/多个组件并各自更新 — 副本必然互相不一致。
- **派生状态不要保存而要计算**: 能从其他状态计算出的值(合计·过滤结果·标志)不要单独保存,而要从源头派生。单独保存派生值会产生与源头同步的责任,很快就会被破坏。
- **变更只通过 action**: 状态只通过既定的变更通道(action/reducer/方法)改变。组件不直接修改全局状态对象。把变更点集中到一处才能进行追踪·测试。
- **读取以订阅方式进行,保持响应性**: 组件只订阅并读取所需的状态。取出状态使用时不要切断响应性(变更时随之更新的连接) — 把值复制一次并固化,则之后的变更不会反映到画面上。
- **最小化持久化范围(排除敏感信息)**: 只挑选刷新后仍需存活的值(主题、非易失的用户设置等)进行持久化。令牌·密码等敏感信息,以及从服务器重新获取即可的缓存性数据,不要随意留在永久存储中。
- **状态模块化**: 不要把全局状态作为一整块,而要按领域单位(用户·购物车·设置等)拆分。让每个模块都有明确的职责和一致的命名。

## 3. 规则

### 3-1. 先决定放在本地还是全局

- 只在一个组件内使用的输入值·开/关·临时标志放在本地。
- 只有被多个画面共享、且路由跳转后仍需保持的值才提升为全局。

```text
// ❌ 禁止 — 把只被一个组件使用的开关提升到全局 store
globalStore.isDropdownOpen = true   // 与其他画面无关却污染全局

// ✅ 推荐 — 局部值放本地,只有共享值放全局
local:  isDropdownOpen          // 只在本组件内
global: currentUser, cart       // 多个画面共享
```

### 3-2. 同一数据采用单一来源

- 不要把同一值分别放在两个 store/两个组件并各自更新。
- 其他地方需要时,让它订阅来源,或只持有标识符并从来源查询。

```text
// ❌ 禁止 — 同一 user 由两处各自保管 → 很快不一致
authStore.user   = {...}
profileStore.user = {...}        // 副本,更新遗漏时不一致

// ✅ 推荐 — 一处拥有,其余引用/订阅
authStore.user = {...}           // 单一来源
profileView → reads authStore.user
```

### 3-3. 派生状态不要保存而要计算

- 从其他状态导出的值不要作为保存字段,而以派生(计算)表达。
- 若执意保存,则每次源头变更都需要手动同步,而该同步会被遗漏。

```text
// ❌ 禁止 — 单独保存合计 → 每当 items 变更都必须直接更新
state: { items, totalCount }     // totalCount 可能与 items 不一致

// ✅ 推荐 — 合计从 items 派生
state:   { items }
derived: totalCount = items.length
```

### 3-4. 变更只通过既定的 action

- 全局状态只通过 action/reducer/方法等既定通道变更。
- 组件不要通过直接给全局状态对象的字段赋值来更改。

```text
// ❌ 禁止 — 组件直接修改全局状态
component:  store.cart.items.push(item)   // 变更点散落四方

// ✅ 推荐 — 通过一处 action 变更
store.addToCart(item)                     // 可追踪·测试·记录日志
```

### 3-5. 读取时保持响应性

- 组件只挑选所需的状态进行订阅(过度订阅会引发不必要的更新)。
- 取出状态使用时不要切断响应连接 — 复制并固化那一时刻的值,则之后的变更不会反映。

```text
// ❌ 禁止 — 复制某一时刻的值 → 之后的变更不会到达画面
const u = store.user          // 被固化的快照

// ✅ 推荐 — 在保持响应连接的同时订阅
const u = subscribe(store, s => s.user)   // user 变更时随之更新
```

### 3-6. 持久化只保留确有必要的,排除敏感信息

- 只把刷新后确需保持的值明确指定为持久化对象。
- 令牌·密码等敏感信息,以及从服务器重新获取即可的缓存性数据,不要留在永久存储中。

```text
// ❌ 禁止 — 将整个 store 整体持久化(含敏感信息)
persist: entire authStore        // 连 accessToken 都留在磁盘上

// ✅ 推荐 — 只挑选非敏感·需保持的项目进行持久化
persist: [theme, locale]         // 排除令牌(内存/安全存储)
```

### 3-7. 按领域单位将状态模块化

- 不要把全局状态放在一个巨大的对象里,而要按领域拆分。
- 为每个模块设定一致的命名规则,使“哪里有什么”可预测。

```text
// ❌ 禁止 — 一切都在一整块里
globalStore = { user, cart, theme, modalOpen, searchText, ... }

// ✅ 推荐 — 按领域分离 + 一致命名
userStore / cartStore / settingsStore
```

## 4. 常见错误

- **全部提升到全局** → 连只被一个组件使用的值也提升到全局,耦合度飙升、难以追踪。只有需要共享·保持的值才放全局。
- **将同一值复制到多处** → 违反单一来源并各自更新,产生不一致。一处拥有,其余引用。
- **把派生值作为保存字段** → 与源头的同步被遗漏而不一致。派生以计算表达。
- **组件直接修改全局状态** → 变更点散落,调试不可能。只通过 action 更改。
- **切断响应性并复制值** → 不订阅而固化某一时刻的值,之后的变更不会反映到画面上。
- **持久化整个 store** → 把敏感信息(令牌等)和缓存性数据也留在磁盘上。只持久化确需保持的非敏感项目。
- **巨大的单一 store** → 把所有状态放进一整块,职责纠缠在一起。按领域单位拆分。
- **把服务器数据扛进全局状态** → 把从服务器获取即可的数据手动复制·同步到全局。服务器状态尽可能交给数据获取/缓存层(`server-state`),全局状态仅限定为真正的客户端状态。

## 5. 检查清单

- [ ] 是否以标准(共享·路由跳转后保持)决定了这个值是**本地还是全局**?
- [ ] 同一数据是否以**单一来源**只在一处拥有(禁止复制后各自更新)?
- [ ] 是否把能用其他状态计算的值设为**派生(计算)**(禁止作为保存字段重复)?
- [ ] 状态变更是否只通过**既定的 action/通道**进行(禁止组件直接修改)?
- [ ] 组件是否只订阅所需状态,并在不切断**响应性**的情况下读取?
- [ ] 持久化是否只以**需保持·非敏感的项目**为对象(排除令牌等敏感信息)?
- [ ] 是否将全局状态**按领域单位模块化**并遵循一致命名?
- [ ] 是否没有把从服务器获取的数据手动复制到全局状态?

## 附录: 各技术栈示例

> 以下是供参考的实现示例。请按照相同的模式,补充符合团队所用技术栈(例如 React/Redux Toolkit·Zustand、Vue/Pinia、Angular/NgRx、Svelte stores 等)的示例。上面 1~5 的目的·原则·规则是标准,附录只是其应用案例。(服务器状态缓存是另一个主题,请参考 `server-state`。)

### Vue 3 (Pinia)

> 正文 2~3 的原则·规则是标准。在 Pinia 中以如下映射应用 — 与其凭空写新代码,不如只点明哪个 API 实现哪个原则。

- **领域模块化(正文 3-7)** → 用 `defineStore` 拆分 store 并遵循 `use[Name]Store` 命名(文件放在 `src/store` 等)。
- **最小化持久化范围(正文 3-6)** → 用 `pinia-plugin-persistedstate` 只持久化需保持的非敏感值(排除令牌)。
- **保持响应性(正文 3-5)** → 解构 store 值时用 `storeToRefs` 使响应连接不被切断。

```javascript
import { useUserStore } from '@/utils/user-store'
import { storeToRefs } from 'pinia'

const userStore = useUserStore()
const { user, sessionId } = storeToRefs(userStore)   // 保持响应性
```
