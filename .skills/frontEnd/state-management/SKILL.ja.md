# 🗂️ 状態管理 (State Management)

> どの値をローカルに置くかグローバルに置くか境界を明確にし、同じデータの出所を一つにまとめ(単一の出所)、派生値は保存せず計算し、変更は定められたアクションを通じてのみ行い、永続化は本当に必要な範囲だけに(機密情報を除く)する。グローバル状態を設計するとき、状態をどこに置くか・どう購読・永続化するかを決めるときに読む。特定のフレームワーク/ライブラリに依存しない汎用標準である。

## 1. 目的

クライアント状態があちこちに複製され、同じ値が二箇所で別々に更新され、コンポーネントがグローバルストアを直接かき回すと、デバッグが不可能になる。この標準は「何をどこに置き、誰がどう変え、何を残すか」の共通基準を定め、状態の流れを予測可能にする。ライブラリの選択(グローバルストアでもコンテキストでもシグナルでも)に関わらず適用される。

## 2. 核心原則

- **ローカル vs グローバルの境界**: 一つのコンポーネント(またはその子)だけが使う値はローカルに置く。複数の画面・複数のコンポーネントが共有し、画面遷移後も生きている必要がある値だけをグローバルに上げる。「念のため」でグローバルに上げない。
- **単一の出所(single source of truth)**: 同じデータは一箇所だけで所有する。同じ値を複数のストア/複数のコンポーネントに複製しておいてそれぞれで更新しない — 複製は必ず互いに食い違う。
- **派生状態は保存せず計算する**: 他の状態から計算できる値(合計・フィルタ結果・フラグ)は別途保存せず、元から派生させる。派生値を別に保存すると元と同期する責任が生じ、すぐに壊れる。
- **変更はアクションを通じてのみ**: 状態は定められた変更経路(アクション/リデューサ/メソッド)でのみ変える。コンポーネントがグローバル状態オブジェクトを直接変更しない。変更地点を一箇所に集めてこそ追跡・テストが可能になる。
- **読み取りは購読で、リアクティビティを保つ**: コンポーネントは必要な状態だけを購読して読む。状態を取り出して使うときにリアクティビティ(変更時に更新される結び付き)が切れないようにする — 値を一度コピーして固定すると、以降の変更が画面に反映されない。
- **永続化範囲の最小化(機密情報を除く)**: リロード後も生き残る必要がある値(テーマ、不揮発性のユーザー設定など)だけを選んで永続化する。トークン・パスワードのような機密情報、そしてサーバーから再取得すればよいキャッシュ性データは永続ストレージにみだりに残さない。
- **状態のモジュール化**: グローバル状態を一塊で置かず、ドメイン単位(ユーザー・カート・設定など)で分ける。モジュールごとに明確な責任と一貫した名前を持たせる。

## 3. ルール

### 3-1. ローカルに置くかグローバルに置くかをまず決める

- 一つのコンポーネント内だけで使う入力値・開閉・一時フラグはローカルに置く。
- 複数の画面が共有し、ルート遷移後も維持されるべき値だけをグローバルに上げる。

```text
// ❌ 禁止 — 一つのコンポーネントだけが使うトグルをグローバルストアに上げる
globalStore.isDropdownOpen = true   // 他の画面と無関係なのにグローバルを汚染

// ✅ 推奨 — ローカル値はローカル、共有値だけグローバル
local:  isDropdownOpen          // このコンポーネント内だけ
global: currentUser, cart       // 複数の画面が共有
```

### 3-2. 同じデータは単一の出所で

- 同じ値を二つのストア/二つのコンポーネントにそれぞれ持って別々に更新しない。
- 他の場所で必要なら出所を購読させるか、識別子だけ持って出所から照会する。

```text
// ❌ 禁止 — 同じ user を二箇所がそれぞれ保管 → すぐに不一致
authStore.user   = {...}
profileStore.user = {...}        // 複製、更新漏れ時に食い違う

// ✅ 推奨 — 一箇所が所有、残りは参照/購読
authStore.user = {...}           // 単一の出所
profileView → reads authStore.user
```

### 3-3. 派生状態は保存せず計算する

- 他の状態から導かれる値は保存フィールドにせず、派生(計算)で表現する。
- あえて保存すると元の変更のたびに手動同期が必要になり、その同期は漏れる。

```text
// ❌ 禁止 — 合計を別途保存 → items が変わるたびに直接更新しなければならない
state: { items, totalCount }     // totalCount が items と食い違いうる

// ✅ 推奨 — 合計は items から派生
state:   { items }
derived: totalCount = items.length
```

### 3-4. 変更は定められたアクションを通じてのみ

- グローバル状態はアクション/リデューサ/メソッドのような定められた経路でのみ変更する。
- コンポーネントがグローバル状態オブジェクトのフィールドを直接代入して変えない。

```text
// ❌ 禁止 — コンポーネントがグローバル状態を直接変更
component:  store.cart.items.push(item)   // 変更地点が四方に散る

// ✅ 推奨 — アクション一箇所を通じて変更
store.addToCart(item)                     // 追跡・テスト・ロギング可能
```

### 3-5. 読み取り時にリアクティビティを保つ

- コンポーネントは必要な状態だけを選んで購読する(過剰購読は不要な更新を招く)。
- 状態を取り出して使うときにリアクティブな結び付きが切れないようにする — その時点の値をコピーして固定すると以降の変更が反映されない。

```text
// ❌ 禁止 — 時点の値をコピー → 以降の変更が画面に来ない
const u = store.user          // 固定されたスナップショット

// ✅ 推奨 — リアクティブな結び付きを保ったまま購読
const u = subscribe(store, s => s.user)   // user 変更時に更新される
```

### 3-6. 永続化は本当に必要なものだけ、機密情報は除く

- リロード後の維持が本当に必要な値だけを永続化対象として明示する。
- トークン・パスワードなど機密情報、サーバーから再取得すればよいキャッシュ性データは永続ストレージに残さない。

```text
// ❌ 禁止 — ストア全体を丸ごと永続化(機密情報を含む)
persist: entire authStore        // accessToken までディスクに残る

// ✅ 推奨 — 非機密・維持が必要な項目だけを選んで永続化
persist: [theme, locale]         // トークンは除く(メモリ/セキュアストレージ)
```

### 3-7. 状態をドメイン単位でモジュール化する

- グローバル状態を一つの巨大なオブジェクトに置かず、ドメイン別に分ける。
- モジュールごとに一貫した命名規則を定め、どこに何があるかを予測可能にする。

```text
// ❌ 禁止 — すべてが一塊
globalStore = { user, cart, theme, modalOpen, searchText, ... }

// ✅ 推奨 — ドメイン別に分離 + 一貫した名前
userStore / cartStore / settingsStore
```

## 4. よくある間違い

- **全部グローバルに上げる** → 一つのコンポーネントだけが使う値までグローバルに上げ、結合度が跳ね上がり追跡が難しくなる。共有・維持が必要な値だけグローバルに。
- **同じ値を複数箇所に複製** → 単一の出所に反してそれぞれ更新し不一致が起きる。一箇所が所有し残りは参照する。
- **派生値を保存フィールドにする** → 元との同期が漏れて食い違う。派生は計算で表現する。
- **コンポーネントがグローバル状態を直接変更** → 変更地点が散らばってデバッグが不可能。アクションを通じてのみ変える。
- **リアクティビティを切って値をコピー** → 購読なしに時点の値を固定し、以降の変更が画面に反映されない。
- **ストア全体を永続化** → 機密情報(トークンなど)とキャッシュ性データまでディスクに残す。維持が本当に必要な非機密項目だけを永続化する。
- **巨大な単一ストア** → すべての状態を一塊に入れて責任が絡み合う。ドメイン単位で分ける。
- **サーバーデータをグローバル状態に抱え込む** → サーバーから取得すればよいデータを手動でグローバルに複製・同期する。サーバー状態はできればデータフェッチ/キャッシュ層(`server-state`)に任せ、グローバル状態は本当のクライアント状態に限定する。

## 5. チェックリスト

- [ ] この値が**ローカルかグローバルか**を基準(共有・ルート遷移後の維持)で決めたか
- [ ] 同じデータを**単一の出所**として一箇所だけで所有しているか(複製後それぞれ更新は禁止)
- [ ] 他の状態で計算可能な値を**派生(計算)**にしたか(保存フィールドとして重複は禁止)
- [ ] 状態変更を**定められたアクション/経路**でのみ行うか(コンポーネントの直接変更は禁止)
- [ ] コンポーネントが必要な状態だけを購読し、**リアクティビティ**が切れないように読むか
- [ ] 永続化は**維持が必要・非機密の項目だけ**を対象にしたか(トークンなど機密情報を除く)
- [ ] グローバル状態を**ドメイン単位でモジュール化**し一貫した名前に従ったか
- [ ] サーバーから取得するデータをグローバル状態に手動複製していないか

## 付録: スタック別の例

> 以下は参考用の実装例である。チームが使うスタック(例: React/Redux Toolkit・Zustand、Vue/Pinia、Angular/NgRx、Svelte stores など)に合った例を同じパターンで追加する。上の 1~5 の目的・原則・ルールが標準であり、付録はその適用事例にすぎない。(サーバー状態のキャッシュは別の主題であり `server-state` を参照する。)

### Vue 3 (Pinia)

> 本文 2~3 の原則・ルールが標準である。Pinia では次のマッピングで適用する — 新しいコードを作り出すより、どの API がどの原則を実装するかだけを押さえる。

- **ドメインモジュール化(本文 3-7)** → `defineStore` でストアを分け、`use[Name]Store` 命名に従う(ファイルは `src/store` など)。
- **永続化範囲の最小化(本文 3-6)** → `pinia-plugin-persistedstate` で維持が必要な非機密値だけを永続化する(トークンを除く)。
- **リアクティビティの維持(本文 3-5)** → ストア値を分割代入するときに `storeToRefs` でリアクティブな結び付きが切れないようにする。

```javascript
import { useUserStore } from '@/utils/user-store'
import { storeToRefs } from 'pinia'

const userStore = useUserStore()
const { user, sessionId } = storeToRefs(userStore)   // リアクティビティ維持
```
