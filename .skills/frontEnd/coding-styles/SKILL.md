# 🎨 코딩 스타일 표준 (Coding Styles)

> 누가 작성해도 코드가 같은 모양으로 읽히도록 명명·포맷·주석·커밋 메시지의 보편 규칙을 정한다. 새 코드를 작성하거나 스타일을 정할 때 읽는다. 특정 언어/프레임워크에 종속되지 않는 범용 표준이며, 브랜치 전략·PR·머지 같은 Git 협업 흐름 자체는 `git-workflow` 스킬을 본다.

## 1. 핵심 원칙
- **일관성이 곧 가독성**: 개인 취향보다 팀이 합의한 한 가지 방식을 따른다. "맞는 스타일"보다 "같은 스타일"이 중요하다.
- **명명은 의도를 드러낸다**: 이름만 보고 무엇인지·어떤 종류인지 알 수 있게 한다. 대상 종류(변수·함수·타입·상수·파일)별로 정해진 케이스를 일관되게 적용한다.
- **포맷은 도구로 강제한다**: 들여쓰기·세미콜론·따옴표 같은 포맷은 사람이 눈으로 맞추지 말고 포매터/린터로 자동 정렬한다 — 리뷰에서 포맷을 논쟁하지 않는다.
- **주석은 '왜'를 적는다**: 코드가 말해주는 '무엇'을 반복하지 말고, 코드로 드러나지 않는 의도·배경·트레이드오프를 적는다.
- **커밋 메시지는 규약을 따른다**: 메시지 첫 줄을 정해진 타입 규약(예: `feat`/`fix`/...)으로 시작해, 이력만 훑어도 변경 성격이 보이게 한다.
- **스택 종속 규칙은 분리한다**: 언어/프레임워크 전용 관례(파일 블록 순서, 컴포넌트 작성 방식 등)는 이 보편 규칙과 섞지 말고, 해당 스택 스킬 또는 아래 부록으로 분리한다.

## 2. 규칙

### 2-1. 명명 케이스 (Naming)
- 대상 종류별로 표준 케이스를 일관되게 쓴다. 한 프로젝트 안에서 같은 종류는 항상 같은 케이스로.
  - **변수·함수**: `camelCase` (예: `userEmail`, `getUserInfo`)
  - **타입·클래스·컴포넌트**: `PascalCase` (예: `UserLogin`, `OrderService`)
  - **상수(불변 전역값)**: `SCREAMING_SNAKE_CASE` (예: `MAX_RETRIES`)
  - **파일·디렉터리**: 기본 `kebab-case` (예: `user-profile`). 스택 관례가 다른 경우(예: 타입/컴포넌트 파일을 `PascalCase`로)는 부록의 스택 규칙을 따른다.
- 의미 없는 약어·한 글자 이름(루프 변수 제외)·타입을 이름에 박는 헝가리안 표기는 피한다.

```text
// ❌ 금지 — 종류와 케이스가 뒤섞임
const UserEmail = ...      // 변수인데 PascalCase
function GetUser() {}      // 함수인데 PascalCase
const maxRetries = 3       // 불변 상수인데 camelCase

// ✅ 권장 — 종류별 표준 케이스
const userEmail = ...
function getUser() {}
const MAX_RETRIES = 3
class OrderService {}
```

### 2-2. 포맷팅 (도구로 자동화)
- 들여쓰기·세미콜론·따옴표·최대 줄길이 같은 포맷 규칙은 팀이 한 번 정하고, 포매터/린터 설정 파일로 저장소에 커밋해 자동 적용한다.
- 포맷 차이만 있는 변경(공백·줄바꿈)으로 진짜 변경이 묻히지 않게 한다 — 포맷은 도구에 맡기고 리뷰에서 다투지 않는다.

```text
// ❌ 금지 — 한 저장소 안에서 포맷이 제각각
let a = "x"
let b = 'y' ;

// ✅ 권장 — 합의된 규칙(예: 세미콜론 사용·홑따옴표·일정 들여쓰기)을 도구로 강제
const a = 'x';
const b = 'y';
```

### 2-3. 주석 ('왜'를 적고, 부채는 추적 가능하게)
- 코드가 이미 말하는 '무엇'을 되풀이하지 말고, 왜 그렇게 했는지(배경·제약·의도)를 적는다.
- 임시 처리·미완은 검색 가능한 형식으로 남긴다: `// TODO(담당자, 날짜): 내용`. 누가·언제·무엇을 알 수 있게 해 방치된 부채를 추적한다.

```text
// ❌ 금지 — 코드가 말하는 걸 그대로 반복 / 추적 불가능한 TODO
i = i + 1   // i에 1을 더한다
// TODO: 나중에 고치기

// ✅ 권장 — '왜' + 추적 가능한 부채
// 외부 API가 0-based 페이지를 쓰므로 +1 보정
page = page + 1
// TODO(김개발, 2026-06-17): 페이지 보정은 API v2에서 1-based로 바뀌면 제거
```

### 2-4. 커밋 메시지 규약
- 커밋 메시지 첫 줄을 정해진 타입으로 시작하는 규약을 따른다(예: Conventional Commits / Angular 규칙). 이력만 훑어도 변경 성격이 분류된다.
  - 대표 타입: `feat`(기능), `fix`(버그), `docs`(문서), `style`(포맷), `refactor`(리팩토링), `test`(테스트), `chore`(빌드/설정).
- 한 커밋은 한 가지 논리적 변경만 담고, 제목은 명령형으로 간결하게 쓴다.
- 브랜치 전략·PR 크기·머지/리베이스·충돌 해결 등 커밋 메시지 밖의 협업 흐름은 `git-workflow` 스킬을 따른다.

```text
// ❌ 금지 — 타입 없는 모호한 메시지
update
fix bug

// ✅ 권장 — 타입(scope) + 명령형 요약
feat(order): 주문 취소 사유 입력 필드 추가
fix(auth): 토큰 만료 시 무한 리다이렉트 수정
```

## 3. 흔한 실수
- **종류와 케이스 혼용** → 변수를 `PascalCase`로, 상수를 `camelCase`로 쓰는 식으로 섞여 의도가 흐려진다. 종류별 표준 케이스로 통일한다.
- **포맷을 사람이 눈으로 맞춤** → 리뷰가 공백·줄바꿈 논쟁으로 샌다. 포매터/린터 설정을 커밋해 자동화한다.
- **'무엇'을 설명하는 주석** → 코드를 그대로 옮긴 주석은 노이즈다. '왜'만 남긴다.
- **추적 불가능한 TODO** → `// TODO 나중에`는 영원히 안 고쳐진다. `// TODO(담당자, 날짜): 내용`으로.
- **타입 없는 커밋 메시지** → `update`, `fix bug` 식이면 이력에서 변경 성격을 못 가린다. 타입 규약을 따른다.
- **보편 규칙에 스택 전용 규칙을 섞음** → 특정 FW의 파일 블록 순서·컴포넌트 작성법을 공통 규칙에 박으면 다른 스택에서 못 쓴다. 부록/전용 스킬로 분리한다.

## 4. 체크리스트
- [ ] 변수·함수는 `camelCase`, 타입·클래스·컴포넌트는 `PascalCase`, 상수는 `SCREAMING_SNAKE_CASE`로 작성했는가
- [ ] 파일·디렉터리 명명이 팀 규칙(기본 `kebab-case`, 스택별 예외는 부록)을 따르는가
- [ ] 들여쓰기·세미콜론·따옴표 등 포맷을 포매터/린터 설정으로 자동 강제하는가
- [ ] 주석이 '무엇'이 아니라 '왜'를 설명하는가
- [ ] 임시·미완을 `// TODO(담당자, 날짜): 내용` 형식으로 추적 가능하게 남겼는가
- [ ] 커밋 메시지가 정해진 타입 규약(feat/fix/docs 등)을 따르는가
- [ ] 스택 전용 관례(블록 순서·컴포넌트 작성법 등)는 보편 규칙과 섞지 않고 부록/전용 스킬로 분리했는가

## 부록: 스택별 예시

> 아래는 특정 스택에서 위 보편 규칙을 적용한 예시다. 팀이 쓰는 스택에 맞게 같은 패턴으로 추가한다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다.

### Vue 3 (SFC)

> 본문 1~4의 보편 규칙(명명 케이스·포맷·주석·TODO)을 Vue 3 SFC에 적용할 때의 **스택 전용 관례만** 적는다. 본문과 겹치는 명명·포맷·주석 규칙은 본문을 따른다. `.vue` 블록 순서·script 내부 정렬의 상세 표준은 `vue-sfc-structure` 스킬을 함께 본다.

#### 파일명 (스택 예외)
- Vue 컴포넌트 파일은 본문 기본값(`kebab-case`) 대신 `PascalCase`를 권장한다(예: `UserLogin.vue`). 그 외 명명 케이스는 본문 2-1을 따른다.

#### 컴포넌트 작성 원칙
- **Composition API 우선**: Options API보다 `<script setup>` 기반의 훅 중심 작성을 권장한다.
- **파일 구조 순서**: `.vue` 파일 내 블록 순서를 일관되게 지킨다 — `<script setup>` → `<template>` → `<style>`.

#### 포맷 (스택 구체값)
- 본문 2-2의 "포매터/린터로 강제" 원칙을 따르며, Vue 프로젝트의 구체값은 2 spaces 들여쓰기·세미콜론 사용·홑따옴표(`'`)다.

> 매직 넘버 금지·디자인 토큰 등 값 상수화 규칙은 `design-system`을 본다.
