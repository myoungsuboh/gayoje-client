# 🎨 编码风格标准 (Coding Styles)

> 制定命名·格式·注释·提交信息的通用规则，使无论谁编写代码都读起来是相同的样子。在编写新代码或制定风格时阅读。这是不依赖特定语言/框架的通用标准，而分支策略·PR·合并这类 Git 协作流程本身参见 `git-workflow` 技能。

## 1. 核心原则
- **一致性即可读性**: 比起个人喜好，遵循团队达成共识的一种方式。"正确的风格"不如"相同的风格"重要。
- **命名揭示意图**: 让人仅看名字就能知道是什么·属于哪种。对每种对象(变量·函数·类型·常量·文件)一致地应用既定的大小写。
- **格式用工具强制**: 缩进·分号·引号等格式不要靠人眼对齐，用格式化工具/检查器自动对齐 — 不在评审中争论格式。
- **注释写'为什么'**: 不要重复代码已说明的'是什么'，写出代码无法体现的意图·背景·权衡。
- **提交信息遵循约定**: 让信息首行以既定的类型约定(如 `feat`/`fix`/...)开头，仅浏览历史就能看出变更性质。
- **栈相关规则要分离**: 语言/框架专用惯例(文件块顺序、组件写法等)不要与这些通用规则混在一起，分离到相应的栈技能或下方附录中。

## 2. 规则

### 2-1. 命名大小写 (Naming)
- 对每种对象一致地使用标准大小写。在同一项目中同种对象始终使用相同大小写。
  - **变量·函数**: `camelCase` (如: `userEmail`, `getUserInfo`)
  - **类型·类·组件**: `PascalCase` (如: `UserLogin`, `OrderService`)
  - **常量(不可变的全局值)**: `SCREAMING_SNAKE_CASE` (如: `MAX_RETRIES`)
  - **文件·目录**: 默认 `kebab-case` (如: `user-profile`)。栈惯例不同的情况(如将类型/组件文件用 `PascalCase`)遵循附录的栈规则。
- 避免无意义的缩写·单字母名称(循环变量除外)·将类型嵌入名字的匈牙利命名法。

```text
// ❌ 禁止 — 种类与大小写混在一起
const UserEmail = ...      // 是变量却用 PascalCase
function GetUser() {}      // 是函数却用 PascalCase
const maxRetries = 3       // 是不可变常量却用 camelCase

// ✅ 推荐 — 每种对象的标准大小写
const userEmail = ...
function getUser() {}
const MAX_RETRIES = 3
class OrderService {}
```

### 2-2. 格式化 (用工具自动化)
- 缩进·分号·引号·最大行长这类格式规则由团队定一次，作为格式化工具/检查器的配置文件提交到仓库以自动应用。
- 不要让仅有格式差异的变更(空白·换行)淹没真正的变更 — 格式交给工具，不在评审中争论。

```text
// ❌ 禁止 — 在同一仓库内格式各不相同
let a = "x"
let b = 'y' ;

// ✅ 推荐 — 用工具强制达成共识的规则(如: 使用分号·单引号·固定缩进)
const a = 'x';
const b = 'y';
```

### 2-3. 注释 (写'为什么'，让债务可追踪)
- 不要重复代码已经说明的'是什么'，写出为什么这样做(背景·约束·意图)。
- 临时处理·未完成以可搜索的形式留下: `// TODO(负责人, 日期): 内容`。让谁·何时·什么可知，以追踪被搁置的债务。

```text
// ❌ 禁止 — 原样重复代码所说的 / 无法追踪的 TODO
i = i + 1   // 给 i 加 1
// TODO: 以后再修

// ✅ 推荐 — '为什么' + 可追踪的债务
// 外部 API 使用 0-based 页码，所以 +1 修正
page = page + 1
// TODO(金开发, 2026-06-17): 当 API v2 改为 1-based 时移除页码修正
```

### 2-4. 提交信息约定
- 遵循让提交信息首行以既定类型开头的约定(如 Conventional Commits / Angular 规则)。仅浏览历史就能将变更性质分类。
  - 代表性类型: `feat`(功能), `fix`(缺陷), `docs`(文档), `style`(格式), `refactor`(重构), `test`(测试), `chore`(构建/配置)。
- 一次提交只包含一个逻辑变更，标题用祈使句简洁书写。
- 分支策略·PR 大小·合并/变基·冲突解决等提交信息之外的协作流程遵循 `git-workflow` 技能。

```text
// ❌ 禁止 — 没有类型的模糊信息
update
fix bug

// ✅ 推荐 — 类型(scope) + 祈使句摘要
feat(order): 添加订单取消理由输入字段
fix(auth): 修复令牌过期时的无限重定向
```

## 3. 常见错误
- **种类与大小写混用** → 像把变量写成 `PascalCase`、把常量写成 `camelCase` 那样混在一起，使意图变模糊。用每种对象的标准大小写统一。
- **靠人眼对齐格式** → 评审滑向空白·换行的争论。提交格式化工具/检查器配置以自动化。
- **解释'是什么'的注释** → 原样照搬代码的注释是噪音。只留'为什么'。
- **无法追踪的 TODO** → `// TODO 以后` 永远不会被修。用 `// TODO(负责人, 日期): 内容`。
- **没有类型的提交信息** → 像 `update`、`fix bug` 那样就无法从历史辨别变更性质。遵循类型约定。
- **将栈专用规则混入通用规则** → 把特定 FW 的文件块顺序·组件写法嵌入公共规则，在其他栈就无法使用。分离到附录/专用技能。

## 4. 检查清单
- [ ] 变量·函数是否写成 `camelCase`，类型·类·组件是否写成 `PascalCase`，常量是否写成 `SCREAMING_SNAKE_CASE`？
- [ ] 文件·目录命名是否遵循团队规则(默认 `kebab-case`，栈别例外在附录)？
- [ ] 缩进·分号·引号等格式是否用格式化工具/检查器配置自动强制？
- [ ] 注释是否解释'为什么'而非'是什么'？
- [ ] 是否将临时·未完成以 `// TODO(负责人, 日期): 内容` 形式留为可追踪？
- [ ] 提交信息是否遵循既定的类型约定(feat/fix/docs 等)？
- [ ] 栈专用惯例(块顺序·组件写法等)是否未与通用规则混合而分离到附录/专用技能？

## 附录: 各栈示例

> 以下是在特定栈中应用上述通用规则的示例。按相同模式添加以匹配团队使用的栈。上述1~4的原则·规则是标准，附录只是其应用案例。

### Vue 3 (SFC)

> 仅写出将正文1~4的通用规则(命名大小写·格式·注释·TODO)应用于 Vue 3 SFC 时的**栈专用惯例**。与正文重叠的命名·格式·注释规则遵循正文。`.vue` 块顺序·script 内部排列的详细标准请一并参见 `vue-sfc-structure` 技能。

#### 文件名 (栈例外)
- Vue 组件文件推荐用 `PascalCase`(如 `UserLogin.vue`)而非正文默认值(`kebab-case`)。其他命名大小写遵循正文 2-1。

#### 组件编写原则
- **Composition API 优先**: 比起 Options API，推荐基于 `<script setup>` 的以钩子为中心的编写。
- **文件结构顺序**: 一致地遵守 `.vue` 文件内的块顺序 — `<script setup>` → `<template>` → `<style>`。

#### 格式 (栈具体值)
- 遵循正文 2-2 的"用格式化工具/检查器强制"原则，Vue 项目的具体值是 2 spaces 缩进·使用分号·单引号(`'`)。

> 禁止魔法数字·设计令牌等值常量化规则请参见 `design-system`。
