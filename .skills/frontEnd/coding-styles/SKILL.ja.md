# 🎨 コーディングスタイル標準 (Coding Styles)

> 誰が書いてもコードが同じ形に読めるよう、命名・フォーマット・コメント・コミットメッセージの普遍ルールを定める。新しいコードを書くときやスタイルを定めるときに読む。特定の言語/フレームワークに依存しない汎用標準であり、ブランチ戦略・PR・マージのような Git 協業フロー自体は `git-workflow` スキルを見る。

## 1. 核心原則
- **一貫性こそ可読性**: 個人の好みよりチームが合意した一つのやり方に従う。「正しいスタイル」より「同じスタイル」が重要だ。
- **命名は意図を表す**: 名前を見ただけで何か・どの種類かが分かるようにする。対象の種類(変数・関数・型・定数・ファイル)ごとに定められたケースを一貫して適用する。
- **フォーマットはツールで強制する**: インデント・セミコロン・引用符などのフォーマットは人が目で揃えず、フォーマッター/リンターで自動整列する — レビューでフォーマットを論争しない。
- **コメントは'なぜ'を書く**: コードが語る'何を'を繰り返さず、コードでは見えない意図・背景・トレードオフを書く。
- **コミットメッセージは規約に従う**: メッセージの1行目を定められたタイプ規約(例: `feat`/`fix`/...)で始め、履歴をざっと見るだけで変更の性格が見えるようにする。
- **スタック依存ルールは分離する**: 言語/フレームワーク専用の慣習(ファイルのブロック順序、コンポーネントの書き方など)はこの普遍ルールと混ぜず、当該スタックスキルまたは下記の付録に分離する。

## 2. ルール

### 2-1. 命名ケース (Naming)
- 対象の種類ごとに標準ケースを一貫して使う。一つのプロジェクト内で同じ種類は常に同じケースで。
  - **変数・関数**: `camelCase` (例: `userEmail`, `getUserInfo`)
  - **型・クラス・コンポーネント**: `PascalCase` (例: `UserLogin`, `OrderService`)
  - **定数(不変のグローバル値)**: `SCREAMING_SNAKE_CASE` (例: `MAX_RETRIES`)
  - **ファイル・ディレクトリ**: 基本 `kebab-case` (例: `user-profile`)。スタック慣習が異なる場合(例: 型/コンポーネントファイルを `PascalCase` に)は付録のスタックルールに従う。
- 意味のない略語・一文字の名前(ループ変数を除く)・型を名前に埋め込むハンガリアン記法は避ける。

```text
// ❌ 禁止 — 種類とケースが入り混じる
const UserEmail = ...      // 変数なのに PascalCase
function GetUser() {}      // 関数なのに PascalCase
const maxRetries = 3       // 不変の定数なのに camelCase

// ✅ 推奨 — 種類ごとの標準ケース
const userEmail = ...
function getUser() {}
const MAX_RETRIES = 3
class OrderService {}
```

### 2-2. フォーマット (ツールで自動化)
- インデント・セミコロン・引用符・最大行長のようなフォーマットルールはチームが一度定め、フォーマッター/リンターの設定ファイルとしてリポジトリにコミットして自動適用する。
- フォーマットの差だけの変更(空白・改行)で本当の変更が埋もれないようにする — フォーマットはツールに任せ、レビューで争わない。

```text
// ❌ 禁止 — 一つのリポジトリ内でフォーマットがバラバラ
let a = "x"
let b = 'y' ;

// ✅ 推奨 — 合意したルール(例: セミコロン使用・シングルクォート・一定のインデント)をツールで強制
const a = 'x';
const b = 'y';
```

### 2-3. コメント ('なぜ'を書き、負債は追跡可能に)
- コードが既に語る'何を'を繰り返さず、なぜそうしたか(背景・制約・意図)を書く。
- 一時処理・未完は検索可能な形式で残す: `// TODO(担当者, 日付): 内容`。誰が・いつ・何をが分かるようにして放置された負債を追跡する。

```text
// ❌ 禁止 — コードが語ることをそのまま繰り返す / 追跡不能な TODO
i = i + 1   // i に 1 を足す
// TODO: 後で直す

// ✅ 推奨 — 'なぜ' + 追跡可能な負債
// 外部 API が 0-based ページを使うので +1 補正
page = page + 1
// TODO(キム開発, 2026-06-17): ページ補正は API v2 で 1-based に変わったら削除
```

### 2-4. コミットメッセージ規約
- コミットメッセージの1行目を定められたタイプで始める規約に従う(例: Conventional Commits / Angular ルール)。履歴をざっと見るだけで変更の性格が分類される。
  - 代表的なタイプ: `feat`(機能), `fix`(バグ), `docs`(ドキュメント), `style`(フォーマット), `refactor`(リファクタリング), `test`(テスト), `chore`(ビルド/設定)。
- 1コミットは一つの論理的変更だけを含み、タイトルは命令形で簡潔に書く。
- ブランチ戦略・PR サイズ・マージ/リベース・コンフリクト解決などコミットメッセージ外の協業フローは `git-workflow` スキルに従う。

```text
// ❌ 禁止 — タイプのない曖昧なメッセージ
update
fix bug

// ✅ 推奨 — タイプ(scope) + 命令形の要約
feat(order): 注文キャンセル理由の入力フィールドを追加
fix(auth): トークン失効時の無限リダイレクトを修正
```

## 3. よくあるミス
- **種類とケースの混用** → 変数を `PascalCase` で、定数を `camelCase` で書くように混ざって意図が曖昧になる。種類ごとの標準ケースで統一する。
- **フォーマットを人が目で揃える** → レビューが空白・改行の論争に流れる。フォーマッター/リンター設定をコミットして自動化する。
- **'何を'を説明するコメント** → コードをそのまま写したコメントはノイズだ。'なぜ'だけ残す。
- **追跡不能な TODO** → `// TODO 後で` は永遠に直らない。`// TODO(担当者, 日付): 内容` で。
- **タイプのないコミットメッセージ** → `update`, `fix bug` 式だと履歴で変更の性格を見分けられない。タイプ規約に従う。
- **普遍ルールにスタック専用ルールを混ぜる** → 特定 FW のファイルブロック順序・コンポーネントの書き方を共通ルールに埋め込むと他のスタックで使えない。付録/専用スキルに分離する。

## 4. チェックリスト
- [ ] 変数・関数は `camelCase`、型・クラス・コンポーネントは `PascalCase`、定数は `SCREAMING_SNAKE_CASE` で書いたか
- [ ] ファイル・ディレクトリの命名がチームルール(基本 `kebab-case`、スタック別の例外は付録)に従っているか
- [ ] インデント・セミコロン・引用符などのフォーマットをフォーマッター/リンター設定で自動強制しているか
- [ ] コメントが'何を'ではなく'なぜ'を説明しているか
- [ ] 一時・未完を `// TODO(担当者, 日付): 内容` 形式で追跡可能に残したか
- [ ] コミットメッセージが定められたタイプ規約(feat/fix/docs など)に従っているか
- [ ] スタック専用の慣習(ブロック順序・コンポーネントの書き方など)は普遍ルールと混ぜず付録/専用スキルに分離したか

## 付録: スタック別の例

> 以下は特定のスタックで上記の普遍ルールを適用した例だ。チームが使うスタックに合わせて同じパターンで追加する。上記1〜4の原則・ルールが標準であり、付録はその適用事例にすぎない。

### Vue 3 (SFC)

> 本文1〜4の普遍ルール(命名ケース・フォーマット・コメント・TODO)を Vue 3 SFC に適用するときの**スタック専用の慣習だけ**を書く。本文と重なる命名・フォーマット・コメントルールは本文に従う。`.vue` ブロック順序・script 内部の整列の詳細標準は `vue-sfc-structure` スキルも併せて見る。

#### ファイル名 (スタック例外)
- Vue コンポーネントファイルは本文の既定値(`kebab-case`)の代わりに `PascalCase` を推奨する(例: `UserLogin.vue`)。それ以外の命名ケースは本文 2-1 に従う。

#### コンポーネント作成原則
- **Composition API 優先**: Options API より `<script setup>` ベースのフック中心の作成を推奨する。
- **ファイル構造順序**: `.vue` ファイル内のブロック順序を一貫して守る — `<script setup>` → `<template>` → `<style>`。

#### フォーマット (スタック具体値)
- 本文 2-2 の「フォーマッター/リンターで強制」原則に従い、Vue プロジェクトの具体値は 2 spaces インデント・セミコロン使用・シングルクォート(`'`)だ。

> マジックナンバー禁止・デザイントークンなど値の定数化ルールは `design-system` を見る。
