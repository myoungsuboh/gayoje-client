# 🚩 피처 플래그 (Feature Flag)

> 코드 병합(배포)과 기능 공개(릴리스)를 분리한다. 새 기능을 점진 공개하거나 A/B 테스트·kill switch가 필요할 때, 플래그 수명·정리·관리 방식을 정할 때 읽는다. 특정 언어/프레임워크에 종속되지 않는 범용 표준이다.

## 1. 핵심 원칙
- **배포와 릴리스를 분리한다**: 새 기능은 플래그로 감싸 배포한다. 코드 병합이 곧 기능 공개가 아니다 — 언제 누구에게 켤지는 플래그가 결정한다.
- **플래그마다 수명을 정의한다**: 임시(릴리스)·실험·영구(운영) 중 무엇인지 정하고, 역할을 다한 임시 플래그는 죽은 분기로 남기지 말고 즉시 제거한다. 만들 때 제거 조건/티켓을 함께 정한다.
- **상태는 코드 밖에서 관리한다**: 플래그 on/off 값은 빌드 산출물에 박지 말고, 환경 변수나 원격 설정 서비스 같은 외부 소스에서 읽어 재배포 없이 바꿀 수 있게 한다.
- **진입점에 집중한다**: 분기는 라우트·컨테이너 같은 진입점에서 한 번만 하고, 화면/모듈 깊숙이 같은 분기를 반복하지 않는다 — 분산될수록 제거가 어려워진다.
- **on/off 양쪽을 모두 테스트한다**: 켠 경로와 끈(롤백) 경로를 모두 검증한다. 끈 경로를 안 보면 kill switch가 정작 필요할 때 깨진다.
- **이름·목적을 문서화한다**: 무슨 플래그가 왜 있는지, 누가 소유하고 언제 제거하는지 추적 가능하게 남긴다.

## 2. 규칙

### 2-1. 새 기능은 플래그로 감싸 배포/릴리스를 분리한다
- 미완성·점진 공개가 필요한 기능은 플래그 뒤에 두고 병합한다. 병합과 공개 시점을 분리해 안전하게 출시·롤백한다.

```text
// ❌ 금지 — 병합 = 즉시 전체 공개 (롤백하려면 되돌려 재배포해야 함)
merge(newCheckout)  → 모든 사용자에게 바로 노출

// ✅ 권장 — 플래그 뒤에 두고 병합, 공개 시점은 플래그로 제어
merge(newCheckout behind flag "new-checkout")  // 기본 off
flag "new-checkout" on → 점진 공개 / off → 즉시 롤백
```

### 2-2. 플래그 유형과 수명을 정한다
- 만들 때 유형(수명)을 분류하고, 임시 플래그에는 제거 조건/티켓을 붙인다.

| 유형 | 수명 | 예시 |
|---|---|---|
| 릴리스 토글 | 단기(출시 후 제거) | 새 결제 플로우 |
| 실험 토글 | 중기(A/B 종료 후 제거) | 버튼 문구 실험 |
| 운영 토글 | 영구 | Kill switch |

### 2-3. 상태는 코드 밖(환경/원격)에서 읽는다
- 플래그 값을 소스에 하드코딩하지 말고, 환경 변수(단순)나 원격 설정 서비스(점진 공개·타겟팅·실시간 변경)에서 읽는다.
- 플래그 조회는 한 곳(설정 모듈/래퍼)으로 모아, 소비하는 쪽이 출처를 몰라도 되게 한다.

```text
// ❌ 금지 — 값이 코드에 박혀 재배포 없이는 못 바꿈
const newCheckout = true

// ✅ 권장 — 외부 소스에서 읽어 한 곳으로 모음
flags.newCheckout = readFlag("new-checkout")   // env 또는 원격 설정
```

### 2-4. 분기는 진입점에 집중한다 (분산 금지)
- 같은 플래그 분기를 화면/모듈마다 반복하지 말고, 라우트·컨테이너 같은 진입점에서 한 번만 갈라 구현체를 선택한다.

```text
// ❌ 금지 — 곳곳에 같은 플래그 분기가 반복됨 (제거가 어려움)
if (flags.newCheckout) showA() else showB()   // 여러 화면에 흩뿌려짐

// ✅ 권장 — 진입점에서 한 번만 분기해 구현체를 고름
Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

### 2-5. on/off 양쪽 경로를 테스트한다
- 켠 경로만 보지 말고, 끈(레거시·롤백) 경로도 테스트로 고정한다. 플래그를 토글해도 양쪽이 동작함을 보장한다.

```text
// ✅ 권장 — 양쪽 경로를 모두 검증
test("flag on  → 새 플로우 노출")
test("flag off → 레거시 플로우 노출")  // 롤백 안전망
```

### 2-6. 완료된 플래그를 정리한다
- 출시가 굳어졌거나 실험이 끝난 임시 플래그는 분기와 죽은 코드(쓰지 않게 된 쪽)를 함께 제거한다.
- 제거를 "나중에"로 미루지 말고, 2-2에서 정한 제거 조건/티켓에 따라 처리한다.

```text
// 출시 확정 후
remove flag "new-checkout"
delete LegacyCheckout (더 이상 닿지 않는 분기)
```

## 3. 흔한 실수
- **완료된 플래그 방치** → 코드에 죽은 분기가 쌓여 복잡도가 늘고, 어떤 게 살아있는 분기인지 헷갈린다.
- **플래그를 화면/모듈마다 분산** → 나중에 제거할 때 모든 분기를 찾아 지워야 해 누락이 잦다. 진입점으로 모은다.
- **off(롤백) 경로 미테스트** → 정작 롤백/kill switch가 필요한 순간에 끈 경로가 깨져 있다.
- **값을 코드에 하드코딩** → 재배포 없이는 끄지 못해 즉시 대응이 불가능하다. 외부 소스에서 읽는다.
- **이름·목적·소유자 미문서화** → 무슨 플래그인지, 제거해도 되는지 아무도 모른다.
- **수명 미분류** → 임시여야 할 플래그가 영구처럼 눌러앉는다. 만들 때 유형과 제거 조건을 정한다.

## 4. 체크리스트
- [ ] 새 기능을 플래그로 감싸 **배포/릴리스를 분리**했는가
- [ ] 플래그 **유형(수명)** 을 정하고, 임시 플래그는 제거 조건/티켓을 만들었는가
- [ ] 플래그 값을 **코드 밖(환경/원격)** 에서 읽고, 조회를 한 곳으로 모았는가
- [ ] 플래그 분기를 **진입점에 집중**했는가 (화면/모듈에 분산하지 않았는가)
- [ ] **on/off 양쪽 경로**를 테스트했는가 (롤백 안전망 포함)
- [ ] 역할을 다한 **임시 플래그와 죽은 분기를 제거**했는가
- [ ] 플래그 **이름·목적·소유자**를 문서화했는가

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(예: React, Svelte, Angular, 서버측 SDK 등)에 맞는 예시를 같은 패턴으로 추가한다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다. 클라이언트 입력 검증은 `validation-bean`(입력값 검증 표준)을 참고한다.

### Vue 3 + Vite

Vite의 `import.meta.env`로 환경 기반 플래그를 읽고, 원격 관리는 GrowthBook으로, 소비는 composable로 감싼다.

#### 환경 변수 기반 (단순)

```js
// config/flags.js
export const flags = {
  newCheckout: import.meta.env.VITE_FLAG_NEW_CHECKOUT === 'true',
}
```

#### 원격 관리 (GrowthBook 예시)

```js
import { GrowthBook } from '@growthbook/growthbook'
const gb = new GrowthBook({ apiHost: '...', clientKey: '...' })
await gb.loadFeatures()
const enabled = gb.isOn('new-checkout')
```

#### 진입점 집중 (분산 금지)

```js
// ❌ 금지 — 컴포넌트 곳곳에 플래그 분기가 반복됨 (제거가 어려움)
if (flags.newCheckout) showA(); else showB()

// ✅ 권장 — 라우트/컨테이너에서 한 번만 분기
const Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

#### composable

```js
export function useFlag(key) {
  const enabled = ref(getFlag(key))   // 캐시 또는 원격
  return { enabled }
}
```
