# 🚩 フィーチャーフラグ (Feature Flag)

> コードのマージ(デプロイ)と機能の公開(リリース)を分離する。新機能を段階的に公開したり、A/B テスト・kill switch が必要なとき、フラグの寿命・整理・管理方法を定めるときに読む。特定の言語/フレームワークに依存しない汎用標準である。

## 1. 中核となる原則
- **デプロイとリリースを分離する**: 新機能はフラグで包んでデプロイする。コードのマージが即ち機能の公開ではない — いつ誰に対して on にするかはフラグが決める。
- **フラグごとに寿命を定義する**: 一時的(リリース)・実験・恒久(運用)のどれかを定め、役割を終えた一時フラグは死んだ分岐として残さず即座に削除する。作成時に削除条件/チケットも併せて定める。
- **状態はコードの外で管理する**: フラグの on/off 値はビルド成果物に埋め込まず、環境変数やリモート設定サービスのような外部ソースから読み、再デプロイなしで変更できるようにする。
- **エントリーポイントに集中する**: 分岐はルート・コンテナのようなエントリーポイントで一度だけ行い、画面/モジュールの奥深くで同じ分岐を繰り返さない — 分散するほど削除が難しくなる。
- **on/off 両方をテストする**: on の経路と off(ロールバック)の経路の両方を検証する。off の経路を見ないと、kill switch がいざ必要なときに壊れている。
- **名前・目的を文書化する**: どのフラグがなぜ存在するか、誰が所有しいつ削除するかを追跡可能に残す。

## 2. ルール

### 2-1. 新機能はフラグで包んでデプロイ/リリースを分離する
- 未完成・段階的公開が必要な機能はフラグの後ろに置いてマージする。マージと公開のタイミングを分離して安全にリリース・ロールバックする。

```text
// ❌ 禁止 — マージ = 即座に全体公開 (ロールバックには差し戻して再デプロイが必要)
merge(newCheckout)  → 全ユーザーにすぐ露出

// ✅ 推奨 — フラグの後ろに置いてマージし、公開タイミングはフラグで制御
merge(newCheckout behind flag "new-checkout")  // デフォルト off
flag "new-checkout" on → 段階的公開 / off → 即座にロールバック
```

### 2-2. フラグの種類と寿命を定める
- 作成時に種類(寿命)を分類し、一時フラグには削除条件/チケットを付ける。

| 種類 | 寿命 | 例 |
|---|---|---|
| リリーストグル | 短期(リリース後に削除) | 新しい決済フロー |
| 実験トグル | 中期(A/B 終了後に削除) | ボタン文言の実験 |
| 運用トグル | 恒久 | Kill switch |

### 2-3. 状態はコードの外(環境/リモート)から読む
- フラグ値をソースにハードコードせず、環境変数(シンプル)やリモート設定サービス(段階的公開・ターゲティング・リアルタイム変更)から読む。
- フラグの参照は一箇所(設定モジュール/ラッパー)にまとめ、消費する側が出所を知らなくてよいようにする。

```text
// ❌ 禁止 — 値がコードに埋め込まれ再デプロイなしでは変えられない
const newCheckout = true

// ✅ 推奨 — 外部ソースから読み一箇所にまとめる
flags.newCheckout = readFlag("new-checkout")   // env またはリモート設定
```

### 2-4. 分岐はエントリーポイントに集中する (分散禁止)
- 同じフラグ分岐を画面/モジュールごとに繰り返さず、ルート・コンテナのようなエントリーポイントで一度だけ分岐して実装を選択する。

```text
// ❌ 禁止 — 至る所に同じフラグ分岐が繰り返される (削除が難しい)
if (flags.newCheckout) showA() else showB()   // 複数の画面に撒き散らされる

// ✅ 推奨 — エントリーポイントで一度だけ分岐して実装を選ぶ
Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

### 2-5. on/off 両方の経路をテストする
- on の経路だけを見ず、off(レガシー・ロールバック)の経路もテストで固定する。フラグをトグルしても両方が動作することを保証する。

```text
// ✅ 推奨 — 両方の経路を検証
test("flag on  → 新フローを露出")
test("flag off → レガシーフローを露出")  // ロールバックの安全網
```

### 2-6. 完了したフラグを整理する
- リリースが固まった、または実験が終わった一時フラグは、分岐と死んだコード(使われなくなった側)を併せて削除する。
- 削除を「後で」と先送りせず、2-2 で定めた削除条件/チケットに従って処理する。

```text
// リリース確定後
remove flag "new-checkout"
delete LegacyCheckout (もう到達しない分岐)
```

## 3. よくある間違い
- **完了したフラグの放置** → コードに死んだ分岐が積もり複雑度が増し、どれが生きている分岐か混乱する。
- **フラグを画面/モジュールごとに分散** → 後で削除するときにすべての分岐を探して消さねばならず漏れが頻発する。エントリーポイントにまとめる。
- **off(ロールバック)経路の未テスト** → いざロールバック/kill switch が必要な瞬間に off の経路が壊れている。
- **値をコードにハードコード** → 再デプロイなしでは off にできず即時対応が不可能。外部ソースから読む。
- **名前・目的・所有者の未文書化** → どのフラグなのか、削除してよいか誰も分からない。
- **寿命の未分類** → 一時であるべきフラグが恒久のように居座る。作成時に種類と削除条件を定める。

## 4. チェックリスト
- [ ] 新機能をフラグで包んで **デプロイ/リリースを分離** したか
- [ ] フラグの **種類(寿命)** を定め、一時フラグは削除条件/チケットを作成したか
- [ ] フラグ値を **コードの外(環境/リモート)** から読み、参照を一箇所にまとめたか
- [ ] フラグ分岐を **エントリーポイントに集中** したか (画面/モジュールに分散していないか)
- [ ] **on/off 両方の経路** をテストしたか (ロールバックの安全網を含む)
- [ ] 役割を終えた **一時フラグと死んだ分岐を削除** したか
- [ ] フラグの **名前・目的・所有者** を文書化したか

## 付録: スタック別の例

> 以下は参考用の実装例である。チームが使うスタック(例: React, Svelte, Angular, サーバーサイド SDK など)に合った例を同じパターンで追加する。上記 1~4 の原則・ルールが標準であり、付録はその適用事例にすぎない。クライアントの入力検証は `validation-bean`(入力値検証の標準)を参照する。

### Vue 3 + Vite

Vite の `import.meta.env` で環境ベースのフラグを読み、リモート管理は GrowthBook で、消費は composable で包む。

#### 環境変数ベース (シンプル)

```js
// config/flags.js
export const flags = {
  newCheckout: import.meta.env.VITE_FLAG_NEW_CHECKOUT === 'true',
}
```

#### リモート管理 (GrowthBook の例)

```js
import { GrowthBook } from '@growthbook/growthbook'
const gb = new GrowthBook({ apiHost: '...', clientKey: '...' })
await gb.loadFeatures()
const enabled = gb.isOn('new-checkout')
```

#### エントリーポイントへの集中 (分散禁止)

```js
// ❌ 禁止 — コンポーネントの至る所にフラグ分岐が繰り返される (削除が難しい)
if (flags.newCheckout) showA(); else showB()

// ✅ 推奨 — ルート/コンテナで一度だけ分岐
const Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

#### composable

```js
export function useFlag(key) {
  const enabled = ref(getFlag(key))   // キャッシュまたはリモート
  return { enabled }
}
```
