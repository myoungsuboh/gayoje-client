# 🚩 功能开关 (Feature Flag)

> 将代码合并(部署)与功能公开(发布)分离。在需要渐进公开新功能、需要 A/B 测试或 kill switch 时，以及在确定开关生命周期、清理、管理方式时阅读。这是不依赖特定语言/框架的通用标准。

## 1. 核心原则
- **将部署与发布分离**: 新功能用开关包裹后部署。合并代码并不等于公开功能 — 何时对谁开启由开关决定。
- **为每个开关定义生命周期**: 确定它属于临时(发布)、实验还是永久(运营)，并将完成使命的临时开关立即移除，而不是留作死分支。创建时一并确定移除条件/工单。
- **在代码之外管理状态**: 不要将开关的 on/off 值写死在构建产物中，而是从环境变量或远程配置服务等外部来源读取，使其无需重新部署即可更改。
- **集中于入口点**: 仅在路由、容器等入口点分支一次，不要在屏幕/模块深处重复相同的分支 — 越分散越难移除。
- **测试 on/off 两侧**: 验证开启路径和关闭(回滚)路径两者。若从不查看关闭路径，kill switch 会恰恰在需要时损坏。
- **记录名称与目的**: 留下可追踪的记录，说明哪个开关因何存在、由谁拥有、何时移除。

## 2. 规则

### 2-1. 用开关包裹新功能以分离部署/发布
- 将未完成或需要渐进公开的功能放在开关之后再合并。分离合并与公开的时机，以便安全发布与回滚。

```text
// ❌ 禁止 — 合并 = 立即全量公开 (回滚需要还原并重新部署)
merge(newCheckout)  → 立即向所有用户暴露

// ✅ 推荐 — 放在开关之后合并，公开时机由开关控制
merge(newCheckout behind flag "new-checkout")  // 默认 off
flag "new-checkout" on → 渐进公开 / off → 立即回滚
```

### 2-2. 确定开关类型与生命周期
- 创建时对类型(生命周期)进行分类，并为临时开关附上移除条件/工单。

| 类型 | 生命周期 | 示例 |
|---|---|---|
| 发布开关 | 短期(发布后移除) | 新支付流程 |
| 实验开关 | 中期(A/B 结束后移除) | 按钮文案实验 |
| 运营开关 | 永久 | Kill switch |

### 2-3. 从代码之外(环境/远程)读取状态
- 不要将开关值硬编码在源码中，而是从环境变量(简单)或远程配置服务(渐进公开、定向、实时更改)读取。
- 将开关查询集中到一处(配置模块/包装器)，使消费方无需知道来源。

```text
// ❌ 禁止 — 值写死在代码里，不重新部署就无法更改
const newCheckout = true

// ✅ 推荐 — 从外部来源读取并集中到一处
flags.newCheckout = readFlag("new-checkout")   // env 或远程配置
```

### 2-4. 将分支集中于入口点 (禁止分散)
- 不要在每个屏幕/模块重复相同的开关分支，而是在路由、容器等入口点仅分支一次以选择实现。

```text
// ❌ 禁止 — 各处重复相同的开关分支 (难以移除)
if (flags.newCheckout) showA() else showB()   // 散布在多个屏幕

// ✅ 推荐 — 在入口点仅分支一次以选择实现
Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

### 2-5. 测试 on/off 两侧路径
- 不要只看开启路径，也用测试固定关闭(遗留·回滚)路径。保证即使切换开关，两侧都能工作。

```text
// ✅ 推荐 — 验证两侧路径
test("flag on  → 暴露新流程")
test("flag off → 暴露遗留流程")  // 回滚安全网
```

### 2-6. 清理已完成的开关
- 对于发布已稳固或实验已结束的临时开关，将分支与死代码(不再使用的一侧)一并移除。
- 不要把移除推迟到"以后"，而是按照 2-2 中确定的移除条件/工单处理。

```text
// 发布确定后
remove flag "new-checkout"
delete LegacyCheckout (不再触达的分支)
```

## 3. 常见错误
- **放任已完成的开关** → 代码中死分支堆积，复杂度上升，难以分辨哪个是存活分支。
- **将开关分散到每个屏幕/模块** → 日后移除时必须找出并删除所有分支，遗漏频发。集中到入口点。
- **未测试 off(回滚)路径** → 在真正需要回滚/kill switch 的瞬间，关闭路径已损坏。
- **将值硬编码在代码中** → 不重新部署就无法关闭，无法即时应对。从外部来源读取。
- **未记录名称、目的、所有者** → 无人知道这是什么开关、是否可以移除。
- **未分类生命周期** → 本应临时的开关像永久般赖着不走。创建时确定类型与移除条件。

## 4. 检查清单
- [ ] 是否用开关包裹新功能以 **分离部署/发布**
- [ ] 是否确定了开关 **类型(生命周期)**，并为临时开关创建了移除条件/工单
- [ ] 是否从 **代码之外(环境/远程)** 读取开关值，并将查询集中到一处
- [ ] 是否将开关分支 **集中于入口点** (没有分散到屏幕/模块)
- [ ] 是否测试了 **on/off 两侧路径** (包含回滚安全网)
- [ ] 是否 **移除了** 完成使命的 **临时开关与死分支**
- [ ] 是否记录了开关的 **名称、目的、所有者**

## 附录: 各技术栈示例

> 以下是供参考的实现示例。请按相同模式为团队所用的技术栈(如 React、Svelte、Angular、服务端 SDK 等)添加示例。上述 1~4 的原则·规则是标准，附录只是其应用案例。客户端输入校验请参考 `validation-bean`(输入值校验标准)。

### Vue 3 + Vite

用 Vite 的 `import.meta.env` 读取基于环境的开关，远程管理用 GrowthBook，消费用 composable 包装。

#### 基于环境变量 (简单)

```js
// config/flags.js
export const flags = {
  newCheckout: import.meta.env.VITE_FLAG_NEW_CHECKOUT === 'true',
}
```

#### 远程管理 (GrowthBook 示例)

```js
import { GrowthBook } from '@growthbook/growthbook'
const gb = new GrowthBook({ apiHost: '...', clientKey: '...' })
await gb.loadFeatures()
const enabled = gb.isOn('new-checkout')
```

#### 集中于入口点 (禁止分散)

```js
// ❌ 禁止 — 组件各处重复出现开关分支 (难以移除)
if (flags.newCheckout) showA(); else showB()

// ✅ 推荐 — 在路由/容器仅分支一次
const Checkout = flags.newCheckout ? NewCheckout : LegacyCheckout
```

#### composable

```js
export function useFlag(key) {
  const enabled = ref(getFlag(key))   // 缓存或远程
  return { enabled }
}
```
