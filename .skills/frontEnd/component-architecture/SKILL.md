# 🧩 컴포넌트 아키텍처 (Component Architecture)

> UI를 작고 책임이 분명한 컴포넌트로 나누고, 데이터는 props로 내려보내고 변경은 이벤트로 올려보내(단방향), 표현과 로직을 분리해 재사용·테스트·교체가 쉬운 구조를 만든다. 새 컴포넌트를 만들거나 비대해진 컴포넌트를 쪼갤 때, 컴포넌트 간 데이터·이벤트 흐름과 상태 위치를 정할 때 읽는다. 특정 언어/프레임워크에 종속되지 않는 범용 표준이다.

## 1. 핵심 원칙
- **단일 책임 컴포넌트**: 하나의 컴포넌트는 한 가지 일만 한다. "보여주기 + 가져오기 + 가공하기 + 라우팅"을 한 컴포넌트가 다 하면 쪼갠다. 설명에 "그리고"가 여러 번 들어가면 분리 신호다.
- **props 입력 / 이벤트 출력 (단방향 데이터 흐름)**: 데이터는 부모 → 자식으로 props(입력)로만 내려가고, 자식의 변경 요구는 부모로 이벤트(출력)로만 올라간다. 자식이 부모의 데이터를 직접 고치지 않는다.
- **표현(Presentational) / 컨테이너(Container) 분리**: "어떻게 보이나"(표현, props만 받아 렌더)와 "어떻게 동작하나"(컨테이너, 데이터 조회·상태·부수효과)를 다른 컴포넌트로 나눈다.
- **작고 재사용 가능한 단위**: 도메인에 묶이지 않은 표현 컴포넌트는 어디서든 재사용되게 만든다. 같은 마크업·동작이 두 번 반복되면 컴포넌트로 추출한다.
- **상태는 필요한 가장 낮은 곳에, 공유되면 끌어올린다**: 상태는 그것을 쓰는 컴포넌트 가까이 둔다. 둘 이상이 같은 상태를 공유해야 하면 가장 가까운 공통 부모로 **끌어올려(lifting state up)** 단일 출처(single source of truth)로 삼는다.
- **공개 API(인터페이스)를 명시적으로**: 컴포넌트가 받는 입력(props)과 내보내는 이벤트는 타입·이름을 분명히 선언해, 사용처가 내부 구현을 몰라도 쓸 수 있게 한다.
- **표준·공용 컴포넌트 우선**: 디자인 시스템/UI 라이브러리의 표준 컴포넌트를 먼저 쓰고, 도메인 특화가 필요한 경우에만 최소한으로 커스텀한다. (→ `design-system` 참고)

## 2. 규칙

### 2-1. 한 컴포넌트 = 한 책임
- 비대해진 컴포넌트(데이터 조회 + 상태 + 복잡한 마크업 + 분기)는 책임 단위로 쪼갠다.
- "이 컴포넌트가 하는 일"을 한 문장으로 못 적으면 너무 많은 일을 하는 것이다.

```text
// ❌ 금지 — 한 컴포넌트가 조회·상태·표현·이벤트를 다 함
UserDashboard:
  fetch users from API
  hold filter/sort/pagination state
  render table markup, rows, cells
  handle row click, edit, delete

// ✅ 권장 — 책임별로 분리
UserDashboardContainer:   // 데이터 조회 + 상태 보유
  fetch users; hold filter state
  render <UserTable users=... onRowSelect=...>
UserTable (presentational): // props만 받아 렌더, 변경은 이벤트로
  render rows from props.users
  emit "rowSelect" on click
```

### 2-2. 데이터는 props로 내리고, 변경은 이벤트로 올린다 (단방향)
- 자식은 부모가 준 데이터를 직접 수정하지 않는다. 바꿔야 하면 이벤트로 "바꿔달라"고 알린다.
- 부모가 상태의 주인(source of truth)이고, 자식은 받아서 보여주고 의도를 전달만 한다.

```text
// ❌ 금지 — 자식이 받은 데이터를 직접 변형
Child(props.item):
  props.item.done = true        // 부모 상태를 몰래 바꿈 (양방향·암묵 변형)

// ✅ 권장 — 자식은 이벤트로 의도만 전달, 변경은 부모가
Child(props.item):
  onClick → emit "toggle", props.item.id
Parent:
  <Child item=... onToggle=(id) => updateItem(id) >
```

### 2-3. 표현(Presentational)과 컨테이너(Container)를 나눈다
- 표현 컴포넌트: props만 받아 렌더, 부수효과·데이터 조회 없음 → 재사용·테스트 쉬움.
- 컨테이너 컴포넌트: 데이터 조회·상태·부수효과 담당, 표현 컴포넌트에 props로 넘김.

```text
// ❌ 금지 — 표현 안에 데이터 조회가 박힘 (재사용·테스트 불가)
PriceTag:
  price = fetch("/api/price")   // 표현 컴포넌트가 네트워크에 의존
  render price

// ✅ 권장 — 조회는 컨테이너, 표현은 값만 받는다
PriceTagContainer:
  price = usePrice()            // 조회/상태
  render <PriceTag value=price>
PriceTag(value):                // 순수 표현, 어디서든 재사용
  render value
```

### 2-4. 반복되는 마크업·동작은 컴포넌트로 추출 (재사용)
- 같은 UI 조각이 두 곳 이상 반복되면 공용 컴포넌트로 뽑아 한 곳에서 관리한다.
- 도메인에 묶이지 않게 설계하면(이름·props를 일반화) 재사용 범위가 넓어진다.

```text
// ❌ 금지 — 같은 배지 마크업을 화면마다 복붙
(여러 화면에 동일한 status-badge 마크업이 흩어짐)

// ✅ 권장 — 공용 컴포넌트로 추출, props로 변형
StatusBadge(status, label):  // 한 곳에 정의해 재사용
  render colored badge by status
```

### 2-5. 상태는 낮게 두고, 공유되면 끌어올린다 (Lifting State Up)
- 한 컴포넌트만 쓰는 상태는 그 안에 둔다. 형제 둘이 같은 값을 공유해야 하면 공통 부모로 끌어올려 한 곳에서 관리하고 props/이벤트로 연결한다.
- 같은 의미의 상태를 여러 곳에 복제하지 않는다(동기화 버그의 원천). 단일 출처를 둔다.
- 앱 전역에서 멀리 떨어진 컴포넌트들이 공유하는 상태는 전역 상태 관리로 분리한다. (→ `state-management` 참고)

```text
// ❌ 금지 — 형제가 각자 상태를 들고 따로 논다 (동기화 깨짐)
FilterInput:  holds its own "query"
ResultList:   holds its own "query"   // 둘이 어긋남

// ✅ 권장 — 공통 부모로 끌어올려 단일 출처
SearchPage:
  query state here
  <FilterInput value=query onChange=setQuery>
  <ResultList query=query>
```

### 2-6. 공개 인터페이스(props/이벤트)를 명시적으로 선언
- 컴포넌트가 받는 props와 내보내는 이벤트의 이름·타입을 분명히 선언한다. 암묵적·문서 없는 인터페이스를 두지 않는다.
- 입력은 명사형(`items`, `disabled`), 출력 이벤트는 동사형/사건명(`submit`, `select`)으로 일관되게 이름 짓는다.

```text
// ❌ 금지 — 무엇을 받는지·내보내는지 불명확
Component(...anything):  // 임의 객체를 통째로 받고 내부에서 추측

// ✅ 권장 — 입력/출력 계약을 선언
Component:
  props:  items: Item[], disabled: boolean
  events: select(id), submit(payload)
```

## 3. 흔한 실수
- **만능 컴포넌트(God component)**: 조회·상태·표현·라우팅을 한 컴포넌트가 다 함 → 책임별로 쪼갠다.
- **자식이 부모 데이터를 직접 변형**: 단방향 흐름이 깨져 추적 불가한 버그 발생 → 변경은 이벤트로 올린다.
- **표현 컴포넌트에 데이터 조회가 박힘**: 재사용·테스트가 막힘 → 조회는 컨테이너로 분리.
- **마크업 복붙**: 같은 UI를 여러 곳에 복제 → 공용 컴포넌트로 추출.
- **상태 중복 보유**: 같은 값을 여러 컴포넌트가 각자 들고 있어 어긋남 → 공통 부모로 끌어올려 단일 출처.
- **prop drilling 남발**: 깊은 트리에 props를 손으로 줄줄이 내림 → 정말 전역적인 값은 전역 상태로 (→ `state-management`).
- **암묵적 인터페이스**: props/이벤트를 선언 없이 통째로 넘김 → 입력·출력 계약을 명시.
- **무조건 커스텀**: 표준 컴포넌트가 있는데 직접 만듦 → 디자인 시스템/표준 우선 (→ `design-system`).

## 4. 체크리스트
- [ ] 이 컴포넌트의 책임을 한 문장으로 설명할 수 있는가 (아니면 분리)
- [ ] 데이터는 props로 내려가고, 변경은 이벤트로 올라가는가 (자식이 부모 데이터를 직접 안 고치는가)
- [ ] 표현 컴포넌트와 데이터 조회/상태(컨테이너)를 분리했는가
- [ ] 반복되는 마크업·동작을 공용 컴포넌트로 추출했는가
- [ ] 상태를 필요한 가장 낮은 곳에 두고, 공유되는 상태는 공통 부모로 끌어올렸는가 (단일 출처)
- [ ] 같은 의미의 상태를 여러 곳에 중복 보유하지 않는가
- [ ] props/이벤트 등 공개 인터페이스를 이름·타입과 함께 명시적으로 선언했는가
- [ ] 표준/디자인 시스템 컴포넌트를 우선하고 커스텀은 최소화했는가

## 부록: 스택별 예시

> 아래는 참고용 구현 예시다. 팀이 쓰는 스택(예: React, Svelte, Angular 등)에 맞는 예시를 같은 패턴으로 추가한다. 위 1~4의 원칙·규칙이 표준이고, 부록은 그 적용 사례일 뿐이다.

### Vue 3 + Vuetify

> 본문 1~4의 원칙·규칙이 표준이다. Vue 3 + Vuetify에서는 다음 매핑으로 적용한다 — 새 코드를 지어내기보다 어떤 API가 어떤 원칙을 구현하는지만 짚는다. (Vue SFC 구조는 `vue-sfc-structure`, Vuetify 컴포넌트 활용은 `ui-vuetify`, 표준 컴포넌트 우선은 `design-system` 참고)

- **공개 인터페이스 선언(본문 2-6)** → `defineProps`/`defineEmits`로 props 입력·emits 출력을 명시한다.
- **표준 컴포넌트 우선(본문 핵심 원칙)** → Vuetify 표준(`VCard`/`VBtn`/`VDataTable` 등)을 먼저 쓰고 커스텀은 최소화한다.
- **작성 관례(스택 전용)** → 모든 컴포넌트는 `.vue` + `<script setup>`, 파일명·태그는 `PascalCase`. 아이콘은 `VIcon`의 `icon` 속성으로 Iconify를 일관 적용한다.
