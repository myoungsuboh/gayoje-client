# 🧩 コンポーネントアーキテクチャ (Component Architecture)

> UIを小さく責任の明確なコンポーネントに分け、データはpropsで下ろし変更はイベントで上げ(一方向)、表現とロジックを分離して再利用・テスト・差し替えが容易な構造を作る。新しいコンポーネントを作る、または肥大化したコンポーネントを分割するとき、コンポーネント間のデータ・イベントフローと状態の位置を決めるときに読む。特定の言語/フレームワークに依存しない汎用標準である。

## 1. コア原則
- **単一責任コンポーネント**: 一つのコンポーネントは一つのことだけを行う。「見せる + 取得する + 加工する + ルーティング」を一つのコンポーネントが全部やるなら分割する。説明に「そして」が何度も入るなら分離の合図だ。
- **props入力 / イベント出力 (一方向データフロー)**: データは親 → 子へ props(入力)としてのみ下り、子の変更要求は親へ イベント(出力)としてのみ上る。子が親のデータを直接書き換えない。
- **表現(Presentational) / コンテナ(Container) の分離**: 「どう見えるか」(表現、propsだけ受けてレンダー)と「どう動くか」(コンテナ、データ取得・状態・副作用)を別のコンポーネントに分ける。
- **小さく再利用可能な単位**: ドメインに縛られない表現コンポーネントはどこでも再利用できるようにする。同じマークアップ・動作が二度繰り返されたらコンポーネントとして抽出する。
- **状態は必要な最も低い場所に、共有されるなら引き上げる**: 状態はそれを使うコンポーネントの近くに置く。二つ以上が同じ状態を共有する必要があれば最も近い共通の親へ**引き上げて(lifting state up)**単一の出所(single source of truth)とする。
- **公開API(インターフェース)を明示的に**: コンポーネントが受け取る入力(props)と発するイベントは型・名前をはっきり宣言し、使う側が内部実装を知らなくても使えるようにする。
- **標準・共用コンポーネント優先**: デザインシステム/UIライブラリの標準コンポーネントをまず使い、ドメイン特化が必要な場合にのみ最小限にカスタマイズする。(→ `design-system` 参照)

## 2. ルール

### 2-1. 一つのコンポーネント = 一つの責任
- 肥大化したコンポーネント(データ取得 + 状態 + 複雑なマークアップ + 分岐)を責任単位で分割する。
- 「このコンポーネントがやること」を一文で書けないなら、やりすぎている。

```text
// ❌ 禁止 — 一つのコンポーネントが取得・状態・表現・イベントを全部やる
UserDashboard:
  fetch users from API
  hold filter/sort/pagination state
  render table markup, rows, cells
  handle row click, edit, delete

// ✅ 推奨 — 責任別に分離
UserDashboardContainer:   // データ取得 + 状態保持
  fetch users; hold filter state
  render <UserTable users=... onRowSelect=...>
UserTable (presentational): // propsだけ受けてレンダー、変更はイベントで
  render rows from props.users
  emit "rowSelect" on click
```

### 2-2. データはpropsで下ろし、変更はイベントで上げる (一方向)
- 子は親が与えたデータを直接修正しない。変えるべきならイベントで「変えてほしい」と知らせる。
- 親が状態の持ち主(source of truth)であり、子は受けて見せて意図を伝えるだけだ。

```text
// ❌ 禁止 — 子が受けたデータを直接変形
Child(props.item):
  props.item.done = true        // 親の状態を密かに変える (双方向・暗黙変形)

// ✅ 推奨 — 子はイベントで意図だけ伝え、変更は親が
Child(props.item):
  onClick → emit "toggle", props.item.id
Parent:
  <Child item=... onToggle=(id) => updateItem(id) >
```

### 2-3. 表現(Presentational)とコンテナ(Container)を分ける
- 表現コンポーネント: propsだけ受けてレンダー、副作用・データ取得なし → 再利用・テストが容易。
- コンテナコンポーネント: データ取得・状態・副作用を担当し、表現コンポーネントにpropsで渡す。

```text
// ❌ 禁止 — 表現の中にデータ取得が埋め込まれる (再利用・テスト不可)
PriceTag:
  price = fetch("/api/price")   // 表現コンポーネントがネットワークに依存
  render price

// ✅ 推奨 — 取得はコンテナ、表現は値だけ受ける
PriceTagContainer:
  price = usePrice()            // 取得/状態
  render <PriceTag value=price>
PriceTag(value):                // 純粋表現、どこでも再利用
  render value
```

### 2-4. 繰り返すマークアップ・動作はコンポーネントに抽出 (再利用)
- 同じUI断片が二箇所以上繰り返されたら共用コンポーネントに取り出し一箇所で管理する。
- ドメインに縛られないよう設計すると(名前・propsを一般化)再利用範囲が広がる。

```text
// ❌ 禁止 — 同じバッジマークアップを画面ごとにコピペ
(複数の画面に同一の status-badge マークアップが散らばる)

// ✅ 推奨 — 共用コンポーネントに抽出、propsで変形
StatusBadge(status, label):  // 一箇所に定義して再利用
  render colored badge by status
```

### 2-5. 状態は低く置き、共有されるなら引き上げる (Lifting State Up)
- 一つのコンポーネントだけが使う状態はその中に置く。兄弟二つが同じ値を共有する必要があれば共通の親へ引き上げ一箇所で管理しprops/イベントで繋ぐ。
- 同じ意味の状態を複数箇所に複製しない(同期バグの原因)。単一の出所を置く。
- アプリ全域で遠く離れたコンポーネントが共有する状態はグローバル状態管理に分離する。(→ `state-management` 参照)

```text
// ❌ 禁止 — 兄弟がそれぞれ状態を持ち別々に動く (同期が壊れる)
FilterInput:  holds its own "query"
ResultList:   holds its own "query"   // 二つがずれる

// ✅ 推奨 — 共通の親へ引き上げて単一の出所
SearchPage:
  query state here
  <FilterInput value=query onChange=setQuery>
  <ResultList query=query>
```

### 2-6. 公開インターフェース(props/イベント)を明示的に宣言
- コンポーネントが受け取るpropsと発するイベントの名前・型をはっきり宣言する。暗黙的・文書のないインターフェースを置かない。
- 入力は名詞形(`items`, `disabled`)、出力イベントは動詞形/事象名(`submit`, `select`)で一貫して名付ける。

```text
// ❌ 禁止 — 何を受け・発するか不明確
Component(...anything):  // 任意のオブジェクトを丸ごと受け内部で推測

// ✅ 推奨 — 入力/出力の契約を宣言
Component:
  props:  items: Item[], disabled: boolean
  events: select(id), submit(payload)
```

## 3. よくある間違い
- **万能コンポーネント(God component)**: 取得・状態・表現・ルーティングを一つのコンポーネントが全部やる → 責任別に分割する。
- **子が親データを直接変形**: 一方向フローが壊れ追跡不能なバグが発生 → 変更はイベントで上げる。
- **表現コンポーネントにデータ取得が埋め込まれる**: 再利用・テストが塞がれる → 取得はコンテナに分離。
- **マークアップのコピペ**: 同じUIを複数箇所に複製 → 共用コンポーネントに抽出。
- **状態の重複保持**: 同じ値を複数のコンポーネントがそれぞれ持ちずれる → 共通の親へ引き上げて単一の出所。
- **prop drilling の濫用**: 深いツリーにpropsを手作業で延々下ろす → 本当にグローバルな値はグローバル状態へ (→ `state-management`)。
- **暗黙的インターフェース**: props/イベントを宣言なしに丸ごと渡す → 入力・出力の契約を明示。
- **無条件のカスタム**: 標準コンポーネントがあるのに自作 → デザインシステム/標準優先 (→ `design-system`)。

## 4. チェックリスト
- [ ] このコンポーネントの責任を一文で説明できるか (できなければ分離)
- [ ] データはpropsで下り、変更はイベントで上るか (子が親データを直接いじらないか)
- [ ] 表現コンポーネントとデータ取得/状態(コンテナ)を分離したか
- [ ] 繰り返すマークアップ・動作を共用コンポーネントに抽出したか
- [ ] 状態を必要な最も低い場所に置き、共有される状態は共通の親へ引き上げたか (単一の出所)
- [ ] 同じ意味の状態を複数箇所に重複保持していないか
- [ ] props/イベントなど公開インターフェースを名前・型とともに明示的に宣言したか
- [ ] 標準/デザインシステムコンポーネントを優先しカスタムを最小化したか

## 付録: スタック別の例

> 以下は参考用の実装例である。チームが使うスタック(例: React, Svelte, Angularなど)に合った例を同じパターンで追加する。上記1〜4の原則・ルールが標準であり、付録はその適用事例にすぎない。

### Vue 3 + Vuetify

> 本文1〜4の原則・ルールが標準だ。Vue 3 + Vuetifyでは次のマッピングで適用する — 新しいコードを作り上げるよりどのAPIがどの原則を実装するかだけを指摘する。(Vue SFC構造は `vue-sfc-structure`、Vuetifyコンポーネント活用は `ui-vuetify`、標準コンポーネント優先は `design-system` 参照)

- **公開インターフェース宣言(本文2-6)** → `defineProps`/`defineEmits`でprops入力・emits出力を明示する。
- **標準コンポーネント優先(本文コア原則)** → Vuetify標準(`VCard`/`VBtn`/`VDataTable` など)をまず使いカスタムを最小化する。
- **作成慣例(スタック専用)** → すべてのコンポーネントは `.vue` + `<script setup>`、ファイル名・タグは `PascalCase`。アイコンは `VIcon` の `icon` 属性でIconifyを一貫適用する。
