# 📎 ファイルアップロードと検証

> クライアント側のファイル検証と複数アップロード UX を標準化するための文書。ファイルアップロード機能を作るとき、または容量・拡張子の検証、進捗表示を扱うときに読みます。

## 1. 中核となる原則
- アップロード前にクライアントで容量と拡張子をまず検証する。
- ファイル選択の直後に即座に検証結果をフィードバックする。
- 複数アップロードは Queue・Progress・Status を UI で明確に露出する。

## 2. ルール

### 2-1. クライアント側の検証
- **容量制限**: `file.size` をチェックしてサーバー負荷を防止します。
- **拡張子制限**: `file.name` から拡張子を抽出して許可された形式か確認します。
- **即時フィードバック**: ファイル選択の直後に検証結果をユーザーに案内します。

### 2-2. 実装パターン
```javascript
const validateFile = (file) => {
  const maxSize = 5 * 1024 * 1024; // 5MB
  const allowedTypes = ['image/png', 'image/jpeg', 'application/pdf'];

  if (file.size > maxSize) return '파일 용량이 너무 큽니다.';
  if (!allowedTypes.includes(file.type)) return '지원하지 않는 파일 형식입니다.';
  
  return null;
};
```

### 2-3. 複数ファイルアップロードの管理
- **Queue**: アップロード待ちのリストを UI で露出します。
- **Progress**: 個別ファイルまたは全体の進捗をパーセント(%)で表示します。
- **Status**: 成功/失敗の有無を明確に区別して表示します。

## 3. よくある間違い
- ❌ **クライアント検証のみを信頼** → サーバーで必ず再検証(MIME・サイズ・マジックナンバー)。クライアント検証は UX 補助にすぎず、簡単に回避される。
- ❌ 拡張子(`file.name`)だけでタイプを判断 → 偽造可能。実際のコンテンツ(マジックナンバー/`Content-Type`)で確認。
- ❌ 元のファイル名をそのまま保存 → パス traversal・上書き・エンコーディングの問題。サーバーが生成した識別子(UUID など)で保存。
- ❌ 同期アップロードで大容量を送信 → タイムアウト・UI の停止。マルチパート/チャンク + 非同期キュー。
- ❌ 進捗率・失敗状態の未表示 → ユーザーが停止/失敗を知れない。Queue・Progress・Status を露出。
- ❌ 画像の再エンコード・EXIF 除去なしで信頼 → 悪性ペイロード・位置情報(EXIF)の漏洩。
- ❌ サイズ上限をクライアントだけに置く → サーバー・ゲートウェイにも上限を設定して回避を防ぐ。

## 4. チェックリスト
- [ ] アップロード前に `file.size` で容量制限を検証する
- [ ] `allowedTypes` で許可されたファイル形式のみ通過させる
- [ ] ファイル選択の直後に即座に検証結果をユーザーに案内する
- [ ] 複数アップロード時に待ち Queue を UI で露出する
- [ ] 個別/全体の進捗をパーセント(%)で表示する
- [ ] 各ファイルの成功/失敗 Status を明確に区別して表示する
