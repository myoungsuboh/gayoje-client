# 🏗️ 프로젝트 아키텍처

> 이 프로젝트의 일관된 구조를 유지하기 위한 지침을 제공한다. 새 파일·모듈을 추가하거나 디렉토리 위치를 결정할 때 읽는다.

## 1. 핵심 원칙
- `@core`는 수정보다 상속/확장하여 사용한다.
- 레이아웃, API, 플러그인, 유틸리티는 각각 정해진 디렉토리에서만 관리한다.
- 모든 API 호출 함수는 `Api` 접미사로 export한다.
- 컴포넌트는 재사용 단위(`src/components`)와 페이지 단위(`src/views`)로 구분한다.

## 2. 규칙

### 2-1. 주요 디렉토리 구조
- **`@core`**: 템플릿 레이아웃과 핵심 유틸리티가 위치합니다. 기본적으로 수정보다는 상속/확장하여 사용합니다.
- **`src/@layouts`**: 레이아웃 컴포넌트(Header, Sidebar, Footer 등) 전용 공간입니다.
- **`src/api`**: 모든 외부 API 호출 함수를 모듈별로 관리합니다.
- **`src/plugins`**: Vite, Vue, Iconify 등 각종 플러그인 설정이 위치합니다.
- **`src/utils`**: 비즈니스 로직과 무관한 순수 유틸리티 함수(Date 포맷팅 등)를 관리합니다.

### 2-2. 모듈 관리 원칙
- 모든 API 호출은 개별 파일별로 `Api` 접미사를 붙여 export 합니다.
- 컴포넌트는 `src/components`와 `src/views`로 나누어 재사용성과 페이지 단위를 구분합니다.

## 3. 흔한 실수
- ❌ `@core`를 직접 수정 → 템플릿 업데이트 시 충돌. 상속/확장으로 쓴다.
- ❌ API 호출을 컴포넌트에 인라인 → `src/api` 모듈로 분리하고 `Api` 접미사로 export.
- ❌ 재사용 컴포넌트와 페이지를 한 폴더에 → `components`(재사용)/`views`(페이지) 구분.
- ❌ `utils`에 비즈니스 로직 혼입 → 순수 유틸만 두고 도메인 로직은 분리.
- ❌ 모듈 간 순환 의존 → 단방향 의존을 유지한다.

## 4. 체크리스트
- [ ] `@core`를 직접 수정하지 않고 상속/확장으로 사용했는가
- [ ] 레이아웃 컴포넌트를 `src/@layouts`에 두었는가
- [ ] API 호출 함수를 `src/api`에 두고 `Api` 접미사로 export했는가
- [ ] 플러그인 설정을 `src/plugins`에 두었는가
- [ ] 순수 유틸리티 함수를 `src/utils`에 두었는가
- [ ] 컴포넌트를 재사용용(`src/components`)·페이지용(`src/views`)으로 구분했는가
