# 🏗️ プロジェクトアーキテクチャ

> このプロジェクトの一貫した構造を維持するための指針を提供する。新しいファイル・モジュールを追加したり、ディレクトリの配置を決めたりするときに読む。

## 1. 基本原則
- `@core`は修正よりも継承/拡張して使用する。
- レイアウト、API、プラグイン、ユーティリティはそれぞれ定められたディレクトリでのみ管理する。
- すべてのAPI呼び出し関数は`Api`接尾辞でexportする。
- コンポーネントは再利用単位(`src/components`)とページ単位(`src/views`)に分ける。

## 2. ルール

### 2-1. 主要なディレクトリ構造
- **`@core`**: テンプレートレイアウトとコアユーティリティが配置されます。基本的に修正よりも継承/拡張して使用します。
- **`src/@layouts`**: レイアウトコンポーネント(Header, Sidebar, Footerなど)専用のスペースです。
- **`src/api`**: すべての外部API呼び出し関数をモジュール別に管理します。
- **`src/plugins`**: Vite, Vue, Iconifyなど各種プラグイン設定が配置されます。
- **`src/utils`**: ビジネスロジックと無関係な純粋なユーティリティ関数(Dateフォーマットなど)を管理します。

### 2-2. モジュール管理原則
- すべてのAPI呼び出しは個別ファイルごとに`Api`接尾辞を付けてexportします。
- コンポーネントは`src/components`と`src/views`に分けて、再利用性とページ単位を区別します。

## 3. よくある間違い
- ❌ `@core`を直接修正 → テンプレート更新時に衝突。継承/拡張で使う。
- ❌ API呼び出しをコンポーネントにインライン → `src/api`モジュールに分離し、`Api`接尾辞でexport。
- ❌ 再利用コンポーネントとページを同じフォルダに → `components`(再利用)/`views`(ページ)を区別。
- ❌ `utils`にビジネスロジックを混入 → 純粋なユーティリティのみを置き、ドメインロジックは分離。
- ❌ モジュール間の循環依存 → 単方向の依存を維持する。

## 4. チェックリスト
- [ ] `@core`を直接修正せず継承/拡張で使用したか
- [ ] レイアウトコンポーネントを`src/@layouts`に置いたか
- [ ] API呼び出し関数を`src/api`に置き、`Api`接尾辞でexportしたか
- [ ] プラグイン設定を`src/plugins`に置いたか
- [ ] 純粋なユーティリティ関数を`src/utils`に置いたか
- [ ] コンポーネントを再利用用(`src/components`)・ページ用(`src/views`)に区別したか
