# 🤖 AIソリューション連携 (AI Integration)

> LLMおよびAI機能をフロントエンドに統合する際に従う設計標準です。AI API連携、ストリーミング応答UI、プロンプト管理構造を設計するときに読みます。

## 1. 核心原則

- **AIに渡すデータは構造化されたJSONに整形**し、ページの状態・過去の活動をコンテキストとして含める。
- **長い応答はストリーミング(SSE / Fetch HTTP Streaming)で処理**し、待ち時間を解消する。
- **プロンプトはビジネスロジックに直接書かず、専用の管理ファイルへ分離**する。

## 2. ルール

### 2-1. データパイプライン

- **データ整形**: UIで生成されたデータを、AIが理解しやすい構造化されたJSON形式に変換してAPIへ渡します。
- **コンテキスト抽出**: 現在のページの状態やユーザーの過去の活動履歴をコンテキストとして含め、回答の正確性を高めます。

### 2-2. ストリーミング処理 (Streaming Response)

- LLMの長い応答待ち時間を解消するため、SSE(Server-Sent Events)またはFetch HTTP Streaming APIによる処理を推奨します。
- リアルタイムで更新されるテキストを処理するための専用Composableを実装して使用します。

### 2-3. プロンプト管理

- ビジネスロジック内にプロンプト文字列を直接書かず、専用の管理ファイル(`プロンプト専用モジュール`)などを通じて体系的に管理します。

## 3. よくある間違い
- ❌ **APIキーをフロントエンドに露出** → バックエンドプロキシを経由し、キーはサーバーのみに置く。
- ❌ ストリーミング中の取り消し(abort)未実装 → ユーザーが画面を離れてもリクエスト・トークンが消費され続ける。`AbortController`で中断する。
- ❌ エラー・タイムアウト・rate limit(429)の未処理 → 空白画面。リトライ・フォールバックメッセージで処理する。
- ❌ LLM応答をそのままマークダウン/HTMLレンダリング → XSS。出力はサニタイズする。
- ❌ プロンプトをコンポーネントにハードコーディング → 専用モジュールへ分離(バージョン・多言語・再利用)。
- ❌ コンテキストの無限蓄積 → トークン上限超過・コスト急増。ウィンドウ制限・要約する。
- ❌ ローディング表示なしの長い待機 → ストリーミングトークン表示またはスケルトンで体感待ち時間を減らす。
