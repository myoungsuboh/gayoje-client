# 🤖 AI 솔루션 연동 (AI Integration)

> LLM 및 AI 기능을 프론트엔드에 통합할 때 따르는 설계 표준입니다. AI API 연동, 스트리밍 응답 UI, 프롬프트 관리 구조를 설계할 때 읽습니다.

## 1. 핵심 원칙

- **AI에 넘기는 데이터는 구조화된 JSON으로 정제**하고, 페이지 상태·이전 활동을 컨텍스트로 포함한다.
- **긴 응답은 스트리밍(SSE / Fetch HTTP Streaming)으로 처리**해 대기 시간을 해소한다.
- **프롬프트는 비즈니스 로직에 직접 작성하지 않고 전용 관리 파일로 분리**한다.

## 2. 규칙

### 2-1. 데이터 파이프라인

- **데이터 정제**: UI에서 생성된 데이터를 AI가 이해하기 쉬운 구조화된 JSON 형태로 변환하여 API에 전달합니다.
- **컨텍스트 추출**: 현재 페이지의 상태나 사용자의 이전 활동 기록을 컨텍스트로 포함시켜 답변의 정확도를 높입니다.

### 2-2. 스트리밍 처리 (Streaming Response)

- LLM의 긴 응답 대기 시간을 해소하기 위해 SSE(Server-Sent Events) 또는 Fetch HTTP Streaming API 처리를 권장합니다.
- 실시간으로 갱신되는 텍스트를 처리하기 위한 전용 Composable을 구현하여 사용합니다.

### 2-3. 프롬프트 관리

- 비즈니스 로직 내에 프롬프트 문자열을 직접 작성하지 않고, 전용 관리 파일(`프롬프트 전용 모듈`) 등을 통해 체계적으로 관리합니다.

## 3. 흔한 실수
- ❌ **API 키를 프론트엔드에 노출** → 백엔드 프록시를 경유하고 키는 서버에만 둔다.
- ❌ 스트리밍 중 취소(abort) 미구현 → 사용자가 화면을 떠나도 요청·토큰이 계속 소모. `AbortController`로 중단.
- ❌ 에러·타임아웃·rate limit(429) 미처리 → 빈 화면. 재시도·폴백 메시지로 처리.
- ❌ LLM 응답을 그대로 마크다운/HTML 렌더 → XSS. 출력은 새니타이즈한다.
- ❌ 프롬프트를 컴포넌트에 하드코딩 → 전용 모듈로 분리(버전·다국어·재사용).
- ❌ 컨텍스트 무한 누적 → 토큰 한도 초과·비용 급증. 윈도우 제한·요약.
- ❌ 로딩 표시 없이 긴 대기 → 스트리밍 토큰 표시 또는 스켈레톤으로 체감 대기를 줄인다.

## 4. 체크리스트

- [ ] AI에 전달하는 데이터를 구조화된 JSON으로 정제했는가
- [ ] 페이지 상태/사용자 활동 기록을 컨텍스트로 포함했는가
- [ ] 긴 응답을 SSE 또는 Fetch HTTP Streaming으로 처리했는가
- [ ] 실시간 텍스트 갱신용 전용 Composable을 구현했는가
- [ ] 프롬프트 문자열을 전용 관리 파일(`프롬프트 전용 모듈` 등)로 분리했는가
