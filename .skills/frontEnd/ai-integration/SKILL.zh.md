# 🤖 AI 解决方案集成 (AI Integration)

> 将 LLM 及 AI 功能集成到前端时所遵循的设计标准。在设计 AI API 集成、流式响应 UI、提示词管理结构时阅读。

## 1. 核心原则

- **传递给 AI 的数据要整理为结构化 JSON**，并将页面状态·先前活动作为上下文包含进去。
- **长响应使用流式（SSE / Fetch HTTP Streaming）处理**以消除等待时间。
- **不要在业务逻辑中直接编写提示词，应分离到专用的管理文件中**。

## 2. 规则

### 2-1. 数据管道

- **数据整理**：将 UI 中生成的数据转换为 AI 易于理解的结构化 JSON 形式，传递给 API。
- **上下文提取**：将当前页面的状态或用户先前的活动记录作为上下文包含进去，以提高回答的准确性。

### 2-2. 流式处理 (Streaming Response)

- 为消除 LLM 长响应的等待时间，推荐使用 SSE（Server-Sent Events）或 Fetch HTTP Streaming API 处理。
- 实现并使用专用 Composable 来处理实时更新的文本。

### 2-3. 提示词管理

- 不要在业务逻辑内直接编写提示词字符串，而是通过专用的管理文件（`提示词专用模块`）等进行系统化管理。

## 3. 常见错误
- ❌ **在前端暴露 API 密钥** → 经由后端代理，密钥仅放在服务器上。
- ❌ 流式过程中未实现取消（abort） → 即使用户离开页面，请求·令牌仍持续消耗。用 `AbortController` 中断。
- ❌ 未处理错误·超时·rate limit（429） → 空白页面。用重试·回退消息处理。
- ❌ 将 LLM 响应直接渲染为 markdown/HTML → XSS。输出要做净化处理。
- ❌ 将提示词硬编码到组件中 → 分离到专用模块（版本·多语言·复用）。
- ❌ 上下文无限累积 → 超出令牌上限·成本激增。限制窗口·摘要化。
- ❌ 无加载提示的长时间等待 → 用流式令牌显示或骨架屏减少体感等待。

## 4. 检查清单

- [ ] 是否将传递给 AI 的数据整理为结构化 JSON
- [ ] 是否将页面状态/用户活动记录作为上下文包含进去
- [ ] 是否用 SSE 或 Fetch HTTP Streaming 处理长响应
- [ ] 是否实现了实时文本更新的专用 Composable
- [ ] 是否将提示词字符串分离到专用管理文件（`提示词专用模块`等）
