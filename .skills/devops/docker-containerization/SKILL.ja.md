# 📦 コンテナ化標準 (Containerization)

> 小さく安全で再現可能なコンテナイメージを作り、一貫したローカル開発環境を提供する。サービスをイメージにパッケージングする、またはイメージサイズ・攻撃面・シークレット注入を決めるときに読む。特定のビルドツール/ランタイム(Docker、Podman、Buildah、containerd など)に依存しない OCI 汎用標準である。

## 1. 核心原則
- **イメージは小さく**: 最終イメージにはアプリケーションを実行するのに必要なものだけを残す。ビルドツール・ソースコード・キャッシュ・ドキュメントはランタイムイメージから除去し、サイズと攻撃面を減らす。
- **ビルドとランタイムを分離(マルチステージ)**: ビルド段階と実行段階を分け、コンパイラ・パッケージマネージャ・中間生成物が最終イメージに漏れないようにする。
- **non-root で実行**: コンテナプロセスは root ではなく専用の非特権ユーザーで動かす。コンテナエスケープ・権限昇格の被害範囲を減らす。
- **レイヤーキャッシュを意図的に活用**: あまり変わらないもの(依存関係マニフェスト)を先に、よく変わるもの(ソースコード)を後にコピーしてキャッシュ命中率を高める。一度に全部コピーするとコード一行の変更でも依存関係を再取得する。
- **ビルドコンテキストを減らす**: 無視ファイル(`.dockerignore` など)で VCS メタデータ・依存関係ディレクトリ・シークレット・テストファイルをビルドコンテキストから除外する。ビルドが速くなり機微なファイルがイメージに紛れない。
- **タグは不変(immutable)に**: `latest` のような浮動タグの代わりに具体的なバージョンまたはダイジェスト(SHA)で固定し、再現可能なビルドを保証する。
- **シークレットはランタイムに注入**: API キー・パスワードをイメージレイヤー(ビルド引数/環境変数)に焼き込まない。ランタイム環境変数・シークレットマウントで外部から注入する — イメージレイヤーは誰でも覗ける。
- **イメージは不変・使い捨てに**: 実行中のコンテナを修正して状態を溜めない。状態は外部(ボリューム・DB)に置き、イメージはいつ破棄して立て直しても同じ動作になるようにする。
- **脆弱性をスキャンする**: ベースイメージと依存関係の既知の脆弱性(CVE)をビルドパイプラインでスキャンし、ベースイメージを定期的に更新する。

## 2. ルール

### 2-1. マルチステージでビルド生成物だけを残す
- ビルド段階でコンパイル/バンドルし、ランタイム段階にはその生成物と実行依存関係だけをコピーする。
- コンパイラ・ビルドキャッシュ・ソースツリーを最終イメージに残さない。

```text
# ❌ 禁止 — 単一ステージにビルドツール・ソース・キャッシュが全部残る
build-tools + source + deps + artifact  → 最終イメージ (巨大・攻撃面大)

# ✅ 推奨 — ビルド生成物だけをランタイムステージにコピー
stage(build):   build-tools + source → artifact
stage(runtime): runtime-deps + artifact   # ビルドツール・ソースなし
```

### 2-2. non-root ユーザーで実行する
- 専用の非特権ユーザーを作り、そのユーザーでプロセスを起動する。
- デフォルト(root)で動かさない。必要なディレクトリにのみ権限を与える。

```text
# ❌ 禁止 — デフォルト root で実行
(USER 指定なし) → root 権限でアプリ起動

# ✅ 推奨 — 専用の非特権ユーザーで実行
create user appuser → run as appuser
```

### 2-3. レイヤーキャッシュを活かす順序でコピーする
- 依存関係マニフェストを先にコピー・インストールし、その後にソースコードをコピーする。
- ソースだけが変わったときに依存関係インストールレイヤーがキャッシュから再利用されるようにする。

```text
# ❌ 禁止 — 全部一度にコピー → コード一行だけ変えても依存関係を再インストール
copy(everything) → install deps → build

# ✅ 推奨 — マニフェスト先(キャッシュ)、ソースは後
copy(manifest) → install deps   # マニフェストが変わらなければキャッシュ命中
copy(source)   → build
```

### 2-4. ビルドコンテキストから不要・機微なファイルを除外する
- 無視ファイルで VCS メタデータ・依存関係ディレクトリ・環境ファイル・テスト・ドキュメントを除外する。
- シークレットがイメージに紛れる事故を根本から防ぎ、ビルドコンテキストの転送量を減らす。

```text
# ✅ 推奨 — 無視リストに含める (例)
.git              # VCS メタデータ
.env              # シークレット
<deps-dir>        # 再インストールされる依存関係 (例: node_modules, .venv)
build/ dist/      # 生成物
tests/ *.test.*   # テスト
```

### 2-5. イメージタグを不変に固定する
- ベースイメージとデプロイイメージの両方を具体的なバージョンまたはダイジェストで固定する。
- `latest` のように指す対象が変わるタグに依存しない — ビルド再現性が壊れる。

```text
# ❌ 禁止 — 浮動タグ
FROM runtime:latest

# ✅ 推奨 — バージョン/ダイジェストで固定
FROM runtime:20.11.1
FROM runtime@sha256:<digest>
```

### 2-6. シークレットはイメージに焼き込まずランタイムに注入する
- パスワード・トークン・キーをビルド引数やイメージ環境変数に埋め込まない。
- ランタイム環境変数・シークレットマウントで注入する。イメージレイヤーは誰でも抽出して見られる。

```text
# ❌ 禁止 — シークレットがイメージレイヤーに永久に残る
ENV API_KEY=sk-live-12345

# ✅ 推奨 — ランタイムに外部から注入
(イメージには値なし) → 実行時に環境変数/シークレットマウントで渡す
```

### 2-7. ヘルスチェックと読み取り専用ファイルシステムを検討する
- コンテナの「生存」と「リクエスト処理可能」をオーケストレーターが分かるようヘルスチェックを定義する。
- 可能ならルートファイルシステムを読み取り専用にし、書き込みが必要なパスだけをボリューム/tmpfs で開く。

```text
# ✅ 推奨
healthcheck: アプリ状態エンドポイント/コマンドで準備状態を判断
read-only root fs + 書き込みが必要なパスだけマウント
```

### 2-8. ビルドパイプラインで脆弱性をスキャンする
- ベースイメージ・依存関係の既知の脆弱性(CVE)を自動スキャンし、ベースイメージを定期的に更新する。
- スキャンはツール非依存の段階だ — チームが使うスキャナ(Trivy、Grype、クラウドレジストリスキャンなど)で CI に組み込む。

```text
# ✅ 推奨 — CI 段階にスキャンを含める
build image → scan(image) → (脆弱性のしきい値超過時にビルド失敗)
```

## 3. よくある失敗
- **単一ステージでビルド** → コンパイラ・ソース・キャッシュが最終イメージに残り肥大化し攻撃面が大きくなる。マルチステージで生成物だけを残す。
- **root で実行** → コンテナエスケープ時の被害が大きい。専用の非特権ユーザーで動かす。
- **全部一度にコピー** → コード一行変えても依存関係を再取得しビルドが遅くなる。マニフェスト先、ソースは後。
- **`latest` タグ依存** → 昨日通ったビルドが今日壊れる。バージョン/ダイジェストで固定する。
- **シークレットをイメージに焼き込み** → レイヤーから抽出可能。ランタイム注入に変える。
- **無視ファイル欠落** → `.env`・依存関係ディレクトリ・`.git` がビルドコンテキストとイメージに紛れる。無視リストを管理する。
- **ヘルスチェックなし** → オーケストレーターが死んだ/準備できていないコンテナにトラフィックを送る。
- **脆弱性スキャン省略** → 既知の CVE がそのままデプロイされる。CI にスキャンを組み込む。
- **状態をコンテナ内に溜める** → 再起動/再デプロイ時にデータが消える。状態は外部(ボリューム・DB)に。

## 4. チェックリスト
- [ ] **マルチステージビルド**で最終イメージに生成物だけを残したか(ビルドツール・ソース除去)
- [ ] **non-root** 非特権ユーザーで実行するか
- [ ] 依存関係マニフェストを先にコピーして**レイヤーキャッシュ**を活かしたか
- [ ] 無視ファイル(`.dockerignore` など)で**不要・機微なファイルをビルドコンテキストから除外**したか
- [ ] イメージタグが**固定バージョン/ダイジェスト**か(`latest` 禁止)
- [ ] シークレットを**ランタイムに注入**するか(ビルド引数/イメージ環境変数へのハードコード禁止)
- [ ] パッケージインストール時に**キャッシュを除去**してレイヤーを減らしたか
- [ ] **ヘルスチェック**を定義したか
- [ ] **読み取り専用ファイルシステム**を検討したか(書き込みが必要なパスだけ開放)
- [ ] ビルドパイプラインで**脆弱性をスキャン**するか(ベースイメージの定期更新)

## 付録: スタック別の例

> 以下は Docker を基準にした参考実装例だ。チームが使うランタイム/ビルドツール(例: Podman、Buildah、containerd、または別のベースイメージ)に合う例を同じパターンで追加する。上記 1〜4 の原則・ルールが標準であり、付録はその適用事例にすぎない。検証モデル/エラー応答統一などの入力検証は別スキル(`入力値検証標準`)を参照する。

### Docker

#### マルチステージ Dockerfile (Node.js)
```dockerfile
# Stage 1: Build — devDependencies を含む全体をインストールしてビルド
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                       # 全依存関係 (build に devDependencies が必要)
COPY . .
RUN npm run build

# Stage 2: Production — ビルド生成物 + ランタイム依存関係のみ
FROM node:20-alpine AS production
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev            # ランタイム依存関係のみインストール
COPY --from=builder /app/dist ./dist   # ビルド生成物のみコピー
USER appuser
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

#### マルチステージ Dockerfile (Python / FastAPI)
```dockerfile
FROM python:3.12-slim AS base
RUN adduser --disabled-password --no-create-home appuser
WORKDIR /app

# Build — ホイールをビルドしてランタイム依存関係を集める (ビルドツールは最終イメージから除外)
FROM base AS builder
RUN pip install --no-cache-dir build
COPY requirements.txt .
RUN pip wheel --no-cache-dir --wheel-dir /wheels -r requirements.txt

# Production — 事前ビルドしたランタイム依存関係のみインストール
FROM base AS production
COPY --from=builder /wheels /wheels
COPY requirements.txt .
RUN pip install --no-cache-dir --no-index --find-links=/wheels -r requirements.txt
COPY . .
USER appuser
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### Docker Compose (ローカル開発)
```yaml
services:
  app:
    build: { context: ., target: development }
    volumes: ["./src:/app/src"]   # Hot reload
    environment:
      DATABASE_URL: postgresql://postgres:password@db:5432/mydb
    depends_on:
      db: { condition: service_healthy }

  db:
    image: postgres:16-alpine
    environment: { POSTGRES_PASSWORD: password }
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      retries: 5
    volumes: ["postgres-data:/var/lib/postgresql/data"]

volumes:
  postgres-data:
```
