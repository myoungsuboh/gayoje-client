# 🚨 故障响应 & 事后复盘 (Incident Response)

> 快速缓解和恢复生产故障，结束后用无指责的复盘让同样的事情不再发生。当故障发生时、搭建值班体系时、撰写事后复盘时阅读。

故障响应陷入混乱的原因大多相同 — **不清楚谁在指挥，先去追查原因导致恢复延误，结束后责怪人而留不下真正的教训。** 好的响应不是来自一位英雄，而是来自**既定的角色、优先级、记录**。两点核心: 故障中**缓解先于根因**，故障后**修系统而不责怪人**。检测与可观测性基础参见 `observability`，恢复设计参见 `error-handling-resilience`。

## 1. 核心原则

- **按严重级别(SEV)确定响应规模** — 不以相同强度对待所有故障。级别决定升级、告警、召集的范围。
- **分离角色** — 划分指挥(Incident Commander)、沟通(Comms)、作业(Ops)、记录(Scribe)。一个人边修边报告边记录会让两者都滞后 — 尤其时间线记录不要由 IC 兼任，交给 Scribe(让指挥专注)。
- **缓解先于根因** — 先恢复服务(回滚、故障转移、关闭功能)。根因分析(RCA)在灭火之后再做。
- **实时记录时间线** — 留下发生、检测、缓解、解决的时刻以及所采取的行动。记忆不可信。
- **事后复盘不指责** — 修复的是允许该错误发生的系统与流程，而非犯错的人。惩罚会让人下次隐瞒。
- **把教训变成可追踪的行动** — 把复盘的改进措施变成有负责人和期限的工单。把重复出现的响应文档化为运行手册。

## 2. 规则

### 2-1. 先定义严重级别(SEV)

各组织有所不同，但要以影响为基准，事先就级别和响应达成一致。

| 级别 | 影响(示例) | 响应 |
|---|---|---|
| **SEV1** | 全面中断、数据丢失、安全入侵 | 立即召集，向高层报告，全员集中 |
| **SEV2** | 核心功能下降、部分用户受影响 | 值班+负责团队，快速缓解 |
| **SEV3** | 轻微、可绕过、非核心 | 在正常工作时间内处理 |

- 级别以**影响(用户、营收、数据)**为基准来评定。不是"原因大不大"而是"损害大不大"。
- 部分组织会进一步细分到 SEV4、5(轻微、信息性)。**安全入侵**涉及证据保全、法务、监管上报，流程与一般故障不同，因此按**单独的轨道**处理(与 `owasp-top10` 及安全负责人一起)。

### 2-2. 分离角色 (禁止一人多职)

```text
❌ 禁止 — 一个人同时调试 + 状态报告 + 决策 → 两者都延误/遗漏
✅ 推荐 — Incident Commander(指挥/决策) / Comms(内部/客户通告) / Ops(实际处置) / Scribe(时间线记录)
         小故障可以兼任，但指挥者始终明确为一人
```

### 2-3. 缓解先于根因

```text
❌ 禁止 — 花30分钟调试"为什么会这样?" → 这期间用户持续遭遇故障
✅ 推荐 — 先恢复: 回滚上一次部署 / 把流量故障转移 / 关闭出问题的功能(feature flag)
         服务恢复后再进行根因分析(RCA) (回滚见 `git-workflow`，数据丢失、故障转移、恢复目标(RTO/RPO)见 `backup-dr`)
```

### 2-4. 实时时间线 + 状态共享

```text
✅ 在一个频道里带时刻只记录事实:
   13:58 部署新支付模块          (发生)
   14:02 支付错误率激增告警      (检测)
   14:08 判断为上一次部署，开始回滚 (开始缓解)
   14:15 确认错误率恢复正常          (解决)
- 内部/客户通告由 Comms 按既定周期发出。即使"调查中"也好过沉默。
```

### 2-5. 无指责的事后复盘 — PIR (SEV1、2 为必须)

即使原因是某人的行为，要修复的对象也是**使该行为成为可能的系统**。

```text
事后复盘模板(推荐):
- 影响: 什么、多少、对谁 (持续时间、用户数)
- 时间线: 发生→检测→缓解→解决
- 根因/贡献因素: 真正的原因而非表面症状 (贡献因素通常是多个 — 禁止寻找单一罪魁，`systematic-debugging`)
- 做得好的点 / 运气好的点 / 遗憾的点
- 行动项: 有负责人和期限的改进措施 (防止再次发生)

❌ 禁止 — 以"A 犯了错"结束 → 下次会隐瞒错误
✅ 推荐 — 追问"为什么一个人的错误蔓延成全面故障?"并增加护栏
```

### 2-6. 行动项追踪 + 运行手册化

```text
❌ 禁止 — 只写事后复盘文档而行动项不了了之
✅ 推荐 — 把行动项登记为工单(负责人、期限)并追踪到完成
         把重复出现的响应流程整理成运行手册: 症状 → 检查命令 → 处置步骤
```

## 3. 常见错误

- ❌ 没有指挥者而多人同时动手，彼此冲突(重复回滚、相互矛盾的处置)
- ❌ 先做根因查明而非缓解，导致故障时长(MTTR=平均恢复时间)增加
- ❌ 没有严重级别标准，琐碎之事召集全员 / 重大之事响应拖沓
- ❌ 未记录时间线 → 复盘时无法重建"何时做了什么"
- ❌ 责怪人的复盘 → 信任与心理安全崩塌，下一次事故被隐瞒
- ❌ 不追踪行动项，导致同样的故障再次发生
- ❌ 故障中保持沉默 → 客户与内部的不安与猜测被放大

> **应用提示**: 在故障发生之前，事先就 SEV 级别、角色、联络体系、基本运行手册达成一致。无法在半夜临时制定定义。检测与告警基础见 `observability`，弹性设计(重试、断路器)见 `error-handling-resilience`，数据恢复与故障转移见 `backup-dr`。

## 4. 检查清单

- [ ] 是否事先定义了严重级别(SEV)及各级别的响应
- [ ] 故障时指挥(IC)、沟通、作业角色是否分离 (指挥者明确为1人)
- [ ] 是否先做服务缓解/恢复而非根因查明
- [ ] 是否实时记录了发生、检测、缓解、解决的时间线
- [ ] 是否为 SEV1、2 撰写了无指责的事后复盘 (影响、时间线、根因、行动)
- [ ] 行动项是否以有负责人和期限的工单追踪，重复响应是否运行手册化
