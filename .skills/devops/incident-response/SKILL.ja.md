# 🚨 障害対応 & ポストモーテム (Incident Response)

> 本番障害を素早く緩和・復旧し、終わった後は非難なき振り返りで同じことが二度と起きないようにする。障害が発生したとき、オンコール体制を整えるとき、事後の振り返りを書くときに読む。

障害対応が混乱に陥る理由はたいてい同じだ — **誰が指揮するのか不明確で、原因から探って復旧が遅れ、終わった後に人を責めて本当の教訓を残せない。** よい対応は一人の英雄ではなく、**定められた役割・優先順位・記録**から生まれる。核心は二つ: 障害中は **緩和が根本原因より先**、障害後は **システムを直すが人を責めない**。検知・観測の基盤は `observability`、回復設計は `error-handling-resilience` と併せて見る。

## 1. 基本原則

- **深刻度(SEV)で対応規模を決める** — すべての障害を同じ強度で扱わない。等級がエスカレーション・通知・招集範囲を決める。
- **役割を分離する** — 指揮(Incident Commander)・コミュニケーション(Comms)・作業(Ops)・記録(Scribe)に分ける。一人が修正しながら報告・記録までやると両方遅れる — 特にタイムライン記録は IC が兼ねず Scribe に任せる(指揮に集中)。
- **緩和が根本原因より先** — まずサービスを回復させる(ロールバック・フェイルオーバー・機能オフ)。原因分析(RCA)は火を消してから。
- **タイムラインをリアルタイムで記録する** — 発生・検知・緩和・解決の時刻と取った行動を残す。記憶は信用できない。
- **ポストモーテムは非難なしで** — 人のミスではなく、それを許したシステム・プロセスを直す。罰すると次は隠す。
- **教訓を追跡可能なアクションに** — 振り返りの改善策を担当・期限付きのチケットに。繰り返される対応はランブックとして文書化する。

## 2. ルール

### 2-1. 深刻度(SEV)等級をまず定義する

組織ごとに異なるが、影響を基準に等級と対応を事前に合意する。

| 等級 | 影響(例) | 対応 |
|---|---|---|
| **SEV1** | 全体停止・データ損失・セキュリティ侵害 | 即時招集、経営層への報告、全員集中 |
| **SEV2** | 主要機能の低下・一部ユーザーへの影響 | オンコール+担当チーム、迅速な緩和 |
| **SEV3** | 軽微・回避可能・非中核 | 通常時間内に処理 |

- 等級は **影響(ユーザー・売上・データ)** を基準に付ける。「原因が大きいか」ではなく「被害が大きいか」。
- 組織によっては SEV4・5(軽微・情報提供)まで細分化することもある。**セキュリティ侵害**は証拠保全・法務・規制報告が絡み、一般障害と手順が異なるため **別トラック**で扱う(`owasp-top10`・セキュリティ担当と共に)。

### 2-2. 役割を分離する (一人多役の禁止)

```text
❌ 禁止 — 一人がデバッグ + 状態報告 + 意思決定を同時に → 両方とも遅延・漏れ
✅ 推奨 — Incident Commander(指揮・決定) / Comms(社内・顧客告知) / Ops(実際の対応) / Scribe(タイムライン記録)
         小さな障害は兼任できるが、指揮者は常に一人で明確に
```

### 2-3. 緩和を根本原因より先に行う

```text
❌ 禁止 — 「なぜこうなった?」というデバッグに30分 → その間ユーザーはずっと障害
✅ 推奨 — まず回復: 直前のデプロイをロールバック / トラフィックをフェイルオーバー / 問題の機能をオフ(feature flag)
         サービス復旧後に根本原因分析(RCA)を進める (ロールバックは `git-workflow`、データ損失・フェイルオーバー・復旧目標(RTO/RPO)は `backup-dr`)
```

### 2-4. リアルタイムタイムライン + 状態共有

```text
✅ 一つのチャンネルに時刻とともに事実だけを記録:
   13:58 新規決済モジュールをデプロイ          (発生)
   14:02 決済エラー率の急増アラート      (検知)
   14:08 直前のデプロイと判断、ロールバック開始 (緩和着手)
   14:15 エラー率の正常化を確認          (解決)
- 社内/顧客告知は Comms が定められた周期で。「調査中」でも沈黙よりはよい。
```

### 2-5. 非難なきポストモーテム — PIR (SEV1・2は必須)

原因が人の行動であっても、直す対象は **その行動を可能にしたシステム**だ。

```text
ポストモーテム様式(推奨):
- 影響: 何が・どれだけ・誰に (継続時間、ユーザー数)
- タイムライン: 発生→検知→緩和→解決
- 根本原因・寄与要因: 表面の症状ではなく本当の原因 (たいてい寄与要因は複数 — 単一の犯人探しは禁止、`systematic-debugging`)
- よかった点 / 運がよかった点 / 残念だった点
- アクションアイテム: 担当・期限付きの改善策 (再発防止)

❌ 禁止 — 「Aがミスした」で終わり → 次はミスを隠す
✅ 推奨 — 「なぜ一人のミスが全体障害に広がったのか?」を問いガードレールを追加
```

### 2-6. アクションアイテム追跡 + ランブック化

```text
❌ 禁止 — ポストモーテム文書だけ書いてアクションアイテムがうやむや
✅ 推奨 — アクションをチケット(担当・期限)として登録し完了まで追跡
         繰り返される対応手順はランブックに: 症状 → 確認コマンド → 対応ステップ
```

## 3. よくある間違い

- ❌ 指揮者なしで複数が同時に手を出し互いに衝突(重複ロールバック・相反する対応)
- ❌ 緩和より根本原因の究明を先に行い障害時間(MTTR=平均復旧時間)が増加
- ❌ 深刻度基準がなく些細なことに全員招集 / 大事に対応が遅い
- ❌ タイムライン未記録 → 振り返り時に「いつ何をしたか」を再構成できない
- ❌ 人を責める振り返り → 信頼・心理的安全の崩壊、次の事故は隠蔽
- ❌ アクションアイテムを追跡せず同じ障害が再発
- ❌ 障害中の沈黙 → 顧客・社内の不安と憶測が増幅

> **適用のヒント**: 障害が起きる前に SEV 等級・役割・連絡体制・基本ランブックを事前に合意しておく。真夜中に定義を作ることはできない。検知・通知の基盤は `observability`、回復力設計(リトライ・サーキットブレーカー)は `error-handling-resilience`、データ復旧・フェイルオーバーは `backup-dr`。
