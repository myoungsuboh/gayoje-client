# 🚨 장애 대응 & 포스트모템 (Incident Response)

> 운영 장애를 빠르게 완화·복구하고, 끝난 뒤 비난 없는 회고로 같은 일이 다시 안 일어나게 한다. 장애가 터졌을 때, 온콜 체계를 세울 때, 사고 후 회고를 쓸 때 읽는다.

장애 대응이 혼란스러워지는 이유는 대개 같다 — **누가 지휘하는지 불명확하고, 원인부터 캐다가 복구가 늦고, 끝난 뒤 사람을 탓하다 진짜 교훈을 못 남긴다.** 좋은 대응은 영웅 한 명이 아니라 **정해진 역할·우선순위·기록**에서 나온다. 핵심 둘: 장애 중엔 **완화가 근본원인보다 먼저**, 장애 후엔 **시스템을 고치되 사람을 탓하지 않는다**. 탐지·관측 기반은 `observability`, 회복 설계는 `error-handling-resilience`와 함께 본다.

## 1. 핵심 원칙

- **심각도(SEV)로 대응 규모를 정한다** — 모든 장애를 같은 강도로 다루지 않는다. 등급이 에스컬레이션·알림·소집 범위를 결정한다.
- **역할을 분리한다** — 지휘(Incident Commander)·소통(Comms)·작업(Ops)·기록(Scribe)을 나눈다. 한 명이 고치면서 보고·기록까지 하면 둘 다 늦는다 — 특히 타임라인 기록은 IC가 겸하지 말고 Scribe에게 맡긴다(지휘 집중).
- **완화가 근본원인보다 먼저** — 먼저 서비스를 회복시킨다(롤백·페일오버·기능 끄기). 원인 분석(RCA)은 불 끄고 나서.
- **타임라인을 실시간 기록한다** — 발생·탐지·완화·해결 시각과 한 행동을 남긴다. 기억은 못 믿는다.
- **포스트모템은 비난 없이** — 사람의 실수가 아니라 그것을 허용한 시스템·프로세스를 고친다. 처벌하면 다음엔 숨긴다.
- **교훈을 추적 가능한 액션으로** — 회고의 개선책을 담당·기한 있는 티켓으로. 반복되는 대응은 런북으로 문서화한다.

## 2. 규칙

### 2-1. 심각도(SEV) 등급을 먼저 정의한다

조직마다 다르지만, 영향 기준으로 등급과 대응을 미리 합의한다.

| 등급 | 영향(예시) | 대응 |
|---|---|---|
| **SEV1** | 전체 중단·데이터 유실·보안 침해 | 즉시 소집, 경영진 보고, 전원 집중 |
| **SEV2** | 핵심 기능 저하·일부 사용자 영향 | 온콜+담당팀, 신속 완화 |
| **SEV3** | 경미·우회 가능·비핵심 | 정규 시간 내 처리 |

- 등급은 **영향(사용자·매출·데이터)** 기준으로 매긴다. "원인이 큰가"가 아니라 "피해가 큰가".
- 조직에 따라 SEV4·5(경미·정보성)까지 세분하기도 한다. **보안 침해**는 증거 보전·법무·규제 신고가 얽혀 일반 장애와 절차가 다르므로 **별도 트랙**으로 다룬다(`owasp-top10`·보안 담당과 함께).

### 2-2. 역할을 분리한다 (1인 다역 금지)

```text
❌ 금지 — 한 사람이 디버깅 + 상태 보고 + 의사결정 동시에 → 둘 다 지연·누락
✅ 권장 — Incident Commander(지휘·결정) / Comms(내부·고객 공지) / Ops(실제 조치) / Scribe(타임라인 기록)
         작은 장애는 겸할 수 있으나, 지휘자는 항상 한 명으로 명확히
```

### 2-3. 완화를 근본원인보다 먼저 한다

```text
❌ 금지 — "왜 이렇게 됐지?" 디버깅에 30분 → 그동안 사용자는 계속 장애
✅ 권장 — 먼저 회복: 직전 배포 롤백 / 트래픽 페일오버 / 문제 기능 끄기(feature flag)
         서비스 복구 후 근본 원인 분석(RCA)을 진행 (롤백은 `git-workflow`, 데이터 유실·페일오버·복구 목표(RTO/RPO)는 `backup-dr`)
```

### 2-4. 실시간 타임라인 + 상태 공유

```text
✅ 한 채널에 시각과 함께 사실만 기록:
   13:58 신규 결제 모듈 배포          (발생)
   14:02 결제 오류율 급증 알림      (탐지)
   14:08 직전 배포로 판단, 롤백 시작 (완화 착수)
   14:15 오류율 정상화 확인          (해결)
- 내부/고객 공지는 Comms가 정해진 주기로. "조사 중"이라도 침묵보다 낫다.
```

### 2-5. 비난 없는 포스트모템 — PIR (SEV1·2는 필수)

원인이 사람의 행동이어도, 고치는 대상은 **그 행동을 가능하게 한 시스템**이다.

```text
포스트모템 양식(권장):
- 영향: 무엇이·얼마나·누구에게 (지속 시간, 사용자 수)
- 타임라인: 발생→탐지→완화→해결
- 근본 원인·기여 요인: 표면 증상이 아니라 진짜 원인 (대개 기여 요인은 복수 — 단일 범인 찾기 금지, `systematic-debugging`)
- 잘된 점 / 운이 좋았던 점 / 아쉬운 점
- 액션 아이템: 담당·기한 있는 개선책 (재발 방지)

❌ 금지 — "A가 실수함"으로 끝 → 다음엔 실수를 숨김
✅ 권장 — "왜 한 사람의 실수가 전체 장애로 번졌나?"를 물어 가드레일을 추가
```

### 2-6. 액션 아이템 추적 + 런북화

```text
❌ 금지 — 포스트모템 문서만 쓰고 액션 아이템이 흐지부지
✅ 권장 — 액션을 티켓(담당·기한)으로 등록해 완료까지 추적
         반복되는 대응 절차는 런북으로: 증상 → 확인 명령 → 조치 단계
```

## 3. 흔한 실수

- ❌ 지휘자 없이 여럿이 동시에 손대 서로 충돌(중복 롤백·상충 조치)
- ❌ 완화보다 근본원인 규명을 먼저 하다 장애 시간(MTTR=평균 복구 시간) 증가
- ❌ 심각도 기준이 없어 사소한 일에 전원 소집 / 큰 일에 늑장
- ❌ 타임라인 미기록 → 회고 때 "언제 뭘 했는지" 재구성 불가
- ❌ 사람을 탓하는 회고 → 신뢰·심리적 안전 붕괴, 다음 사고는 은폐
- ❌ 액션 아이템을 추적 안 해 같은 장애가 재발
- ❌ 장애 중 침묵 → 고객·내부의 불안과 추측 증폭

> **적용 팁**: 장애가 나기 전에 SEV 등급·역할·연락 체계·기본 런북을 미리 합의해 둔다. 한밤중에 정의를 만들 순 없다. 탐지·알림 기반은 `observability`, 회복탄력 설계(재시도·회로차단기)는 `error-handling-resilience`, 데이터 복구·페일오버는 `backup-dr`.

## 4. 체크리스트

- [ ] 심각도(SEV) 등급과 등급별 대응이 사전에 정의돼 있는가
- [ ] 장애 시 지휘(IC)·소통·작업 역할이 분리되는가 (지휘자 1명 명확)
- [ ] 근본원인 규명보다 서비스 완화·복구를 먼저 했는가
- [ ] 발생·탐지·완화·해결 타임라인을 실시간 기록했는가
- [ ] SEV1·2에 비난 없는 포스트모템을 작성했는가 (영향·타임라인·근본원인·액션)
- [ ] 액션 아이템이 담당·기한 있는 티켓으로 추적되고, 반복 대응은 런북화됐는가
