# ⚡ 应用体积 & 性能优化

> 这是优化移动应用体积·启动时间·帧·内存·电量的通用标准。不靠猜测而靠测量找到瓶颈,把重活移到延迟·后台,移除未使用的代码·资源,列表·图片进行复用·降采样。在应用沉重或缓慢时、发布前点检·剖析性能时阅读。这是不依赖于特定平台(iOS/Android)或框架的共通概念,平台专有工具在附录中讨论。

## 1. 目的

- 把用户体感的**启动速度·响应性·发热·应用容量**保证在一定水平以上。
- 把性能用**测量值(启动时间·帧·内存·电量)**而非“感觉”来管理,并将发布前点检标准化。
- 把即使平台不同也同样通用的**共通原则**汇聚在一处,并将工具映射为概念。

## 2. 核心原则

- **先测量,禁止猜测**: 优化要在用剖析器确认瓶颈之后进行。不测量就不动手。(分别测量启动时间·内存·帧·电量。)
- **把启动时间(冷启动)放在最优先**: 测量从 进程创建 → 应用初始化 → 首屏渲染 的全过程,并把启动路径上的重初始化移到**延迟(lazy)·后台·按需**。
- **腾空主(UI)线程**: 把解码·解析·I/O·加密等重活从 UI 线程移出,守住**帧预算(60fps ≈ 16ms/frame)**。
- **减小二进制体积**: 移除未使用的代码·资源(代码缩减·混淆),让每个设备只下载所需的内容(应用包/切片),并以高效的格式·分辨率缩小图片。
- **限定内存**: 列表**复用**视图,图片按显示尺寸**降采样**,用剖析器排查泄漏。
- **节省电量**: 最小化后台定位·网络轮询·唤醒,并以**批处理·延迟·合并**来处理任务。
- **共通概念优先,工具做映射**: 原则以平台中立方式确定,实际工具(例如 iOS Instruments·App Thinning、Android Profiler·R8/App Bundle)视为该概念的实现。

## 3. 规则 (✅/❌)

### 3-1. 不测量就不优化

- 先用剖析器确认哪里慢,改完后再次测量以验证效果。

```text
// ❌ 禁止 — 不测量凭“感觉”动手
"列表好像慢,先多加点缓存吧"

// ✅ 推荐 — 测量 → 定位瓶颈 → 修改 → 重新测量
用剖析器确认掉帧点 → 移除原因(主线程解码) → 重新测量确认改善
```

### 3-2. 启动路径上的重初始化移到延迟·后台

- 在冷启动路径(应用初始化 → 首屏)上,不要同步运行眼下不需要的 SDK·配置·缓存预热。
- 只立即做绘制首屏所需的内容,其余推迟到**延迟加载·后台·按需**。

```text
// ❌ 禁止 — 应用启动时同步初始化所有 SDK(首屏变慢)
应用启动:
  analytics.init(); ads.init(); crash.init(); imageCache.warmAll();  // 全部同步
  showFirstScreen()

// ✅ 推荐 — 只做首屏所需,其余延迟/后台
应用启动:
  crash.init()                    // 只做立即必需的最小项
  background { analytics.init(); ads.init() }   // 移到启动路径之外
  lazy { imageCache }             // 首次使用时初始化
  showFirstScreen()
```

### 3-3. 不在主(UI)线程做重活

- 把图片解码·大体量解析·磁盘/网络 I/O·加密之类的工作从 UI 线程移出,防止掉帧。
- 像绑定列表单元格的路径这样**被频繁·快速调用的点**,越要避免放置重活。

```text
// ❌ 禁止 — 在单元格绑定(每帧路径)中同步解码 → 掉帧
bindListItem(item):
  image = decodeFullSizeImage(item.path)   // 阻塞 UI 线程
  view.image = image

// ✅ 推荐 — 后台解码 + 降采样到显示尺寸 + 异步应用
bindListItem(item):
  loadImageAsync(item.path, targetW, targetH) into view.image
```

### 3-4. 二进制体积靠“移除 + 拆分 + 高效格式”

- 在构建阶段移除未使用的代码·资源(代码缩减·混淆),让每个设备只下载所需的资源(应用包/切片)。
- 图片以足够显示的分辨率与高效格式缩小(禁止随附不必要的原图)。

```text
// ❌ 禁止 — 把全设备用资源全塞进一个二进制,未使用的代码也原样保留
所有分辨率的图片 + 所有 ABI/架构 + 含死代码 → 下载容量臃肿

// ✅ 推荐 — 移除·拆分·高效格式
移除未使用代码/资源 + 按设备拆分(包/切片) + 高效的图片格式·分辨率
```

### 3-5. 列表·图片靠复用·降采样限定内存

- 长列表只为屏幕上可见的数量创建视图并**复用**(不要一次性把整体载入内存)。
- 图片不按原图分辨率,而按**实际显示尺寸**降采样,以节省内存·带宽。

```text
// ❌ 禁止 — 一次性创建全部项,并把原图分辨率图片原样放进内存
buildAllRowsAtOnce(items)         // 一次性 1 万个
view.image = load(originalHugeImage)   // 把 4000x3000 放进小单元格

// ✅ 推荐 — 只复用可见数量 + 降采样到显示尺寸
recycleVisibleRows(items)
view.image = load(path, downsampleTo = cellSize)
```

### 3-6. 电量: 后台任务最小化·批处理·延迟

- 最小化后台定位追踪·频繁的网络轮询·反复唤醒。
- 确有必要的任务以**汇总(batch)·延迟·适当的触发(条件/周期)**处理。

```text
// ❌ 禁止 — 以短周期不断唤醒进行轮询·定位采集(发热·耗电)
every 5s: pollServer(); readPreciseLocation()

// ✅ 推荐 — 合并、推迟、并按条件
batched/deferred sync (网络·充电条件等) + 仅以所需精度/周期请求定位
```

## 4. 常见错误

- **不测量就优化** → 改错地方且无法验证效果。先用剖析器找到瓶颈。
- **启动时全部同步初始化** → 冷启动变长。只保留首屏所需,其余延迟·后台。
- **在主线程做解码·解析·I/O** → 滚动·动画卡顿(掉帧)。重活移到后台。
- **把原图分辨率图片原样放进小视图** → 内存激增·OOM。降采样到显示尺寸。
- **一次性创建整个列表** → 内存·卡顿。只为可见数量复用视图。
- **把未使用代码·资源·全设备资源随附进一个二进制** → 下载容量臃肿。移除 + 按设备拆分。
- **滥用后台轮询·定位·唤醒** → 发热·耗电。批处理·延迟·按条件。
- **放任内存泄漏** → 反复进入画面时用量持续上升。用剖析器排查泄漏。

## 5. 检查清单

- [ ] 是否用**剖析器测量**了启动时间·内存·帧·电量,并在修改后重新测量确认效果?
- [ ] 是否测量了冷启动(应用初始化 → 首屏),并把重初始化移到**延迟·后台**?
- [ ] 是否通过把解码·解析·I/O 等重活从主(UI)线程移除来保持 **60fps**?
- [ ] 是否移除了未使用代码·资源(代码缩减·混淆),并应用了按设备**拆分(应用包/切片)**?
- [ ] 是否把图片以**高效格式·显示尺寸**缩小?
- [ ] 列表是否**复用**视图、图片是否**降采样**以限定内存?
- [ ] 是否用剖析器排查了**内存泄漏**?
- [ ] 是否对后台定位·网络轮询·唤醒做了**最小化·批处理·延迟**处理?

> 为不与输入值校验·网络·错误处理等其他标准冲突,也要一并确认性能优化是否破坏了正确性·安全性。

### Android (Kotlin)

#### 启动时间优化

```
冷启动阶段: 进程创建 → Application onCreate → 首屏渲染
目标: < 2秒 (Android Vitals 推荐)

策略:
- 延迟 onCreate 中的重 SDK 初始化 (App Startup·lazy)
- 启动屏 → 首屏数据预取
- 移除主线程阻塞工作
```

#### 二进制体积缩减

技法·工具请参考上面「各平台工具映射」表(二进制拆分 / 移除未使用代码·资源 / 高效图片 各行)。Android 用 App Bundle(AAB)按设备拆分 APK,用 R8/ProGuard 移除未使用代码·资源,用矢量 drawable·WebP 缩小图片。

#### 帧性能 (60fps)

```kotlin
// ❌ 禁止 — 在主线程做重活 (掉帧)
fun onBindViewHolder(...) {
    val bitmap = decodeHugeImage(path)  // 阻塞主线程 → 掉帧
}

// ✅ 推荐 — 后台解码 + 降采样
fun onBindViewHolder(...) {
    Glide.with(context).load(path)
        .override(targetW, targetH)  // 降采样
        .into(holder.image)
}
```
