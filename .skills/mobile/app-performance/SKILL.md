# ⚡ 앱 사이즈 & 성능 최적화

> 모바일 앱의 크기·시작 시간·프레임·메모리·배터리를 최적화하는 범용 표준이다. 추측이 아니라 측정으로 병목을 찾고, 무거운 작업은 지연·백그라운드로 옮기며, 미사용 코드·리소스를 제거하고, 리스트·이미지는 재활용·다운샘플링한다. 앱이 무겁거나 느릴 때, 출시 전 성능을 점검·프로파일링할 때 읽는다. 특정 플랫폼(iOS/Android)이나 프레임워크에 종속되지 않는 공통 개념이며, 플랫폼별 도구는 부록에서 다룬다.

## 1. 목적

- 사용자가 체감하는 **시작 속도·반응성·발열·앱 용량**을 일정 수준 이상으로 보장한다.
- 성능을 "감"이 아니라 **측정값(시작 시간·프레임·메모리·배터리)**으로 관리하고, 출시 전 점검을 표준화한다.
- 플랫폼이 달라도 동일하게 통하는 **공통 원칙**을 한곳에 모으고, 도구는 개념으로 매핑한다.

## 2. 핵심 원칙

- **측정 먼저, 추측 금지**: 최적화는 프로파일러로 병목을 확인한 뒤에 한다. 측정 없이 손대지 않는다. (시작 시간·메모리·프레임·배터리를 각각 측정한다.)
- **시작 시간(콜드 스타트)을 최우선으로**: 프로세스 생성 → 앱 초기화 → 첫 화면 렌더까지를 측정하고, 시작 경로의 무거운 초기화는 **지연(lazy)·백그라운드·온디맨드**로 옮긴다.
- **메인(UI) 스레드를 비운다**: 디코딩·파싱·I/O·암호화 등 무거운 작업을 UI 스레드에서 빼서 **프레임 예산(60fps ≈ 16ms/frame)**을 지킨다.
- **바이너리 크기를 줄인다**: 미사용 코드·리소스를 제거하고(코드 축소·난독화), 기기별로 필요한 것만 내려받게 하며(앱 번들/슬라이싱), 이미지는 효율적 포맷·해상도로 줄인다.
- **메모리를 한정한다**: 리스트는 뷰를 **재활용**하고, 이미지는 표시 크기에 맞게 **다운샘플링**하며, 누수를 프로파일러로 점검한다.
- **배터리를 아낀다**: 백그라운드 위치·네트워크 폴링·웨이크업을 최소화하고, 작업을 **배치·지연·합치기**로 처리한다.
- **공통 개념 우선, 도구는 매핑**: 원칙은 플랫폼 중립으로 정하고, 실제 도구(예: iOS Instruments·App Thinning, Android Profiler·R8/App Bundle)는 그 개념의 구현으로 본다.

## 3. 규칙 (✅/❌)

### 3-1. 측정 없이 최적화하지 않는다

- 어디가 느린지 프로파일러로 먼저 확인하고, 바꾼 뒤 다시 측정해 효과를 검증한다.

```text
// ❌ 금지 — 측정 없이 "느낌"으로 손대기
"리스트가 느린 것 같으니 일단 캐시를 더 붙이자"

// ✅ 권장 — 측정 → 병목 특정 → 수정 → 재측정
프로파일러로 프레임 드롭 지점 확인 → 원인(메인 스레드 디코딩) 제거 → 다시 측정해 개선 확인
```

### 3-2. 시작 경로의 무거운 초기화는 지연·백그라운드로

- 콜드 스타트 경로(앱 초기화 → 첫 화면)에서 당장 필요 없는 SDK·설정·캐시 워밍을 동기로 돌리지 않는다.
- 첫 화면을 그리는 데 필요한 것만 즉시 하고, 나머지는 **지연 로딩·백그라운드·온디맨드**로 미룬다.

```text
// ❌ 금지 — 앱 시작 시 모든 SDK를 동기 초기화 (첫 화면이 늦어짐)
앱 시작:
  analytics.init(); ads.init(); crash.init(); imageCache.warmAll();  // 전부 동기
  showFirstScreen()

// ✅ 권장 — 첫 화면에 필요한 것만, 나머지는 지연/백그라운드
앱 시작:
  crash.init()                    // 즉시 필요한 최소만
  background { analytics.init(); ads.init() }   // 시작 경로 밖으로
  lazy { imageCache }             // 처음 쓸 때 초기화
  showFirstScreen()
```

### 3-3. 메인(UI) 스레드에서 무거운 작업을 하지 않는다

- 이미지 디코딩·대용량 파싱·디스크/네트워크 I/O·암호화 같은 작업은 UI 스레드에서 빼서 프레임 드롭을 막는다.
- 리스트 셀을 바인딩하는 경로처럼 **자주·빠르게 호출되는 지점**일수록 무거운 작업을 두지 않는다.

```text
// ❌ 금지 — 셀 바인딩(매 프레임 경로)에서 동기 디코딩 → 프레임 드롭
bindListItem(item):
  image = decodeFullSizeImage(item.path)   // UI 스레드 블로킹
  view.image = image

// ✅ 권장 — 백그라운드 디코딩 + 표시 크기로 다운샘플 + 비동기 적용
bindListItem(item):
  loadImageAsync(item.path, targetW, targetH) into view.image
```

### 3-4. 바이너리 크기는 "제거 + 분할 + 효율 포맷"으로

- 미사용 코드·리소스를 빌드 단계에서 제거하고(코드 축소·난독화), 기기별로 필요한 리소스만 내려받게 한다(앱 번들/슬라이싱).
- 이미지는 표시에 충분한 해상도와 효율적 포맷으로 줄인다(불필요한 원본 동봉 금지).

```text
// ❌ 금지 — 전 기기용 리소스를 한 바이너리에 다 담고, 미사용 코드도 그대로
모든 해상도 이미지 + 모든 ABI/아키텍처 + 죽은 코드 포함 → 다운로드 용량 비대

// ✅ 권장 — 제거·분할·효율 포맷
미사용 코드/리소스 제거 + 기기별 분할(번들/슬라이싱) + 효율적 이미지 포맷·해상도
```

### 3-5. 리스트·이미지는 재활용·다운샘플링으로 메모리를 한정한다

- 긴 리스트는 화면에 보이는 만큼만 뷰를 만들어 **재활용**한다(전체를 한꺼번에 메모리에 올리지 않는다).
- 이미지는 원본 해상도가 아니라 **실제 표시 크기**로 다운샘플링해 메모리·대역폭을 아낀다.

```text
// ❌ 금지 — 전체 항목을 한 번에 만들고, 원본 해상도 이미지를 그대로 메모리에
buildAllRowsAtOnce(items)         // 1만 개를 한꺼번에
view.image = load(originalHugeImage)   // 4000x3000을 작은 셀에

// ✅ 권장 — 보이는 만큼만 재활용 + 표시 크기로 다운샘플
recycleVisibleRows(items)
view.image = load(path, downsampleTo = cellSize)
```

### 3-6. 배터리: 백그라운드 작업은 최소화·배치·지연

- 백그라운드 위치 추적·잦은 네트워크 폴링·반복 웨이크업을 최소화한다.
- 꼭 필요한 작업은 **모아서(batch)·지연시켜·적절한 트리거(조건/주기)**로 처리한다.

```text
// ❌ 금지 — 짧은 주기로 계속 깨워서 폴링·위치 수집 (발열·배터리 소모)
every 5s: pollServer(); readPreciseLocation()

// ✅ 권장 — 합치고 늦추고 조건부로
batched/deferred sync (네트워크·충전 조건 등) + 필요한 정확도/주기로만 위치 요청
```

## 4. 흔한 실수

- **측정 없이 최적화** → 엉뚱한 곳을 고치고 효과도 검증 못 한다. 프로파일러로 병목을 먼저 찾는다.
- **시작 시 전부 동기 초기화** → 콜드 스타트가 길어진다. 첫 화면에 필요한 것만 두고 나머지는 지연·백그라운드.
- **메인 스레드에서 디코딩·파싱·I/O** → 스크롤·애니메이션이 끊긴다(프레임 드롭). 무거운 작업은 백그라운드로.
- **원본 해상도 이미지를 작은 뷰에 그대로** → 메모리 급증·OOM. 표시 크기로 다운샘플링한다.
- **리스트 전체를 한꺼번에 생성** → 메모리·렉. 보이는 만큼만 뷰 재활용.
- **미사용 코드·리소스·전 기기 리소스를 한 바이너리에 동봉** → 다운로드 용량 비대. 제거 + 기기별 분할.
- **백그라운드 폴링·위치·웨이크업 남발** → 발열·배터리 소모. 배치·지연·조건부로.
- **메모리 누수 방치** → 화면 반복 진입 시 사용량이 계속 증가. 프로파일러로 누수 점검.

## 5. 체크리스트

- [ ] 시작 시간·메모리·프레임·배터리를 **프로파일러로 측정**하고, 수정 후 재측정으로 효과를 확인했는가
- [ ] 콜드 스타트(앱 초기화 → 첫 화면)를 측정했고, 무거운 초기화는 **지연·백그라운드**로 옮겼는가
- [ ] 메인(UI) 스레드에서 디코딩·파싱·I/O 등 무거운 작업을 제거해 **60fps**를 유지하는가
- [ ] 미사용 코드·리소스를 제거(코드 축소·난독화)하고, 기기별 **분할(앱 번들/슬라이싱)**을 적용했는가
- [ ] 이미지를 **효율적 포맷·표시 크기**로 줄였는가
- [ ] 리스트는 뷰를 **재활용**하고, 이미지는 **다운샘플링**해 메모리를 한정했는가
- [ ] 프로파일러로 **메모리 누수**를 점검했는가
- [ ] 백그라운드 위치·네트워크 폴링·웨이크업을 **최소화·배치·지연** 처리했는가

> 입력값 검증·네트워크·에러 처리 등 다른 표준과 충돌하지 않게, 성능 최적화가 정합성·보안을 깨뜨리지 않는지도 함께 확인한다.

### 플랫폼별 도구 매핑 (개념 → 구현)

| 개념 | Android | iOS |
|------|---------|-----|
| 바이너리 분할(기기별 전달) | App Bundle (AAB) | App Thinning (슬라이싱) |
| 미사용 코드·리소스 제거 | R8 / ProGuard | (컴파일러 최적화·데드코드 제거) |
| 효율적 이미지 | 벡터 드로어블 · WebP | 적절한 포맷·해상도(@x 자원) |
| 시작 시간 측정 | Macrobenchmark | Instruments (App Launch) |
| 메모리 측정·누수 | Memory Profiler | Instruments (Allocations/Leaks) |
| 프레임 측정 | Profile GPU Rendering | Instruments (Core Animation) |
| 배터리 측정 | Battery Historian | Instruments (Energy Log) |

### Android (Kotlin)

#### 시작 시간 최적화

```
콜드 스타트 단계: 프로세스 생성 → Application onCreate → 첫 화면 렌더
목표: < 2초 (Android Vitals 권장)

전략:
- onCreate에서 무거운 SDK 초기화 지연 (App Startup·lazy)
- 스플래시 → 첫 화면 데이터 프리페치
- 메인 스레드 블로킹 작업 제거
```

#### 바이너리 크기 축소

기법·도구는 위 「플랫폼별 도구 매핑」 표(바이너리 분할 / 미사용 코드·리소스 제거 / 효율적 이미지 행)를 참고한다. Android는 App Bundle(AAB)로 기기별 APK를 분할하고, R8/ProGuard로 미사용 코드·리소스를 제거하며, 벡터 드로어블·WebP로 이미지를 줄인다.

#### 프레임 성능 (60fps)

```kotlin
// ❌ 금지 — 메인 스레드에서 무거운 작업 (프레임 드롭)
fun onBindViewHolder(...) {
    val bitmap = decodeHugeImage(path)  // 메인 스레드 블로킹 → 프레임 드롭
}

// ✅ 권장 — 백그라운드 디코딩 + 다운샘플링
fun onBindViewHolder(...) {
    Glide.with(context).load(path)
        .override(targetW, targetH)  // 다운샘플
        .into(holder.image)
}
```
