# ⚡ アプリサイズ & パフォーマンス最適化

> モバイルアプリのサイズ・起動時間・フレーム・メモリ・バッテリーを最適化する汎用標準である。推測ではなく測定でボトルネックを見つけ、重い処理は遅延・バックグラウンドへ移し、未使用コード・リソースを除去し、リスト・画像は再利用・ダウンサンプリングする。アプリが重い・遅いとき、リリース前にパフォーマンスを点検・プロファイリングするときに読む。特定のプラットフォーム(iOS/Android)やフレームワークに依存しない共通概念であり、プラットフォーム別のツールは付録で扱う。

## 1. 目的

- ユーザーが体感する**起動速度・反応性・発熱・アプリ容量**を一定水準以上に保証する。
- パフォーマンスを「勘」ではなく**測定値(起動時間・フレーム・メモリ・バッテリー)**で管理し、リリース前の点検を標準化する。
- プラットフォームが異なっても同じく通用する**共通原則**を一箇所に集め、ツールは概念にマッピングする。

## 2. 核心原則

- **測定が先、推測は禁止**: 最適化はプロファイラでボトルネックを確認した後に行う。測定せずに手を付けない。(起動時間・メモリ・フレーム・バッテリーをそれぞれ測定する。)
- **起動時間(コールドスタート)を最優先に**: プロセス生成 → アプリ初期化 → 最初の画面レンダリングまでを測定し、起動経路の重い初期化は**遅延(lazy)・バックグラウンド・オンデマンド**へ移す。
- **メイン(UI)スレッドを空ける**: デコード・パース・I/O・暗号化などの重い処理を UI スレッドから外し、**フレーム予算(60fps ≈ 16ms/frame)**を守る。
- **バイナリサイズを減らす**: 未使用コード・リソースを除去し(コード縮小・難読化)、端末ごとに必要なものだけをダウンロードさせ(アプリバンドル/スライシング)、画像は効率的なフォーマット・解像度で減らす。
- **メモリを限定する**: リストはビューを**再利用**し、画像は表示サイズに合わせて**ダウンサンプリング**し、リークをプロファイラで点検する。
- **バッテリーを節約する**: バックグラウンドの位置情報・ネットワークポーリング・ウェイクアップを最小化し、処理を**バッチ・遅延・まとめ**で扱う。
- **共通概念を優先、ツールはマッピング**: 原則はプラットフォーム中立で定め、実際のツール(例: iOS Instruments・App Thinning、Android Profiler・R8/App Bundle)はその概念の実装と見なす。

## 3. ルール (✅/❌)

### 3-1. 測定なしに最適化しない

- どこが遅いかをプロファイラでまず確認し、変えた後に再度測定して効果を検証する。

```text
// ❌ 禁止 — 測定なしに「感覚」で手を付ける
"リストが遅い気がするので、とりあえずキャッシュをもっと足そう"

// ✅ 推奨 — 測定 → ボトルネック特定 → 修正 → 再測定
プロファイラでフレームドロップ地点を確認 → 原因(メインスレッドのデコード)を除去 → 再測定して改善を確認
```

### 3-2. 起動経路の重い初期化は遅延・バックグラウンドへ

- コールドスタート経路(アプリ初期化 → 最初の画面)で、今すぐ必要のない SDK・設定・キャッシュウォーミングを同期で回さない。
- 最初の画面を描くのに必要なものだけ即時に行い、残りは**遅延ロード・バックグラウンド・オンデマンド**へ先送りする。

```text
// ❌ 禁止 — アプリ起動時にすべての SDK を同期初期化(最初の画面が遅れる)
アプリ起動:
  analytics.init(); ads.init(); crash.init(); imageCache.warmAll();  // 全部同期
  showFirstScreen()

// ✅ 推奨 — 最初の画面に必要なものだけ、残りは遅延/バックグラウンド
アプリ起動:
  crash.init()                    // 即時必要な最小限だけ
  background { analytics.init(); ads.init() }   // 起動経路の外へ
  lazy { imageCache }             // 最初に使うときに初期化
  showFirstScreen()
```

### 3-3. メイン(UI)スレッドで重い処理をしない

- 画像デコード・大容量パース・ディスク/ネットワーク I/O・暗号化のような処理は UI スレッドから外し、フレームドロップを防ぐ。
- リストセルをバインドする経路のように**頻繁・高速に呼ばれる地点**ほど重い処理を置かない。

```text
// ❌ 禁止 — セルバインド(毎フレーム経路)で同期デコード → フレームドロップ
bindListItem(item):
  image = decodeFullSizeImage(item.path)   // UI スレッドブロッキング
  view.image = image

// ✅ 推奨 — バックグラウンドデコード + 表示サイズへダウンサンプル + 非同期適用
bindListItem(item):
  loadImageAsync(item.path, targetW, targetH) into view.image
```

### 3-4. バイナリサイズは「除去 + 分割 + 効率的フォーマット」で

- 未使用コード・リソースをビルド段階で除去し(コード縮小・難読化)、端末ごとに必要なリソースだけをダウンロードさせる(アプリバンドル/スライシング)。
- 画像は表示に十分な解像度と効率的なフォーマットで減らす(不要な原本同梱は禁止)。

```text
// ❌ 禁止 — 全端末用リソースを一つのバイナリに全部詰め込み、未使用コードもそのまま
全解像度の画像 + 全 ABI/アーキテクチャ + デッドコードを含む → ダウンロード容量が肥大

// ✅ 推奨 — 除去・分割・効率的フォーマット
未使用コード/リソース除去 + 端末別分割(バンドル/スライシング) + 効率的な画像フォーマット・解像度
```

### 3-5. リスト・画像は再利用・ダウンサンプリングでメモリを限定する

- 長いリストは画面に見える分だけビューを作って**再利用**する(全体を一度にメモリに載せない)。
- 画像は原本解像度ではなく**実際の表示サイズ**にダウンサンプリングしてメモリ・帯域を節約する。

```text
// ❌ 禁止 — 全項目を一度に作り、原本解像度の画像をそのままメモリに
buildAllRowsAtOnce(items)         // 1万個を一度に
view.image = load(originalHugeImage)   // 4000x3000 を小さなセルに

// ✅ 推奨 — 見える分だけ再利用 + 表示サイズへダウンサンプル
recycleVisibleRows(items)
view.image = load(path, downsampleTo = cellSize)
```

### 3-6. バッテリー: バックグラウンド処理は最小化・バッチ・遅延

- バックグラウンドの位置情報追跡・頻繁なネットワークポーリング・反復ウェイクアップを最小化する。
- どうしても必要な処理は**まとめて(batch)・遅延させ・適切なトリガ(条件/周期)**で扱う。

```text
// ❌ 禁止 — 短い周期で起こし続けてポーリング・位置収集(発熱・バッテリー消耗)
every 5s: pollServer(); readPreciseLocation()

// ✅ 推奨 — まとめて遅らせて条件付きで
batched/deferred sync (ネットワーク・充電条件など) + 必要な精度/周期だけで位置要求
```

## 4. よくある間違い

- **測定なしに最適化** → 見当違いの場所を直し効果も検証できない。プロファイラでボトルネックをまず見つける。
- **起動時に全部同期初期化** → コールドスタートが長くなる。最初の画面に必要なものだけ置き、残りは遅延・バックグラウンド。
- **メインスレッドでデコード・パース・I/O** → スクロール・アニメーションがカクつく(フレームドロップ)。重い処理はバックグラウンドへ。
- **原本解像度の画像を小さなビューにそのまま** → メモリ急増・OOM。表示サイズへダウンサンプリングする。
- **リスト全体を一度に生成** → メモリ・もたつき。見える分だけビュー再利用。
- **未使用コード・リソース・全端末リソースを一つのバイナリに同梱** → ダウンロード容量が肥大。除去 + 端末別分割。
- **バックグラウンドポーリング・位置・ウェイクアップの乱発** → 発熱・バッテリー消耗。バッチ・遅延・条件付きで。
- **メモリリーク放置** → 画面反復進入時に使用量が増え続ける。プロファイラでリーク点検。

## 5. チェックリスト

- [ ] 起動時間・メモリ・フレーム・バッテリーを**プロファイラで測定**し、修正後の再測定で効果を確認したか
- [ ] コールドスタート(アプリ初期化 → 最初の画面)を測定し、重い初期化は**遅延・バックグラウンド**へ移したか
- [ ] メイン(UI)スレッドからデコード・パース・I/O など重い処理を除去して**60fps**を維持しているか
- [ ] 未使用コード・リソースを除去(コード縮小・難読化)し、端末別**分割(アプリバンドル/スライシング)**を適用したか
- [ ] 画像を**効率的フォーマット・表示サイズ**で減らしたか
- [ ] リストはビューを**再利用**し、画像は**ダウンサンプリング**してメモリを限定したか
- [ ] プロファイラで**メモリリーク**を点検したか
- [ ] バックグラウンドの位置情報・ネットワークポーリング・ウェイクアップを**最小化・バッチ・遅延**処理したか

> 入力値検証・ネットワーク・エラー処理など他の標準と衝突しないよう、パフォーマンス最適化が整合性・セキュリティを壊さないかも併せて確認する。

## 付録: スタック別の例

> 以下は参考用の実装例である。上の 2~5 の原則・ルールが標準であり、付録はその適用事例にすぎない。チームが使うプラットフォーム/ツールに合った例を同じパターンで追加する。

### プラットフォーム別ツールマッピング (概念 → 実装)

| 概念 | Android | iOS |
|------|---------|-----|
| バイナリ分割(端末別配信) | App Bundle (AAB) | App Thinning (スライシング) |
| 未使用コード・リソース除去 | R8 / ProGuard | (コンパイラ最適化・デッドコード除去) |
| 効率的な画像 | ベクタードロアブル · WebP | 適切なフォーマット・解像度(@x リソース) |
| 起動時間測定 | Macrobenchmark | Instruments (App Launch) |
| メモリ測定・リーク | Memory Profiler | Instruments (Allocations/Leaks) |
| フレーム測定 | Profile GPU Rendering | Instruments (Core Animation) |
| バッテリー測定 | Battery Historian | Instruments (Energy Log) |

### Android (Kotlin)

#### 起動時間最適化

```
コールドスタート段階: プロセス生成 → Application onCreate → 最初の画面レンダリング
目標: < 2秒 (Android Vitals 推奨)

戦略:
- onCreate での重い SDK 初期化を遅延 (App Startup・lazy)
- スプラッシュ → 最初の画面データプリフェッチ
- メインスレッドブロッキング処理を除去
```

#### バイナリサイズ縮小

技法・ツールは上の「プラットフォーム別ツールマッピング」表(バイナリ分割 / 未使用コード・リソース除去 / 効率的な画像の行)を参考にする。Android は App Bundle(AAB)で端末別 APK を分割し、R8/ProGuard で未使用コード・リソースを除去し、ベクタードロアブル・WebP で画像を減らす。

#### フレーム性能 (60fps)

```kotlin
// ❌ 禁止 — メインスレッドで重い処理 (フレームドロップ)
fun onBindViewHolder(...) {
    val bitmap = decodeHugeImage(path)  // メインスレッドブロッキング → フレームドロップ
}

// ✅ 推奨 — バックグラウンドデコード + ダウンサンプリング
fun onBindViewHolder(...) {
    Glide.with(context).load(path)
        .override(targetW, targetH)  // ダウンサンプル
        .into(holder.image)
}
```

> プロファイリングツール(起動時間・メモリ・フレーム・バッテリー)は上の「プラットフォーム別ツールマッピング」表を参考にする。

### iOS (Swift)

> 上の 1~5 の原則・ルールは iOS・Android 共通である。以下の Swift 実装例はチームが同じパターンで追加する(プロファイリングツールは「プラットフォーム別ツールマッピング」表の iOS 列を参考)。
