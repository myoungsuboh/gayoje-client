# 📋 명세 작성 — 무엇을 만들지 합의하기

> 코드를 쓰기 전에 **'무엇을 만들고, 어디까지가 완료인가'**를 글로 못 박는다. 새 기능·변경을 시작하기 전, 특히 AI 에이전트에게 일을 시키기 전에 읽는다.

가장 비싼 실수는 잘못 짠 코드가 아니라 **엉뚱한 것을 정확히 만드는 것**이다. AI 에이전트는 모호한 요청을 받으면 빈칸을 '그럴듯하게' 추측으로 채운다 — 그 추측이 틀리면 코드는 멀쩡한데 결과물이 쓸모없어진다. 명세가 구체적일수록 추측이 줄고, AI 재작업도 줄어든다. 이 스킬은 **무엇/완료 기준(what)**을 다룬다. **어떻게/단계 계획(how)**은 `implementation-plan` 스킬에서 다룬다.

## 1. 핵심 원칙

- 코딩 전에 '무엇을' 만들지 먼저 합의한다 — 작은 명세가 큰 재작업을 막는다.
- 완료 기준을 **검증 가능하게** 적는다 — 사람·AI 모두 "다 됐다"를 같은 뜻으로 읽어야 한다.
- 범위(In)와 비범위(Out)를 함께 적는다 — 안 하는 것을 못 박아야 스코프가 안 샌다.
- 정상 흐름뿐 아니라 **엣지·에러 케이스**를 명세에 포함한다 — AI는 적지 않으면 만들지 않는다.
- 모호어("빠르게·적절히·잘")를 측정 가능한 수치·조건으로 바꾼다.
- 명세는 '무엇/완료 기준'까지만. '어떻게 단계별로 만들지'는 `implementation-plan`으로 분리한다.

## 2. 규칙

### 2-1. 사용자 스토리로 '무엇/왜'를 적는다

기능명만 적지 말고 **누가·무엇을·왜**를 한 문장으로.

```
❌ "검색 기능"
   → AI가 대상·범위·정렬·권한을 전부 추측한다.

✅ As a 로그인 사용자,
   I want 내 주문 목록을 주문번호로 검색하기를,
   so that 과거 주문을 빨리 다시 찾을 수 있다.
```

'왜(so that)'가 있으면 AI가 트레이드오프를 옳은 방향으로 판단한다.

### 2-2. 수용 기준은 Given-When-Then으로

"완료"를 **관찰 가능한 행동**으로 못 박는다. 통과/실패를 따질 수 없으면 수용 기준이 아니다.

```
❌ "검색이 잘 동작해야 한다"  (잘 = 무슨 뜻? 검증 불가)

✅ Given 주문이 3건 있는 사용자가
   When 존재하는 주문번호로 검색하면
   Then 해당 주문 1건만 목록에 표시된다.

   Given 동일 사용자가
   When 없는 주문번호로 검색하면
   Then "검색 결과 없음" 안내가 표시되고 에러는 나지 않는다.
```

이 형식은 그대로 테스트 케이스가 된다. (테스트는 `test-strategy` / `unit-testing` 스킬 참조.)

### 2-3. 범위(In)와 비범위(Out)를 함께 적는다

```
✅ ## 범위 (In Scope)
   - 주문번호 완전일치 검색
   - 본인 주문만 조회

   ## 비범위 (Out of Scope) — 이번엔 안 함
   - 상품명·부분일치 검색  (→ 별도 티켓)
   - 다른 사용자 주문 조회 (권한상 금지)
```

비범위를 비워두면 AI가 친절하게 '알아서' 만들어 스코프와 리스크가 커진다. **안 하는 것을 명시**한다.

### 2-4. 엣지·에러 케이스를 명세에 포함한다

정상 흐름만 적으면 AI는 정상 흐름만 만든다. 빈 값·초과·중복·권한·실패를 미리 정한다.

```
✅ ## 엣지/에러 케이스
   - 빈 검색어: 검색 미실행, 입력 안내 표시
   - 결과 0건: "결과 없음" 안내 (빈 화면 금지)
   - 외부 조회 실패: 재시도 안내, 사용자 데이터 변경 없음
   - 권한 없는 주문 접근: 403, 타인 데이터 노출 금지
```

### 2-5. 모호어를 측정 가능하게 바꾼다

```
❌ 빠르게 / 적절히 / 사용자 친화적으로 / 많은 데이터도

✅ 결과를 1초 이내(p95) 표시
✅ 한 번에 20건씩, 더 보기로 추가 로드
✅ 입력 오류 시 해당 필드 아래 빨간 안내문 표시
```

수치·조건으로 적을 수 없다면 아직 합의가 덜 된 것이다 — 그 자리에서 질문해 좁힌다.

### 2-6. 명세 ↔ 계획의 경계

| 구분 | 이 스킬 (명세) | `implementation-plan` (계획) |
|---|---|---|
| 답하는 질문 | 무엇을? 완료 기준은? | 어떻게? 어떤 순서로? |
| 산출물 | 사용자 스토리·수용 기준·범위 | 단계(Phase)·진행 상태·일정 |
| 시점 | 가장 먼저 (합의) | 명세 합의 후 |

명세에 구현 방법(파일·라이브러리·단계)을 섞지 않는다. 그건 계획으로 넘긴다.

## 3. 흔한 실수

- ❌ 기능명 한 줄만 던지고 바로 코딩/프롬프트 시작 → AI가 빈칸을 추측으로 채움
- ❌ 정상 흐름만 적고 빈 값·실패·권한 케이스 누락 → 운영에서 터짐
- ❌ "빠르게·잘·적절히" 같은 모호어를 그대로 둠 → 검수 때 "이게 완료냐" 논쟁
- ❌ 비범위를 안 적어 AI가 요청 안 한 기능까지 만듦 (스코프 크리프)
- ❌ 명세에 구현 단계·파일 구조를 섞음 → `implementation-plan`과 역할 중복
- ❌ 명세를 한 번 쓰고 방치 → 합의가 바뀌면 명세부터 고친다 (코드보다 먼저)

> **바이브 코딩 팁**: 에이전트에게 "코드부터 짜줘" 대신 **명세(스토리·수용 기준·범위)를 먼저 주고** 코드를 생성시키면 환각·재작업이 크게 준다. 구체적인 명세 = AI에게 주는 가장 강력한 컨텍스트다. (`vibe-coding-workflow` 스킬과 함께 쓴다.)

## 4. 체크리스트

- [ ] 사용자 스토리로 '누가·무엇을·왜'를 적었는가
- [ ] 수용 기준을 Given-When-Then 등 검증 가능한 형태로 적었는가
- [ ] 범위(In)와 비범위(Out)를 둘 다 명시했는가
- [ ] 엣지·에러 케이스(빈 값·실패·권한·중복)를 포함했는가
- [ ] "빠르게·잘·적절히" 같은 모호어를 수치·조건으로 바꿨는가
- [ ] 구현 방법(단계·파일)을 섞지 않고 `implementation-plan`으로 분리했는가
