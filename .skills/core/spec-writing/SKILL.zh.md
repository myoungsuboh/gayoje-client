# 📋 规格编写 — 就要做什么达成共识

> 在写代码前，以文字钉死 **『要做什么，到哪里算完成』**。在开始新功能·变更前，尤其在让 AI 代理执行任务前阅读。

最昂贵的错误不是写错的代码，而是 **精确地做出了不该做的东西**。AI 代理收到含糊请求时，会用『看似合理』的推测填补空白 — 一旦推测错误，代码没问题但成果无用。规格越具体，推测越少，AI 返工也越少。本技能处理 **做什么/完成标准（what）**。**如何/分步计划（how）** 在 `implementation-plan` 技能中处理。

## 1. 核心原则

- 在编码前先就『做什么』达成共识 — 小的规格能避免大的返工。
- 把完成标准写得 **可验证** — 人和 AI 都必须把“做完了”读成同一个意思。
- 把范围（In）和非范围（Out）一起写 — 钉死不做的事，范围才不会泄漏。
- 不仅正常流程，也把 **边缘·错误用例** 纳入规格 — AI 不写就不做。
- 把含糊词（“快速·适当·好”）换成可测量的数值·条件。
- 规格只到『做什么/完成标准』。『如何分步去做』分离到 `implementation-plan`。

## 2. 规则

### 2-1. 用用户故事写『做什么/为什么』

不要只写功能名，用一句话写出 **谁·做什么·为什么**。

```
❌ "搜索功能"
   → AI 把对象·范围·排序·权限全都靠推测。

✅ As a 登录用户，
   I want 用订单号搜索我的订单列表，
   so that 能快速重新找到过去的订单。
```

有了『为什么（so that）』，AI 才能把权衡判断到正确方向。

### 2-2. 验收标准用 Given-When-Then

把“完成”钉死为 **可观察的行为**。无法判定通过/失败的就不是验收标准。

```
❌ "搜索应当好好工作"  (好好 = 什么意思? 无法验证)

✅ Given 一个有 3 笔订单的用户
   When 用一个存在的订单号搜索
   Then 列表中只显示该 1 笔订单。

   Given 同一用户
   When 用一个不存在的订单号搜索
   Then 显示 "无搜索结果" 提示且不报错。
```

这种格式可直接成为测试用例。（测试参见 `test-strategy` / `unit-testing` 技能。）

### 2-3. 把范围（In）和非范围（Out）一起写

```
✅ ## 范围 (In Scope)
   - 按订单号完全匹配搜索
   - 仅查询本人订单

   ## 非范围 (Out of Scope) — 这次不做
   - 商品名·部分匹配搜索  (→ 另开工单)
   - 查询其他用户的订单 (权限上禁止)
```

把非范围留空，AI 会“自作主张”地热心去做，让范围和风险膨胀。**明示不做的事**。

### 2-4. 把边缘·错误用例纳入规格

只写正常流程，AI 就只做正常流程。预先确定空值·超出·重复·权限·失败。

```
✅ ## 边缘/错误用例
   - 空搜索词: 不执行搜索，显示输入提示
   - 结果 0 条: "无结果" 提示 (禁止空白屏)
   - 外部查询失败: 重试提示，不改动用户数据
   - 访问无权限的订单: 403，禁止暴露他人数据
```

### 2-5. 把含糊词换成可测量的

```
❌ 快速 / 适当 / 用户友好 / 即使数据很多

✅ 在 1 秒内（p95）显示结果
✅ 每次 20 条，通过“查看更多”追加加载
✅ 输入出错时在对应字段下方显示红色提示文字
```

若无法以数值·条件写出，说明共识还不够 — 当场提问以收窄。

### 2-6. 规格 ↔ 计划的边界

| 区分 | 本技能（规格） | `implementation-plan`（计划） |
|---|---|---|
| 回答的问题 | 做什么? 完成标准? | 如何? 以什么顺序? |
| 产物 | 用户故事·验收标准·范围 | 阶段（Phase）·进度状态·日程 |
| 时点 | 最先（共识） | 规格达成共识后 |

不要把实现方法（文件·库·步骤）混进规格。那些交给计划。

## 3. 常见错误

- ❌ 只丢一行功能名就立刻开始编码/提示 → AI 用推测填补空白
- ❌ 只写正常流程，遗漏空值·失败·权限用例 → 在生产环境爆雷
- ❌ 把 "快速·好·适当" 之类含糊词原样保留 → 验收时为“这算完成吗”争论
- ❌ 不写非范围，AI 连未请求的功能都做出来 (范围蔓延)
- ❌ 把实现步骤·文件结构混进规格 → 与 `implementation-plan` 角色重叠
- ❌ 规格写一次就放着不管 → 共识变了就先改规格 (先于代码)

> **氛围编程提示**: 与其对代理说“先写代码”，不如 **先给它规格（故事·验收标准·范围）** 再让它生成代码，幻觉·返工会大幅减少。具体的规格 = 给 AI 的最强上下文。（与 `vibe-coding-workflow` 技能配合使用。）

## 4. 检查清单

- [ ] 是否用用户故事写了『谁·做什么·为什么』
- [ ] 是否以 Given-When-Then 等可验证形式写了验收标准
- [ ] 是否同时明示了范围（In）和非范围（Out）
- [ ] 是否纳入了边缘·错误用例（空值·失败·权限·重复）
- [ ] 是否把 "快速·好·适当" 之类含糊词换成了数值·条件
- [ ] 是否未把实现方法（步骤·文件）混入，而分离到 `implementation-plan`
