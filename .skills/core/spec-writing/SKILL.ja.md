# 📋 仕様作成 — 何を作るか合意する

> コードを書く前に **『何を作り、どこまでが完了か』** を文章で釘づけする。新機能・変更を始める前、特に AI エージェントに仕事を任せる前に読む。

最も高くつく失敗は、誤って書いたコードではなく **見当違いのものを正確に作ること** だ。AI エージェントは曖昧な要求を受けると、空欄を『それらしく』推測で埋める — その推測が外れると、コードは正常なのに成果物が無用になる。仕様が具体的なほど推測が減り、AI の手戻りも減る。このスキルは **何/完了基準（what）** を扱う。**どう/段階計画（how）** は `implementation-plan` スキルで扱う。

## 1. 中核原則

- コーディング前に『何を』作るかをまず合意する — 小さな仕様が大きな手戻りを防ぐ。
- 完了基準を **検証可能に** 書く — 人も AI も「できた」を同じ意味で読まねばならない。
- 範囲（In）と範囲外（Out）を併記する — やらないことを釘づけしてこそスコープが漏れない。
- 正常フローだけでなく **エッジ・エラーケース** を仕様に含める — AI は書かなければ作らない。
- 曖昧語（「速く・適切に・うまく」）を測定可能な数値・条件に変える。
- 仕様は『何/完了基準』まで。『どう段階的に作るか』は `implementation-plan` に分離する。

## 2. ルール

### 2-1. ユーザーストーリーで『何/なぜ』を書く

機能名だけ書かず **誰が・何を・なぜ** を一文で。

```
❌ "検索機能"
   → AI が対象・範囲・並び順・権限をすべて推測する。

✅ As a ログイン済みユーザー、
   I want 自分の注文一覧を注文番号で検索することを、
   so that 過去の注文をすばやく再び見つけられる。
```

『なぜ（so that）』があると AI がトレードオフを正しい方向に判断する。

### 2-2. 受け入れ基準は Given-When-Then で

「完了」を **観察可能な振る舞い** で釘づけする。合否を判定できなければ受け入れ基準ではない。

```
❌ "検索がうまく動くべき"  (うまく = どういう意味? 検証不能)

✅ Given 注文が 3 件あるユーザーが
   When 存在する注文番号で検索すると
   Then 該当する注文 1 件のみが一覧に表示される。

   Given 同一ユーザーが
   When 存在しない注文番号で検索すると
   Then "検索結果なし" の案内が表示され、エラーは出ない。
```

この形式はそのままテストケースになる。（テストは `test-strategy` / `unit-testing` スキル参照。）

### 2-3. 範囲（In）と範囲外（Out）を併記する

```
✅ ## 範囲 (In Scope)
   - 注文番号の完全一致検索
   - 本人の注文のみ照会

   ## 範囲外 (Out of Scope) — 今回はやらない
   - 商品名・部分一致検索  (→ 別チケット)
   - 他ユーザーの注文照会 (権限上禁止)
```

範囲外を空にすると AI が親切に『自分で』作り、スコープとリスクが膨らむ。**やらないことを明示** する。

### 2-4. エッジ・エラーケースを仕様に含める

正常フローだけ書くと AI は正常フローだけ作る。空値・超過・重複・権限・失敗をあらかじめ定める。

```
✅ ## エッジ/エラーケース
   - 空の検索語: 検索を実行せず、入力案内を表示
   - 結果 0 件: "結果なし" 案内 (空画面禁止)
   - 外部照会失敗: 再試行案内、ユーザーデータの変更なし
   - 権限のない注文へのアクセス: 403、他人のデータ露出禁止
```

### 2-5. 曖昧語を測定可能に変える

```
❌ 速く / 適切に / ユーザーフレンドリーに / 大量のデータでも

✅ 結果を 1 秒以内（p95）に表示
✅ 一度に 20 件ずつ、もっと見るで追加ロード
✅ 入力エラー時に該当フィールドの下に赤い案内文を表示
```

数値・条件で書けないなら、まだ合意が不十分だ — その場で質問して絞る。

### 2-6. 仕様 ↔ 計画の境界

| 区分 | このスキル（仕様） | `implementation-plan`（計画） |
|---|---|---|
| 答える質問 | 何を? 完了基準は? | どう? どの順序で? |
| 成果物 | ユーザーストーリー・受け入れ基準・範囲 | 段階（Phase）・進行状態・スケジュール |
| 時点 | 一番先（合意） | 仕様合意後 |

仕様に実装方法（ファイル・ライブラリ・段階）を混ぜない。それは計画に渡す。

## 3. よくある誤り

- ❌ 機能名一行だけ投げてすぐコーディング/プロンプト開始 → AI が空欄を推測で埋める
- ❌ 正常フローだけ書いて空値・失敗・権限ケースを欠落 → 運用で破綻
- ❌ "速く・うまく・適切に" のような曖昧語をそのまま残す → 検収時に「これが完了か」論争
- ❌ 範囲外を書かず AI が要求していない機能まで作る (スコープクリープ)
- ❌ 仕様に実装段階・ファイル構造を混ぜる → `implementation-plan` と役割重複
- ❌ 仕様を一度書いて放置 → 合意が変わったら仕様から直す (コードより先に)

> **バイブコーディングのヒント**: エージェントに「コードからまず書いて」と言う代わりに **仕様（ストーリー・受け入れ基準・範囲）を先に与えて** コードを生成させると、ハルシネーション・手戻りが大きく減る。具体的な仕様 = AI に与える最も強力なコンテキストだ。（`vibe-coding-workflow` スキルと併用する。）
