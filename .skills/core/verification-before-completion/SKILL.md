# ✅ 완료 전 검증 — 증거 기반 보고

> 이 스킬은 본 사내 하네스의 자체 표준이다(외부 'superpowers' 번들의 동명 스킬과 무관).
>
> "완료/고침"을 주장하기 전에 반드시 실제 실행·테스트로 증거를 확보한다. 작업을 끝냈다고 보고하거나 커밋·PR을 만들기 직전에 읽는다.

AI 에이전트가 가장 흔히 저지르는 신뢰 사고는 **실행해 보지 않고 "다 됐다 / 고쳤다 / 통과한다"고 단언하는 것**이다. 코드를 바꾼 직후 "아마 될 것"이라는 추측을 성공 보고로 포장하면, 사용자는 깨진 결과를 떠안고 신뢰는 무너진다. 규칙은 단순하다 — **증거(명령 출력)가 없으면 성공이라고 말하지 않는다.**

## 1. 핵심 원칙

- 증거가 먼저, 주장은 나중. 성공 단언 앞에는 항상 그 근거가 되는 실제 명령 출력이 있어야 한다.
- "완료"는 느낌이 아니라 기준(DoD)이다 — 빌드 통과·테스트 통과·lint 통과·수용 기준 충족을 모두 만족해야 완료다.
- 변경 범위를 전수 확인한다 — 내가 건드린 모든 파일·경로·케이스가 실제로 동작하는지 확인하고, 안 본 부분을 "됐을 것"으로 넘기지 않는다.
- 실패는 실패로 정직하게 보고한다 — 부분 성공, 미확인, 막힌 지점을 숨기거나 미화하지 않는다.
- 추측 금지 — "아마 될 것", "보통 이러면 동작함" 같은 표현으로 검증을 대체하지 않는다.
- 못 돌렸으면 "못 돌렸다"고 말한다 — 환경 제약으로 실행이 불가능하면 그 사실과 한계를 명시한다(검증한 척 금지).

## 2. 규칙

### 2-1. 증거 없이 성공 단언 금지

성공을 주장하려면 그 직전에 실제로 실행한 명령과 출력이 있어야 한다.

```
❌ 금지 — 추측을 결과로 보고
"수정 완료했습니다. 이제 정상 동작할 겁니다."   (실행 안 함)

✅ 권장 — 실행 → 출력 확인 → 그 근거로 보고
$ <빌드/테스트 명령>
 ... 42 passed, 0 failed
"테스트 42개 전부 통과 확인 후 완료 보고합니다."
```

### 2-2. 완료 기준(DoD)을 명시적으로 충족

아래를 전부 만족해야 "완료"다. 하나라도 미확인이면 완료가 아니다.

| 항목 | 확인 방법 |
|---|---|
| 빌드 통과 | 빌드/컴파일 명령이 에러 없이 끝남 |
| 테스트 통과 | 관련 테스트 실행, 0 fail (테스트 작성·범위는 `test-strategy` 참조) |
| 정적 검사 통과 | lint·타입 체크·포맷 검사 통과 |
| 수용 기준 충족 | 요청한 기능·버그 수정이 실제로 요구대로 동작 |

### 2-3. 변경 범위 전수 확인

```
❌ 금지 — 일부만 보고 전체를 단정
"파일 3개 고쳤고 하나 돌려보니 되니까 나머지도 될 겁니다."

✅ 권장 — 바꾼 곳·영향 받는 곳을 모두 확인
- 변경한 모든 파일/함수의 호출부·케이스를 점검
- 회귀 가능 지점(공유 모듈·경계 케이스)까지 실행으로 확인
```

### 2-4. 실패·미확인은 정직하게

```
❌ 금지 — 부분 성공을 완료로 포장 / 실패 은폐
"전부 해결했습니다."   (실제로는 1건 미해결)

✅ 권장 — 상태를 있는 그대로
"3건 중 2건 수정·검증 완료. 1건은 X 원인으로 미해결 — 다음 단계 제안: …"
"환경 제약으로 통합 테스트는 미실행. 단위 테스트만 통과 확인함."
```

## 3. 흔한 실수

AI가 자주 만드는 것 — 보고 전 스스로 거른다.

- ❌ 코드를 바꾼 직후 실행 없이 "완료/수정됨"이라고 단언
- ❌ "아마 동작할 것", "보통 이러면 됨"으로 실제 검증을 대체
- ❌ 빌드·테스트 중 하나만 보고 나머지는 "괜찮을 것"으로 생략
- ❌ 테스트가 실패했는데 "사소한 문제"라며 통과한 것처럼 넘어감
- ❌ 명령 출력을 확인하지 않고(또는 실행 자체를 안 하고) 성공이라 보고
- ❌ 일부 케이스만 확인하고 변경 범위 전체를 동작한다고 단정

> **적용 팁**: 보고 직전 한 문장으로 자문한다 — "방금 이걸 증명하는 명령을 내가 실제로 돌렸고, 그 출력을 봤는가?" 아니라면 아직 완료가 아니다. (코드 품질 점검은 `code-review`, 테스트 범위·전략은 `test-strategy`와 함께 쓴다)

## 4. 체크리스트

- [ ] 성공을 주장하는 근거가 되는 실제 명령 출력을 확인했는가
- [ ] 빌드·테스트·lint·수용 기준(DoD)을 모두 충족했는가
- [ ] 변경한 모든 파일·영향 범위를 실행으로 확인했는가 (일부만 보고 단정하지 않았는가)
- [ ] 실패·부분 성공·미확인 항목을 숨기지 않고 정직하게 보고했는가
- [ ] "아마 될 것" 같은 추측 표현으로 검증을 대체하지 않았는가
- [ ] 실행이 불가능했다면 그 한계를 명시했는가 (검증한 척하지 않았는가)
