# ✅ 完成前验证 — 基于证据的报告

> 本技能是本公司内部 harness 的自有标准（与外部 'superpowers' 套件中同名技能无关）。
>
> 在声称"完成/修复"之前，必须通过实际执行与测试获取证据。在报告工作完成或创建提交·PR 之前阅读。

AI 智能体最常犯的信任事故是**不实际运行就断言"全做好了 / 修好了 / 通过了"**。在改完代码后立刻把"大概能行"的推测包装成成功报告，用户就会承受损坏的结果，信任随之崩塌。规则很简单——**没有证据（命令输出）就不说成功。**

## 1. 核心原则

- 证据在先，主张在后。成功断言之前必须始终有作为依据的实际命令输出。
- 「完成」不是感觉而是标准（DoD）——构建通过·测试通过·lint 通过·验收标准满足全部达成才算完成。
- 全数确认变更范围——确认自己触碰的每一个文件·路径·用例都实际运行，不把未查看的部分当作「应该没问题」蒙混过去。
- 失败就如实报告为失败——不隐藏或美化部分成功、未确认、受阻之处。
- 禁止推测——不用「大概能行」「通常这样就能运行」之类的说法替代验证。
- 没跑成就说「没跑成」——若因环境约束无法执行，则明确说明该事实与局限（禁止假装验证过）。

## 2. 规则

### 2-1. 禁止无证据的成功断言

要主张成功，其紧前必须有实际执行过的命令及其输出。

```
❌ 禁止 — 把推测当作结果报告
"已修复完成。现在应该能正常运行了。"   (未执行)

✅ 推荐 — 执行 → 确认输出 → 以此为据报告
$ <构建/测试命令>
 ... 42 passed, 0 failed
"确认 42 个测试全部通过后报告完成。"
```

### 2-2. 显式满足完成标准（DoD）

必须满足以下全部才算"完成"。哪怕一项未确认就不算完成。

| 项目 | 确认方法 |
|---|---|
| 构建通过 | 构建/编译命令无错误地结束 |
| 测试通过 | 运行相关测试，0 fail（测试编写·范围参见 `test-strategy`） |
| 静态检查通过 | lint·类型检查·格式检查通过 |
| 验收标准满足 | 所请求的功能·缺陷修复实际按要求运行 |

### 2-3. 全数确认变更范围

```
❌ 禁止 — 只看一部分就断定整体
"我改了 3 个文件，跑了一个能行，所以其余的也应该能行。"

✅ 推荐 — 把改动处·受影响处全部确认
- 检查所有改动的文件/函数的调用处·用例
- 用执行确认到可能回归之处（共享模块·边界用例）
```

### 2-4. 失败·未确认要诚实

```
❌ 禁止 — 把部分成功包装为完成 / 隐瞒失败
"全都解决了。"   (实际还有 1 项未解决)

✅ 推荐 — 如实陈述状态
"3 项中 2 项已修复·验证完成。1 项因 X 原因未解决——建议下一步: …"
"因环境约束未运行集成测试。仅确认单元测试通过。"
```

## 3. 常见错误

AI 经常造出的——报告前自行过滤掉。

- ❌ 改完代码后不运行就断言"完成/已修复"
- ❌ 用"大概能行""通常这样就行"替代实际验证
- ❌ 只看构建·测试中的一个，其余以"应该没问题"略过
- ❌ 测试失败了却当作"小问题"像通过了一样略过
- ❌ 不确认命令输出（或根本不运行）就报告成功
- ❌ 只确认部分用例就断定整个变更范围都能运行

> **应用提示**：报告前用一句话自问——"刚才证明这一点的命令，我是否真的跑过、并看了它的输出？"若否，就还没完成。（代码质量检查与 `code-review`、测试范围·策略与 `test-strategy` 一起使用）

## 4. 检查清单

- [ ] 是否确认了作为成功主张依据的实际命令输出
- [ ] 是否满足了构建·测试·lint·验收标准（DoD）全部
- [ ] 是否用执行确认了所有改动文件·影响范围（没有只看一部分就断定）
- [ ] 是否未隐藏失败·部分成功·未确认项而如实报告
- [ ] 是否未用"大概能行"之类的推测说法替代验证
- [ ] 若无法执行，是否明确说明了该局限（没有假装验证过）
