# ✅ 完了前の検証 — 証拠に基づく報告

> このスキルは本社内ハーネスの独自標準である（外部'superpowers'バンドルの同名スキルとは無関係）。
>
> 「完了/修正」を主張する前に、必ず実際の実行・テストで証拠を確保する。作業を終えたと報告する、またはコミット・PRを作成する直前に読む。

AIエージェントが最も頻繁に起こす信頼事故は、**実行してみずに「全部できた / 直した / 合格する」と断言すること**だ。コードを変えた直後に「たぶん動く」という推測を成功報告として包装すると、ユーザーは壊れた結果を抱え込み、信頼は崩れる。ルールは単純だ — **証拠（コマンド出力）がなければ成功とは言わない。**

## 1. 核心原則

- 証拠が先、主張は後。成功の断言の前には常にその根拠となる実際のコマンド出力がなければならない。
- 「完了」は感覚ではなく基準（DoD）である — ビルド合格・テスト合格・lint合格・受け入れ基準充足をすべて満たして初めて完了だ。
- 変更範囲を全数確認する — 自分が触れたすべてのファイル・パス・ケースが実際に動作するか確認し、見ていない部分を「できているだろう」で済ませない。
- 失敗は失敗として正直に報告する — 部分的成功、未確認、行き詰まった箇所を隠したり美化したりしない。
- 推測禁止 — 「たぶん動く」「普通はこうすれば動く」のような表現で検証を代替しない。
- 実行できなかったなら「実行できなかった」と言う — 環境制約で実行が不可能なら、その事実と限界を明記する（検証したふり禁止）。

## 2. ルール

### 2-1. 証拠なしの成功断言禁止

成功を主張するには、その直前に実際に実行したコマンドと出力がなければならない。

```
❌ 禁止 — 推測を結果として報告
"修正完了しました。これで正常に動作するはずです。"   (実行していない)

✅ 推奨 — 実行 → 出力確認 → その根拠で報告
$ <ビルド/テストコマンド>
 ... 42 passed, 0 failed
"テスト42個すべて合格を確認のうえ完了報告します。"
```

### 2-2. 完了基準（DoD）を明示的に充足

以下をすべて満たして初めて「完了」だ。一つでも未確認なら完了ではない。

| 項目 | 確認方法 |
|---|---|
| ビルド合格 | ビルド/コンパイルコマンドがエラーなく終わる |
| テスト合格 | 関連テストを実行、0 fail（テスト作成・範囲は `test-strategy` 参照） |
| 静的検査合格 | lint・型チェック・フォーマット検査が合格 |
| 受け入れ基準充足 | 要求した機能・バグ修正が実際に要求どおり動作 |

### 2-3. 変更範囲の全数確認

```
❌ 禁止 — 一部だけ見て全体を断定
"ファイル3個を直して一つ動かしてみたら動いたので、残りも動くはずです。"

✅ 推奨 — 変えた箇所・影響を受ける箇所をすべて確認
- 変更したすべてのファイル/関数の呼び出し部・ケースを点検
- 回帰の可能性がある箇所（共有モジュール・境界ケース）まで実行で確認
```

### 2-4. 失敗・未確認は正直に

```
❌ 禁止 — 部分的成功を完了として包装 / 失敗隠蔽
"全部解決しました。"   (実際には1件未解決)

✅ 推奨 — 状態をありのままに
"3件中2件を修正・検証完了。1件はX原因で未解決 — 次のステップ提案: …"
"環境制約で統合テストは未実行。単体テストのみ合格を確認した。"
```

## 3. よくある間違い

AIがよく作るもの — 報告前に自分で漉し取る。

- ❌ コードを変えた直後に実行なしで「完了/修正済み」と断言
- ❌ 「たぶん動く」「普通はこうすれば動く」で実際の検証を代替
- ❌ ビルド・テストのうち一つだけ見て残りは「大丈夫だろう」と省略
- ❌ テストが失敗したのに「些細な問題」として合格したかのように通過
- ❌ コマンド出力を確認せず（または実行自体せず）成功と報告
- ❌ 一部のケースだけ確認して変更範囲全体が動作すると断定

> **適用のヒント**: 報告直前に一文で自問する — 「今これを証明するコマンドを自分が実際に走らせ、その出力を見たか？」そうでなければまだ完了ではない。（コード品質の点検は `code-review`、テスト範囲・戦略は `test-strategy` と一緒に使う）

## 4. チェックリスト

- [ ] 成功を主張する根拠となる実際のコマンド出力を確認したか
- [ ] ビルド・テスト・lint・受け入れ基準（DoD）をすべて充足したか
- [ ] 変更したすべてのファイル・影響範囲を実行で確認したか（一部だけ見て断定していないか）
- [ ] 失敗・部分的成功・未確認項目を隠さず正直に報告したか
- [ ] 「たぶん動く」のような推測表現で検証を代替していないか
- [ ] 実行が不可能だったなら、その限界を明記したか（検証したふりをしていないか）
