# 🌿 Git ワークフロー — ブランチ戦略 & 協業

> ブランチを短く・小さく保ち、レビュー・CI を通過しなければ main に入れないように固め、誰が作業しても履歴が綺麗でコンフリクトが少なくなるようにする。新しいブランチを切ったり、PR を上げたり、マージ・コンフリクトを扱うときに読む。

AI エージェントと人が最もよく犯す協業ミスは、**main に直接 push し、巨大な変更を一度に上げ、共有ブランチを force-push で塗り替えること**だ。最初は速く見えても、レビュー・差し戻し・原因追跡が急激に難しくなる。フローをルールに固めれば、エージェントもその枠の中で安全にコミットする。

## 1. 中核原則

- トランクベースで — main は常にデプロイ可能な状態に保ち、作業は短命(理想的には1~2日)ブランチで行い素早くマージする。長生きするブランチはコンフリクトを育てる。
- main に直接 push しない — すべての変更はブランチ → PR を経る。main はブランチ保護で直接 push・force-push を防ぐ。
- PR は小さく — 1 つの PR は 1 つのことだけ。小さな PR はレビューが速くバグを隠しにくい。
- マージ前のゲートは両方通過 — 最低 1 人のレビュー承認 + CI(ビルド・テスト・リント)グリーン。どちらか一方でも赤のままマージしない。
- 共有ブランチは履歴を上書きしない — 他人が見るブランチ(main・共同作業ブランチ)に force-push 禁止。rebase は自分のローカルブランチでのみ。
- シークレット・ビルド成果物はコミットしない — `.gitignore` で最初から除外する。一度上がった秘密は漏れたものとみなす。

## 2. ルール

### 2-1. ブランチ命名 & 短命に保つ

```
# ❌ 禁止 — 意味のない/長生きするブランチ
git checkout -b temp
git checkout -b my-work        # 数週間放置 → マージ地獄

# ✅ 推奨 — タイプ/イシュー/要約、作業が終わったらすぐマージ・削除
git checkout -b feat/142-login-rate-limit
git checkout -b fix/payment-null-amount
git checkout -b chore/bump-deps
```

- 形式の例: `<タイプ>/<イシュー番号>-<短い要約>` (タイプ: feat・fix・chore・docs・refactor など)。小文字・kebab-case。
- 1 ブランチ = 1 作業。終わったらマージ後に削除して一覧を綺麗に保つ。

### 2-2. main 保護 & PR 経由

```
# ❌ 禁止 — main に直接コミット・push
git switch main && git commit -m "..." && git push

# ✅ 推奨 — ブランチで作業 → push → PR 作成
git switch -c feat/142-login-rate-limit
git push -u origin feat/142-login-rate-limit
# → PR を開いてレビュー・CI を受ける
```

- リポジトリ設定で main(および release ブランチ)に **ブランチ保護** をかける: 直接 push 禁止、PR 必須、レビュー・CI 通過必須。

### 2-3. 小さな PR & マージ前ゲート

```
# ❌ 禁止 — 機能・リファクタ・フォーマットを 1 つの PR に 90 ファイルで
# ✅ 推奨 — 目的は 1 つ、可能な限り小さく。レビュアーが一度に読める分量
```

- レビュー承認 1+ & CI グリーンの **両方** を満たしてマージ。失敗した CI を「あとで直す」とマージしない(セルフ承認禁止)。
- 上げる前のセルフレビュー(デバッグ痕跡の除去など)は `code-review`、完了の証明(DoD・CI グリーン確認)は `verification-before-completion` に従う。

### 2-4. rebase vs merge & コンフリクト解決

```
# ✅ 自分のローカルブランチを最新 main の上に整理 (綺麗な直線履歴)
git switch feat/142-login-rate-limit
git fetch origin
git rebase origin/main
# コンフリクト時: ファイル修正 → git add <file> → git rebase --continue
#               戻すなら git rebase --abort

# ✅ 共有履歴の保存が重要なら merge (マージコミットを残す)
git merge origin/main
```

- **rebase**: まだ共有前の自分のブランチ整理に。履歴が直線になる。
- **merge**: すでに共有された履歴を保存するとき。チームルール(直線 vs マージコミット)に従う。
- コンフリクトは決して一方を無闇に上書きせず、両方の意図を確認して解決する。

### 2-5. force-push 禁止 (共有ブランチ)

```
# ❌ 絶対禁止 — 共有ブランチ履歴を破壊 (他人の作業消失)
git push --force origin main

# ✅ 許容 — 自分の個人ブランチで、それも安全オプションで
git push --force-with-lease origin feat/142-login-rate-limit
```

- `--force-with-lease` はその間に他人が push していれば拒否 → 上書き事故を防ぐ。共有ブランチにはそれでも使わない。

### 2-6. .gitignore — シークレット・成果物除外

```gitignore
# ❌ コミットされてはいけないもの (例)
.env
.env.*
*.key
*.pem
secrets/

# ビルド成果物・依存・ローカルキャッシュ
dist/
build/
node_modules/
*.log
.DS_Store
```

- 秘密(.env・キー・トークン)とビルド成果物・依存フォルダは最初から追跡しない。
- 誤って秘密をコミットしたら: 即座にキーを失効・交換する(履歴から消しても既に露出したものとみなす)。

### 2-7. 頻繁に・意味単位でコミット

```
# ❌ 禁止 — 一日中溜めてから巨大な一発コミット
# ✅ 推奨 — 動く小さな単位で頻繁にコミット → push
```

- 小さく頻繁なコミットはレビュー・差し戻し・原因追跡(`git bisect`)を容易にする。
- コミットメッセージ形式(タイプ・タイトル・本文ルール)は `coding-styles` に従う。

## 3. よくあるミス

レビュー時にふるい落とす。

- ❌ main に直接 push、または保護なしで放置
- ❌ 数百ファイル・複数目的が混ざった巨大 PR
- ❌ CI 赤・レビューなしでマージ
- ❌ 共有ブランチを `--force` で塗り替えて他人のコミット消失
- ❌ コンフリクトを両方の意図確認なしに一方へ上書き
- ❌ `.env`・キー・`node_modules`・`dist` をコミット
- ❌ マージ済みブランチを削除せず一覧が数百個に積み上がる

> **適用のコツ**: リポジトリホスティング(GitHub/GitLab など)にブランチ保護・必須レビュー・必須 CI をオンにしておけば、ルールを人・エージェントがうっかり忘れてもシステムが防いでくれる。メッセージ規約は `coding-styles`、変更レビュー基準は `architecture-layering` と一緒に見るとよい。

## 4. チェックリスト

- [ ] 作業を main ではなく短命ブランチで行い、ブランチ名が `タイプ/要約` ルールに従っているか
- [ ] PR が 1 つの目的で小さく分割されているか
- [ ] レビュー承認 1+ と CI(ビルド・テスト・リント)グリーンの両方を通過したか
- [ ] 共有ブランチに force-push していないか (個人ブランチは `--force-with-lease`)
- [ ] `.env`・キー・トークン・ビルド成果物がコミットに混ざっていないか (`.gitignore` 確認)
- [ ] マージ後に作業ブランチを削除したか
