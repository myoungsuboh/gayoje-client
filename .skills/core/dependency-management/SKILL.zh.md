# 📦 依赖管理 — 采用判断 & 版本卫生

> 不把依赖看作「免费」而当作成本,从而谨慎添加、可复现地固定版本,并把依赖树保持得小而新。在新引入或更新库、评审依赖文件时阅读。

AI 代理最常犯的依赖错误,是**即使是琐碎的事也当场安装新包,以及编造并不存在的包名**。每个依赖都连带引入包体积、维护负担、供应链攻击面和许可证义务。把添加决策钉成规则,连代理也会先问「真的需要吗?」。
> 本文档只涉及**是否添加的判断和版本卫生**。漏洞(CVE)扫描、自动打补丁、CI 审计门禁由 `dependency-scanning` 负责。

## 1. 核心原则

- 依赖是成本 — 添加前权衡包体积、维护、安全面、许可证,先看标准库或已在用的依赖能否解决。
- 保持最小依赖 — 不要为一两个函数引入大包,不再使用的依赖立即移除。
- 保证可复现性 — 始终提交锁文件,版本用固定(或窄范围)钉死,消除「在我电脑上能跑」。
- 定期小步更新 — 频繁做小更新,不要把数年的量一次性堆上去。
- 确认许可证 — 采用前查看许可证,公司政策上有风险的(如强 copyleft)不引入。
- 警惕幻觉包名 — AI 生成的安装命令里的包名,要在官方注册表确认实际存在且正确后再安装。

## 2. 规则

### 2-1. 添加前先权衡成本(真的需要吗?)

新依赖必须通过以下才添加。

| 检查 | 问题 |
|---|---|
| 替代 | 标准库/语言特性、已有的依赖做不到吗? |
| 规模 | 相对于所用功能,包及其传递依赖树是否过大? |
| 健康度 | 近期更新、下载、issue 响应是否还活着?(规避被弃置的包) |
| 许可证 | 符合公司政策吗? |

```text
// ❌ 禁止 — 为一行工具函数添加包
npm install is-even        // 直接 n % 2 === 0 就够了

// ✅ 推荐 — 优先标准/已有手段,确有必要时才用可信赖的包
const isEven = (n) => n % 2 === 0
```

### 2-2. 提交锁文件 + 版本固定(可复现性)

```text
// ❌ 禁止 — 锁文件未提交 + 无限范围 → 每次安装版本都不同
把 lockfile 加到 .gitignore
"some-lib": "*"   // 或 "latest"

// ✅ 推荐 — 提交锁文件 + 固定版本(或窄范围)
git add <lockfile>          // package-lock.json、poetry.lock、gradle.lockfile 等
"some-lib": "4.2.1"         // 固定,或按政策用 "~4.2.1"
```

- 锁文件固定真正安装的版本(含传递依赖)— **务必提交**。
- 清单(package.json 等)里的范围要窄。`*`、`latest` 会破坏构建可复现性,故禁止。
- CI/部署以锁文件为基准安装(`npm ci` 等),防止重新解读清单。

### 2-3. 定期小规模更新

```text
// ❌ 禁止 — 弃置 2 年后一次性大版本跳跃 → 变更点爆炸,调试地狱
一次性几十个大版本升级

// ✅ 推荐 — 小而频繁,一次一批 + 用测试验证
定期先做 patch/minor,大版本看变更日志逐个处理
```

- 频繁做小更新 → 每次变化都小,易于查出回归。
- 更新后务必用测试验证。(相关: `ai-generated-code-review`)
- 安全补丁自动化(Dependabot/Renovate 等)按上述边界遵循 `dependency-scanning`。

### 2-4. 移除未使用依赖 & 分离

```text
// ❌ 禁止 — 不用了却留在清单里的依赖,prod/dev 混在一起
import 没了依赖却还在
把构建工具放进运行时依赖

// ✅ 推荐 — 清理未使用 + 分离运行时/开发依赖
移除无使用处的依赖(如 depcheck 类检查)
运行时放 dependencies,构建/测试放 devDependencies(或等价分组)
```

### 2-5. 注意幻觉包名(供应链安全)

```text
// ❌ 禁止 — 不经确认就安装 AI 建议的看似合理的名字(错别字、幽灵包 = slopsquatting 风险)
npm install react-http-client   // 是否实际存在未确认

// ✅ 推荐 — 在官方注册表/官方文档确认准确的名字、所有者后再安装
在官方页面确认包名、下载量、仓库 → 安装
```

## 3. 常见错误

AI 常造出来的 — 评审时剔除。

- ❌ 为一行逻辑添加包(不必要的传递依赖树、攻击面)
- ❌ 锁文件未提交或用 `*`/`latest` 范围 → 不可复现的构建
- ❌ 拖延更新直到一次性大版本爆炸 → 巨型 PR、无法追踪回归
- ❌ 放任未使用依赖,不分离运行时/开发依赖
- ❌ 未确认许可证就采用 → 部署时才发现违反政策
- ❌ 把不存在或拼错的包名照原样安装(供应链风险)

> **应用技巧**: 在 AGENTS.md / 规则文件里钉上一行「添加新依赖需正当化,优先标准库,包名安装前确认」,代理每次都会守住这个标准。(与 `agent-rules-file` 技能一起用效果更佳)

## 4. 检查清单

- [ ] 新依赖是否真的需要 — 是否无法用标准库/已有依赖替代
- [ ] 是否提交了锁文件,并把版本固定(或窄范围)钉死(没有 `*`/`latest`)
- [ ] 是否小而频繁地更新,更新后用测试验证了
- [ ] 是否移除了未使用依赖并分离了运行时/开发依赖
- [ ] 采用的库的许可证是否符合公司政策
- [ ] 安装的包名是否是官方注册表里实际存在的准确名字(非幻觉、非错别字)
