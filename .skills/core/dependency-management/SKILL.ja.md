# 📦 依存関係管理 — 採用判断 & バージョン衛生

> 依存を「タダ」とみなさずコストとして扱い、追加は慎重に、バージョンは再現可能に固定し、ツリーを小さく最新に保つ。ライブラリを新しく入れたり更新したり依存ファイルをレビューするときに読む。

AI エージェントが最もよく犯す依存の間違いは、**些細なことでも新しいパッケージをその場でインストールし、存在しないパッケージ名をでっち上げること**だ。すべての依存はバンドルサイズ・保守負担・サプライチェーン攻撃表面・ライセンス義務を一緒に連れてくる。追加の判断をルールで釘付けにすれば、エージェントも「本当に必要か?」をまず検討する。
> このドキュメントは**追加の可否判断とバージョン衛生**だけを扱う。脆弱性(CVE)スキャン・自動パッチ・CI 監査ゲートは `dependency-scanning` が担当する。

## 1. 核心原則

- 依存はコストだ — 追加前にバンドルサイズ・保守・セキュリティ表面・ライセンスを検討し、標準ライブラリやすでに使っている依存で解決できるかをまず見る。
- 最小依存を保つ — 1つ2つの関数のために大きなパッケージを入れず、使わなくなった依存は即座に除去する。
- 再現性を保証する — ロックファイルを常にコミットし、バージョンはピン(または狭い範囲)で固定して「私のPCでは動いた」をなくす。
- 定期的に少しずつ更新する — 小さな更新を頻繁にし、数年分を一度にまとめて上げない。
- ライセンスを確認する — 採用前にライセンスを見て、会社方針上リスクのあるもの(例: 強いコピーレフト)は入れない。
- 幻覚パッケージ名を警戒する — AI が生成したインストールコマンドのパッケージ名は公式レジストリで実在・正確かを確認してからインストールする。

## 2. ルール

### 2-1. 追加前にコストを検討する(本当に必要か?)

新しい依存は以下を通過してから追加する。

| 点検 | 質問 |
|---|---|
| 代替 | 標準ライブラリ/言語機能、既存の依存ではできないか? |
| 規模 | 使う機能に比べてパッケージ・推移的ツリーが過大か? |
| 健全度 | 最近の更新・ダウンロード・イシュー応答が生きているか?(放置パッケージ回避) |
| ライセンス | 会社方針に合うか? |

```text
// ❌ 禁止 — 一行のユーティリティのためにパッケージ追加
npm install is-even        // 直接 n % 2 === 0 で十分

// ✅ 推奨 — 標準/既存の手段を優先、本当に必要なときだけ信頼できるパッケージ
const isEven = (n) => n % 2 === 0
```

### 2-2. ロックファイルのコミット + バージョンピン(再現性)

```text
// ❌ 禁止 — ロックファイル未コミット + 無制限範囲 → インストールのたびにバージョンが変わる
.gitignore に lockfile を追加
"some-lib": "*"   // または "latest"

// ✅ 推奨 — ロックファイルのコミット + バージョン固定(または狭い範囲)
git add <lockfile>          // package-lock.json・poetry.lock・gradle.lockfile など
"some-lib": "4.2.1"         // ピン、または方針に応じて "~4.2.1"
```

- ロックファイルが本当のインストールバージョン(推移的含む)を固定する — **必ずコミット**する。
- マニフェスト(package.json など)の範囲は狭く。`*`・`latest` はビルド再現性を壊すので禁止。
- CI/デプロイはロックファイル基準でインストール(`npm ci` など)してマニフェストの再解釈を防ぐ。

### 2-3. 定期的な小規模更新

```text
// ❌ 禁止 — 2年放置後に一度にメジャージャンプ → 変更点が爆発、デバッグ地獄
一度に数十個のメジャーアップグレード

// ✅ 推奨 — 小さく頻繁に、一度に一塊ずつ + テストで検証
定期的に patch/minor を先に、メジャーは変更ログを見て個別に処理
```

- 小さな更新を頻繁に → 各変化が小さく回帰を見つけやすい。
- 更新後は必ずテストで検証する。(関連: `ai-generated-code-review`)
- セキュリティパッチ自動化(Dependabot/Renovate など)は上記の境界どおり `dependency-scanning` に従う。

### 2-4. 未使用依存の除去 & 分離

```text
// ❌ 禁止 — 使っていないのにマニフェストに残った依存、prod/dev が混在
import が消えたのに依存はそのまま
ビルドツールをランタイム依存に置く

// ✅ 推奨 — 未使用の整理 + ランタイム/開発依存の分離
使用箇所のない依存を除去(例: depcheck 系の点検)
ランタイムは dependencies、ビルド/テストは devDependencies(または同等グループ)
```

### 2-5. 幻覚パッケージ名に注意(サプライチェーン安全)

```text
// ❌ 禁止 — AI が提案したもっともらしい名前を確認なしでインストール(タイポ・幽霊パッケージ = スロップスクワッティングのリスク)
npm install react-http-client   // 実在の有無未確認

// ✅ 推奨 — 公式レジストリ/公式ドキュメントで正確な名前・所有者を確認してからインストール
公式ページでパッケージ名・ダウンロード・リポジトリを確認 → インストール
```

## 3. よくある間違い

AI がよく作るもの — レビュー時に取り除く。

- ❌ 一行のロジックのためにパッケージ追加(不要な推移的ツリー・攻撃表面)
- ❌ ロックファイル未コミットまたは `*`/`latest` 範囲 → 再現不能なビルド
- ❌ 更新を先延ばしにして一度にメジャー爆発 → 巨大 PR、回帰の追跡不能
- ❌ 使っていない依存の放置、ランタイム/開発依存の未分離
- ❌ ライセンス未確認のまま採用 → デプロイ時点で方針違反を発見
- ❌ 存在しないかタイポしたパッケージ名をそのままインストール(サプライチェーンのリスク)

> **適用のコツ**: AGENTS.md / ルールファイルに「新しい依存の追加は正当化が必要、標準ライブラリを優先、パッケージ名はインストール前に確認」の一行を埋め込んでおくと、エージェントが毎回この基準を守る。(`agent-rules-file` スキルと一緒に使うと効果的)

## 4. チェックリスト

- [ ] 新しい依存が本当に必要か — 標準ライブラリ/既存の依存で代替不能か
- [ ] ロックファイルをコミットし、バージョンをピン(または狭い範囲)で固定したか(`*`/`latest` なし)
- [ ] 更新を小さく頻繁にし、更新後にテストで検証したか
- [ ] 未使用の依存を除去しランタイム/開発依存を分離したか
- [ ] 採用したライブラリのライセンスが会社方針に合うか
- [ ] インストールしたパッケージ名が公式レジストリに実在する正確な名前か(幻覚・タイポでない)
