# 📦 의존성 관리 — 채택 판단 & 버전 위생

> 의존성을 "공짜"로 보지 않고 비용으로 다뤄, 추가는 신중히 하고 버전은 재현 가능하게 고정하며 트리를 작고 최신으로 유지한다. 라이브러리를 새로 넣거나 갱신하거나 의존성 파일을 검토할 때 읽는다.

AI 에이전트가 가장 흔히 저지르는 의존성 실수는 **사소한 일에도 새 패키지를 즉석에서 설치하고, 존재하지 않는 패키지명을 지어내는 것**이다. 모든 의존성은 번들 크기·유지보수 부담·공급망 공격 표면·라이선스 의무를 함께 들여온다. 추가 결정을 규칙으로 못 박으면 에이전트도 "정말 필요한가?"를 먼저 따진다.
> 이 문서는 **추가 여부 판단과 버전 위생**만 다룬다. 취약점(CVE) 스캔·자동 패치·CI 감사 게이트는 `dependency-scanning`이 담당한다.

## 1. 핵심 원칙

- 의존성은 비용이다 — 추가 전 번들 크기·유지보수·보안 표면·라이선스를 따지고, 표준 라이브러리나 이미 쓰는 의존성으로 해결되는지 먼저 본다.
- 최소 의존을 유지한다 — 한두 함수 때문에 큰 패키지를 들이지 않고, 안 쓰게 된 의존성은 즉시 제거한다.
- 재현성을 보장한다 — 잠금 파일을 항상 커밋하고, 버전은 핀(또는 좁은 범위)으로 고정해 "내 PC에선 됐다"를 없앤다.
- 정기적으로 조금씩 갱신한다 — 작은 갱신을 자주 하고, 몇 년 치를 한 번에 몰아 올리지 않는다.
- 라이선스를 확인한다 — 채택 전 라이선스를 보고, 회사 정책상 위험한 것(예: 강한 카피레프트)은 들이지 않는다.
- 환각 패키지명을 경계한다 — AI가 생성한 설치 명령의 패키지명은 공식 레지스트리에서 실재·정확 여부를 확인한 뒤 설치한다.

## 2. 규칙

### 2-1. 추가 전 비용을 따진다 (정말 필요한가?)

새 의존성은 아래를 통과해야 추가한다.

| 점검 | 질문 |
|---|---|
| 대안 | 표준 라이브러리/언어 기능, 이미 있는 의존성으로 안 되나? |
| 규모 | 쓰는 기능에 비해 패키지·트랜지티브 트리가 과한가? |
| 건강도 | 최근 갱신·다운로드·이슈 응답이 살아 있나? (방치 패키지 회피) |
| 라이선스 | 회사 정책에 맞나? |

```text
// ❌ 금지 — 한 줄짜리 유틸을 위해 패키지 추가
npm install is-even        // 직접 n % 2 === 0 이면 충분

// ✅ 권장 — 표준/기존 수단 우선, 꼭 필요할 때만 신뢰할 만한 패키지
const isEven = (n) => n % 2 === 0
```

### 2-2. 잠금 파일 커밋 + 버전 핀 (재현성)

```text
// ❌ 금지 — 잠금 파일 미커밋 + 무제한 범위 → 설치할 때마다 버전이 달라짐
.gitignore 에 lockfile 추가
"some-lib": "*"   // 또는 "latest"

// ✅ 권장 — 잠금 파일 커밋 + 버전 고정(또는 좁은 범위)
git add <lockfile>          // package-lock.json·poetry.lock·gradle.lockfile 등
"some-lib": "4.2.1"         // 핀, 또는 정책에 따라 "~4.2.1"
```

- 잠금 파일이 진짜 설치 버전(트랜지티브 포함)을 고정한다 — **반드시 커밋**한다.
- 매니페스트(package.json 등)의 범위는 좁게. `*`·`latest`는 빌드 재현성을 깨므로 금지.
- CI/배포는 잠금 파일 기준으로 설치(`npm ci` 등)해 매니페스트 재해석을 막는다.

### 2-3. 정기적 소규모 갱신

```text
// ❌ 금지 — 2년 방치 후 한 번에 메이저 점프 → 변경점 폭증, 디버깅 지옥
한꺼번에 수십 개 메이저 업그레이드

// ✅ 권장 — 작게 자주, 한 번에 한 묶음씩 + 테스트로 검증
주기적으로 patch/minor 먼저, 메이저는 변경 로그 보고 개별 처리
```

- 작은 갱신을 자주 → 각 변화가 작아 회귀를 찾기 쉽다.
- 갱신 후 반드시 테스트로 검증한다. (관련: `ai-generated-code-review`)
- 보안 패치 자동화(Dependabot/Renovate 등)는 위 경계대로 `dependency-scanning`을 따른다.

### 2-4. 미사용 의존성 제거 & 분리

```text
// ❌ 금지 — 안 쓰는데 매니페스트에 남은 의존성, prod/dev 뒤섞임
import 사라졌는데 의존성은 그대로
빌드 도구를 런타임 의존성에 둠

// ✅ 권장 — 미사용 정리 + 런타임/개발 의존성 분리
사용처 없는 의존성 제거(예: depcheck 류 점검)
런타임은 dependencies, 빌드/테스트는 devDependencies(또는 동등 그룹)
```

### 2-5. 환각 패키지명 주의 (공급망 안전)

```text
// ❌ 금지 — AI가 제안한 그럴듯한 이름을 확인 없이 설치 (오타·유령 패키지 = 슬롭스쿼팅 위험)
npm install react-http-client   // 실재 여부 미확인

// ✅ 권장 — 공식 레지스트리/공식 문서에서 정확한 이름·소유자 확인 후 설치
공식 페이지에서 패키지명·다운로드·리포지토리 확인 → 설치
```

## 3. 흔한 실수

AI가 자주 만드는 것 — 검토 시 거른다.

- ❌ 한 줄 로직 때문에 패키지 추가 (불필요한 트랜지티브 트리·공격 표면)
- ❌ 잠금 파일 미커밋 또는 `*`/`latest` 범위 → 재현 불가능한 빌드
- ❌ 갱신을 미루다 한 번에 메이저 폭증 → 거대 PR, 회귀 추적 불가
- ❌ 안 쓰는 의존성 방치, 런타임/개발 의존성 미분리
- ❌ 라이선스 미확인 채 채택 → 배포 시점에 정책 위반 발견
- ❌ 존재하지 않거나 오타 난 패키지명을 그대로 설치 (공급망 위험)

> **적용 팁**: AGENTS.md / 규칙 파일에 "새 의존성 추가는 정당화 필요, 표준 라이브러리 우선, 패키지명은 설치 전 확인" 한 줄을 박아두면 에이전트가 매번 이 기준을 지킨다. (`agent-rules-file` 스킬과 함께 쓰면 효과적)

## 4. 체크리스트

- [ ] 새 의존성이 정말 필요한가 — 표준 라이브러리/기존 의존성으로 대체 불가한가
- [ ] 잠금 파일을 커밋했고, 버전을 핀(또는 좁은 범위)으로 고정했는가 (`*`/`latest` 없음)
- [ ] 갱신을 작게 자주 했고, 갱신 후 테스트로 검증했는가
- [ ] 미사용 의존성을 제거하고 런타임/개발 의존성을 분리했는가
- [ ] 채택한 라이브러리의 라이선스가 회사 정책에 맞는가
- [ ] 설치한 패키지명이 공식 레지스트리에 실재하는 정확한 이름인가 (환각·오타 아님)
