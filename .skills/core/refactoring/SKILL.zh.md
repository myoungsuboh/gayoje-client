# ♻️ 重构 — 保持行为的改进

> 让外在行为保持原样，只安全地改变内部结构。在清理代码坏味道或在添加功能前后整理结构时阅读。

AI 智能体在重构中最常犯的错误是 **改结构的同时偷偷把行为也改了**，以及 **一次推平太多**。重构的核心是「保持行为」。用测试铺好安全网，拆成小步频繁验证，AI 也只会产出安全的变更。

## 1. 核心原则

- 重构是 **保持行为** — 输入/输出·副作用·公开 API 必须保持不变。行为变了那就不是重构而是功能变更。
- **测试是安全网** — 重构前确认有覆盖该行为的测试，没有就先补上再开始。
- **小步进行，频繁验证** — 一次只改一处，每一步都用测试·构建确认绿灯。
- **不要把功能变更和重构混在同一次提交** — 会让审查·回退·追溯原因变得不可能。把提交分开。
- **童子军规则** — 让动过的代码比原来稍微干净一点再离开。但与当前作业无关的大规模整理要单独拆出去。
- **避免大爆炸式重写** — 与其整体重写不如渐进改进。当下还不上的技术债不要藏在代码里，用工单可视化。

## 2. 规则

### 2-1. 先用测试把行为钉死

重构前，确认覆盖待改行为的测试是绿灯。没有就先补上(特征化测试)。(编写测试见 `test-strategy`，完成前的验证流程见 `verification-before-completion`)

```
// ❌ 禁止 — 没有测试就直接改结构 → 无从知道是否改坏了
拆函数、改名字…… 「大概没问题吧」

// ✅ 推荐 — 先铺安全网
1) 确认既有行为的测试(没有就补上) → 绿灯
2) 重构
3) 重新跑同样的测试 → 仍然绿灯
```

### 2-2. 拆成小步并每一步验证

一次提交一处变更。大型重构拆成一连串安全的小变换。

```
// ❌ 禁止 — 一次提交里同时做函数拆分 + 改名 + 逻辑整理 + 替换依赖
//          坏了就找不到在哪坏的

// ✅ 推荐 — 每一步确认绿灯后再进入下一步
第1步: 只把变量/函数名改成有意义的  → 测试通过
第2步: 从长函数里抽出一块            → 测试通过
第3步: 去除重复                      → 测试通过
```

### 2-3. 功能变更与重构分开提交

即便在同一个 PR 里，也把「仅重构」的提交和「行为变更」的提交分开。审查者能精确看到行为变更点。

```
// ❌ 禁止
commit: "整理支付模块并应用折扣率 5%→7%"   ← 结构变更里藏着行为变更

// ✅ 推荐
commit 1: "refactor: 把支付计算逻辑抽成函数 (行为相同)"
commit 2: "feat: 折扣率 5%→7% 变更"
```

### 2-4. 把代码坏味道当作信号

看到下面的信号就是重构候选。但不是「有味道 = 无条件修」，而是以正在动的代码为主进行整理。

| 坏味道 | 信号 | 常见处方 |
|---|---|---|
| 重复代码 | 同样的逻辑复制粘贴在多处 | 抽取为公共函数/模块 |
| 长函数 | 一个函数超出屏幕·做好几件事 | 按意义单位抽取函数 |
| 强耦合 | 改一处就把远处弄坏 | 用边界·接口分离 (参见 `architecture-layering`) |
| 大参数列表 / 重复条件判断 | 参数 5 个以上，同一分支重复 | 打包成对象·多态/提前返回 |

### 2-5. 用渐进代替大爆炸式重写，把债务可视化

```
// ❌ 禁止 — 「这个模块整体重写更快」之后数周停摆·回归暴增

// ✅ 推荐 — 一点点替换(例如: 让旧代码与新代码并行一段时间)，每一步保持可部署状态
//          当下还不上的债务不要藏在代码里，用工单:  // 不是 TODO → 登记为 issue #1234
```

## 3. 常见错误

AI 经常制造的 — 审查时筛掉。

- ❌ 没有测试就重构并以「整理好了」收尾 (未验证保持行为)
- ❌ 在结构改进的同时偷偷改了行为·返回值·错误处理
- ❌ 在一次提交/一个 PR 里一次性推平几十个文件导致无法审查
- ❌ 超出请求范围「顺便」做大规模重写
- ❌ 改了公开 API/签名而弄坏调用方 (那不是重构)
- ❌ 把债务只埋在代码注释里，放任其无法追踪

## 4. 检查清单

- [ ] 改过的代码行为是否原样 (输入/输出·副作用·公开 API 相同)?
- [ ] 重构前覆盖该行为的测试是否为绿灯 (没有的话是否先补上了)?
- [ ] 是否拆成小步并每一步用构建·测试验证?
- [ ] 功能变更与重构是否按提交分离?
- [ ] 是否没有夹带超出请求范围的大规模重写?
- [ ] 留下的技术债是否用工单/issue 可视化了?
