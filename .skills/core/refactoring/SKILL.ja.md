# ♻️ リファクタリング — 動作保存の改善

> 見かけの動作はそのままに内部構造だけを安全に変える。コードの臭いを整理したり機能追加の前後に構造を整えたりするときに読む。

AI エージェントがリファクタリングで最もよく犯すミスは、**構造を変えながら動作までこっそり変えること**、そして **一度に変えすぎること**だ。リファクタリングの核心は「動作保存」だ。テストで安全網を敷き、小さなステップに分けて頻繁に検証すれば、AI も安全な変更だけを作り出す。

## 1. 中核原則

- リファクタリングは **動作保存** だ — 入力/出力・副作用・公開 API がそのままでなければならない。動作が変わればそれはリファクタリングではなく機能変更だ。
- **テストが安全網** だ — リファクタリング前にその動作を覆うテストがあるか確認し、なければ先に追加してから始める。
- **小さなステップで、頻繁に検証** する — 一度に一つずつ変え、毎ステップでテスト・ビルドで青信号を確認する。
- **機能変更とリファクタリングを同じコミットに混ぜない** — レビュー・差し戻し・原因追跡が不可能になる。コミットを分ける。
- **ボーイスカウトルール** — 触ったコードは最初より少し綺麗にして去る。ただし今の作業と無関係な大規模整理は別に切り出す。
- **ビッグバン書き直しを避ける** — 丸ごと書き直すより段階的に改善する。今すぐ返せない技術的負債はコードに隠さずチケットで可視化する。

## 2. ルール

### 2-1. テストで動作を先に固定する

リファクタリング前、変える動作を覆うテストが青信号か確認する。なければ先に追加する(特性化テスト)。(テスト作成は `test-strategy`、完了前の検証手順は `verification-before-completion` 参照)

```
// ❌ 禁止 — テストなしでいきなり構造変更 → 壊れたか知る術がない
関数を分けて名前を変えて… 「たぶんうまくいくだろう」

// ✅ 推奨 — 安全網が先
1) 既存動作のテスト確認(なければ追加) → 青信号
2) リファクタリング
3) 同じテストを再実行 → 依然青信号
```

### 2-2. 小さなステップに分けて毎ステップ検証する

1 コミットに変更一つ。大きなリファクタリングは安全な小さな変換の連続に分ける。

```
// ❌ 禁止 — 1 コミットに関数分離 + 名前変更 + ロジック整理 + 依存差し替えを一度に
//          壊れたらどこで壊れたか見つけられない

// ✅ 推奨 — ステップごとに青信号確認後、次へ
ステップ1: 変数/関数の名前だけ意味のあるものに  → テスト通過
ステップ2: 長い関数から一塊を抽出              → テスト通過
ステップ3: 重複を除去                          → テスト通過
```

### 2-3. 機能変更とリファクタリングはコミットを分ける

同じ PR でも「リファクタリングのみ」コミットと「動作変更」コミットを分ける。レビュアーが動作変更地点を正確に見られる。

```
// ❌ 禁止
commit: "決済モジュールを整理して割引率 5%→7% を反映"   ← 構造変更に動作変更が隠れる

// ✅ 推奨
commit 1: "refactor: 決済計算ロジックを関数抽出 (動作同一)"
commit 2: "feat: 割引率 5%→7% 変更"
```

### 2-4. コードの臭いを信号にする

下記の信号が見えたらリファクタリング候補だ。ただし「臭う = 無条件に直す」ではなく、今触っているコードを中心に整理する。

| 臭い | 信号 | よくある処方 |
|---|---|---|
| 重複コード | 同じロジックが複数箇所にコピペ | 共通関数/モジュールへ抽出 |
| 長い関数 | 一関数が画面を超える・複数の仕事をする | 意味単位で関数抽出 |
| 密結合 | 一箇所直すと遠くが壊れる | 境界・インターフェースで分離 (`architecture-layering` 参照) |
| 大きな引数リスト / 重複した条件分岐 | 引数 5 個以上、同じ分岐の繰り返し | オブジェクトに束ねる・多態性/早期リターン |

### 2-5. ビッグバン書き直しの代わりに段階的に、負債は可視化する

```
// ❌ 禁止 — 「このモジュールは丸ごと書き直す方が速いです」の後、数週間動作停止・回帰急増

// ✅ 推奨 — 少しずつ差し替え(例: 旧コードと新コードをしばらく並行)、毎ステップでデプロイ可能な状態を維持
//          今返せない負債はコードに隠さずチケットで:  // TODO ではなく → イシュー #1234 として登録
```

## 3. よくあるミス

AI がよく作るもの — レビュー時にふるい落とす。

- ❌ テストなしでリファクタリングして「整理しました」で終わり (動作保存未検証)
- ❌ 構造改善とともに動作・戻り値・エラー処理をこっそり変える
- ❌ 1 コミット/1 PR に数十ファイルを一度に塗り替えてレビュー不能
- ❌ 依頼していない範囲まで「ついでに」大規模書き直し
- ❌ 公開 API/シグネチャを変えて呼び出し側を壊す (それはリファクタリングではない)
- ❌ 負債をコードコメントだけに埋めて追跡不能に放置

## 4. チェックリスト

- [ ] 変えたコードの動作はそのままか (入力/出力・副作用・公開 API 同一)
- [ ] リファクタリング前にその動作を覆うテストが青信号だったか (なければ先に追加したか)
- [ ] 小さなステップに分けて毎ステップをビルド・テストで検証したか
- [ ] 機能変更とリファクタリングがコミットで分離されているか
- [ ] 依頼範囲を超える大規模書き直しを差し込んでいないか
- [ ] 残した技術的負債をチケット/イシューで可視化したか
