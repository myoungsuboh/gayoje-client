# ♻️ 리팩토링 — 동작 보존 개선

> 겉보기 동작은 그대로 두고 내부 구조만 안전하게 바꾼다. 코드 냄새를 정리하거나 기능 추가 전후로 구조를 다듬을 때 읽는다.

AI 에이전트가 리팩토링에서 가장 흔히 저지르는 실수는 **구조를 바꾸면서 동작까지 슬쩍 바꾸는 것**, 그리고 **한 번에 너무 많이 갈아엎는 것**이다. 리팩토링의 핵심은 "동작 보존"이다. 테스트로 안전망을 깔고, 작은 단계로 쪼개 자주 검증하면 AI도 안전한 변경만 만들어낸다.

## 1. 핵심 원칙

- 리팩토링은 **동작 보존**이다 — 입력/출력·부수효과·공개 API가 그대로여야 한다. 동작이 바뀌면 그건 리팩토링이 아니라 기능 변경이다.
- **테스트가 안전망**이다 — 리팩토링 전에 해당 동작을 덮는 테스트가 있는지 확인하고, 없으면 먼저 추가한 뒤 시작한다.
- **작은 단계로, 자주 검증**한다 — 한 번에 하나씩 바꾸고 매 단계마다 테스트·빌드로 초록불을 확인한다.
- **기능 변경과 리팩토링을 같은 커밋에 섞지 않는다** — 리뷰·되돌리기·원인 추적이 불가능해진다. 커밋을 분리한다.
- **보이스카웃 룰** — 만진 코드는 처음보다 조금 더 깨끗하게 두고 떠난다. 단, 지금 작업과 무관한 대규모 정리는 별도로 뺀다.
- **빅뱅 재작성을 지양**한다 — 통째로 새로 쓰기보다 점진적으로 개선한다. 당장 못 갚을 기술부채는 코드에 숨기지 말고 티켓으로 가시화한다.

## 2. 규칙

### 2-1. 테스트로 동작을 먼저 고정한다

리팩토링 전, 바꿀 동작을 덮는 테스트가 초록불인지 확인한다. 없으면 먼저 추가한다(특성화 테스트). (테스트 작성은 `test-strategy`, 완료 전 검증 절차는 `verification-before-completion` 참조)

```
// ❌ 금지 — 테스트 없이 바로 구조 변경 → 깨졌는지 알 길이 없음
함수를 쪼개고 이름 바꾸고… "아마 잘 되겠지"

// ✅ 권장 — 안전망 먼저
1) 기존 동작 테스트 확인(없으면 추가) → 초록불
2) 리팩토링
3) 같은 테스트 다시 실행 → 여전히 초록불
```

### 2-2. 작은 단계로 쪼개고 매 단계 검증한다

한 커밋에 변경 하나. 큰 리팩토링은 안전한 작은 변환의 연속으로 나눈다.

```
// ❌ 금지 — 한 커밋에 함수 분리 + 이름 변경 + 로직 정리 + 의존성 교체 한꺼번에
//          깨지면 어디서 깨졌는지 못 찾음

// ✅ 권장 — 단계마다 초록불 확인 후 다음으로
1단계: 변수/함수 이름만 의미 있게  → 테스트 통과
2단계: 긴 함수에서 한 덩어리 추출   → 테스트 통과
3단계: 중복 제거                    → 테스트 통과
```

### 2-3. 기능 변경과 리팩토링은 커밋을 분리한다

같은 PR이라도 "리팩토링만" 커밋과 "동작 변경" 커밋을 나눈다. 리뷰어가 동작 변경 지점을 정확히 볼 수 있다.

```
// ❌ 금지
commit: "결제 모듈 정리하고 할인율 5%→7% 반영"   ← 구조 변경에 동작 변경이 숨음

// ✅ 권장
commit 1: "refactor: 결제 계산 로직 함수 추출 (동작 동일)"
commit 2: "feat: 할인율 5%→7% 변경"
```

### 2-4. 코드 냄새를 신호로 삼는다

아래 신호가 보이면 리팩토링 후보다. 단, "냄새가 난다 = 무조건 고친다"가 아니라 지금 만지는 코드 위주로 정리한다.

| 냄새 | 신호 | 흔한 처방 |
|---|---|---|
| 중복 코드 | 같은 로직이 여러 곳에 복붙 | 공통 함수/모듈로 추출 |
| 긴 함수 | 한 함수가 화면을 넘김·여러 일을 함 | 의미 단위로 함수 추출 |
| 강결합 | 한 곳 고치면 먼 곳이 깨짐 | 경계·인터페이스로 분리 (`architecture-layering` 참조) |
| 큰 매개변수 목록 / 중복 조건문 | 인자 5개 이상, 같은 분기 반복 | 객체로 묶기·다형성/조기 반환 |

### 2-5. 빅뱅 재작성 대신 점진적으로, 부채는 가시화한다

```
// ❌ 금지 — "이 모듈은 통째로 새로 짜는 게 빠릅니다" 후 몇 주간 동작 정지·회귀 폭증

// ✅ 권장 — 조금씩 교체(예: 옛 코드와 새 코드를 한동안 병행), 매 단계 배포 가능 상태 유지
//          지금 못 갚는 부채는 코드에 숨기지 말고 티켓으로:  // TODO 아님 → 이슈 #1234 로 등록
```

## 3. 흔한 실수

AI가 자주 만드는 것 — 검토 시 거른다.

- ❌ 테스트 없이 리팩토링하고 "정리했습니다"로 끝 (동작 보존 미검증)
- ❌ 구조 개선과 함께 동작·반환값·에러 처리를 슬쩍 바꿈
- ❌ 한 커밋/한 PR에 수십 파일을 한꺼번에 갈아엎어 리뷰 불가
- ❌ 요청하지 않은 범위까지 "겸사겸사" 대규모 재작성
- ❌ 공개 API/시그니처를 바꿔 호출부를 깨뜨림 (그건 리팩토링이 아님)
- ❌ 부채를 코드 주석으로만 묻어두고 추적 불가능하게 방치

## 4. 체크리스트

- [ ] 바꾼 코드의 동작이 그대로인가 (입력/출력·부수효과·공개 API 동일)
- [ ] 리팩토링 전 해당 동작을 덮는 테스트가 초록불이었는가 (없으면 먼저 추가했는가)
- [ ] 작은 단계로 나눠 매 단계 빌드·테스트로 검증했는가
- [ ] 기능 변경과 리팩토링이 커밋으로 분리되어 있는가
- [ ] 요청 범위를 넘는 대규모 재작성을 끼워 넣지 않았는가
- [ ] 남긴 기술부채를 티켓/이슈로 가시화했는가
