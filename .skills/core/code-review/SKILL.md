# 👀 코드 리뷰 (Code Review)

> 셀프리뷰·리뷰 관점·심각도 표기·말투를 통일해, 리뷰가 빠르고 건설적으로 끝나게 한다. PR을 올리기 전, 남의 PR을 리뷰할 때, 리뷰 코멘트에 답할 때 읽는다. (PR 크기·머지 기준은 `git-workflow`가 소유한다.)

리뷰가 느려지는 이유는 대개 같다 — PR이 너무 크고, 셀프 리뷰 없이 올라오고, blocker와 취향(nit)이 섞여 있고, 피드백이 코드가 아닌 사람을 향한다. 리뷰는 결함을 거르는 동시에 지식을 나누는 자리다. 아래 규칙으로 이 마찰을 줄인다.

## 1. 핵심 원칙

- 리뷰는 사람이 아니라 **코드**를 본다 — 피드백은 코드와 동작을 향하고, 관찰·근거·제안을 함께 적는다.
- PR은 **리뷰 가능한 크기**로 작게 — 한 PR = 한 가지 변경. 작을수록 리뷰가 빠르고 결함이 더 걸린다.
- 올리기 전 **셀프 리뷰**를 먼저 — diff를 직접 읽고 군더더기·디버그 흔적·빠진 테스트를 작성자가 먼저 거른다.
- 모든 코멘트에 **심각도(blocker/nit)** 를 표기 — 막는 문제와 취향 제안을 명확히 나눈다.
- 작성자는 방어하지 않고 **근거로 응답** — 반영하거나, 왜 그대로 두는지 이유를 적는다.
- 머지는 **승인 + CI 통과** 가 모두 충족될 때만.

## 2. 규칙

### 2-1. 올리기 전 셀프 리뷰

PR을 올리기 전에 작성자가 자기 diff를 한 번 읽는다 — 리뷰어가 사소한 것에 시간을 쓰지 않게.

- ✅ diff를 직접 훑고 디버그 로그·주석 처리된 코드·임시 변수를 제거, 빠진 테스트와 엣지 케이스를 점검한 뒤 올린다.
- ❌ 로컬에서 동작만 확인하고 diff는 안 본 채 바로 리뷰 요청 → `console.log`, 죽은 코드, 오타가 그대로 올라간다.
- PR 설명에 **무엇을·왜·어떻게 검증했는지**를 적어 리뷰어의 맥락 파악을 돕는다.

### 2-2. 작은 PR (리뷰 가능한 크기)

한 PR = 한 가지 변경. 작을수록 리뷰가 빠르고 결함이 더 걸린다 — 변경이 크면 여러 PR로 쪼갠다. PR 크기·브랜치·머지 기준의 규칙은 `git-workflow`를 따른다.

### 2-3. 리뷰 관점 (무엇을 보나)

다음 관점을 순서대로 본다. 스타일은 자동화(linter/formatter)에 맡기고 사람은 윗단에 집중한다.

| 관점 | 확인 질문 |
|---|---|
| 정확성 | 의도대로 동작하나? 엣지 케이스·에러 처리는? |
| 가독성 | 6개월 뒤 봐도 읽히나? 이름은 명확한가? |
| 설계 | 적절한 위치에 있나? 더 단순한 방법은 없나? |
| 테스트 | 핵심 경로·실패 경로에 테스트가 있나? |
| 보안 | 입력 검증·권한·민감정보 노출은 괜찮나? |
| 일관성 | 기존 컨벤션·아키텍처와 맞나? |

- 정확성·설계 > 가독성 > 스타일 순으로 비중을 둔다. 스타일 지적에 매몰되지 않는다.
- 잘한 점도 짧게 남긴다 — 리뷰는 결함 색출만이 아니다.

### 2-4. blocker vs nit (심각도 표기)

모든 코멘트 앞에 심각도를 붙여, 작성자가 무엇을 꼭 고쳐야 하는지 즉시 안다.

- ✅ `[blocker]` 머지를 막는 결함(버그·보안·설계 결함) / `[nit]` 고치면 좋지만 선택인 사소한 제안.
- ❌ 심각도 없이 "이거 이상한데요" → 작성자는 필수인지 취향인지 몰라 헤맨다.
- nit만 남았으면 승인하고 넘긴다 — nit이 머지를 막지 않는다. 막아야 하면 blocker로 명시한다.

### 2-5. 구체적·건설적 피드백

피드백은 코드를 향하고, 근거와 대안을 함께 적는다. 명령이 아니라 제안과 질문으로.

- ✅ "이 루프는 입력이 클 때 O(n²)입니다. Map으로 바꾸면 O(n)이 됩니다. (근거+대안)"
- ✅ "여기서 null이 들어오면 어떻게 되나요?" (질문으로 맥락을 끌어냄)
- ❌ "코드를 왜 이렇게 짰어요?" / "이건 틀렸어요." (사람 비난·근거 없음)
- 같은 지적을 반복하지 말고 한 번 패턴으로 묶어 말한다.

### 2-6. 작성자의 응답 — 방어 대신 근거

리뷰는 공격이 아니다. 작성자는 반영하거나, 그대로 두는 이유를 근거로 답한다.

- ✅ "반영했습니다" / "여기는 X 이유로 그대로 두겠습니다 — 어떻게 보시나요?"
- ❌ 정당화·방어로 일관하거나, 코멘트를 말없이 무시하고 닫는다.
- 의견이 갈려 길어지면 댓글 핑퐁 대신 **짧은 대화(통화/회의)** 로 옮긴다.

### 2-7. 리뷰 측 머지 조건

- 미해결 `[blocker]` 코멘트가 없어야 한다. nit은 남아 있어도 된다.
- 승인 + CI 그린 게이트와 셀프 머지 금지는 `git-workflow`(머지 기준)·`verification-before-completion`(완료 증명)을 따른다.

## 3. 흔한 실수

- ❌ 수백~수천 줄짜리 거대 PR — 리뷰어가 대충 보고 "LGTM"만 찍게 된다.
- ❌ 셀프 리뷰 없이 올려 디버그 로그·죽은 코드가 그대로 남음.
- ❌ blocker와 nit을 섞어, 사소한 취향 때문에 머지가 막힘.
- ❌ "왜 이렇게 했냐"는 식의 사람 비난 — 근거·대안 없는 코멘트.
- ❌ 작성자가 모든 코멘트를 방어하거나, 답 없이 닫아버림.
- ❌ 스타일 지적에 매몰되어 정확성·설계 문제를 놓침(스타일은 자동화에 위임).

> **AI가 만든 코드를 리뷰할 때**는 일반 리뷰와 초점이 다르다 — 그럴듯하지만 틀린 코드, 환각 API, 과도한 추상화, 검증 없는 의존성을 추가로 의심해야 한다. 그 차이와 점검 항목은 `ai-generated-code-review` 스킬로 위임한다.

## 4. 체크리스트

- [ ] PR이 한 가지 변경만 담고 리뷰 가능한 크기인가
- [ ] 올리기 전 셀프 리뷰로 디버그 흔적·죽은 코드를 거르고, 설명에 무엇·왜·검증을 적었나
- [ ] 정확성·가독성·설계·테스트·보안·일관성 관점을 봤나
- [ ] 모든 코멘트에 `[blocker]`/`[nit]` 심각도를 표기했나
- [ ] 피드백이 사람이 아닌 코드를 향하고, 근거·대안을 담았나
- [ ] 작성자가 방어 대신 반영하거나 근거로 응답했나
- [ ] 미해결 `[blocker]` 코멘트가 없나 (승인·CI 그린 게이트는 `git-workflow`)
