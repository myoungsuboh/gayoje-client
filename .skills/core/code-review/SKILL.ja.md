# 👀 コードレビュー (Code Review)

> セルフレビュー・レビュー観点・重大度表記・口調を統一して、レビューが速く建設的に終わるようにする。PR を出す前、他人の PR をレビューするとき、レビューコメントに返信するときに読む。(PR の大きさ・マージ基準は `git-workflow` が所有する。)

レビューが遅くなる理由はたいてい同じだ — PR が大きすぎ、セルフレビューなしで出され、blocker と好み(nit)が混ざり、フィードバックがコードではなく人に向かう。レビューは欠陥を取り除くと同時に知識を分かち合う場だ。以下のルールでこの摩擦を減らす。

## 1. 核心原則

- レビューは人ではなく**コード**を見る — フィードバックはコードと挙動に向け、観察・根拠・提案をそろえて書く。
- PR は**レビュー可能な大きさ**に小さく — 1 PR = 1 つの変更。小さいほどレビューが速く、欠陥がより多く見つかる。
- 出す前にまず**セルフレビュー** — diff を自分で読み、余分なもの・デバッグ痕跡・欠けたテストを作者が先に取り除く。
- すべてのコメントに**重大度(blocker/nit)**を表記 — 妨げる問題と好みの提案を明確に分ける。
- 作者は防御せず**根拠で応答**する — 反映するか、なぜそのままにするのか理由を書く。
- マージは**承認 + CI 通過**の両方が満たされたときだけ。

## 2. ルール

### 2-1. 出す前のセルフレビュー

PR を出す前に作者が自分の diff を一度読む — レビュアーが些細なことに時間を使わないように。

- ✅ diff を自分で目を通し、デバッグログ・コメントアウトされたコード・一時変数を除去し、欠けたテストとエッジケースを点検してから出す。
- ❌ ローカルで動作だけ確認して diff は見ずにすぐレビュー依頼 → `console.log`、デッドコード、タイポがそのまま上がる。
- PR 説明に**何を・なぜ・どう検証したか**を書き、レビュアーの文脈把握を助ける。

### 2-2. 小さな PR(レビュー可能な大きさ)

1 PR = 1 つの変更。小さいほどレビューが速く、欠陥がより多く見つかる — 変更が大きければ複数の PR に分ける。PR の大きさ・ブランチ・マージ基準のルールは `git-workflow` に従う。

### 2-3. レビュー観点(何を見るか)

次の観点を順に見る。スタイルは自動化(linter/formatter)に任せ、人は上位に集中する。

| 観点 | 確認する質問 |
|---|---|
| 正確性 | 意図どおり動くか? エッジケース・エラー処理は? |
| 可読性 | 6 か月後に見ても読めるか? 名前は明確か? |
| 設計 | 適切な場所にあるか? もっと単純な方法はないか? |
| テスト | 核心経路・失敗経路にテストがあるか? |
| セキュリティ | 入力検証・権限・機密情報の露出は大丈夫か? |
| 一貫性 | 既存の慣習・アーキテクチャと合うか? |

- 正確性・設計 > 可読性 > スタイルの順に比重を置く。スタイルの指摘に埋没しない。
- うまくできた点も短く残す — レビューは欠陥探しだけではない。

### 2-4. blocker vs nit(重大度表記)

すべてのコメントの前に重大度を付け、作者が何を必ず直すべきかを即座に分かるようにする。

- ✅ `[blocker]` マージを妨げる欠陥(バグ・セキュリティ・設計上の欠陥) / `[nit]` 直せば良いが任意の些細な提案。
- ❌ 重大度なしで「これ変ですね」 → 作者は必須か好みか分からず迷う。
- nit だけ残ったら承認して進める — nit はマージを妨げない。妨げるべきなら blocker と明示する。

### 2-5. 具体的・建設的なフィードバック

フィードバックはコードに向け、根拠と代案をそろえて書く。命令ではなく提案と質問で。

- ✅ 「このループは入力が大きいとき O(n²) です。Map に変えれば O(n) になります。(根拠+代案)」
- ✅ 「ここで null が入ってきたらどうなりますか?」(質問で文脈を引き出す)
- ❌ 「コードをなぜこう書いたんですか?」 / 「これは間違っています。」(人への非難・根拠なし)
- 同じ指摘を繰り返さず、一度パターンとしてまとめて言う。

### 2-6. 作者の応答 — 防御ではなく根拠

レビューは攻撃ではない。作者は反映するか、そのままにする理由を根拠で答える。

- ✅ 「反映しました」 / 「ここは X の理由でそのままにします — どう思われますか?」
- ❌ 正当化・防御に終始するか、コメントを黙って無視して閉じる。
- 意見が分かれて長引いたら、コメントのピンポンの代わりに**短い会話(通話/会議)**に移す。

### 2-7. レビュー側のマージ条件

- 未解決の `[blocker]` コメントがあってはならない。nit は残っていてよい。
- 承認 + CI グリーンのゲートとセルフマージ禁止は `git-workflow`(マージ基準)・`verification-before-completion`(完了の証明)に従う。

## 3. よくある間違い

- ❌ 数百〜数千行の巨大な PR — レビュアーがざっと見て「LGTM」だけ押すことになる。
- ❌ セルフレビューなしで出し、デバッグログ・デッドコードがそのまま残る。
- ❌ blocker と nit を混ぜ、些細な好みのためにマージが妨げられる。
- ❌ 「なぜこうしたのか」式の人への非難 — 根拠・代案のないコメント。
- ❌ 作者がすべてのコメントを防御するか、返信なしで閉じてしまう。
- ❌ スタイルの指摘に埋没して正確性・設計の問題を見落とす(スタイルは自動化に委任)。

> **AI が作ったコードをレビューするとき**は、通常のレビューと焦点が異なる — もっともらしいが間違ったコード、幻覚 API、過度な抽象化、検証なしの依存性をさらに疑わなければならない。その違いと点検項目は `ai-generated-code-review` スキルに委任する。

## 4. チェックリスト

- [ ] PR が 1 つの変更だけを含み、レビュー可能な大きさか
- [ ] 出す前のセルフレビューでデバッグ痕跡・デッドコードを取り除き、説明に何を・なぜ・検証を書いたか
- [ ] 正確性・可読性・設計・テスト・セキュリティ・一貫性の観点を見たか
- [ ] すべてのコメントに `[blocker]`/`[nit]` の重大度を表記したか
- [ ] フィードバックが人ではなくコードに向き、根拠・代案を含んでいるか
- [ ] 作者が防御の代わりに反映するか根拠で応答したか
- [ ] 未解決の `[blocker]` コメントがないか(承認・CI グリーンのゲートは `git-workflow`)
