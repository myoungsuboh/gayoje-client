# 👀 代码评审 (Code Review)

> 通过统一自评审、评审视角、严重度标注和语气,让评审快速且建设性地结束。在提交 PR 之前、评审他人的 PR 时、回复评审意见时阅读。(PR 大小与合并标准由 `git-workflow` 所有。)

评审变慢的原因大多相同 — PR 太大、没有自评审就提交、blocker 与口味(nit)混在一起、反馈指向人而非代码。评审既是剔除缺陷也是分享知识的场合。以下规则用来减少这种摩擦。

## 1. 核心原则

- 评审看的是**代码**而非人 — 反馈指向代码和行为,并附上观察、依据和建议。
- PR 要**小到可评审** — 一个 PR = 一项变更。越小评审越快,越能查出缺陷。
- 提交前先**自评审** — 作者亲自读 diff,先行剔除冗余、调试痕迹和缺失的测试。
- 每条意见都标注**严重度(blocker/nit)** — 明确区分阻塞性问题和口味建议。
- 作者不辩护而是**以依据回应** — 要么采纳,要么写出为何保持原样的理由。
- 只有在**批准 + CI 通过**都满足时才合并。

## 2. 规则

### 2-1. 提交前的自评审

提交 PR 之前作者把自己的 diff 读一遍 — 别让评审者把时间花在琐碎事上。

- ✅ 亲自浏览 diff,移除调试日志、注释掉的代码、临时变量,检查缺失的测试和边界情况后再提交。
- ❌ 只在本地确认能跑、不看 diff 就直接请求评审 → `console.log`、死代码、错别字原样上去。
- 在 PR 说明里写**做了什么、为什么、如何验证**,帮助评审者把握上下文。

### 2-2. 小 PR(可评审的大小)

一个 PR = 一项变更。越小评审越快,越能查出缺陷 — 变更大就拆成多个 PR。PR 大小、分支、合并标准的规则遵循 `git-workflow`。

### 2-3. 评审视角(看什么)

按顺序看以下视角。把样式交给自动化(linter/formatter),人专注于上层。

| 视角 | 确认的问题 |
|---|---|
| 正确性 | 是否如意图运行?边界情况、错误处理呢? |
| 可读性 | 6 个月后再看也读得懂吗?命名清晰吗? |
| 设计 | 放在了恰当的位置吗?有没有更简单的办法? |
| 测试 | 核心路径、失败路径有测试吗? |
| 安全 | 输入校验、权限、敏感信息暴露没问题吗? |
| 一致性 | 与现有约定、架构契合吗? |

- 按正确性、设计 > 可读性 > 样式的顺序分配比重。不要陷在样式挑刺里。
- 做得好的地方也简短留言 — 评审不只是搜寻缺陷。

### 2-4. blocker vs nit(严重度标注)

在每条意见前加上严重度,让作者立刻知道哪些必须改。

- ✅ `[blocker]` 阻塞合并的缺陷(bug、安全、设计缺陷) / `[nit]` 改了更好但可选的琐碎建议。
- ❌ 不带严重度的「这里有点怪」 → 作者不知道是必须还是口味而犯难。
- 只剩 nit 就批准放行 — nit 不阻塞合并。该阻塞就明确标为 blocker。

### 2-5. 具体、建设性的反馈

反馈指向代码,并附上依据和替代方案。用建议和提问,而非命令。

- ✅ 「这个循环在输入很大时是 O(n²)。改成 Map 就变成 O(n)。(依据+替代)」
- ✅ 「这里如果传入 null 会怎样?」(用提问引出上下文)
- ❌ 「代码为什么这么写?」 / 「这是错的。」(指责人、没有依据)
- 不要重复同样的指摘,把它一次性归纳为一种模式来说。

### 2-6. 作者的回应 — 以依据代替辩护

评审不是攻击。作者要么采纳,要么以依据回答保持原样的理由。

- ✅ 「已采纳」 / 「这里出于 X 的理由保持原样 — 您怎么看?」
- ❌ 一味正当化、辩护,或者默默无视意见并关闭。
- 意见分歧拖长时,与其评论乒乓,不如转为**简短的对话(通话/会议)**。

### 2-7. 评审侧的合并条件

- 不得有未解决的 `[blocker]` 意见。nit 留着也行。
- 批准 + CI 绿灯门禁和禁止自合并遵循 `git-workflow`(合并标准)、`verification-before-completion`(完成证明)。

## 3. 常见错误

- ❌ 数百到数千行的巨型 PR — 评审者草草一看就只盖个「LGTM」。
- ❌ 不做自评审就提交,调试日志、死代码原样留存。
- ❌ 把 blocker 和 nit 混在一起,因琐碎口味而阻塞合并。
- ❌ 「为什么这么做」式的指责人 — 没有依据、替代方案的意见。
- ❌ 作者辩护每一条意见,或不回复就关闭。
- ❌ 陷在样式挑刺中而漏掉正确性、设计问题(样式委托给自动化)。

> **评审 AI 生成的代码时**,焦点与普通评审不同 — 还要额外怀疑看似合理却错误的代码、幻觉 API、过度抽象、未经验证的依赖。那个差异和检查项委托给 `ai-generated-code-review` 技能。

## 4. 检查清单

- [ ] PR 是否只含一项变更且为可评审的大小
- [ ] 提交前的自评审是否剔除了调试痕迹、死代码,说明里是否写了做什么、为什么、验证
- [ ] 是否看了正确性、可读性、设计、测试、安全、一致性视角
- [ ] 是否给每条意见标注了 `[blocker]`/`[nit]` 严重度
- [ ] 反馈是否指向代码而非人,并包含依据、替代方案
- [ ] 作者是否以采纳或依据回应而非辩护
- [ ] 是否没有未解决的 `[blocker]` 意见(批准、CI 绿灯门禁是 `git-workflow`)
