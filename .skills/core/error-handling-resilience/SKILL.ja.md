# 🛡️ エラー処理 & 回復力 (Error Handling & Resilience)

> 例外を捕捉し・伝播し・回復する方式をルールとして固め、失敗してもシステムが崩れず原因が露出するようにする。外部依存を呼び出したり try/catch・例外ポリシーを記述・レビューするときに読む。

AI エージェントが最もよく犯すミスは、**エラーを静かに握りつぶすこと**(空 catch、ログなしの無視)と、**すべての失敗を同じに扱うこと**だ。そうすると障害が隠れ、一時的なエラーでも全体が止まる。失敗を「プログラミングエラー」と「外部依存の失敗」に分け、それぞれに合ったポリシーをルールに固めておけば、AI もその枠の中で安全なコードを生成する。

## 1. 中核原則

- **エラーを握りつぶさない** — 捕まえたら処理する、できないなら投げ直す。ログ・処理のない空 catch は禁止。
- **失敗の種類を区別する** — プログラミングエラー(バグ・不正な入力)は **fail-fast**(即時中断・露出)、外部依存の失敗(ネットワーク・DB・外部 API)は **graceful degradation**(フォールバック・部分動作)で。
- **一時的な失敗はリトライするが安全に** — 指数バックオフ + ジッター、タイムアウト、リトライ上限を常に併用する。無限・即時リトライは禁止。
- **リトライには冪等性が必要** — 副作用のある処理は冪等キーで重複を防ぐ (`idempotency` 参照)。
- **障害を隔離する** — 繰り返し失敗する依存にはサーキットブレーカーで呼び出しを断ち、フォールバックする。一部の失敗が全体に広がらないようにする。
- **メッセージは相手別に** — ユーザーには親切で安全なメッセージ、ログには詳細なコンテキスト(ただしパスワード・トークン・個人情報は除く)。

## 2. ルール

### 2-1. エラーを握りつぶさない

```text
# ❌ 禁止 — 静かに握りつぶす: 障害が隠れデバッグ不能
try { charge(order) } catch (e) { /* 何もしない */ }
try { ... } catch (e) { return null }   // 原因消失

# ✅ 推奨 — 処理するか、コンテキストを付けて投げ直す
try {
  charge(order)
} catch (e) {
  log.error("charge 失敗 orderId=%s", order.id, e)   // 原因(e)を保存
  throw new PaymentError("決済処理失敗", cause=e)    // 上位に判断させる
}
```

- 捕まえた例外は **必ず** ログするか再伝播する。「あとで見る」空 catch は禁止。
- 投げ直すときは **元の原因(cause/stack)を保存** する — 新しい例外で上書きしない。
- 広範な catch(`catch (Exception)`/`except:`)ですべてを一度に握りつぶさない — 捕まえる例外を具体的に。

### 2-2. fail-fast vs graceful degradation

| 失敗の種類 | 例 | ポリシー |
|---|---|---|
| プログラミングエラー | null 参照、不正な引数、壊れた設定 | **fail-fast** — 即時中断・露出、隠さない |
| 不正な外部入力 | 検証に失敗したリクエスト | 拒否 + 明確な 4xx (入力検証は `input-validation`) |
| 外部依存の失敗 | DB・外部 API・キューのタイムアウト | **graceful degradation** — リトライ・フォールバック・部分動作 |

```text
# ✅ 起動時に必須設定が欠落 → fail-fast (静かにデフォルトを使わない)
if (config.apiKey == null) throw new ConfigError("API_KEY 必須")

# ✅ レコメンドサービスがダウン → 中核フローはフォールバックで継続
recs = tryGet(() => recoApi.fetch(user), fallback=[])  // 空リストで進める
```

### 2-3. リトライ + バックオフ + タイムアウト + 上限

```text
# ❌ 禁止 — タイムアウト・上限なしの即時無限リトライ (障害増幅)
while (true) { try { return call() } catch (e) { /* すぐリトライ */ } }

# ✅ 推奨 — タイムアウト + 指数バックオフ + ジッター + 回数上限
for (attempt in 1..MAX_RETRIES) {        // 上限 (例: 3)
  try {
    return call(timeout=2s)              // 呼び出しごとのタイムアウト
  } catch (e) {
    if (!isTransient(e) || attempt == MAX_RETRIES) throw e
    sleep(min(base * 2^attempt, cap) + random_jitter)  // 指数バックオフ + ジッター
  }
}
```

- **タイムアウトのない外部呼び出しは禁止** — 無限待機はスレッド・コネクション枯渇につながる。
- リトライは **一時的なエラー**(ネットワーク・5xx・タイムアウト)にのみ。4xx・検証エラーはリトライしても無意味。
- バックオフに **ジッター** を加えて同時リトライの殺到(thundering herd)を防ぐ。
- 副作用のある処理(決済・生成)をリトライするには **冪等性** が必須 (`idempotency` 参照)。

### 2-4. サーキットブレーカー & フォールバック

```text
# ✅ 繰り返し失敗したら回路を開いて素早く失敗させ、フォールバックで応答
if (breaker.isOpen()) return cachedOrDefault()   // 死んだ依存を叩き続けない
try { r = call(); breaker.onSuccess(); return r }
catch (e) { breaker.onFailure(); return cachedOrDefault() }
```

- 依存が失敗し続けるなら回路を **開いて** 即座に失敗させる — 資源の浪費・連鎖障害を防止。
- フォールバックは **安全なデフォルト/キャッシュ/縮退機能** で。フォールバックが別の障害を呼ばないように。

### 2-5. 部分失敗の処理

```text
# ❌ 禁止 — 100件中1件の失敗で全体中断・ロールバック
for (item in batch) process(item)   // 一つでも throw → 残りが消失

# ✅ 推奨 — 件ごとに隔離、成功/失敗を集計して報告
results = batch.map(item => trySettle(() => process(item)))
failed = results.filter(isFailure)
if (failed) log.warn("部分失敗 %d/%d", failed.size, batch.size)
// 失敗分は再処理キューへ (全部捨てず、全部止めもしない)
```

- バッチ・ファンアウト呼び出しは **件ごとに隔離** し、一件の失敗が全体を止めないようにする。
- 全部成功でなければ全部失敗が **正しい場合**(原子的トランザクション)と **部分許容** が正しい場合を区別する。

### 2-6. メッセージ: ユーザー vs ログ

```text
# ❌ 禁止 — 生の例外・スタック・内部情報をユーザーに露出
return Response(500, e.toString())   // SQL・パス・トークン漏洩のリスク

# ✅ 推奨 — ユーザーには親切+相関ID、ログには詳細(機微情報を除く)
log.error("注文作成失敗 traceId=%s userId=%s", traceId, userId, e)
return Response(500, { message: "しばらくしてから再度お試しください", traceId })
```

- ユーザーメッセージに内部構造・スタック・生のエラーを露出しない。相関 ID でログと紐づける。
- ログには十分なコンテキスト(識別子・入力要約)を残すが **パスワード・トークン・個人情報はマスク/除外**。

## 3. よくあるミス

AI がよく作るもの — レビュー時にふるい落とす。

- ❌ 空 catch / `catch (e) {}` / ログも再伝播もない無視
- ❌ `catch (Exception)`・`except:` ですべての例外をまとめて握りつぶす
- ❌ 外部呼び出しにタイムアウトなし → 無限待機・資源枯渇
- ❌ タイムアウト・上限なしのリトライ、または 4xx までリトライ
- ❌ 冪等性なしで決済・生成処理をリトライ → 重複実行
- ❌ 新しい例外で包んで元の cause/stack を消失
- ❌ ユーザーに生のスタック・SQL・内部パスを露出、ログにトークン・個人情報をそのまま記録
- ❌ プログラミングのバグを try/catch で隠して「正常」のように見せる (静かな失敗)

> **適用のコツ**: AGENTS.md / ルールファイルに「空 catch 禁止、外部呼び出しにはタイムアウト、リトライにはバックオフ+上限」の一行を固めておけば、エージェントが生成のたびに守る。この文書は **処理・回復ポリシー** のみを扱い、エラー収集・集計・通知は `error-monitoring`(FE)・`async-error-handling`(Mobile)に委譲する(入力検証は `input-validation`)。
