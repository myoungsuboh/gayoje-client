# 🔬 체계적 디버깅 (Systematic Debugging)

> 이 스킬은 본 사내 하네스의 자체 표준이다(외부 'superpowers' 번들의 동명 스킬과 무관).
>
> 버그를 추측으로 더듬지 않고 재현·근거·검증으로 근본 원인까지 좁힌다. 원인이 불분명한 버그·테스트 실패·이상 동작을 만났을 때, 고치기 전에 읽는다.

AI 에이전트가 디버깅에서 가장 흔히 저지르는 실수는 **증상을 보자마자 그럴듯한 수정을 추측으로 갈아끼우는 것**이다. 운 좋게 증상이 사라져도 근본 원인은 남아 다른 형태로 다시 터지고, 그 사이 코드엔 의미 없는 변경이 쌓인다. 재현 → 근거 수집 → 가설을 하나씩 검증하는 절차를 규칙으로 못 박으면 AI도 그 틀 안에서 원인을 좁힌다.

## 1. 핵심 원칙

- 추측으로 고치지 않는다 — 먼저 **재현**한다. 100% 재현되는 최소 케이스가 없으면 아직 고칠 단계가 아니다.
- 증상이 아니라 **근본 원인**을 추적한다. 에러 메시지를 지우는 게 아니라 왜 그 상태가 됐는지를 찾는다.
- 한 번에 하나씩 — 가설을 하나 세우고, 그것만 검증하고, 변경도 한 번에 하나만 한다.
- 머리로 단정하지 말고 **근거로 좁힌다** — 로그·관측·이분 탐색으로 사실을 확인한다.
- 고치면 끝이 아니다 — 같은 버그를 다시 잡아낼 **회귀 테스트**를 남긴다.
- 막히면 더 작게 쪼갠다 — 문제 범위를 절반씩 줄여 원인 구간을 가둔다.

## 2. 규칙

### 2-1. 추측 금지 — 먼저 최소 재현 케이스

원인을 모를 때 코드부터 바꾸지 않는다. **항상 똑같이 재현되는 가장 작은 케이스**를 먼저 만든다.

```text
❌ 금지 — "아마 null 때문이겠지" 하고 곳곳에 옵셔널/널 체크를 추가
   → 증상은 가려지고 원인은 그대로, 변경만 쌓임

✅ 권장 — 실패하는 입력·환경을 단계적으로 제거해 최소 재현 케이스 확보
   재현 절차: 입력 X → 단계 Y → 기대 A, 실제 B (매번 동일)
   → 이 케이스가 그대로 회귀 테스트가 된다 (2-5)
```

### 2-2. 증상이 아니라 근본 원인

눈에 보이는 증상 지점이 원인이 아닐 때가 많다. "왜?"를 원인에 닿을 때까지 되묻는다.

```text
❌ 금지 — 화면에 NaN이 떠서 표시 직전에 `|| 0`으로 가림
✅ 권장 — NaN이 어디서 처음 생겼는지 거슬러 올라감
         → 입력 파싱이 빈 문자열을 숫자로 바꾼 지점이 진짜 원인
```

- 표면 수정으로 증상만 사라지면, 같은 원인이 다른 곳에서 또 터진다.

### 2-3. 가설 → 검증, 한 번에 하나

여러 추측을 한꺼번에 적용하면 무엇이 효과였는지 알 수 없다.

```text
❌ 금지 — 캐시 끄고, 라이브러리 올리고, 설정도 바꾼 뒤 "이제 되네"
   → 무엇이 원인이고 무엇이 무관한지 끝까지 모름

✅ 권장 — 가설 1개 → 그것만 바꿔 검증 → 결과 기록 → 다음 가설
   "X가 원인이다" → X만 바꿔 확인 → 아니면 원복하고 다음으로
```

- 검증 후 효과 없던 변경은 **원복**한다. 디버깅 흔적을 코드에 남기지 않는다.

### 2-4. 근거 수집 — 이분 탐색 · 로그/관측

머리로 짚지 말고 사실로 좁힌다.

```text
✅ 이분 탐색 — 어느 변경부터 깨졌는지 모르면 git bisect로 커밋을 절반씩 가른다
✅ 로그/관측 — 의심 구간 앞뒤로 입력·상태·분기 값을 찍어 실제 흐름을 확인
✅ 범위 좁히기 — 코드/입력/환경을 절반씩 잘라 원인 구간을 가둔다
```

- 외부 의존(네트워크·시간·동시성)이 의심되면, 그 변수를 고정/격리해 재현되는지 본다.

### 2-5. 수정 후 회귀 테스트

원인을 고쳤으면 **고치기 전엔 실패하고, 고친 뒤엔 통과하는** 테스트를 남긴다.

```text
✅ 권장 — 2-1의 최소 재현 케이스를 테스트로 박제
   1) 먼저 그 테스트가 실제로 실패하는지 확인 (원인을 맞게 짚었다는 증거)
   2) 수정 후 통과 확인
   → 같은 버그의 재발을 자동으로 막는다
```

> 수정이 "진짜 끝났는지"(빌드·테스트·실제 동작) 판단하는 절차는 `verification-before-completion` 스킬을 따른다.

## 3. 흔한 실수

AI가 자주 저지르는 것 — 검토 시 거른다.

- ❌ 재현도 안 해보고 "아마 이거일 것" 하며 추측 패치부터 던짐
- ❌ 에러를 try/catch로 삼키거나 표시 직전에 가려 증상만 제거 (근본 원인 방치)
- ❌ 한 번에 여러 군데를 바꿔서 무엇이 효과였는지 모름
- ❌ "고친 것 같다"로 종료 — 재현 케이스로 재확인하지 않음
- ❌ 효과 없던 디버그 코드(로그·임시 분기)를 그대로 커밋에 남김
- ❌ 고친 뒤 회귀 테스트를 추가하지 않아 같은 버그가 재발

> **적용 팁**: 막히면 범위를 절반으로 쪼개는 것을 기본 동작으로 삼는다. "어디까지 정상인가"를 이분 탐색으로 확인하면 추측 없이 원인 구간이 좁혀진다.

## 4. 체크리스트

- [ ] 추측으로 고치기 전에 100% 재현되는 최소 케이스를 확보했는가
- [ ] 증상이 아니라 근본 원인까지 거슬러 올라갔는가
- [ ] 가설을 하나씩 세우고 변경도 한 번에 하나만 했는가 (효과 없던 변경은 원복)
- [ ] 추측 대신 로그·관측·이분 탐색(git bisect)으로 근거를 확보했는가
- [ ] 고치기 전 실패하고 고친 뒤 통과하는 회귀 테스트를 남겼는가
- [ ] 임시 디버그 코드를 정리했는가
