# 🔬 体系的デバッグ (Systematic Debugging)

> このスキルは本社内ハーネスの独自標準である（外部 'superpowers' バンドルの同名スキルとは無関係）。
>
> バグを推測で手探りせず、再現・根拠・検証で根本原因まで絞り込む。原因が不明なバグ・テスト失敗・異常動作に出会ったとき、直す前に読む。

AI エージェントがデバッグで最もよく犯す誤りは **症状を見るやいなや、それらしい修正を推測で差し替えること** だ。運よく症状が消えても根本原因は残り、別の形で再び壊れ、その間にコードには意味のない変更が積み上がる。再現 → 根拠収集 → 仮説を一つずつ検証する手順をルールとして釘づけすれば、AI もその枠の中で原因を絞る。

## 1. 中核原則

- 推測で直さない — まず **再現** する。100% 再現する最小ケースがなければ、まだ直す段階ではない。
- 症状ではなく **根本原因** を追跡する。エラーメッセージを消すのではなく、なぜその状態になったかを見つける。
- 一度に一つずつ — 仮説を一つ立て、それだけを検証し、変更も一度に一つだけ行う。
- 頭で断定せず **根拠で絞る** — ログ・観測・二分探索で事実を確認する。
- 直して終わりではない — 同じバグを再び捕まえる **回帰テスト** を残す。
- 詰まったらもっと小さく割る — 問題範囲を半分ずつ減らして原因区間を囲い込む。

## 2. ルール

### 2-1. 推測禁止 — まず最小再現ケース

原因が分からないときコードから変えない。**常に同じように再現する最も小さなケース** をまず作る。

```text
❌ 禁止 — 「たぶん null のせいだろう」と各所にオプショナル/ヌルチェックを追加
   → 症状は隠れ原因はそのまま、変更だけ積み上がる

✅ 推奨 — 失敗する入力・環境を段階的に取り除いて最小再現ケースを確保
   再現手順: 入力 X → 段階 Y → 期待 A、実際 B (毎回同じ)
   → このケースがそのまま回帰テストになる (2-5)
```

### 2-2. 症状ではなく根本原因

目に見える症状の地点が原因でないことが多い。「なぜ?」を原因に届くまで問い返す。

```text
❌ 禁止 — 画面に NaN が出るので表示直前に `|| 0` で隠す
✅ 推奨 — NaN がどこで最初に生じたか遡る
         → 入力パースが空文字列を数値に変えた地点が真の原因
```

- 表面的な修正で症状だけが消えると、同じ原因が別の場所でまた壊れる。

### 2-3. 仮説 → 検証、一度に一つ

複数の推測を一度に適用すると、何が効果だったか分からない。

```text
❌ 禁止 — キャッシュを切り、ライブラリを上げ、設定も変えた後「これで動くね」
   → 何が原因で何が無関係か最後まで分からない

✅ 推奨 — 仮説 1 個 → それだけ変えて検証 → 結果記録 → 次の仮説
   「X が原因だ」 → X だけ変えて確認 → 違えば原状復帰して次へ
```

- 検証後、効果のなかった変更は **原状復帰** する。デバッグの痕跡をコードに残さない。

### 2-4. 根拠収集 — 二分探索 · ログ/観測

頭で当てず事実で絞る。

```text
✅ 二分探索 — どの変更から壊れたか分からなければ git bisect でコミットを半分ずつ割る
✅ ログ/観測 — 疑わしい区間の前後で入力・状態・分岐値を出力し実際の流れを確認
✅ 範囲を絞る — コード/入力/環境を半分ずつ切って原因区間を囲い込む
```

- 外部依存（ネットワーク・時間・並行性）が疑わしければ、その変数を固定/隔離して再現するか見る。

### 2-5. 修正後の回帰テスト

原因を直したら **直す前は失敗し、直した後は通る** テストを残す。

```text
✅ 推奨 — 2-1 の最小再現ケースをテストとして固定する
   1) まずそのテストが実際に失敗するか確認 (原因を正しくつかんだ証拠)
   2) 修正後に通ることを確認
   → 同じバグの再発を自動で防ぐ
```

> 修正が「本当に終わったか」（ビルド・テスト・実動作）を判断する手順は `verification-before-completion` スキルに従う。

## 3. よくある誤り

AI がよく犯すもの — レビュー時に弾く。

- ❌ 再現もしないで「たぶんこれだろう」と推測パッチをまず投げる
- ❌ エラーを try/catch で握りつぶす、または表示直前に隠して症状だけ除去 (根本原因放置)
- ❌ 一度に複数箇所を変えて何が効果だったか分からない
- ❌ 「直ったようだ」で終了 — 再現ケースで再確認しない
- ❌ 効果のなかったデバッグコード（ログ・一時分岐）をそのままコミットに残す
- ❌ 直した後に回帰テストを追加せず同じバグが再発

> **適用のヒント**: 詰まったら範囲を半分に割ることを基本動作にする。「どこまで正常か」を二分探索で確認すれば、推測なしで原因区間が絞られる。

- [ ] 推測の代わりにログ・観測・二分探索（git bisect）で根拠を確保したか
- [ ] 直す前は失敗し直した後は通る回帰テストを残したか
