# 🔬 系统化调试 (Systematic Debugging)

> 本技能是本公司内部 harness 的自有标准（与外部 'superpowers' 套件中同名技能无关）。
>
> 不靠猜测去摸索缺陷，而以复现·证据·验证收窄到根本原因。在遇到原因不明的缺陷·测试失败·异常行为时，在动手修之前阅读。

AI 代理在调试中最常犯的错误是 **一看到症状就靠猜测换上一个看似合理的修复**。即便症状侥幸消失，根本原因仍在，会以另一种形式再次爆发，期间代码里还堆积了无意义的变更。把 复现 → 收集证据 → 一个一个验证假设 的流程钉为规则，AI 也能在这个框架内收窄原因。

## 1. 核心原则

- 不靠猜测去修 — 先 **复现**。没有 100% 复现的最小用例，就还没到该修的阶段。
- 追踪 **根本原因** 而非症状。不是擦掉错误信息，而是找出为何会进入那种状态。
- 一次一个 — 立一个假设，只验证它，变更也一次只做一个。
- 不靠脑子下定论而 **靠证据收窄** — 用日志·观测·二分查找确认事实。
- 修好不算结束 — 留下能再次抓住同一缺陷的 **回归测试**。
- 卡住就再切小 — 把问题范围一半一半地缩小，圈定原因区间。

## 2. 规则

### 2-1. 禁止猜测 — 先做最小复现用例

不知道原因时不要先改代码。先做出 **每次都同样复现的最小用例**。

```text
❌ 禁止 — “大概是 null 吧”就到处加可选/空值检查
   → 症状被遮住、原因照旧，只堆积变更

✅ 推荐 — 逐步剔除失败的输入·环境，确保一个最小复现用例
   复现步骤: 输入 X → 步骤 Y → 期望 A、实际 B (每次相同)
   → 这个用例可直接成为回归测试 (2-5)
```

### 2-2. 根本原因而非症状

肉眼可见的症状点往往不是原因。一直追问“为什么?”直到触及原因。

```text
❌ 禁止 — 屏幕上出现 NaN，就在显示之前用 `|| 0` 遮住
✅ 推荐 — 回溯 NaN 最初是在哪里产生的
         → 输入解析把空字符串变成数字的那一点才是真正的原因
```

- 表面修复只让症状消失，同一原因会在别处再次爆发。

### 2-3. 假设 → 验证，一次一个

一次性套用多个猜测，就分不清是哪一个起了作用。

```text
❌ 禁止 — 关掉缓存、升级库、连配置也改了之后“现在行了”
   → 到头来不知道什么是原因、什么无关

✅ 推荐 — 1 个假设 → 只改它来验证 → 记录结果 → 下一个假设
   “X 是原因” → 只改 X 来确认 → 不是就还原再换下一个
```

- 验证后，对没有效果的变更要 **还原**。不要把调试痕迹留在代码里。

### 2-4. 收集证据 — 二分查找 · 日志/观测

别靠脑子猜，靠事实收窄。

```text
✅ 二分查找 — 不知道从哪个变更开始坏掉，就用 git bisect 把提交一半一半地切
✅ 日志/观测 — 在可疑区间前后打印输入·状态·分支值，确认实际流程
✅ 收窄范围 — 把代码/输入/环境一半一半地切，圈定原因区间
```

- 若怀疑外部依赖（网络·时间·并发），就固定/隔离该变量，看是否仍然复现。

### 2-5. 修复后的回归测试

修好原因后，留下一个 **修复前失败、修复后通过** 的测试。

```text
✅ 推荐 — 把 2-1 的最小复现用例固化为测试
   1) 先确认该测试确实会失败 (你正确抓住原因的证据)
   2) 修复后确认通过
   → 自动防止同一缺陷再发
```

> 判断修复是否“真正完成”（构建·测试·实际行为）的流程，遵循 `verification-before-completion` 技能。

## 3. 常见错误

AI 常犯的 — 评审时予以拦截。

- ❌ 连复现都没做就“大概是这个吧”先丢出猜测补丁
- ❌ 用 try/catch 把错误吞掉，或在显示之前遮住，只除去症状 (放任根本原因)
- ❌ 一次改多处，导致不知道什么起了作用
- ❌ 以“好像修好了”收尾 — 没有用复现用例重新确认
- ❌ 把没有效果的调试代码（日志·临时分支）原样留在提交里
- ❌ 修好后不补回归测试，导致同一缺陷再发

> **应用提示**: 把卡住时将范围一半一半地切，作为默认动作。用二分查找确认“到哪里为止是正常的”，无需猜测即可收窄原因区间。

## 4. 检查清单

- [ ] 在靠猜测去修之前，是否确保了一个 100% 复现的最小用例
- [ ] 是否回溯到了根本原因而非症状
- [ ] 是否一个一个地立假设、变更也一次只做一个 (没有效果的变更已还原)
- [ ] 是否用日志·观测·二分查找（git bisect）取得证据而非猜测
- [ ] 是否留下了修复前失败、修复后通过的回归测试
- [ ] 是否清理了临时调试代码
