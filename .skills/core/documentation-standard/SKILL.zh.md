# 📚 文档标准 — README · ADR · 注释 · 同步

> 把文档写在哪里、写什么、怎么写钉死,让新成员和 AI 代理一次搜索就获得上下文。在开启新仓库、添加或整理文档、留下重要决策时阅读。

文档是与代码同等重要的产物。尤其 AI 代理无法直接运行代码,因此**依赖 README、决策记录、注释里写的上下文**来工作。文档缺失或陈旧时,人和 AI 都会做出错误假设,重复同样的提问、同样的错误。最危险的是**错误文档** — 陈旧或虚假的文档比无文档更糟(因为读者会相信它)。

## 1. 核心原则

- 文档随代码一起变更 — 改变行为的 PR 要在同一个 PR 里修好相关文档。「以后」不会到来。
- 错误文档 > 无文档之害 — 陈旧或虚假的文档要删除、修正。没把握的内容不靠猜测写下。
- 以新人、AI 代理为基准来写 — 应能在没有前置知识的情况下只读 README 就到达构建、运行。
- 注释是「为什么」而非「是什么」 — 不要重复代码所说的,写下代码说不出的意图、背景、权衡。
- 文档位置保持一致 — 同类文档始终放在同一处。散落的文档等于没有文档。
- 重要技术决策用 ADR 留存 — 仅看代码无法知道「为什么这么做」的选择,连同依据一起记录。

## 2. 规则

### 2-1. README 必备章节

仓库根目录的 `README.md` 要具备以下章节。顺序自由,但**禁止缺漏**。

| 章节 | 必须回答的问题 |
|---|---|
| 概述 | 这个项目是什么,为何存在 |
| 要求 | 需要哪些运行时、工具、版本 |
| 安装 | 如何获取依赖 |
| 运行 | 如何在本地启动并测试 |
| 项目结构 | 主要目录、入口点在哪里 |
| 贡献 | 分支、提交、PR 规则是什么 |

```md
✅ 推荐 — 用可复制粘贴的命令
## 运行
\`\`\`
<包管理器> install
<运行命令>        # http://localhost:<端口>
<测试命令>
\`\`\`

❌ 禁止 — 抽象叙述(新人无法照做)
## 运行
安装依赖后,像平常一样启动服务器即可。
```

### 2-2. ADR — 架构决策记录

难以回退或对团队影响大的决策(框架选择、数据存储方式、认证策略等)用一张 ADR 留存。位置像 `docs/adr/NNNN-标题.md` 那样集中一处,并编号。

```md
✅ 推荐 — 连决策的「为什么」与替代、结果都写上
# ADR-0007: 采用 Redis 作为会话存储

- 状态: 已批准 (2026-06-16)
- 背景: 实例增至多台,内存会话失效。
- 决策: 引入 Redis 作为外部会话存储。
- 替代: DB 会话(延迟↑)、粘性会话(扩展性↓) — 明示否决理由。
- 结果: 新增 1 个运维组件,可水平扩展。

❌ 禁止 — 只有结论,没有依据、替代
用 Redis。
```

> 一旦记录的 ADR **不再修改**,只改状态(如 `已被取代 → ADR-0012`)。决策的历史必须留存。

### 2-3. 注释是「为什么」重于「是什么」

```js
// ❌ 禁止 — 照搬代码的注释(信息量为 0,代码一变就成虚假)
i = i + 1; // 给 i 加 1

// ✅ 推荐 — 代码说不出的「为什么」
// 外部 API 用 0-based 页码,故做校正(工单 #482)
page = page + 1;
```

- 不给自明的代码加注释。把名字取好以减少注释。
- 临时凑合、绕行用 `// TODO:`、`// HACK:` 标注,并留下理由、工单。
- 公开函数、模块用 docstring 写出输入、输出、异常等契约(格式遵循栈的惯例)。

### 2-4. 代码变更时同步文档

```
改变行为/配置/结构的 PR  ⟶  在同一个 PR 里同时更新 README、ADR、注释、示例
```

- 改了环境变量、命令、API、目录结构,就把 README 和示例一起修。
- 不再属实的句子立即删掉 — 不要「以防万一」留着。
- 为防漏同步,把「文档是否更新」放进 PR 模板/评审检查清单。

## 3. 常见错误

AI 代理和赶工时常出现的 — 评审时剔除。

- ❌ 只改代码,README、示例还是旧命令、旧变量名(虚假文档)
- ❌ 没有「为什么」的 ADR、缺失决策 → 6 个月后无人知道理由
- ❌ 照搬代码的注释,或放任注释掉的死代码
- ❌ 文档散落在 README、wiki、代码注释、聊天里,没有单一来源
- ❌ 在 README 里重复定义领域术语(术语表通过 `docs-glossary` 技能统一)
- ❌ 把 AI 用规则混进普通文档(代理规则参见 `agent-rules-file` 技能)

> **应用技巧**: 在贡献指南和代理规则文件里钉上一行「改了行为就在同一个 PR 里修文档」,人和 AI 每次都会守住这个同步。

## 4. 检查清单

- [ ] README 是否有全部 6 个章节(概述、要求、安装、运行、项目结构、贡献)
- [ ] 新人是否只看 README 就能到达构建、运行、测试
- [ ] 重要架构决策是否以含「为什么、替代、结果」的 ADR 记录在一处
- [ ] 注释是否说「为什么」而非「是什么」(没有自明注释、死代码)
- [ ] 本次变更是否在同一个 PR 里更新、删除了陈旧文档
- [ ] 同类文档是否集中在一致位置,且角色与术语表/代理规则不重叠
