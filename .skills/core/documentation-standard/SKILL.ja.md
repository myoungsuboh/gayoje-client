# 📚 ドキュメント標準 — README · ADR · コメント · 同期

> ドキュメントをどこに・何を・どう書くかを釘付けにして、新規人員と AI エージェントが検索一度で文脈を得られるようにする。新しいリポジトリを開いたり、ドキュメントを追加・整理したり、重要な決定を残すときに読む。

ドキュメントはコードと同じくらい重要な成果物だ。特に AI エージェントはコードを直接実行してみることができないので、**README・決定記録・コメントに書かれた文脈に依存**して作業する。ドキュメントが無いか古いと、人も AI も誤った仮定をし、同じ質問・同じ間違いを繰り返す。最も危険なのは**誤ったドキュメント**だ — 古いか虚偽のドキュメントは無ドキュメントより悪い(読む人がそれを信じるため)。

## 1. 核心原則

- ドキュメントはコードと共に変更する — 挙動を変える PR は関連ドキュメントも同じ PR で直す。「あとで」は来ない。
- 誤ったドキュメント > 無ドキュメントの害 — 古いか虚偽のドキュメントは削除・修正する。自信のない内容を推測で書かない。
- 新規者・AI エージェント基準で書く — 事前知識なしで README だけ読んでビルド・実行に到達できなければならない。
- コメントは「何を」ではなく「なぜ」 — コードが語ることを繰り返さず、コードが語れない意図・背景・トレードオフを書く。
- ドキュメントの位置を一貫させる — 同じ種類のドキュメントは常に同じ場所に置く。散らばったドキュメントは無いドキュメントだ。
- 重要な技術決定は ADR で残す — コードだけ見ても「なぜこうしたか」が分からない選択は根拠と共に記録する。

## 2. ルール

### 2-1. README 必須セクション

リポジトリルートの `README.md` は以下のセクションを備える。順序は自由だが**欠落は禁止**。

| セクション | 答えるべき質問 |
|---|---|
| 概要 | このプロジェクトは何で、なぜ存在するか |
| 要件 | どんなランタイム・ツール・バージョンが必要か |
| インストール | 依存をどう取得するか |
| 実行 | ローカルでどう起動しテストするか |
| プロジェクト構造 | 主要なディレクトリ・エントリーポイントはどこか |
| 貢献 | ブランチ・コミット・PR ルールは何か |

```md
✅ 推奨 — コピペ可能なコマンドで
## 実行
\`\`\`
<パッケージマネージャ> install
<実行コマンド>        # http://localhost:<ポート>
<テストコマンド>
\`\`\`

❌ 禁止 — 抽象的な叙述(新規者がたどれない)
## 実行
依存をインストールしたあと、いつものようにサーバーを起動すればよいです。
```

### 2-2. ADR — アーキテクチャ決定記録

戻しにくいか、チームへの影響が大きい決定(フレームワーク選択、データ保存方式、認証戦略など)は ADR 一枚で残す。位置は `docs/adr/NNNN-タイトル.md` のように一箇所に集め、番号を振る。

```md
✅ 推奨 — 決定の「なぜ」と代替・結果まで
# ADR-0007: セッションストアとして Redis を採用

- 状態: 承認済み (2026-06-16)
- 文脈: インスタンスが複数台に増えてインメモリセッションが壊れた。
- 決定: 外部セッションストアとして Redis を導入する。
- 代替: DB セッション(遅延↑)、スティッキーセッション(拡張性↓) — 却下理由を明示。
- 結果: 運用コンポーネント1個追加、水平拡張が可能。

❌ 禁止 — 結論だけ、根拠・代替なし
Redis 使います。
```

> 一度記録した ADR は**修正せず**状態だけ変える(例: `置換済み → ADR-0012`)。決定の歴史が残らなければならない。

### 2-3. コメントは「何を」より「なぜ」

```js
// ❌ 禁止 — コードをそのまま移したコメント(情報量0、コードが変わると虚偽になる)
i = i + 1; // i に 1 を足す

// ✅ 推奨 — コードが語れない「なぜ」
// 外部 API が 0-based ページを使うので補正する (チケット #482)
page = page + 1;
```

- 自明なコードにはコメントを付けない。名前をうまく付けてコメントを減らす。
- 一時しのぎ・回避は `// TODO:`・`// HACK:` で示し、理由・チケットを残す。
- 公開関数・モジュールは docstring で入力・出力・例外などの契約を書く(形式はスタックの慣例に従う)。

### 2-4. コード変更時のドキュメント同期

```
挙動/設定/構造を変える PR  ⟶  同じ PR で README・ADR・コメント・例も更新
```

- 環境変数・コマンド・API・ディレクトリ構造を変えたら README と例を一緒に直す。
- もはや事実でない文は即座に消す — 「もしものために」残さない。
- 同期漏れを防ぐため PR テンプレート/レビューチェックリストに「ドキュメント更新の有無」を入れる。

## 3. よくある間違い

AI エージェントと急ぎの作業でよく出るもの — レビューで取り除く。

- ❌ コードだけ変えて README・例は古いコマンド・古い変数名のまま(虚偽ドキュメント)
- ❌ 「なぜ」のない ADR・決定の欠落 → 6か月後に誰も理由を知らない
- ❌ コードをそのまま移したコメント、またはコメントアウトされたデッドコードの放置
- ❌ ドキュメントが README・wiki・コードコメント・チャットに散らばり単一の出典がない
- ❌ ドメイン用語を README に重複定義(用語集は `docs-glossary` スキルで一元化)
- ❌ AI 用ルールを一般ドキュメントに混ぜる(エージェントルールは `agent-rules-file` スキル参照)

> **適用のコツ**: 「挙動を変えたらドキュメントも同じ PR で直す」を貢献ガイドとエージェントルールファイルに一行埋め込んでおくと、人も AI も毎回この同期を守る。

## 4. チェックリスト

- [ ] README に6つのセクション(概要・要件・インストール・実行・プロジェクト構造・貢献)がすべてあるか
- [ ] 新規者が README だけ見てビルド・実行・テストに到達できるか
- [ ] 重要なアーキテクチャ決定が「なぜ・代替・結果」を含む ADR で一箇所に記録されたか
- [ ] コメントが「何を」ではなく「なぜ」を語っているか(自明なコメント・デッドコードがない)
- [ ] 今回の変更で古いドキュメントが同じ PR で更新・削除されたか
- [ ] 同じ種類のドキュメントが一貫した位置に集まり、用語集/エージェントルールと役割が重ならないか
