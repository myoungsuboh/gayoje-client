# 📚 문서화 표준 — README · ADR · 주석 · 동기화

> 문서를 어디에·무엇을·어떻게 쓸지 못 박아, 신규 인원과 AI 에이전트가 검색 한 번으로 맥락을 얻게 한다. 새 저장소를 열거나, 문서를 추가·정리하거나, 중요한 결정을 남길 때 읽는다.

문서는 코드만큼 중요한 산출물이다. 특히 AI 에이전트는 코드를 직접 실행해 보지 못하므로 **README·결정 기록·주석에 적힌 맥락에 의존**해 작업한다. 문서가 없거나 낡으면 사람과 AI 모두 잘못된 가정을 하고, 같은 질문·같은 실수를 반복한다. 가장 위험한 것은 **틀린 문서**다 — 오래되거나 거짓인 문서는 무문서보다 나쁘다(읽는 사람이 그것을 믿기 때문).

## 1. 핵심 원칙

- 문서는 코드와 함께 변경한다 — 동작을 바꾸는 PR은 관련 문서도 같은 PR에서 고친다. "나중에"는 오지 않는다.
- 틀린 문서 > 무문서의 해악 — 낡거나 거짓인 문서는 삭제·수정한다. 자신 없는 내용을 추정으로 적지 않는다.
- 신규자·AI 에이전트 기준으로 쓴다 — 사전 지식 없이 README만 읽고 빌드·실행에 도달할 수 있어야 한다.
- 주석은 '무엇'이 아니라 '왜' — 코드가 말하는 것을 반복하지 말고, 코드가 말 못 하는 의도·배경·트레이드오프를 적는다.
- 문서 위치를 일관되게 — 같은 종류의 문서는 늘 같은 곳에 둔다. 흩어진 문서는 없는 문서다.
- 중요한 기술 결정은 ADR로 남긴다 — 코드만 보면 "왜 이렇게 했는지"를 알 수 없는 선택은 근거와 함께 기록한다.

## 2. 규칙

### 2-1. README 필수 섹션

저장소 루트의 `README.md`는 아래 섹션을 갖춘다. 순서는 자유지만 **누락은 금지**.

| 섹션 | 답해야 할 질문 |
|---|---|
| 개요 | 이 프로젝트는 무엇이고 왜 존재하나 |
| 요구사항 | 어떤 런타임·도구·버전이 필요한가 |
| 설치 | 의존성을 어떻게 받나 |
| 실행 | 로컬에서 어떻게 띄우고 테스트하나 |
| 프로젝트 구조 | 주요 디렉터리·진입점이 어디인가 |
| 기여 | 브랜치·커밋·PR 규칙은 무엇인가 |

```md
✅ 권장 — 복붙 가능한 명령으로
## 실행
\`\`\`
<패키지매니저> install
<실행 명령>        # http://localhost:<포트>
<테스트 명령>
\`\`\`

❌ 금지 — 추상적 서술 (신규자가 따라 할 수 없음)
## 실행
의존성을 설치한 뒤 평소처럼 서버를 띄우면 됩니다.
```

### 2-2. ADR — 아키텍처 결정 기록

되돌리기 어렵거나 팀에 영향이 큰 결정(프레임워크 선택, 데이터 저장 방식, 인증 전략 등)은 ADR 한 장으로 남긴다. 위치는 `docs/adr/NNNN-제목.md`처럼 한 곳에 모으고, 번호를 매긴다.

```md
✅ 권장 — 결정의 '왜'와 대안·결과까지
# ADR-0007: 세션 저장소로 Redis 채택

- 상태: 승인됨 (2026-06-16)
- 맥락: 인스턴스가 여러 대로 늘며 인메모리 세션이 깨짐.
- 결정: 외부 세션 저장소로 Redis를 도입한다.
- 대안: DB 세션(지연↑), 스티키 세션(확장성↓) — 기각 사유 명시.
- 결과: 운영 컴포넌트 1개 추가, 수평 확장 가능.

❌ 금지 — 결론만, 근거·대안 없음
Redis 씁니다.
```

> 한 번 기록한 ADR은 **수정하지 않고** 상태만 바꾼다(예: `대체됨 → ADR-0012`). 결정의 역사가 남아야 한다.

### 2-3. 주석은 '무엇'보다 '왜'

```js
// ❌ 금지 — 코드를 그대로 옮긴 주석 (정보량 0, 코드 바뀌면 거짓이 됨)
i = i + 1; // i에 1을 더한다

// ✅ 권장 — 코드가 못 말하는 '왜'
// 외부 API가 0-based 페이지를 쓰므로 보정한다 (티켓 #482)
page = page + 1;
```

- 자명한 코드에는 주석을 달지 않는다. 이름을 잘 지어 주석을 줄인다.
- 임시방편·우회는 `// TODO:`·`// HACK:`로 표시하고 사유·티켓을 남긴다.
- 공개 함수·모듈은 docstring으로 입력·출력·예외 등 계약을 적는다(형식은 스택 관례를 따른다).

### 2-4. 코드 변경 시 문서 동기화

```
동작/설정/구조를 바꾸는 PR  ⟶  같은 PR에서 README·ADR·주석·예제도 갱신
```

- 환경 변수·명령·API·디렉터리 구조를 바꿨으면 README와 예제를 함께 고친다.
- 더 이상 사실이 아닌 문장은 즉시 지운다 — "혹시 몰라" 남기지 않는다.
- 동기화 누락을 막기 위해 PR 템플릿/리뷰 체크리스트에 "문서 갱신 여부"를 넣는다.

## 3. 흔한 실수

AI 에이전트와 급한 작업에서 자주 나오는 것 — 리뷰에서 거른다.

- ❌ 코드만 바꾸고 README·예제는 옛 명령·옛 변수명 그대로 (거짓 문서)
- ❌ "왜"가 없는 ADR·결정 누락 → 6개월 뒤 아무도 이유를 모름
- ❌ 코드를 그대로 옮긴 주석, 또는 주석 처리된 죽은 코드 방치
- ❌ 문서가 README·위키·코드 주석·채팅에 흩어져 단일 출처가 없음
- ❌ 도메인 용어를 README에 중복 정의 (용어집은 `docs-glossary` 스킬로 일원화)
- ❌ AI용 규칙을 일반 문서에 섞음 (에이전트 규칙은 `agent-rules-file` 스킬 참조)

> **적용 팁**: "동작을 바꾸면 문서도 같은 PR에서 고친다"를 기여 가이드와 에이전트 규칙 파일에 한 줄 박아두면, 사람도 AI도 매번 이 동기화를 지킨다.

## 4. 체크리스트

- [ ] README에 6개 섹션(개요·요구사항·설치·실행·프로젝트 구조·기여)이 모두 있는가
- [ ] 신규자가 README만 보고 빌드·실행·테스트에 도달할 수 있는가
- [ ] 중요한 아키텍처 결정이 '왜·대안·결과'를 담은 ADR로 한곳에 기록됐는가
- [ ] 주석이 '무엇'이 아니라 '왜'를 말하는가 (자명한 주석·죽은 코드 없음)
- [ ] 이번 변경으로 낡은 문서가 같은 PR에서 갱신·삭제됐는가
- [ ] 같은 종류 문서가 일관된 위치에 모여 있고, 용어집/에이전트 규칙과 역할이 겹치지 않는가
