# 🔍 전문 검색 (Full-text Search)

> 자연어·형태소 기반 검색을 특정 엔진에 종속되지 않는 방식으로 설계한다. 검색 기능을 새로 도입하거나, LIKE 쿼리를 걷어내고 전용 검색으로 옮기거나, 한국어 검색 품질·동기화·타임아웃을 정할 때 읽는다.

## 1. 핵심 원칙
- 복잡한 전문 검색(자연어·형태소·부스팅·오타 허용)은 전용 검색엔진을 쓰고, `LIKE '%...%'` 쿼리를 남용하지 않는다.
- 한국어 텍스트는 형태소 분석기를 적용해 형태소 단위로 색인·검색한다 — 공백 분리만으로는 한국어 검색 품질이 떨어진다.
- 검색 인덱스는 원본 DB의 단순 replica가 아니라 별도 동기화 파이프라인(CDC·이벤트·배치)으로 채운다.
- 검색 품질은 측정한다 — 오프라인 지표(nDCG·MRR)와 온라인 지표(클릭률)로 평가하고 개선한다.
- 쿼리에는 타임아웃을 두고, 초과 시 빈 결과로 위장하지 말고 오류와 재시도 경로를 제공한다.

## 2. 규칙

### 2-1. 전용 검색엔진 vs LIKE 남용 금지
- 단순 일치·소량 문서는 DB 내장 기능으로 충분하지만, 자연어/형태소/부스팅/오타 허용이 필요하면 전용 검색엔진으로 옮긴다.
- `LIKE '%키워드%'` 는 선행 와일드카드 때문에 인덱스를 타지 못해 풀스캔이 되고, 형태소·부스팅·하이라이트를 지원하지 못한다.

```text
# ❌ 금지 — 전문 검색 요구를 LIKE 로 떼우기 (풀스캔·형태소 미지원·관련도 없음)
SELECT * FROM docs WHERE title LIKE '%사용자 입력%' OR content LIKE '%사용자 입력%'

# ✅ 권장 — 전용 검색엔진의 형태소 분석 + 필드 부스팅 + 하이라이트로 질의
search(query="사용자 입력", fields=["title^3", "content"], fuzziness="AUTO")
```

엔진 선택은 규모·운영 환경·기능 요구로 결정한다.

| 엔진 | 장점 | 적합 상황 |
|------|------|-----------|
| Elasticsearch | 기능 풍부, 한국어 nori | 대규모·복잡한 검색 |
| OpenSearch | AWS 통합, OSS | AWS 환경 |
| Typesense | 간편 설치, 오타 허용 | 소규모 SaaS |
| PostgreSQL FTS | 추가 인프라 없음 | 소량 문서·단순 검색 |

### 2-2. 한국어 형태소 분석
- 한국어는 조사·어미가 붙어 단어 경계가 공백과 일치하지 않으므로, 형태소 분석기로 토큰화해야 "검색했다 / 검색하는 / 검색" 이 같은 어근으로 매칭된다.
- 엔진별로 대표 분석기를 선택한다: Elasticsearch는 `nori`, 그 외 환경은 `mecab`·`khaiii` 등.
- 색인 시점과 질의 시점에 같은 분석기를 적용해야 토큰이 어긋나지 않는다.

### 2-3. 별도 동기화 파이프라인
- 검색 인덱스는 원본 DB를 그대로 복제한 것이 아니라, 검색에 최적화된 별도 저장소다. DB 변경을 인덱스에 반영하는 동기화 경로를 명시적으로 둔다.
- CDC(예: Debezium)·도메인 이벤트·주기적 배치 중 데이터 신선도 요구에 맞는 방식을 고른다.

```text
# ✅ 권장 — DB 변경을 파이프라인으로 인덱스에 반영
DB 변경 → CDC / 이벤트 발행 → 큐 → 인덱서 워커 → 검색엔진 업서트
```

### 2-4. 검색 품질 지표
- 검색은 "결과가 나온다"로 끝이 아니라 "관련 있는 결과가 위에 있다"가 목표다 — 반드시 측정한다.
- 오프라인: nDCG·MRR 로 랭킹 품질을 정량 평가한다.
- 온라인: 사용자 클릭률(CTR)·재검색률로 실제 만족도를 추적한다.

### 2-5. 쿼리 타임아웃
- 검색 쿼리에 타임아웃(예: 5초)을 설정해 느린 질의가 시스템을 점유하지 못하게 한다.
- 타임아웃 발생 시 빈 결과로 위장하면 사용자는 "결과 없음"으로 오해한다 — 오류 메시지와 재시도 경로를 제공한다.

## 3. 흔한 실수
- 전문 검색 요구를 `LIKE '%...%'` 로 떼움 → 풀스캔·형태소 미지원·관련도 없음.
- 한국어에 형태소 분석기 미적용 → 어미 변화·조사 때문에 검색 누락.
- 색인/질의에 서로 다른 분석기 적용 → 토큰 불일치로 매칭 실패.
- 인덱스를 DB와 동기화하는 경로 없음 → 검색 결과가 최신 데이터와 어긋남.
- 품질 지표 미측정 → 랭킹이 나빠도 인지하지 못함.
- 타임아웃 미설정 또는 초과 시 빈 결과 반환 → 장애가 "결과 없음"으로 둔갑.

## 4. 체크리스트
- [ ] 자연어/형태소 검색 요구를 LIKE 가 아니라 전용 검색엔진으로 처리했는가
- [ ] 규모·운영 환경에 맞는 엔진을 선택했는가
- [ ] 한국어에 형태소 분석기를 적용하고, 색인/질의에 같은 분석기를 썼는가
- [ ] DB → 인덱스 동기화 파이프라인(CDC·이벤트·배치)을 설계했는가
- [ ] 검색 품질을 nDCG·MRR(오프라인)과 클릭률(온라인)로 측정하는가
- [ ] 쿼리 타임아웃을 설정하고, 초과 시 오류·재시도를 제공하는가

## 부록: 스택별 예시

특정 엔진에 종속된 구체 설정이다. 위 본문 원칙을 Elasticsearch로 구현할 때의 참고용이며, 다른 엔진을 쓰면 대응 설정으로 대체한다.

### Elasticsearch 인덱스 설정 (한국어 nori)

```json
{
  "settings": {
    "analysis": {
      "analyzer": {
        "korean": {
          "type": "custom",
          "tokenizer": "nori_tokenizer",
          "filter": ["lowercase", "nori_part_of_speech"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title":   { "type": "text", "analyzer": "korean", "boost": 3 },
      "content": { "type": "text", "analyzer": "korean" },
      "tags":    { "type": "keyword" }
    }
  }
}
```

### Elasticsearch 검색 쿼리 패턴

```json
{
  "query": {
    "multi_match": {
      "query": "사용자 입력",
      "fields": ["title^3", "content"],
      "type": "best_fields",
      "fuzziness": "AUTO"
    }
  },
  "highlight": {
    "fields": { "title": {}, "content": {} }
  }
}
```

### DB → 검색 인덱스 동기화 (구현 예)

```text
DB 변경 → CDC(Debezium) → Kafka → 인덱서 워커 → ES 업서트
```
또는
```text
DB 변경 시 → 이벤트 발행 → 인덱스 업데이트 잡 큐 → ES 업서트
```
