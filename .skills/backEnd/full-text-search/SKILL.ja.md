# 🔍 全文検索 (Full-text Search)

> 自然言語・形態素ベースの検索を特定エンジンに依存しない方式で設計する。検索機能を新たに導入する、LIKE クエリを取り除いて専用検索へ移す、あるいは韓国語の検索品質・同期・タイムアウトを定めるときに読む。

## 1. 核心原則
- 複雑な全文検索(自然言語・形態素・ブースティング・タイポ許容)は専用検索エンジンを使い、`LIKE '%...%'` クエリを乱用しない。
- 韓国語テキストは形態素解析器を適用し形態素単位で索引・検索する — 空白分割だけでは韓国語の検索品質が落ちる。
- 検索インデックスは元 DB の単純なレプリカではなく、別途の同期パイプライン(CDC・イベント・バッチ)で満たす。
- 検索品質は測定する — オフライン指標(nDCG・MRR)とオンライン指標(クリック率)で評価し改善する。
- クエリにはタイムアウトを設け、超過時は空の結果に偽装せずエラーと再試行経路を提供する。

## 2. 規則

### 2-1. 専用検索エンジン vs LIKE 乱用禁止
- 単純一致・少量ドキュメントは DB 内蔵機能で十分だが、自然言語/形態素/ブースティング/タイポ許容が必要なら専用検索エンジンへ移す。
- `LIKE '%キーワード%'` は先頭ワイルドカードのためインデックスを使えずフルスキャンになり、形態素・ブースティング・ハイライトを支援できない。

```text
# ❌ 禁止 — 全文検索要件を LIKE で済ませる(フルスキャン・形態素非対応・関連度なし)
SELECT * FROM docs WHERE title LIKE '%사용자 입력%' OR content LIKE '%사용자 입력%'

# ✅ 推奨 — 専用検索エンジンの形態素解析 + フィールドブースティング + ハイライトで照会
search(query="사용자 입력", fields=["title^3", "content"], fuzziness="AUTO")
```

エンジン選択は規模・運用環境・機能要件で決める。

| エンジン | 長所 | 適した状況 |
|------|------|-----------|
| Elasticsearch | 機能豊富、韓国語 nori | 大規模・複雑な検索 |
| OpenSearch | AWS 統合、OSS | AWS 環境 |
| Typesense | 簡単インストール、タイポ許容 | 小規模 SaaS |
| PostgreSQL FTS | 追加インフラなし | 少量ドキュメント・単純検索 |

### 2-2. 韓国語形態素解析
- 韓国語は助詞・語尾が付いて単語境界が空白と一致しないため、形態素解析器でトークン化する必要があり、それにより「검색했다 / 검색하는 / 검색」が同じ語根でマッチする。
- エンジンごとに代表的な解析器を選ぶ: Elasticsearch は `nori`、その他の環境は `mecab`・`khaiii` など。
- 索引時点とクエリ時点に同じ解析器を適用しないとトークンがずれる。

### 2-3. 別途の同期パイプライン
- 検索インデックスは元 DB をそのまま複製したものではなく、検索に最適化された別のストアだ。DB 変更をインデックスに反映する同期経路を明示的に置く。
- CDC(例: Debezium)・ドメインイベント・定期バッチのうち、データ鮮度要件に合う方式を選ぶ。

```text
# ✅ 推奨 — DB 変更をパイプラインでインデックスに反映
DB 変更 → CDC / イベント発行 → キュー → インデクサワーカー → 検索エンジン upsert
```

### 2-4. 検索品質指標
- 検索は「結果が出る」で終わりではなく「関連ある結果が上にある」が目標だ — 必ず測定する。
- オフライン: nDCG・MRR でランキング品質を定量評価する。
- オンライン: ユーザーのクリック率(CTR)・再検索率で実際の満足度を追跡する。

### 2-5. クエリタイムアウト
- 検索クエリにタイムアウト(例: 5秒)を設定し、遅いクエリがシステムを占有できないようにする。
- タイムアウト発生時に空の結果へ偽装するとユーザーは「結果なし」と誤解する — エラーメッセージと再試行経路を提供する。

## 3. よくある間違い
- 全文検索要件を `LIKE '%...%'` で済ませる → フルスキャン・形態素非対応・関連度なし。
- 韓国語に形態素解析器を未適用 → 語尾変化・助詞のため検索漏れ。
- 索引/クエリに異なる解析器を適用 → トークン不一致でマッチ失敗。
- インデックスを DB と同期する経路なし → 検索結果が最新データとずれる。
- 品質指標を未測定 → ランキングが悪くても気づかない。
- タイムアウト未設定または超過時に空の結果を返す → 障害が「結果なし」に化ける。

## 4. チェックリスト
- [ ] 自然言語/形態素検索の要件を LIKE ではなく専用検索エンジンで処理したか
- [ ] 規模・運用環境に合うエンジンを選択したか
- [ ] 韓国語に形態素解析器を適用し、索引/クエリに同じ解析器を使ったか
- [ ] DB → インデックス同期パイプライン(CDC・イベント・バッチ)を設計したか
- [ ] 検索品質を nDCG・MRR(オフライン)とクリック率(オンライン)で測定するか
- [ ] クエリタイムアウトを設定し、超過時にエラー・再試行を提供するか

## 付録: スタック別の例

特定エンジンに依存した具体的な設定だ。上の本文の原則を Elasticsearch で実装する際の参考であり、他のエンジンを使うなら対応する設定に置き換える。

### Elasticsearch インデックス設定 (韓国語 nori)

```json
{
  "settings": {
    "analysis": {
      "analyzer": {
        "korean": {
          "type": "custom",
          "tokenizer": "nori_tokenizer",
          "filter": ["lowercase", "nori_part_of_speech"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title":   { "type": "text", "analyzer": "korean", "boost": 3 },
      "content": { "type": "text", "analyzer": "korean" },
      "tags":    { "type": "keyword" }
    }
  }
}
```

### Elasticsearch 検索クエリパターン

```json
{
  "query": {
    "multi_match": {
      "query": "사용자 입력",
      "fields": ["title^3", "content"],
      "type": "best_fields",
      "fuzziness": "AUTO"
    }
  },
  "highlight": {
    "fields": { "title": {}, "content": {} }
  }
}
```

### DB → 検索インデックス同期 (実装例)

```text
DB 変更 → CDC(Debezium) → Kafka → インデクサワーカー → ES upsert
```
または
```text
DB 変更時 → イベント発行 → インデックス更新ジョブキュー → ES upsert
```
