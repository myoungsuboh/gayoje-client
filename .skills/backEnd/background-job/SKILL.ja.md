# ⚙️ バックグラウンドジョブ & キュー

> 長時間処理をキューに切り離し、冪等・指数バックオフのリトライ・DLQ で安定的に処理する。非同期ジョブパイプラインを設計するときやリトライ・モニタリングを決めるときに読む。

## 1. 核心原則

- 長時間処理(メール・レポート・外部 API 呼び出し)は HTTP リクエストから切り離してキューに入れ、即座に 202 Accepted を返す。
- 失敗したジョブは指数バックオフ(例: 1m→5m→15m→1h)で最大 N 回リトライし、超過したら Dead Letter Queue へ移す。
- ジョブ定義は冪等に設計する — 同じジョブが二度実行されても副作用が重複しないようにする。
- ジョブごとに timeout を設定し、実行時間が長くなれば heartbeat でアクティブ状態を更新する。
- DLQ を定期的にモニタリングし、手動再処理インターフェースを提供する。

## 2. ルール

### 2-1. アーキテクチャパターン

```
HTTP 요청 → API 서버 → 큐(Redis/RabbitMQ/SQS) → 워커 프로세스
              ↓                                        ↓
          202 + job_id                          결과 저장(DB/Cache)
```

### 2-2. リトライ戦略 (指数バックオフ)

```
시도 1: 즉시
시도 2: 1분 후
시도 3: 5분 후
시도 4: 15분 후
시도 5: 1시간 후
→ 초과: Dead Letter Queue (수동 검토)
```

### 2-3. ジョブの冪等性

```python
# ✅ 권장 — 중복 실행돼도 부작용이 한 번만 발생하도록 가드
def send_welcome_email(user_id: str):
    # 이미 발송됐으면 스킵
    if cache.get(f"welcome_sent:{user_id}"):
        return
    email_service.send(user_id, template="welcome")
    cache.set(f"welcome_sent:{user_id}", 1, ex=86400)
```

| スタック | ライブラリ |
|------|-----------|
| Python | Celery + Redis, ARQ |
| Node.js | BullMQ, pg-boss |
| Java | Spring Batch, Quartz |

### 2-5. モニタリング必須項目

- キュー長アラート(閾値超過時に Slack・PagerDuty へ)
- DLQ ジョブ数ダッシュボード
- ジョブ処理時間ヒストグラム(P95 基準で SLA を設定)
- ワーカープロセス稼働状態のヘルスチェック

## 3. よくあるミス

- 長時間処理を HTTP リクエスト内で同期処理 → タイムアウト・遅いレスポンス。
- リトライなしで一度失敗するとジョブ消失、または無限リトライでキュー過負荷。
- 冪等性の未保証 → リトライ時にメール・決済が重複実行される。
- DLQ を貯めるだけでモニタリング・再処理の手段がなく、失敗ジョブを放置。
