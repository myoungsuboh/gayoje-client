# 🏢 マルチテナンシー設計

> 1 インスタンスが複数の顧客（テナント）にサービスを提供する SaaS の核心。分離・識別・漏洩防止の 3 大軸を決める際に読む。

## 1. 核心原則
- 分離レベルをまず決める — DB-per-tenant（強分離・高コスト）/ スキーマ-per-tenant / 行ベース（tenant_id、低コスト・漏洩リスク）。
- 行ベースならすべてのクエリに tenant_id フィルタを強制する — ORM グローバルフィルタまたは DB Row-Level Security で漏れを根本から遮断する。
- テナント識別は認証トークン（JWT claim）・サブドメインから導出する — クライアントが送った tenant パラメータを信頼しない。
- テナント間データ漏洩は致命的 — 「他テナントのデータへのアクセス試行 → 遮断」を必須テストケースに置く。
- 共用リソース（コネクションプール・キャッシュキー・ファイルパス）に tenant_id をネームスペースとして混ぜ、衝突・漏洩を防ぐ。

## 2. 規則

### 2-1. 分離戦略の比較
| 戦略 | 分離強度 | コスト・拡張 | 特徴 |
|---|---|---|---|
| DB per tenant | 最も強い | テナントが多いと運用費が急増 | バックアップ・復旧・規制対応が容易 |
| スキーマ per tenant | 中間 | 中間 | 1 つの DB 内でスキーマを分離 |
| 行ベース（共有テーブル） | 最も弱い | 最も安価・拡張容易 | tenant_id 欠落時に即漏洩 — RLS・グローバルフィルタ必須 |

### 2-2. テナント識別 — クライアント入力を信頼しない
```text
// ❌ 禁止 — クライアントが送った tenant パラメータをそのまま信頼
tenantId = request.query.tenantId

// ✅ 推奨 — 認証トークン（JWT claim）・サブドメインから導出
tenantId = jwt.claims.tenant_id   // または subdomain → tenant マッピング
```

### 2-3. 行ベースクエリ — tenant_id を強制
```text
// ❌ 禁止 — クエリから tenant_id フィルタが抜けると即漏洩
SELECT * FROM orders WHERE id = :id

// ✅ 推奨 — ORM グローバルフィルタまたは DB Row-Level Security で漏れを根本から遮断
SELECT * FROM orders WHERE id = :id AND tenant_id = :tenantId
```

### 2-4. 共用リソースのネームスペース化
```text
// ❌ 禁止 — キャッシュキー・ファイルパスに tenant の区別がなく衝突・漏洩
cache.get("user:" + userId)

// ✅ 推奨 — tenant_id をネームスペースとして混ぜる
cache.get(tenantId + ":user:" + userId)
```

### 2-5. 安全装置のフロー
- 認証でテナントを確定 → リクエストコンテキストに固定 → データアクセス層で強制適用。
- テナント間アクセステストを回帰スイートに含める（セキュリティ回帰の防止）。

## 3. よくある間違い
- 行ベースなのに一部のクエリで tenant_id フィルタ欠落 → テナント間漏洩。
- クライアントが送った tenant パラメータを信頼 → 改ざんによる他テナントアクセス。
- キャッシュキー・ファイルパスに tenant ネームスペース未適用 → データ衝突・漏洩。
- テナント間アクセステストの不在 → セキュリティ回帰を捕捉できない。

## 4. チェックリスト
- [ ] 分離レベル（DB/スキーマ/行）をまず決めたか
- [ ] 行ベースならすべてのクエリに tenant_id フィルタが強制されるか（グローバルフィルタ・RLS）
- [ ] テナントを認証トークン・サブドメインから導出したか（クライアント入力を信頼しない）
- [ ] 共用リソースに tenant_id ネームスペースを適用したか
- [ ] 「他テナントアクセス試行 → 遮断」テストを回帰スイートに入れたか
