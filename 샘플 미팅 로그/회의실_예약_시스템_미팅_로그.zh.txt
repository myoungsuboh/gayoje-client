### [会议记录 V1] - 启动会：会议室争夺战，再也受不了了
* **时间:** 2026-03-02 (10:00 ~ 11:00)
* **参会者:** 运营组长(以下简称 PO)、产品经理(以下简称 PM)、开发主管(以下简称 DEV)、设计师(以下简称 UX)、总务组负责人(以下简称 总务)
* **PO:** "总部有8间会议室，可每天早上 Slack 上都在转发'下午3点有人要用 B 会议室吗？'这类消息。上个月员工问卷里，有42%的人回答'抢会议室是最大的工作压力'。现在这种在白板上手写登记的方式已经到极限了。"
* **总务:** "说说手写预约的实际情况吧，预约了却不出现的爽约，体感上每3件里就有1件。明明是空着的会议室，可白板上写着名字，谁都不敢进。反过来，不登记就直接用，结果跟预约人撞上、闹得脸红脖子粗的事，每周都有。"
* **PM:** "所以问题归纳起来就三点。1) 现在哪间会议室是空的，谁都不知道；2) 预约了却不用的爽约太多；3) 就算出现预约冲突，也没有可以仲裁的标准。"
* **DEV:** "我也评估过引入外部 SaaS，但它按人头计费，一年要花一千多万韩元，而且没法跟我们公司内部的 SSO 认证打通。就8间会议室这点规模，还不如我们自己做个轻量的。不过范围一定要收窄。一开始就别加视频会议联动那种东西。"
* **UX:** "我同意。核心就是'在走廊里掏出手机，10秒内找到空房并完成预约'。只要把这一个体验做扎实就够了。"
* **PO:** "好。目标就简简单单来吧。一眼看到空会议室，立刻预约，减少爽约。就这三点。下次会议上把功能范围定下来。"
* **[进度: 10% - 启动会完成，三大问题(可见性·爽约·冲突)及目标达成共识]**

### [会议记录 V2] - 核心功能范围敲定：现状看板 · 预约 · 我的预约
* **时间:** 2026-03-05 (14:00 ~ 15:30)
* **参会者:** PM、DEV、UX、总务
* **PM:** "我先把用户场景理了一遍。主角是'此刻马上就需要会议室的员工'。① 在现状看板上确认空房，② 选好时段进行预约，③ 查看·取消我的预约。就这三条动线，没别的了。"
* **UX:** "现状看板我画了个方案：把8间会议室排成卡片，每张卡片显示'正在使用中/空闲'，并用条形图展示今天各时段的占用情况。颜色只用两种。绿色代表空闲，灰色代表已预约。"
* **DEV:** "创建预约时，输入项只收开始时间、结束时间和会议标题就行。一旦开始要求填参会者名单，输入就会变长，谁都不愿意用了。预约完成后，作为结果返回一个预约号，这样在我的预约里就能直接找到。"
* **总务:** "会议标题请设为必填。白板时代也是，只写个名字的话，根本不知道是哪个组、开什么会，也就没法协调。只要有标题和预约人，发生冲突时当事人之间就能直接协调。"
* **PM:** "预约变更怎么处理？如果做修改功能，时间重叠的校验会变复杂。"
* **DEV:** "第一版就用'取消后重新预约'吧。修改功能等看了使用数据再判断也不迟。作为补偿，取消要做得轻量，两次点击就搞定。"
* **UX:** "我的预约页面上，把即将到来的预约按时间顺序展示，每一项都放个取消按钮。过去的预约就折叠隐藏起来。"
* **PM:** "我来总结。第一版范围是 ① 会议室现状看板，② 创建预约(标题·开始·结束必填)，③ 我的预约列表与取消。预约修改和统计推迟到第二版。"
* **[进度: 30% - 第一版功能范围三项敲定，预约修改推迟到第二版]**

### [会议记录 V3] - 画面流程与防重复预订规则
* **时间:** 2026-03-09 (10:00 ~ 11:30)
* **参会者:** PM、DEV、UX
* **UX:** "画面我想先做四个。① 现状看板(主页面，/board)，② 创建预约弹窗(在看板上点击某个时段即可打开)，③ 我的预约(/my)，④ 管理员设置(/admin，用于会议室的注册·修改)。不单独做首页，看板即首页。"
* **DEV:** "访问权限也按画面来定吧。看板·预约·我的预约只要通过公司内部 SSO 认证，谁都能用；管理员设置只有拥有 admin 权限的总务组账号才能进。没有权限的人进 /admin 时，就显示'无权限'提示，并把他退回看板。"
* **PM:** "在创建预约弹窗里时间怎么选？要是自由输入，就会出现9点13分这种预约了。"
* **UX:** "限制为30分钟为单位的时段。在看板上点击空时段，弹窗打开时那个时间已经预填好了，只要选个结束时间就完事。输入项就只剩会议标题这一个。"
* **DEV:** "防重复预订是这套系统的命根子。规则要定准。同一间会议室里，时间区间只要有1分钟重叠，就拒绝预约并显示'这个时间已被预约'。校验必须放在服务端做。画面上看着是空的，但谁要是早按1秒，服务端就该拒绝，这才是正解。"
* **PM:** "连续预约的独占怎么办？比如一个组把一整天都占了的情况。"
* **DEV:** "限制每人每天最多3件、每件最多2小时。总务说的'大型工作坊算例外'，就用管理员在管理画面里直接登记的方式来解决。"
* **UX:** "预约被拒之类的提示文案，我会打磨得不带责备的口气。像'这个时间已被预约。紧接着的下一个时段是空的哦'这样，连同备选方案一起展示。"
* **PM:** "四个画面、权限区分，以及防重复预订·上限规则都敲定了。下一步是爽约对策。"
* **[进度: 50% - 四个画面 + 权限区分(SSO 认证/admin) + 30分钟时段 + 服务端防重复预订 + 预约上限(每天3件·每件2h)敲定]**

### [会议记录 V4] - 爽约对策：QR 签到与10分钟自动释放
* **时间:** 2026-03-12 (14:00 ~ 15:00)
* **参会者:** PO、PM、DEV、UX、总务
* **PM:** "针对爽约我准备了两个方案。A 方案是：开始后10分钟内不签到，就自动释放预约，把会议室还原成空闲；B 方案是：对爽约次数记扣分，累计3次就限制其一周的预约。"
* **总务:** "扣分制反弹会很大。去年试过迟到扣分制，一个月就废了。比起处罚，'不用就自动释放'的 A 方案更自然。"
* **PO:** "就用 A 方案。签到怎么做？"
* **DEV:** "在会议室门口贴 QR 码。进门时用手机一扫，就打开签到画面(/checkin)，点一下'签到'按钮就完事。这样画面就变成五个了。每间会议室都有专属 QR，所以在别的房间扫码作弊签不了到；在自己没有预约的时间扫码，会显示'找不到此时段的预约'。"
* **UX:** "忘记签到的人肯定不少，所以在预约开始前5分钟发个'请签到'提醒。提醒通过公司内部 Slack 机器人私信发送，就不用另搭推送基础设施了。"
* **PM:** "被自动释放的会议室，必须在看板上立刻显示成'空闲'才有意义。释放的那一刻看板实时刷新，这才算一整套。"
* **DEV:** "那个只要让看板每30秒轮询一次就够了。还没到要上 WebSocket 的规模。"
* **PO:** "拍板。QR 签到(加上签到画面)、10分钟未签到自动释放、开始前5分钟 Slack 提醒。扣分制不引入。"
* **[进度: 70% - 爽约政策敲定(QR 签到 + 签到画面(/checkin) + 10分钟自动释放 + 提前5分钟提醒)，扣分制否决]**

### [会议记录 V5] - 数据模型 · API · 错误响应规则
* **时间:** 2026-03-16 (10:00 ~ 11:30)
* **参会者:** PM、DEV
* **DEV:** "数据模型有三个就够了。会议室(Room)有名称(字符串，最多30字)·楼层(整数)·容纳人数(整数，1以上)·QR 码值(字符串)；预约(Reservation)有会议室 ID·预约人 ID·会议标题(字符串，最多50字)·开始/结束时刻(日期+时间)·状态。状态是已预约、已签到、自动释放、已取消四者之一的固定值。用户(User)直接沿用公司内部 SSO 资料(工号·姓名·邮箱)。不另做注册。"
* **PM:** "API 是什么样的？"
* **DEV:** "先做六个。① 会议室列表·今日占用情况查询(看板每30秒调用一次)，② 创建预约 —— 输入是会议室 ID·会议标题·开始/结束时刻，成功则在响应里返回预约号和状态，③ 取消预约，④ 我的预约列表查询，⑤ QR 签到，⑥ 管理员用会议室注册·修改。创建 API 全权负责防重复预订校验和上限(每天3件·每件2小时)校验。"
* **PM:** "为了在失败时让画面能区分情况给出提示，失败响应也得有规则吧？"
* **DEV:** "把错误码定死。未认证的请求返回401；取消别人的预约，或没有 admin 权限却调用会议室注册，返回403；要取消的预约不存在返回404；时间重叠的重复预订返回409；时段单位不对、超出上限、结束早于开始这类校验失败返回422。画面只看这个码来挑提示文案。"
* **PM:** "日历联动的话题老是被提起，怎么办？有人希望日程能自动进到 Outlook 里。"
* **DEV:** "第一版我想搁置。光是日历 API 权限申请和同步冲突处理就是一个月的活儿。作为替代，在预约完成画面放个'添加到日历'按钮，下载标准日程文件(.ics)，这个折中方案一天就能做出来。"
* **PM:** "这个折中方案不错。自动释放是怎么运作的？谁来计这10分钟？"
* **DEV:** "服务端有个每分钟跑一次的批处理作业，找出'开始已过10分钟 + 未签到'的预约，把它改成自动释放状态。同一个作业顺带处理提前5分钟的提醒。结构很简单，没有会出故障的地方。"
* **PM:** "总结一下，模型三种(Room·Reservation·User)、API 六种、错误码五种(401/403/404/409/422)、日历用 .ics 折中、自动释放用每分钟一次的批处理。下次会议把发布范围钉死。"
* **[进度: 85% - 数据模型三种(含属性·类型) + API 六种(含输入/响应) + 错误响应规则(401/403/404/409/422) + 自动释放批处理结构敲定]**
