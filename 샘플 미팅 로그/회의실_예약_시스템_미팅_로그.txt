### [미팅 로그 V3] - 화면 흐름과 더블부킹 방지 규칙
* **일시:** 2026-03-09 (10:00 ~ 11:30)
* **참석자:** PM, DEV, UX
* **UX:** "화면은 네 개로 시작하고 싶습니다. ① 현황 보드(메인, /board), ② 예약 생성 모달(보드에서 시간대를 탭하면 열림), ③ 내 예약(/my), ④ 관리자 설정(/admin, 회의실 등록·수정용). 별도 홈 화면은 만들지 않고 보드가 곧 홈입니다."
* **DEV:** "접근 권한도 화면 단위로 정하죠. 보드·예약·내 예약은 사내 SSO 인증만 통과하면 누구나 쓰고, 관리자 설정은 admin 권한이 있는 총무팀 계정만 들어올 수 있습니다. 권한 없는 사람이 /admin 에 들어오면 '권한 없음' 안내를 보여주고 보드로 돌려보내고요."
* **PM:** "예약 생성 모달에서 시간은 어떻게 고르나요? 자유 입력이면 9시 13분 같은 예약이 생길 텐데요."
* **UX:** "30분 단위 슬롯으로 제한합니다. 보드에서 빈 슬롯을 탭하면 그 시간이 미리 채워진 채로 모달이 열리고, 종료 시간만 고르면 끝나요. 입력 항목은 회의 제목 하나만 남습니다."
* **DEV:** "더블부킹 방지가 이 시스템의 생명입니다. 규칙을 정확히 정하죠. 같은 회의실에서 시간 구간이 1분이라도 겹치면 예약을 거부하고 '이미 예약된 시간이에요'를 보여줍니다. 검증은 반드시 서버에서 합니다. 화면에서 비어 보여도 누가 1초 먼저 누르면 서버가 거부하는 게 정답이에요."
* **PM:** "연속 예약 독점은요? 한 팀이 하루 종일 잡아버리는 경우요."
* **DEV:** "1인당 하루 최대 3건, 건당 최대 2시간으로 제한합시다. 총무팀이 말한 '대형 워크숍은 예외'는 관리자 화면에서 관리자가 직접 등록하는 방식으로 풀고요."
* **UX:** "예약 거부 같은 안내 문구는 비난조가 안 되게 다듬을게요. '이미 예약된 시간이에요. 바로 다음 슬롯은 비어 있어요'처럼 대안을 함께 보여주는 방향으로요."
* **PM:** "화면 4종과 권한 구분, 더블부킹·상한 규칙까지 확정. 다음은 노쇼 대책입니다."
* **[진척도: 50% - 화면 4종 + 권한 구분(SSO 인증/admin) + 30분 슬롯 + 서버측 더블부킹 차단 + 예약 상한(일 3건·건당 2h) 확정]**

### [미팅 로그 V4] - 노쇼 대책: QR 체크인과 10분 자동 해제
* **일시:** 2026-03-12 (14:00 ~ 15:00)
* **참석자:** PO, PM, DEV, UX, 총무팀
* **PM:** "노쇼 대책으로 두 가지 안을 준비했습니다. A안은 시작 후 10분 안에 체크인하지 않으면 예약을 자동 해제하고 방을 비어 있음으로 되돌리는 것, B안은 노쇼 횟수에 벌점을 줘서 3회 누적 시 일주일 예약을 제한하는 것입니다."
* **총무팀:** "벌점제는 반발이 클 겁니다. 작년에 지각 벌점제 시도했다가 한 달 만에 접었어요. 처벌보다는 '안 쓰면 자동으로 풀린다'는 A안이 자연스럽습니다."
* **PO:** "A안으로 갑시다. 체크인은 어떻게 하나요?"
* **DEV:** "회의실 문 앞에 QR 코드를 붙입니다. 입실하면서 폰으로 찍으면 체크인 화면(/checkin)이 열리고, '체크인' 버튼 한 번이면 끝이에요. 화면은 이걸로 다섯 개가 됩니다. 회의실마다 고유 QR이라 다른 방에서 찍는 부정 체크인은 안 되고, 자기 예약이 없는 시간에 찍으면 '이 시간의 예약을 찾을 수 없어요'가 뜹니다."
* **UX:** "체크인을 잊는 사람이 많을 테니, 예약 시작 5분 전에 '체크인해 주세요' 알림을 보내죠. 알림은 사내 슬랙 봇 DM으로 보내면 따로 푸시 인프라가 필요 없습니다."
* **PM:** "자동 해제된 방은 보드에 바로 '비어 있음'으로 떠야 의미가 있습니다. 해제 시점에 보드가 실시간 갱신되는 것까지가 한 세트예요."
* **DEV:** "그건 보드를 30초 주기로 폴링하는 것으로 충분합니다. 웹소켓까지 갈 규모가 아니에요."
* **PO:** "확정합니다. QR 체크인(+체크인 화면), 10분 미체크인 자동 해제, 시작 5분 전 슬랙 알림. 벌점제는 도입하지 않습니다."
* **[진척도: 70% - 노쇼 정책 확정(QR 체크인 + 체크인 화면(/checkin) + 10분 자동 해제 + 5분 전 알림), 벌점제 기각]**

### [미팅 로그 V5] - 데이터 모델 · API · 에러 응답 규칙
* **일시:** 2026-03-16 (10:00 ~ 11:30)
* **참석자:** PM, DEV
* **DEV:** "데이터 모델은 세 개면 충분합니다. 회의실(Room)은 이름(문자열, 최대 30자)·층(정수)·수용 인원(정수, 1 이상)·QR 코드 값(문자열)을 갖고, 예약(Reservation)은 회의실 ID·예약자 ID·회의 제목(문자열, 최대 50자)·시작/종료 시각(날짜+시간)·상태를 갖습니다. 상태는 예약됨, 체크인됨, 자동해제, 취소됨 네 가지 중 하나만 갖는 고정 값이에요. 사용자(User)는 사내 SSO 프로필(사번·이름·이메일)을 그대로 씁니다. 별도 가입은 없습니다."
* **PM:** "API는 어떤 모양인가요?"
* **DEV:** "여섯 개로 시작합니다. ① 회의실 목록·오늘 점유 현황 조회(보드가 30초마다 호출), ② 예약 생성 — 입력은 회의실 ID·회의 제목·시작/종료 시각이고 성공하면 응답으로 예약 번호와 상태를 돌려줍니다, ③ 예약 취소, ④ 내 예약 목록 조회, ⑤ QR 체크인, ⑥ 관리자용 회의실 등록·수정. 생성 API가 더블부킹 검증과 상한(일 3건·건당 2시간) 검증을 모두 책임집니다."
* **PM:** "실패했을 때 화면이 상황을 구분해서 안내하려면 실패 응답에도 규칙이 있어야겠죠?"
* **DEV:** "에러 코드를 박아두죠. 인증이 안 된 요청은 401, 남의 예약을 취소하려 하거나 admin 권한 없이 회의실 등록을 부르면 403, 취소하려는 예약이 미존재면 404, 시간이 겹치는 더블부킹은 409, 슬롯 단위가 안 맞거나 상한 초과·종료가 시작보다 빠른 입력 같은 검증 실패는 422입니다. 화면은 이 코드만 보고 안내 문구를 고릅니다."
* **PM:** "캘린더 연동 얘기가 자꾸 나오는데 어떻게 할까요? 아웃룩에 자동으로 일정이 들어가면 좋겠다는 요구가 있어요."
* **DEV:** "1차에서는 보류하고 싶습니다. 캘린더 API 권한 신청과 동기화 충돌 처리만 한 달짜리 일입니다. 대신 예약 완료 화면에 '캘린더에 추가' 버튼을 둬서 표준 일정 파일(.ics)을 내려주는 절충안이면 하루면 만들어요."
* **PM:** "그 절충안 좋네요. 자동 해제는 어떻게 동작하죠? 누가 10분을 재나요?"
* **DEV:** "서버에서 1분마다 도는 배치 작업이 '시작 10분 경과 + 미체크인' 예약을 찾아서 자동해제 상태로 바꿉니다. 같은 작업이 5분 전 알림도 함께 처리하고요. 구조가 단순해서 장애 날 곳이 없습니다."
* **PM:** "정리하면 모델 3종(Room·Reservation·User), API 6종, 에러 코드 5종(401/403/404/409/422), 캘린더는 .ics 절충, 자동 해제는 1분 주기 배치. 다음 미팅에서 출시 범위를 못 박죠."
* **[진척도: 85% - 데이터 모델 3종(속성·타입 포함) + API 6종(입력/응답 포함) + 에러 응답 규칙(401/403/404/409/422) + 자동해제 배치 구조 확정]**
