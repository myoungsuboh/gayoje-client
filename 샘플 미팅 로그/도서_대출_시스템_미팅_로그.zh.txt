### [会议记录 V1] - 启动会：书总是莫名其妙地消失
* **时间:** 2026-04-06 (10:00 ~ 10:50)
* **参会者:** 总务组长(以下简称 PO)、产品经理(以下简称 PM)、开发主管(以下简称 DEV)、设计师(以下简称 UX)
* **PO:** "公司图书馆的书，一年要丢失30多本。没有记录是谁拿走的，也就无从查找。现在是让大家在入口处的本子上手写登记，可一半人就那么直接拿走了。"
* **PM:** "把问题归纳一下就三点。1) 哪本书现在可以借，谁都不知道；2) 谁借了什么书，无法追踪；3) 没有归还期限，书就回不来。"
* **DEV:** "外部图书馆解决方案是给学校·公共图书馆用的，对我们太重也太贵。藏书才2千本这点规模，还不如我们自己做个轻量的。不过条码扫描仪这类设备的引入，第一版就先去掉吧。"
* **UX:** "核心体验就是'搜一本书借走，在期限内归还'。用工号登录、用手机直接借，这个流程就够了。"
* **PO:** "目标很简单。看到有哪些书，借走，按时收回。下次会议把功能范围定下来。"
* **[进度: 10% - 启动会完成，三大问题(可见性·追踪·逾期)及目标达成共识]**

### [会议记录 V2] - 核心功能范围敲定：搜索 · 借阅 · 我的借阅
* **时间:** 2026-04-09 (14:00 ~ 15:10)
* **参会者:** PM、DEV、UX
* **PM:** "我先把用户场景理了一遍。主角是'需要书来读的员工'。① 在图书列表里搜书，② 借走可借的书，③ 在我的借阅列表里归还。就这三条动线，没别的了。"
* **UX:** "图书列表支持按标题·作者搜索，每张书卡上明确标出'可借/借阅中'。如果是借阅中，就连同预计归还日一起显示。"
* **DEV:** "创建借阅时，输入只收书 ID 就行。借书人由登录的工号自动确定。借阅完成后，在响应里返回归还期限，让人当场就能确认。"
* **PM:** "一个人最多能借几本书？"
* **DEV:** "先从每人同时3本、借期14天开始吧。续借第一版先去掉，看了数据再判断。"
* **UX:** "我的借阅页面把借的书按归还期限顺序展示，每一项都放个归还按钮。过期的书就标红强调。"
* **PM:** "我来总结。第一版范围是 ① 图书搜索·列表，② 借阅(每人3本·14天)，③ 我的借阅列表与归还。续借和预约推迟到第二版。"
* **[进度: 30% - 第一版功能范围三项敲定，借阅续借·预约推迟到第二版]**

### [会议记录 V3] - 画面流程与防重复借阅规则
* **时间:** 2026-04-13 (10:00 ~ 11:00)
* **参会者:** PM、DEV、UX
* **UX:** "画面先做四个。① 图书列表(主页面，/books)，② 图书详情·借阅弹窗(点击书即可打开)，③ 我的借阅(/my)，④ 管理员设置(/admin，用于图书的注册·修改)。不单独做首页，列表即首页。"
* **DEV:** "访问权限按画面来定吧。列表·借阅·我的借阅只要通过公司内部 SSO 认证，谁都能用；管理员设置只有拥有 admin 权限的总务组账号才能进。没有权限的人进 /admin 时，就显示'无权限'，并把他退回列表。"
* **PM:** "要是去借一本已经在借阅中的书呢？"
* **DEV:** "防重复借阅是核心。同一本书如果已在借阅中，就拒绝并显示'这本书已在借阅中'。校验必须放在服务端做。画面上显示'可借'，但谁要是早借1秒，服务端就该拒绝，这才是正解。"
* **PM:** "超过上限还要借的情况呢？"
* **DEV:** "已经借了3本的人去尝试借第4本，就拒绝并显示'已超过可借数量'。这个校验同样由服务端在创建借阅时负责。"
* **UX:** "拒绝文案我会打磨得不带责备的口气。像'已在借阅中。归还后要通知您吗？'这样，朝着建议下一步行动的方向来。"
* **PM:** "四个画面、权限区分，以及防重复借阅·上限规则都敲定了。下一步是逾期对策。"
* **[进度: 50% - 四个画面 + 权限区分(SSO 认证/admin) + 服务端防重复借阅 + 借阅上限(3本)敲定]**

### [会议记录 V4] - 逾期对策：归还提醒与逾期时限制借阅
* **时间:** 2026-04-16 (14:00 ~ 14:50)
* **参会者:** PO、PM、DEV、UX
* **PM:** "针对逾期我提两点。① 在归还期限前一天发提醒，让人提前留意；② 只要有一本逾期的书，就拦住新的借阅。"
* **PO:** "扣分或罚款不行。以公司氛围来看只会激起反弹。靠提醒来留意，有逾期就不让再借，这个程度比较合适。"
* **DEV:** "提醒通过公司内部 Slack 机器人私信发送，就不用另搭推送基础设施。归还前一天和到期当天，发两次。"
* **UX:** "在我的借阅页面把逾期的书标红，并连同'逾期○天'一起显示。尝试新借阅时，就用'请先归还逾期的书'来提示。"
* **DEV:** "逾期判定和提醒由服务端每天跑一次的批处理作业处理。找出'归还期限已过 + 未归还'的借阅标记为逾期，同一个作业顺带把第二天的提醒对象也一并筛出来。结构很简单，没有会出故障的地方。"
* **PO:** "拍板。归还前一天·当天 Slack 提醒，只要有1本逾期就拦住新借阅。扣分制不引入。"
* **[进度: 70% - 逾期政策敲定(归还提醒 + 逾期时拦截借阅 + 每天一次批处理)，扣分制否决]**

### [会议记录 V5] - 数据模型 · API · 错误响应 · MVP 发布范围
* **时间:** 2026-04-20 (10:00 ~ 11:20)
* **参会者:** PO、PM、DEV、UX
* **DEV:** "数据模型有三个就够了。图书(Book)有标题(字符串，最多100字)·作者(字符串)·ISBN(字符串)·状态，状态是可借·借阅中两个固定值。借阅(Loan)有图书 ID·借阅人 ID·借出日·归还期限·归还日·状态，状态是借阅中·已归还·逾期三个固定值。用户(User)直接沿用公司内部 SSO 资料(工号·姓名·邮箱)。不另做注册。"
* **PM:** "API 是什么样的？"
* **DEV:** "先做五个。① 图书列表·搜索查询(标题·作者关键词)，② 创建借阅 —— 输入是图书 ID，成功则在响应里返回借阅号和归还期限，③ 归还处理，④ 我的借阅列表查询，⑤ 管理员用图书注册·修改。创建 API 全权负责防重复借阅校验和上限(3本)·逾期拦截校验。"
* **PM:** "失败响应也得有规则，画面才能区分情况给出提示吧？"
* **DEV:** "把错误码定死。未认证的请求返回401；没有 admin 权限却调用图书注册返回403；要归还的借阅不存在返回404；已在借阅中的书、或因逾期被拦住的新借阅这类状态冲突返回409；像超出上限这样的校验失败返回422。画面只看这个码来挑提示文案。"
* **PM:** "做最终检查。MVP 范围是四个画面 —— 图书列表、借阅弹窗、我的借阅、管理员设置。政策是每人3本·14天借阅、服务端防重复借阅、归还前一天·当天提醒、逾期时拦截新借阅。"
* **PO:** "质量标准也定死吧。列表要是慢，大家就干脆把书直接拿走了。"
* **DEV:** "我提三个非功能目标。1) 图书列表查询响应500ms 以内，2) 同时在线50人也能无延迟处理，3) 工作时间(工作日9~19点)可用性99.9%。借阅·归还记录保留1年，用于计算丢失·逾期指标。"
* **PM:** "成功指标我提三个。1) 上线8周后，图书丢失从每年30本降到5本以下，2) 逾期率维持在10%以下，3) 完成借阅的平均耗时10秒以内。丢失可以用'未归还且长期搁置的借阅'比例来追踪。"
* **PO:** "好。确定5月18日公司内部开放。大家辛苦了。"
* **[进度: 100% - 数据模型三种(Book·Loan·User)·API 五种·错误响应规则(401/403/404/409/422)·MVP 范围(四个画面)·非功能目标三种·成功指标三种·发布日程(5/18)敲定，规划阶段完成]**
