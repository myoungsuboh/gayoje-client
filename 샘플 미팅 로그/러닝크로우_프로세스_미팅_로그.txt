# 🏃 [Running Crow 러닝 트래커 프로젝트] PoC → 운영 가능한 v1.0 고도화 미팅 로그 (V1 ~ V16)

> 본 미팅 로그는 `C:\project\runnig-project` 에 스케치된 Running Crow PoC(Vue 3 + Vuetify + Capacitor Android, 정적 mockData 의존)를 **실제 동작하는 v1.0 러닝 트래커**로 끌어올리는 의사결정 과정을 기록한다. 핵심 고도화 목표는 다음 5가지다:
> 1. 실제 동작하는 로그인 / 회원가입 (JWT 기반 인증)
> 2. 친구(크루) 초대·수락·관리 백엔드 연동
> 3. 큐레이션 + 사용자 생성 추천 코스
> 4. GPS 기반 실측 러닝 히스토리 (mock 제거)
> 5. **지도 위에 실제 뛴 경로 폴리라인 렌더링 + A→B 코스 추종 시 진행률/잔여거리 실시간 계산**

### [미팅 로그 V7] - 러닝 폴리라인 저장 포맷: Encoded Polyline vs GeoJSON LineString vs PostGIS GEOGRAPHY
* **일시:** 2026-06-01 (10:00 ~ 11:30)
* **참석자:** BE, FE, GIS, 분석팀, DB 관리자(이하 DBA)
* **BE:** "러닝 1회당 GPS 포인트 수를 추정해봤습니다. **1초 간격 샘플링, 평균 30분 러닝 = 1800 포인트**. 각 포인트 = (lat, lng, timestamp, accuracy, optional elevation). 1년 사용 시 사용자당 평균 200회 = 36만 포인트. 1만 사용자 = 36억 포인트. 이걸 어떻게 저장하느냐가 핵심입니다."
* **DBA:** "세 가지 옵션입니다. **(A) GPS 포인트 raw row 그대로 - `run_track_point(run_id, seq, lat, lng, ts, accuracy)`**: 정밀도 최대, 쿼리 유연, 용량 큼(36억 row). **(B) Google Encoded Polyline 5자리 정밀도 압축 문자열 - `run.polyline_encoded TEXT`**: 1800 포인트 → ~10KB → 압축 후 ~3KB. 정밀도 ±1.1m, 시각화에 충분. 시간 정보 손실. **(C) PostGIS `GEOGRAPHY(LINESTRINGZM, 4326)` - 시간을 M(measure) 차원에 저장**: PostGIS 공간쿼리 가능 + 시간 보존. 용량 중간."
* **GIS:** "용도 분리가 답입니다. **저장은 (C) PostGIS LINESTRINGZM** - 향후 '이 사용자가 한강공원 근처에서 뛴 횟수' 같은 공간 쿼리, 코스 매칭, 통계가 모두 DB 레벨로 가능. **API 전송은 (B) Encoded Polyline** - 네트워크 비용 1/10. **클라이언트 시각화도 (B)** - MapLibre는 `polyline.decode()`를 거쳐 GeoJSON으로 변환해서 렌더링."
* **DBA:** "동의합니다. 추가로 `run` 테이블 분리: **`run(id, user_id, started_at, ended_at, distance_m, duration_s, avg_pace_s_per_km, calories, polyline_encoded TEXT, geom GEOGRAPHY(LINESTRINGZM,4326), gps_signal_loss_segments JSONB)`**. `polyline_encoded`는 API 응답용 캐시, `geom`은 공간쿼리용 권한 컬럼."
* **BE:** "시간 정보가 필요한 이유는 스플릿 페이스(1km, 5km, 10km 구간별 페이스) 계산 때문입니다. 이건 클라이언트에서 raw 포인트 배열로부터 후처리하는 게 자연스러운데, raw 포인트를 DB에 남길지 폐기할지?"
* **분석팀:** "raw 포인트를 영구 보관할 필요는 없지만 **후처리 통계(스플릿, 심박존, 일시정지 구간) 계산이 끝나기 전까지는 raw가 있어야** 합니다. **러닝 종료 시점에 클라이언트에서 raw 포인트를 백엔드로 업로드 → BE가 폴리라인 인코딩 + 스플릿 계산 + run 테이블에 저장 → raw는 30일 동안만 `run_track_point` 테이블에 보관 후 자동 삭제** 정책을 제안합니다. 30일 안에 사용자가 '구간 분석 다시 보기' 같은 디테일 요청을 하면 raw로 재계산 가능."

### [미팅 로그 V9] - GPS 정확도 필터: Accuracy Threshold + EMA 평활화 + 페이스 계산
* **일시:** 2026-06-08 (10:00 ~ 11:30)
* **참석자:** FE, 모바일, GIS, 분석팀, QA
* **GIS:** "raw GPS는 노이즈가 큽니다. 도시 빌딩 사이에서 정지 상태인데도 좌표가 ±20m씩 흔들리는 'GPS 드리프트'가 일상입니다. 그대로 폴리라인을 그리면 실제 안 뛴 거리가 누적되어 5km 코스가 5.8km로 기록됩니다. 필터링이 필수입니다."
* **분석팀:** "3단계 필터 파이프라인을 제안합니다. **(1) 정확도 임계치 필터**: `accuracy > 30m` 인 포인트는 폐기. **(2) 속도 sanity 필터**: 이전 점과의 instantaneous 속도가 25km/h 초과 시 폐기 (러닝 최고 속도 한참 상회 - 점프 오류). **(3) EMA(지수이동평균) 평활화**: α=0.3, smoothed_pos = α × new_pos + (1-α) × prev_smoothed."
* **FE:** "EMA가 가장 단순하면서 효과가 좋은 옵션이라 채택합니다. 칼만 필터까지 가면 구현 복잡도 대비 이득이 크지 않습니다."
* **GIS:** "거리 계산은 두 점 사이를 **Haversine 공식**으로 계산합니다. `2R × asin(√(sin²(Δφ/2) + cos(φ1)cos(φ2)sin²(Δλ/2)))`. R=6371000m. 평면 근사(Equirectangular)는 짧은 구간에서는 충분하지만 일관성을 위해 Haversine으로 통일합니다."
* **분석팀:** "페이스 계산은 두 단계입니다. **(1) 인스턴트 페이스**: 최근 30초 윈도우의 거리 / 시간. 라이브 화면에 표시. **(2) 평균 페이스**: 총 거리 / 총 활동 시간. '활동 시간'은 일시정지 구간 제외. 일시정지는 사용자가 버튼을 눌렀거나, 자동 일시정지가 감지된 경우(아래 V9-자동 일시정지 항목)."
* **QA:** "자동 일시정지 정의해주세요. 신호등 대기 같은 짧은 정지를 어떻게 감지하나요?"
* **모바일:** "자동 일시정지 알고리즘: **이전 5초 평균 속도 < 0.5m/s (시속 1.8km, 사실상 정지)** 이고 **현재 정확도 양호(< 15m)** 이면 자동 일시정지 진입. 다시 1초 평균 속도 > 1.5m/s 시 자동 재개. 사용자가 수동 일시정지한 경우는 자동 재개 비활성화."
* **GIS:** "고도(elevation)는 GPS의 z 좌표를 그대로 쓰면 노이즈가 심합니다(±10m 흔들림). **누적 상승고도** 계산 시에는 (1) EMA 평활화(α=0.2), (2) 최소 변화 임계치 3m(노이즈 제거), (3) 양수 변화분만 누적. 음수 변화(하강)는 'elevation_loss'로 별도 카운트."
* **분석팀:** "칼로리는 단순 공식으로 갑니다. **calories = MET × weight_kg × hours**. 러닝 평균 MET 9.8 (7~13 범위에서 페이스 따라 조정). 사용자 프로필에 체중이 없으면 70kg 기본값. v1.1에서 심박수 기반 정밀 계산을 추가합니다."
* **QA:** "회귀 테스트 데이터 만들기 - V1에서 합의한 'E2E Run-Replay 시나리오'를 위해 **사전 녹화된 GPS trace 5종**을 준비합니다. (1) 한강 평지 5km, (2) 남산 둘레길 (오르막), (3) 빌딩 사이 (드리프트 많음), (4) 터널 통과 (신호 손실), (5) 트랙(400m × 12바퀴, 곡선 정확도 검증). 각 trace에 대해 필터 적용 후 거리 ±2% 오차 이내가 합격선."

### [미팅 로그 V10] - A→B 코스 추종 진행률: turf.js nearestPointOnLine + 이탈 감지

(2) **진행률 = L / 코스 총 길이 × 100%**. (3) **잔여 거리 = 코스 총 길이 - L**. (4) **이탈 감지: D > 50m 이고 5초 이상 지속 시 'off-route' 플래그 ON.**"
